The Board as a Projected Domain

Design spike (#3575) — model Card as an owl:Class and project the cards API from the model, the same move generate-dal (#3567) just proved. Scopes the exercise; does not build it. · Wren · 2026-06-23

The team's board is the one piece of core infrastructure still built the old way: flat Vikunja string-labels, hand-managed, untyped, unvalidated. Everything else is moving to generate-from-model. This doc asks whether the board should too — and the answer, now that the generator can emit both read and governed-write surfaces, is yes, eventually: it becomes one more projected domain.

1 · The problem: the board is config-in-strings

A card today is a Vikunja task carrying a bag of string labels across nine axes — owner:, priority:, status (the bucket), type:, origin:, domain:, chunk:, sequence:, subproduct:. There is no schema. The axes are a hardcoded map in cards/src/config.ts; a value is valid because a human remembered to add it.

This is not academic. #3432 — landed today — was exactly this failure: the chunk axis silently dropped any value not in the static label map, so cards filed with a real priority chunk landed untagged and invisible in the priorities readout. The bug was only possible because chunk is unmodeled string config rather than a typed relationship to a real Chunk node. The class of bug is structural, not a one-off.

The symptoms of config-in-strings:

  • No validation. Nothing refuses an off-list value at the source — it just silently doesn't stick (#3432) or lands as a typo that fragments a chunk.
  • No relationships. A chunk, a sequence, a domain are strings, not nodes — so you can't ask "what chunks does this product own?" or "which cards block this one?" without scraping labels.
  • The readout is a label-scrape. chorus_priorities_readout is TAGGED-ONLY because tags are the only structure there is; an untagged card is simply invisible.
  • Drift by construction. The axes live in code; changing them is a code edit + deploy, exactly the config-in-code anti-pattern the properties domain exists to retire.

2 · The model: Card as an owl:Class

Model the board the way every other domain is modeled — a class with typed datatype properties, edge relationships to first-class nodes, and a SHACL shape that is the validation floor.

chorus:Card  a owl:Class ;
  # datatype properties (the scalars)
  title        xsd:string     # sh:minCount 1  (the floor)
  status       chorus:Lane    # enum: Next | WIP | Blocked | Done | Won't Do | …
  priority     chorus:Prio    # enum: P1 | P2 | P3
  cardType     chorus:Kind    # enum: new | enhance | fix | chore | swat
  origin       chorus:Origin  # enum: reflective | reactive
  experience   xsd:string     # the Experience floor (gate-product reads it)
  acceptance   xsd:string     # the AC floor

  # edge properties (relationships to REAL nodes, not strings)
  ownedBy      → chorus:Role        # wren | silas | kade | jeff
  inChunk      → chorus:Chunk       # coherent-model, loom-authoring, …  (a NODE)
  inSequence   → chorus:Subproduct  # athena | loom | werk | borg | …
  inDomain     → chorus:Domain      # the existing Domain class
  blockedBy    → chorus:Card        # dependency edges → a real DAG
  partOf       → chorus:Card        # parent/child (epics, #2561-style splits)

The decisive shift is the edge properties. A chunk stops being a string and becomes a chorus:Chunk node — so #3432 cannot recur (you can't "silently drop" an edge to a node the way you drop an unmatched string), the priorities readout becomes a query over typed instances rather than a label-scrape, and "what does this product own / what blocks this" become graph traversals. The completeness floor (title, Experience, AC) becomes sh:minCount on the shape — the gate-product floor enforced at the model, not in bouncer prose.

This is the same containment grammar the repo already uses (ADR-041, #3488): Card relates to Chunk / Subproduct / Domain the way Service relates to Domain relates to Product. The board is not a special case — it's another region of the one graph.

3 · What owl-api projects vs what stays hand-written

The discipline from the platform doc: build the integration, rent the orchestration. Applied here, and now concrete because generate()→RouteTable→serve already projects any class to a live API, and generate-dal (#3567) projects a governed write-edge:

ConcernSourceHow
Cards read API — GET /cards, /cards/:id, /cards?chunk=X, the priorities readout as a queryProjectedowl-api serve already does this for every shaped class. Card becomes one more served class — near-zero new code.
Validation — refuse off-enum status/priority, enforce the title+Experience+AC floorProjectedThe SHACL shape is the validator; the generator reads it (the same mandatory/secured projection used today).
Governed writes — create/move/tag a card with auth + scope + shape-validateProjectedThe generate-dal write-edge landed today. The board's mutations become a governed DAL like any other writer.
Orchestration — lane transitions, WIP limits, the bouncer, gate sequencingHand-writtenThis is logic, not schema — it stays code (the verb layer), reading the typed model rather than scraping labels.
The Kanban UI (Vikunja today)Rented / retiredOnce the model is the source of truth, Vikunja is a view — kept as a board UI, or retired for a generated one. Not the system of record.

So the board splits cleanly: schema + read + validated-write = projected; transition logic = hand-written; UI = rented. The same three-way cut the platform makes everywhere.

4 · Migration cost vs the flat-Vikunja status quo

This is a strangler-fig, identical in shape to the domain build process (Shape → Generate → Land → Migrate → Retire), not a big-bang rewrite:

  • Shape — author chorus:Card + Chunk/Subproduct nodes + the SHACL floor. Small; the axes already exist as strings to lift.
  • Generate — owl-api serve picks up Card as another class. Near-free given today's generator.
  • Migrate data — the real cost: convert ~live Vikunja tasks (labels → typed instances + edges). One-time backfill; the label values become node references. This is the expensive leg.
  • Repoint — the cards CLI + the chorus_cards_* MCP tools read/write the generated API instead of Vikunja directly. Bounded — they're already one chokepoint each (the #3566 pattern).
  • Retire — Vikunja drops to a view or is removed; config.ts's static label maps are deleted (the #3432 source).

Cost is dominated by the data migration + repointing the CLI/MCP, not by the generation — exactly the asymmetry the #3567 spike found for the write-lane (the emit is cheap; moving the consumers is the work).

The honest cost: the board is load-bearing and used by every role every minute. A migration that corrupts or loses card state is a far worse failure than the #3432-class bug it cures. So this is gated on a reversible, dual-write migration (model + Vikunja in parallel until proven), and it must not jump ahead of #2159 — the board hierarchy model (value-stream → domain → sub-product → work-unit) — which decides the very relationships Card would point at. Modeling Card before #2159 settles the containment shape would model it against a moving target.

5 · Recommendation — go/no-go

GO — but sequenced, not now. The capability question is answered: owl-api can project the cards read API (it serves any shaped class today) and the governed write (generate-dal landed today). The board can become a projected domain. This is the natural endpoint of the generate-from-model thesis applied to our own coordination substrate.

The sequencing:

  • First, #2159 (Chorus-native board — the value-stream → domain → sub-product → work-unit hierarchy). That card already owns the relationship model; Card-as-owl:Class is its schema layer. Fold this spike's model into #2159 rather than running a competing board-modeling effort.
  • Then, once the GOVERN write-lane (#3573) matures the generated-DAL pattern on a real consumer, the board's writes ride the same rails — no bespoke board-write code.
  • Don't build a standalone board projection now. #3432 already drained the acute pain (chunks stick at add-time); the board works. Building ahead of #2159 risks modeling against a moving containment shape.

Net: record the answer (yes, the board is projectable; here is the Card model and the three-way cut), fold the model into #2159, and let the write-lane prove out on #3573 first. No new build card off this spike.
