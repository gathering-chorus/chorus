chorus-api tier split (#3080)

Ingestion workers feed the stores; a thin serving API reads them; the data stores are the contract. Decoupled by data, not process. From Jeff's architecture call, 2026-05-25. Designing-step — architecture + decomposition; build cards follow per job.

The problem

chorus-api is one Node process — one event loop — hosting three different kinds of work that the architecture should keep apart:

  • Serving: request handlers (search, cards, context, nudge) that must answer fast.
  • Ingestion: batch jobs (crawl, reindex, embed) that write source data into the stores — heavy, periodic, CPU/IO-bound.
  • Emission: jobs that read store state, derive, and push out (pulse snapshot, andon/role-state, alert relay, eventloop alert). Not request/response, not source-ingestion — a third pattern.

Because all three share one loop, any heavy job freezes all serving for every role + Jeff at once — the shared-fate freeze behind the all-session eventloop.blocked alerts (~90/hr post-deploy, blocks up to 15s, confirmed by #3079's live capture pinning even a 5–7s GET /api/chorus/search). Patching handlers one at a time (#3077: index, embed, search) is whack-a-mole because the design is wrong, not the handlers. The clean cut is serving (on the loop) vs everything off the loop; off-loop splits into two worker classes — ingestion and emission.

The key insight: ingestion has no real dependency on the API

Each ingestion job depends on the data stores, not on chorus-api:

  • reindex reads files → writes SQLite.
  • embed reads SQLite → calls Ollama → writes LanceDB → marks SQLite.
  • crawl reads SQLite/Fuseki/Loki → writes a summary row to SQLite.

None of that needs the API. They touch SQLite, LanceDB, Fuseki, files — directly. chorus-api is just another client of those same stores. The dependency that exists today is incidental: reindex is setInterval'd into server.ts (code placement), and the crawler curls its own API (GET /api/chorus/crawl/:domain — a self-call inversion). They were bolted into the API; they never depended on it. Embed already proves the pattern: #1978 extracted it to a standalone worker (chorus-embed-worker.sh) precisely because its 100+ sequential Ollama calls per cycle froze the loop — it has run as a separate process writing LanceDB + marking SQLite, concurrent with the API reading, since 2026-04-12.

Two tracks, named separately. The evidence splits this into two problems the tier-split does not conflate:
  • Track A — tiering (this card's core). Move ingestion (reindex, crawl) out of the API process. Kills ingestion-caused shared-fate freezes. embed is the shipped precedent.
  • Track B — search offload. The #3079 5–7s block is GET /api/chorus/search — which is request-serving and stays in the thin API by definition. Tiering cannot fix it; the block lives inside the serving tier. Search needs in-tier offload (worker_threads/piscina or child process for the FTS + RRF fusion + query-embed compute), independent of Track A.
Both are P1. Conflating them — "the split fixes the freezes" — would leave the loudest blocker (search) untouched.

AC1 — two tiers: serving-loop vs off-loop workers

The cut is serving (on the loop) vs everything off the loop. Off-loop work is two worker classes, not one — both touch only the stores, never the serving loop:

  • Ingestion — write source data into the stores (crawl, reindex, embed).
  • Emission — read store state, derive, and push out (pulse snapshot, andon/role-state, alert relay, eventloop alert). The store is the emitter's source; emission is store-driven.

The boundary contract: the serving API depends on the data in the stores, not on any worker process. Ingestion writes into the stores; emission reads store state and pushes derived state outward; serving reads on request. None of the three invokes another — the store is the buffer that decouples them. A worker grinding a 5-second job (ingest or emit) cannot block a search, because it is not in the serving process. Today all three share one loop — which is why a heavy reindex, a slow pulse aggregation, and a 5–7s search all show up as the same eventloop.blocked alert.

AC3 — decomposition + migration plan

Track A (off-loop migration) — move both worker classes out of the serving loop. Each job becomes a standalone worker hitting the stores directly, on its own cadence. embed is the proven precedent; the sequence is the jobs still inside the loop, smallest/safest first. Each migration is its own build card with a runtime-outcome AC (a heavy run of that job cannot block a serving request — measured post-migration).

Ingestion class:

  1. reindex — smallest, cleanest (files → SQLite, no external service). Move the setInterval out of server.ts:3329 into a standalone worker — mirror chorus-embed-worker.sh. First, lowest risk. Owner: Silas (infra/service) or Wren (API).
  2. crawl — folds in #3069 (crawlers as declared instances): each collector becomes a worker instance writing the stores directly; stop the self-call (the crawler no longer curls the API). Heaviest, last. Owner: Wren (#3069).
  3. cache-warm — decision, not a move: either keep as serving-tier read-through caches (cheap, in-process) or fold the counts into the reindex worker writing a shared row. Owner: Wren (API).

Emission class (Jeff's AC1 catch — the store is also the emitter):

  1. pulse snapshot — move the compose+write of /tmp/pulse-latest.json into a pulse-emitter worker reading the stores directly; GET /pulse/latest stays serving (reads the file). Owner: Silas (observability).
  2. andon / alert / eventloop emitters — relays that read state and push out (spine, desktop notify, role-state snapshot). Move to the emitter worker; the eventloop alert especially must not ride the loop it's measuring. Owner: Silas (observability).

Track B (search) — two sub-problems, do not conflate (Silas's AC4 caveat). The #3079 5–7s block is not addressed by Track A, and it is itself two things:

  • B1 — shared-fate freeze. Search compute on the main loop freezes everyone else's requests. Fix: move the FTS + RRF fusion + query-embed + SPARQL work off the loop (worker_threads/piscina or child process). This is the tiering-shaped fix for serving — search stays serving, but its CPU leaves the loop.
  • B2 — single-query latency. A search is just slow (5–7s) on the 1.1 GB index.db — a data-size / query-latency problem, not concurrency, and not fixed by B1. B1 stops one slow search from freezing others; making any one search fast is async/streaming/index-tuning, separate work.

Owner: Wren (search/API). File B1 + B2 as their own P1 cards — naming them apart is the point.

AC4 — concurrency assumption: VALIDATED (3 independent proofs). The whole design rests on multiple processes safely sharing the stores while ingestion writes and serving reads. All spikes ran on scratch stores — live index.db never touched.
  • Production precedent: embed has run as a separate process writing LanceDB + marking SQLite, concurrent with the API reading, since 2026-04-12 (#1978) — ~6 weeks, no corruption, no contention deaths.
  • SQLite WAL spike (Silas, 2026-05-25, 20s): 1 writer (153 commits/s) + 4 readers (~43k FTS queries/s each) → 0 SQLITE_BUSY, 0 locked, 0 err, reader p50 0.0ms / p99 0.1ms under continuous write, integrity_check=ok, counts consistent. WAL gives cross-process readers-never-block-writer for free. (/tmp/ac4-sqlite-spike.py)
  • LanceDB spike (Silas, 2026-05-25, 20s): 1 writer + 4 readers → 0 conflict, 0 err, reader p50 8ms / p99 19ms, all appends landed (20000+539). Manifest-versioning handles 1-writer / N-reader cleanly — exactly the design shape (embed is the only Lance writer; serving only reads). (/tmp/ac4-lance-spike.js)
  • Fuseki: already a standalone HTTP server, multi-process by construction — zero new risk.
Conclusion: the decouple-by-data assumption is proven across all three stores. The data-store boundary is a safe buffer between tiers. No blocker to migration.
Why this is the fix, not #3077. #3077's handler patches are stopgaps — real wins (index/embed faster) but still inside the one process, so the next heavy handler freezes everyone again. This removes the category of failure: ingestion can't freeze serving because they're not in the same process. It's also the product bar made structural — the team-model only beats solo Claude Code if the coordination layer can't be frozen by its own pipeline. Solo has no shared loop to freeze; our advantage is negative until this lands.
