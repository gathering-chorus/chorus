chorus-api tier split (#3080)

Ingestion workers feed the stores; a thin serving API reads them; the data stores are the contract. Decoupled by data, not process. From Jeff's architecture call, 2026-05-25. Designing-step — architecture + decomposition; build cards follow per job.

The problem

chorus-api is one Node process — one event loop — hosting two fundamentally different kinds of work:

  • Serving: request handlers (search, cards, context, nudge) that must answer fast.
  • Ingestion: batch jobs (crawl, reindex, embed, cache-warm) that are heavy, periodic, CPU/IO-bound.

Because they share one loop, any heavy ingestion job freezes all serving for every role + Jeff at once — the shared-fate freeze behind the all-session eventloop.blocked alerts (~90/hr post-deploy, blocks up to 15s, confirmed by #3079's live stack capture pinning even a 5–7s GET /api/chorus/search as a symptom). Patching handlers one at a time (#3077: index, embed, search) is whack-a-mole because the design is wrong, not the handlers.

The key insight: ingestion has no real dependency on the API

Each ingestion job depends on the data stores, not on chorus-api:

  • reindex reads files → writes SQLite.
  • embed reads SQLite → calls Ollama → writes LanceDB → marks SQLite.
  • crawl reads SQLite/Fuseki/Loki → writes a summary row to SQLite.

None of that needs the API. They touch SQLite, LanceDB, Fuseki, files — directly. chorus-api is just another client of those same stores. The dependency that exists today is incidental: reindex/embed are setInterval'd into server.ts (code placement), and the crawler curls its own API (GET /api/chorus/crawl/:domain — a self-call inversion). They were bolted into the API; they never depended on it.

AC1 — the two-tier architecture

The boundary contract: the serving API depends on the data in the stores, not on the worker processes. Workers write to the stores; the API reads from them; neither tier invokes the other. A worker grinding a 5-second job cannot block a search — they are not in the same process, and the store is the buffer that decouples them.

AC3 — decomposition + migration plan

Each ingestion job becomes a standalone worker that hits the stores directly, on its own cadence. Sequence smallest/safest first; each migration is its own build card with a runtime-outcome AC (a heavy run of that job cannot block a serving request — measured post-migration).

  1. reindex — smallest, cleanest (files → SQLite, no external service). Move the setInterval out of server.ts into a standalone worker. First, lowest risk.
  2. embed — worker hitting SQLite + Ollama + LanceDB. Second.
  3. crawl — folds in #3069 (crawlers as declared instances): each collector becomes a worker instance writing the stores directly; stop the self-call (the crawler no longer curls the API). Heaviest, last.
  4. cache-warm — either move the counts to the reindex worker (it already touches SQLite) or make them cheap/approximate; the API reads cached values.

The serving tier shrinks to request handlers only. Search stays but gets the #3077 non-blocking treatment (cache first, worker for cold FTS).

AC4 — the one real risk to validate FIRST (a spike, before any migration): concurrent store access across processes. SQLite WAL allows multiple processes to read/write, but we must prove it under realistic load — workers writing while the API reads, no SQLITE_BUSY deaths (note #3073's busy_timeout work), no corruption. Same check for LanceDB multi-process. The entire design rests on this; if it doesn't hold, the buffer isn't the store and we rethink. Measured spike, not assumption.
Why this is the fix, not #3077. #3077's handler patches are stopgaps — real wins (index/embed faster) but still inside the one process, so the next heavy handler freezes everyone again. This removes the category of failure: ingestion can't freeze serving because they're not in the same process. It's also the product bar made structural — the team-model only beats solo Claude Code if the coordination layer can't be frozen by its own pipeline. Solo has no shared loop to freeze; our advantage is negative until this lands.
