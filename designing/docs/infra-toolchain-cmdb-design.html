Infrastructure / Toolchain CMDB — Design of Record

#3560 (proven slice) · ADR-050 · Silas · 2026-06-22 — the operational-context domain ADR-033's heralds project into

1. Why — the morning incident

The electric company swapped the house meter; the brief power loss took Bedroom (M2 Pro) down. Six health nudges fired and I reconstructed the dependency graph by hand — launchctl, ping, arp, port-probes — to answer "what died and why." That graph should already exist as queryable structure. ADR-033 decomposed the crawler into per-facet heralds and declared the operational facets (disk, network, compute, alerts) as Silas-owned with write-home = the graph — but the domain they write into was never modeled. There are no Tool / Host / Volume / Network classes for the heralds to project into. This is a CMDB by any other name, and its canonical failure mode is a hand-curated inventory that drifts into confident wrongness (our 2026-03-12 topology is already stale).
"If Bedroom dies, what dies with it?" should be a runsOn × dependsOn traversal answered before the outage — not reconstructed after.

2. Two domains, both Silas-owned (ADR-045 punning)

chorus:infrastructure
the substrate — hosts, volumes, networks. owl:Class + individual of chorus:Domain.
chorus:toolchain
the technologies that run on it (replaces the #3350 stub's scope). owl:Class + chorus:Domain.
Each is a punned class (so owl-api projects it) and a Domain individual (so it sits in the ADR-041 tree with ownedBy/atStep). Members bind via chorus:contains.

3. The Configuration Items

Infrastructure triad (compute / storage / network)

chorus:Host
compute — a machine. Library (M1, primary), Bedroom (M2 Pro, secondary).
chorus:Volume
storage — a disk / mount. Carries the disk budget (warn 90% / crit 95%, C1–C7).
chorus:Network
the LAN — 192.168.86.0/24, DHCP-volatile.

Toolchain

chorus:Tool
one technology / CI — Fuseki, Loki, Lance, MySQL, Vikunja, MCP, WordPress, act, Prometheus, Grafana.
chorus:Toolchain
a purpose-grouped composition of Tools (persistence, observability, coordination, pipeline, web). Carries chain-level facts: provides, recoveryStrength (weakest-link), chainHealth.

Reuse, don't rebuild (no-competing-implementations): where a LaunchAgent exists, Tool runsAs → Service rather than re-describing it. Tool adds only what Service lacks: dependsOn, backedUp, managedBy, runsOn.

5. Discovered, not declared — the anti-rot loop

ADR-033 herald— crawls live → CI instances in graph→ reconciler flags → model-vs-actual delta→ re-crawl
Instances are projected by the operational heralds (pull, not push) and reconciled by re-crawl — never hand-curated. The model is the schema; reality populates it. Same anti-drift DNA as owl-api-generates-from-model: a hand-maintained CMDB is worse than none — it is trusted and wrong. Operations (backup, restore, probe, compact) are verbs run against a CI (reusing the MCP verb registry), each carrying guarantee (consistent | bounded | idempotent | reversible | off-machine) and verifiedBy (the probe/test that proves it).

6. The proven slice — #3560 Fuseki recoverability

One CI, modeled end-to-end as the exemplar:
Tool: FusekipersistsTo → Volume: ~/.gathering/databackupTarget → Volume onHost: Bedroom (≠ self)
FactValue
holdsState31,032,717 triples (TDB2)
backedUptrue — was false; this card flips it

verifiedByrestore round-trip — 62 GB pulled, opened with tdb2.tdbquery, counted, recovery-on-open clean
schedulecom.gathering.fuseki-backup — daily 03:00
Why APFS snapshot, not $/backup (Thrift dump fails "Unrecognized type 0" on this store) or tdb2.tdbbackup (2nd process fights Fuseki's lock) or raw rsync of the live dir (torn copy): the snapshot is an atomic point-in-time view; rsync copies frozen bytes with no contact with the running store, and TDB2 recovers-on-open.
Companion safety: the raw unguarded DELETE in tag-tests-domain.py is retired behind a bounded guard (_DOMAIN_GRAPH regex) — only urn:chorus:domain:<name> may be cleared, never a system/portfolio graph; proven by a hermetic bats test.

7. What this makes answerable

QuestionBecomes
"If Bedroom dies, what dies with it?"runsOn × dependsOn traversal — the morning's six-nudge scramble as one query
"What holds state with no off-machine backup?"Tool where guarantee=off-machine ∧ status=missing — the incident as a one-liner
"Is the persistence chain recoverable?"Toolchain.recoveryStrength — weakest-link rollup over its Tools

8. Build order (continues ADR-033's)

  1. Schema (ADR-050): classes + edges, modeled, proven through owl-api (renders the domain from real instances).
  2. One facet end-to-end = this #3560 Fuseki slice — discovered live by the storage herald, backup/restore verbs bound, verifiedBy a tested round-trip.
  3. Fan out the remaining heralds (compute, network, the rest of the Tools) onto the proven schema.

9. Not in scope

  • The other unmodeled Borg reflection domains. Model one operational domain well as the exemplar; a deliberate reflection-layer pass is a later, separate decision.
  • Cross-machine harvest engine. The harvest engine stays single-machine (Library); CIs describe both machines.
