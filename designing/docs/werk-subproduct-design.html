Werk — Subproduct Design

Kade · as-is reconcile 2026-06-11 (#3340) · prior 2026-05-16 / 2026-05-25 (#3078) · subproduct-design doc, peer to athena-subproduct-design.html and loom-subproduct-design.html. Werk is the Chorus execution protocol — how the team realizes value: turning intent into running, verified software across Building and Proving.

TL;DR — The work this team does could be prompted into Claude for $200/mo. The reason it's worth more than that — the reason a service built on it can be sold — is that the output arrives proven: running, demonstrated, accountable. Not "I think it works," but "here it is, running, and here's the proof." Werk is the part of the model that makes "proven" true. Solo, you are the proof step yourself — Claude says done, the stale binary is still serving, and you find out when it breaks in front of someone. Werk closes that gap: one trace from /pull to the running cdhash, a check per verb, gates that refuse early — so a green result can't lie. Werk owns Building and Proving; the upstream steps decide what to build. Owner: Kade.

What changed in the 2026-06-11 reconcile (#3340): mechanics brought to as-is from the three reconciled service designs (#3339) — the verb monolith is GONE (ten Rust binaries); the pipeline is ONE go-gated werk.yml on act; GO=accept; three of the four 2026-05 headline gaps CLOSED (try-before-buy → per-card werk-slot demos; deployed≠running → cdhash verify; receipts → shared-trace propagation); gates run as demo-prework subagents + the cold-eyes review; NEW §6: the model layer — query-backwards OWL → owl-api → generated domain APIs (Jeff's 2026-06-11 direction), with roles/kade/ontology/werk-domains.ttl as the first artifact.

1 · Why pay, not just prompt

The question the whole team has to answer. Anyone can point Claude Code at a problem for $200/mo and get code that says it's done. If that's all this is, the subscription wins. The model is the same on both sides of the line — Claude writes the code either way. So Werk, and the team it serves, earns its keep only by delivering the one thing a solo prompt structurally can't: output you can stand behind.

Solo, you are the proof step. A card reads Done while the old binary is still serving; the demo went green over a stale build; the deploy reported success; the work stalls between steps because a human has to carry it across; weeks later no one can say which commit is actually running. Prompting alone, there is no one to catch the false-green but you — you are the architect, the reviewer, and the one who eats it when it breaks in front of your customer. The difference that justifies the cost isn't the code. It's the discipline around it.

Werk is that discipline. It's the protocol that carries a unit of work from pulled to proven and accepted, so the answer to "is it actually done?" is always a yes you can check, not a yes you take on faith. The /werk verbs are the interface; underneath — one trace from /pull to the running cdhash, a check per verb, gates that refuse early — exists for exactly one reason: so a green result can't lie. That is the difference between "I built something" and "I can stand behind it" — and standing behind it is what the service is sold on.

And it is versioned. Every line of every session opens with the protocol version all three roles are executing under — the Werk v1.4 in this turn's header. That stamp isn't decoration: it's how the team stays in sync. If two roles show different numbers, they've drifted, and that's a defect (#2311). You are inside Werk before you have pulled a single card — the protocol is the ground the whole session stands on, and the card pipeline is where it turns that ground into shipped, proven value.

Where value gets realized — Building and Proving: the upstream steps (shaping · designing · directing) produce intent; value is only realized when intent becomes running, verified software. That happens in two steps, and Werk owns the execution of both: Building (turning shaped/designed/directed work into running software on main — Werk's canonical anchor, chorus:werk · atStep Building) and Proving (/demo → /acp — the work shown working and accepted with a coherent receipt). Building makes it real; Proving makes it true. The verbs map straight onto the two: pull→commit→push→build = Building; test→review→demo→GO (merge·sync·deploy·accept) = Proving.

How we know it's working — observed, not asserted. Werk's job is to make output you can stand behind; whether it actually does is measured in Observing Value, the canonical home for value observation. Werk's contribution is the raw data: it emits the spine events (card.pulled / card.accepted / binary.deployed / error + reopen) the flow metrics compute from — cycle time, lead time, escaping defects (the reactive-card rate), and MTTR. The bar: those four trend the right way and false-green acceptances trend to zero. Canonical (Athena v2): chorus:werk · status operating · owner Kade · domains pipelines / version-control / code / tests.

2 · The product

Werk is the execution protocol — how the team turns work into a done, verifiable result. Its users are the three roles building Chorus and Gathering, and Jeff accepting the work — the operators of the method, not its buyers. The external customer never touches Werk; what they buy is a service Werk makes trustworthy. The promise to the operators: every card that enters leaves as merged source on origin/main with a coherent receipt — one trace, /pull to deployed cdhash. Werk does not build artifacts or run them — that's the sibling Build + Deploy (Silas), handed off at the post-/acp release-trigger.

The verb family is the developer interface — ten zero-dependency Rust binaries (werk-pull · commit · push · build · deploy · review · demo · merge · accept plus werk-sync/werk-unpull for recovery and the /pull inverse), each its own flock, typed refusals, witness + spine emit (ADR-032), reachable via typed MCP tools and /pull//unpull. The pipeline is ONE workflow file (werk.yml) run by act — two go-gated jobs with the human GO between them, a stopped pipeline, not a held connection, holding the walk-away bar (#3279/#3193). GO=accept (#3311): Jeff's go at the demo IS the acceptance; there is no second accept moment. The chorus_commit/chorus_acp monolith is deleted, not deprecated. One environment (production, live); the per-card werk slot is the demo variant; gh mirrors each verb as a commit-status. Full mechanics: ci-pipeline-service-design.html (reconciled #3339).

3 · Surfaces — how the team leverages Werk

The entry points the three roles use to drive work through Werk:

  • The session header — every response opens with Werk v1.4: the protocol version all three roles execute under, refreshed each turn and drift-checked across roles (#2311). The most-seen surface of all — present from session start, before any card.
  • The verbs + the one trigger — /pull (card to WIP + fresh werk), chorus_werk (run the pipeline to the demo stop; go:true on your GO lands it end-to-end), /unpull. The primary developer interface.
  • The proving surfaces — 5 demo gates (product/code/quality/arch/ops) run as Explore subagents in demo prework + the cold-eyes review (one adversarial agent, diff+AC only, anti-ceremony-guarded — #3193); recorded on the witness; werk-demo refuses to present un-gated (#3284). The cockpit shows verbs · gates · tests · review · variant at the go/no/more decision.
  • MCP write-paths — one typed tool per verb (werk-pull … werk-review) + chorus_werk; old chorus_*_card names are deprecation aliases (ADR-031). The surface is PINNED by a contract test — adding a verb is a conscious pin edit (#3329).
  • GitHub — the PR + per-verb commit-status checks (chorus/pull..accept); the source-of-truth + process-holder surface.
  • The spine — card.pulled / card.demoed / card.accepted + build/deploy events; the observability surface, queryable by card or trace.
  • The board — the cards CLI (the cards-runner is Wren's, consumed from The Clearing) — where work enters and exits.

(ADR-033 was never landed; its act-as-orchestrator framing is retired.) Each instance conforms to the set's contracts; each owner builds their own.

Set-governance owner: Kade — frameworks, ADRs, act, contracts; NOT the individual instances (cards-runner is Wren's, build+deploy is Silas's). Design doc (the specifics): ci-pipeline-service-design.html — the ci-cd instance: act, the verb binaries, the v8 model.

Card ↔ artifact coherence

The pipelines domain holds 1..n instances; the card pipeline is Wren's cards-runner (Werk consumes it to drive a card Next→WIP→Demo→Done). The artifact lifecycle is not a peer pipeline — it's a coherence concern: the card and the artifact lifecycle share one source-of-truth (the card + its commits), but they observe different state and emit different events, and the two must stay reconciled or the graph drifts behind reality.

Coherence findings — synthesis

1. Two of the four Domains have service-designs; two don't. Version-control and pipelines have canonical docs and graph instances. Code has a markdown service-design that lags behind the wired pre-commit rules. Tests has no service-design at all — its substance is spread across four markdown files. The next service-design Werk needs is tests-service-design.html.
2. Werk-the-subproduct is the locus where work-as-product and the artifact lifecycle meet. Athena gives the structural model; Werk runs the assembly line that produces instances of that model. Each accepted card must advance the artifact lifecycle in step with the card pipeline or the graph drifts behind reality.
3. The pipelines domain's biggest gaps are product gaps, not infrastructure gaps. Customers can't try before they buy (/werk/build + /werk/deploy pre-/acp); receipts don't reconcile (card_id across launchd); the brand promise isn't measurable (no deployed-equals-running). The CI/CD service-design's gap list reads as a punch list; under Werk-as-product framing, these are headline customer-experience defects.
4. Werk does not own build+deploy. The post-/acp release-trigger is the handoff line. Build+Deploy is a sibling subproduct under Silas. Werk's responsibility ends at "card accepted, commit on origin/main, trace_id propagating into the build pipeline."

Alternatives considered & risks

Rejected approaches (why this shape):

  • The chorus_commit / chorus_acp monolith — opaque, no per-step state. Rejected for per-verb binaries under the verb contract (ADR-032): each verb is testable, refusable, and carries its own gh check.
  • Renaming pipelines → ci-cd (#2720) — collapsed the governed set into one instance. Reversed: pipelines is the domain (1..n instances); ci-cd is one instance Kade owns.
  • The "two pipelines" framing — pinned the count at two and mis-typed the artifact lifecycle as a peer pipeline. Replaced by 1..n instances + the card ↔ artifact coherence claim.
  • A preview / staging environment — rejected: there is one environment (production). demo proves werk code live before acp lands it; no separate stack to drift.

Risks / failure modes: the operational risks are the headline gaps in §4 — can't-try-before-you-buy, receipts-don't-reconcile, deployed≠running-unasserted. One structural risk to name outright: model-mirroring — because this doc's outline mirrors the Athena containment graph, anything the ontology can't express (cross-domain concerns, alternatives, this risk list itself) has no natural home; this section is the deliberate slack against that blind spot.

6 · The model layer — query-backwards OWL → generated domain APIs

Jeff's direction (2026-06-11), verbatim intentions: (1) get werk stabilized, observable — all other chorus work relies on werk; (2) model the OWL for werk so owl-api can generate APIs for Kade's domains in a deterministic way; (3) those APIs are really domain APIs that gather data from the graph via queries.

The chain: werk stable+observable → werk's witnesses/spine events land as graph data → the OWL models the four domains → owl-api GENERATES the domain APIs deterministically → the APIs are query surfaces over the graph, never hand-built handlers. This ends the reconcile treadmill structurally: generated surfaces cannot forget to notice that the system changed.

The authoring rule — query-backwards: every class and property must map to graph data that exists or has a defined ingestion path (witnesses in ops/logs/werk-*.jsonl, spine events, verb runs, gate results, review verdicts, cards). A class that cannot answer a query is decoration and gets cut. First artifact: roles/kade/ontology/werk-domains.ttl — the four-domain refinement (cicd, version-control, code, tests), each class annotated with its data source and ingestion status. Grain: extends the canonical chorus: model (silas/ontology/chorus.ttl v0.3.0); the IRI-formation layer is explicitly PROVISIONAL pending Silas's #3256 keystone ADR (model-1) so the namespace grain lands once. Consumers: Silas's model program (model-3 reconcile cross-checks; model-5 owl-api generates); werk-test (#3190) is the first generated-API customer — it queries the tests domain for which-tests-for-which-card.

The DE finding that sequences the work: cicd and version-control are graph-ready today (rich, well-named witness events per verb run, gate, deploy); tests is not — no per-test identity, results transient. The tests OWL therefore ships ingestion-path-first or not at all.

Headline contract: hermetic-by-default with explicit integration gating, AC → tests → code (DEC-1674), run-tests contract named as the load-bearing CI↔tests interface.
  2. Card-driven artifact-status advance. When /acp accepts a card that touches a Service, write the artifact-status transition (chorus:status) through the Athena API as part of the commit. Without this, the artifact lifecycle drifts behind reality and Athena's "graph is whole" claim fails.
  3. /werk/build + /werk/deploy as pre-/acp act jobs. The "try before you buy" gap. Designed in ci-pipeline doc, shipped only post-/acp today. This is the single highest-impact change to Werk-the-product experience.
  4. card_id across the launchd boundary. Plist env override on the kickstart, OR pipeline-side manifest read. Until this lands, post-/acp events fire as role=system and cross-step correlation by card is unbuildable. (Same gap Build+Deploy names as P1 #1.)
  5. Werk-domain instances in Athena. The four Domain instances and their hosted Service instances need to be authored under the recursive-Domain model — today the graph has SubDomain instances mis-typed per Athena's audit. Migration sequenced behind Athena's type-migration card.
