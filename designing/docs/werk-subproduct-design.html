Werk — Subproduct Design

Kade · 2026-05-16, reconciled 2026-05-25 (#3078) · subproduct-design doc, peer to athena-subproduct-design.html and loom-subproduct-design.html. Werk is the engineering pipeline — the line that turns "card in Next" into "running code on origin/main with a coherent receipt."

Werk is the engineering pipeline product. Its customers are the three roles building Chorus and Gathering, and Jeff as the human accepting work. Its promise: every card that enters Werk leaves as merged source on origin/main with a coherent receipt — card_id and trace_id propagated through every step from /pull to deployed cdhash. Four Domains compose it (code, tests, version-control, pipelines) sharing one discipline: typed contracts at every boundary, refuse early, emit observable events. Werk does not build artifacts or run them — that's the sibling Build + Deploy domain, handed off at the post-/acp release-trigger.
Two pipelines, one product. Werk runs two state machines in parallel from a single source of truth (the card and its commits):
  • The card pipeline — the cards-runner, owned by Wren (not Werk). States: Next → WIP → Demo → Done, emitting card.pulled / card.demoed / card.accepted, gated by the five role gates + five pre-merge surfaces. Werk consumes this state; it doesn't run it. This is the assembly line.
  • The artifact lifecycle — what Werk leaves behind, governed by Athena. States: exploring → building → operating → retiring on the Service or Domain the card touched. This is the product on the shelf.
Werk's load-bearing claim: every accepted card advances both pipelines coherently. A card that ships code without advancing the artifact's status is incomplete — the artifact lifecycle is part of the deliverable, not a follow-up.

What Werk is

The engineering pipeline subproduct of Chorus, at the Building value-stream step. Composed of four child Domains. Owner: Kade. The /werk verbs (/pull · /demo · /acp) are its developer interface; act is the local runner beneath them (build/deploy/verify are steps inside demo & acp, not standalone commands).

The pipeline model is v8 (ADR-033): git ⊃ env ⊃ DSL running on act — exactly one environment (production, live), GitHub as source-of-truth (main) + per-card process-holder, and one commit-status check per verb (chorus/pull · commit · build · deploy · verify · accept; all green = every verb ran, push folds under commit). The verbs are decomposing from the chorus_commit / chorus_acp monolith into per-verb binaries under the verb contract (ADR-032): werk-pull shipped (#3045); commit #3056, accept #3057, build #3061, deploy #3062, shared-trace #3063, acp orchestrator #3064 carded. The five pre-merge hooks plus the five role gates are the refusal surfaces. Full model + as-is/to-be diagrams: ci-pipeline-service-design.html.

The four Domains

pipelines BUILDING

What it is: a domain of 1..n pipeline instances, owned per-role — the ci-cd/build pipeline is Kade's; the cards-runner (the Next→WIP→Demo→Done card lifecycle) is Wren's; build+deploy is Silas's. Promise (set-governance): Kade governs how the set behaves, not the instances — every pipeline is uniform via the frameworks: the verb contract (ADR-032), the v8 model (ADR-033), resource/verb/grain (ADR-031), act as the local runner, typed refusals, and one commit-status check per verb. Each instance conforms to the set's contracts; each owner builds their own.

Set-governance owner: Kade — frameworks, ADRs, act, contracts; NOT the individual instances (cards-runner is Wren's, build+deploy is Silas's). Service hosted: act-runner (local LaunchAgent com.chorus.building-pipeline). Design doc: ci-pipeline-service-design.html (the ci-cd instance).

Shipped: orchestration via act + workflow_dispatch + launchd kickstart (#2933, #2939); pre-merge surface 5 (pre-commit) wired; cost-stop / schedule-only main re-run (#2600, DEC-2525); ci-main-red reactive alert. v8 model ratified (ADR-033) + per-verb contract (ADR-032); pull v2 shipped (werk-pull #3045), remaining verbs carded (#3056/#3057/#3061/#3062/#3063/#3064).

Gaps (headline, in product-frame):

  • Can't try before you buy. /werk/build and /werk/deploy as pre-/acp act jobs are designed but not shipped — Jeff accepts before the feature is observable in a running instance. This is the customer-facing #1 gap, not item 6 on a punch list.
  • Receipts don't reconcile. card_id is set in env by /acp but lost across the launchd boundary — pipeline events fire as role=system with no card_id. Cross-step correlation by card is unbuildable from current data. (Build+Deploy doc names the same gap from the other side as P1 #1.)
  • No deployed-equals-running assertion. Spine says binary.deployed; nothing today proves the deployed cdhash matches the running PID's cdhash. The brand promise of invariance isn't measurable yet.

Downstream gaps tracked: drift lane (#2527), sensor consumer registry (#2528), run-tests contract (#2529), ratchet drift signal (#2496), infra-card demo adapter (#2499). (Closed since the 2026-05-16 draft: KNOWN_FAILS allowlist #2497 Done; GHA org-billing alert #2591 Won't-Do.)

version-control OPERATING

Promise: source state on origin/main is the only truth; every mutation is a typed MCP call (chorus_commit, chorus_acp) carrying card_id + trace_id; werks are ephemeral per-card worktrees, never branch-swapped in place; cross-role contamination is refused at four distinct surfaces.

Owner: Kade. Services hosted: git-queue (mutex-locked wrapper, branch-check), werks (chorus-werk add/remove), chorus-hook-shim (PreToolUse refusals). Design doc: version-control-service-design.html.

Shipped: chorus_commit / chorus_acp MCPs with verify-after sequenced steps (#2782); ephemeral per-card worktrees replacing persistent per-role mutation (#2735, #2913); pre-merge surfaces 1–4 (wrong-branch, raw push, dangerous git, cross-worktree contamination — #2580, #2598, #2625); trace propagation through commit trailers (#2931); idempotent /acp re-run path on orphan branches (#2943); pre-push refusal on stale-base silent deletions (#2944).

Gaps: three-domain rollback across version-control / build / deploy (named in Build+Deploy doc as structurally hard, no card). Shared-HEAD race on canonical accepted-residual — closed (#2706 Done). Same-role wrong-card commit-time defense was filed as #2641 but declined Won't-Do in favor of the #2625 PreToolUse worktree-contamination guard plus chorus_commit_status pre-flight.

code BUILDING

Promise: staged code that reaches a commit is type-clean, lint-clean below thresholds, free of duplicate-string accretion (≥5), and below cognitive-complexity 12 — refused at the pre-commit boundary, not at review.

Owner: Kade. Service: quality (gates + lint rules). Design doc: quality-service-design.md.

Shipped: sonarjs error tier wired in pre-commit (no-duplicate-string threshold 5, cognitive-complexity threshold 12), magic-comment override (// cog-override: <reason>) emits audit event (#2603, #2627). No-warn-tier convention asserted.

Gaps: no-warn-tier convention is not self-enforced — nothing today refuses a contributor adding a warn-tier rule. Trust audit for pre-merge maturity (claim that "trust burden moved to pre-merge" needs a check that pre-merge isn't quietly weakening) is uncarded. Ratchet drift signal (#2496) still planned — baseline accretion only caught via manual audit.

tests BUILDING

Promise: tests describe the experience the card was filed to fix, not internal state; pre-commit runs hermetic tier only; integration tier is explicitly gated and runs in /demo. AC items become test cases before code. (DEC-1674.)

Owner: Kade. Service: test-runner. Design doc: not yet authored as a service-design — currently lives across TESTING.md, test-categorization.html, ICD_TEST_STRATEGY.html, zero-test-subdomain-triage.md. Gap: tests has no canonical service-design doc; this is the first thing the domain needs.

Shipped: hermetic-by-default discipline named as principle and enforced in pre-commit (cargo test --lib --bins / jest scope). Integration tests gated in /demo. TDD discipline written into CLAUDE.md (DEC-1674).

Gaps: no run-tests contract (#2529) — the load-bearing CI ↔ tests interface is unnamed. Hermetic/integration tagging on individual tests is partial (some bats files marked, most TS tests rely on filename convention). chorus-hooks/inject test isolation (#2491) has hardcoded paths in some tests. sessions.test.ts pre-existing fail (#2493).

Two pipelines reconciled

The card pipeline and the artifact lifecycle share one source-of-truth (the card + its commits), but they observe different state and emit different events.

The coherence claim: a card that touches a Service (e.g., quality-service, version-control-service) must declare whether its acceptance advances that Service's chorus:status. If the card is moving building → operating, /acp writes that triple through the Athena API as part of the commit. If the card doesn't move status, that's stated explicitly (a maintenance card on an operating service stays operating). Today: no enforcement — artifact status drifts behind reality. Coherence findings — synthesis

1. Two of the four Domains have service-designs; two don't. Version-control and pipelines have canonical docs and graph instances. Code has a markdown service-design that lags behind the wired pre-commit rules. Tests has no service-design at all — its substance is spread across four markdown files. The next service-design Werk needs is tests-service-design.html.
2. Werk-the-subproduct is the locus where work-as-product and the artifact lifecycle meet. Athena gives the structural model; Werk runs the assembly line that produces instances of that model. Both pipelines need to advance per accepted card or the graph drifts behind reality.
3. The pipelines domain's biggest gaps are product gaps, not infrastructure gaps. Reframed: customers can't try before they buy (#/werk/build + /werk/deploy pre-/acp); receipts don't reconcile (card_id across launchd); the brand promise isn't measurable (no deployed-equals-running). The CI/CD service-design's gap list reads as a punch list; under Werk-as-product framing, these are headline customer-experience defects.
4. Werk does not own build+deploy. The post-/acp release-trigger is the handoff line. Build+Deploy is a sibling subproduct under Silas. Werk's responsibility ends at "card accepted, commit on origin/main, trace_id propagating into the build pipeline."

OWL + SHACL — conformance to Athena v2

This subproduct adds no new structural classes. It uses Athena's chorus:Product / chorus:Domain / chorus:Service with the recursive containment model. The Werk-specific shape is:

Athena's invariant-rendering claim ("every stored attribute required, no conditional show-if-defined branches") is the design-time twin of Werk's runtime fitness function ("for every commit on origin/main, all step pairs emit started+completed with same card_id"). Same posture, different layer. Werk satisfies Athena's SHACL at write time; Werk's own pipeline satisfies its fitness function at run time. Both gates refuse early.

Path to close

  1. tests-service-design.html. Werk's only Domain without a canonical service-design. Consolidates TESTING.md / test-categorization.html / ICD_TEST_STRATEGY.html / zero-test-subdomain-triage.md. Headline contract: hermetic-by-default with explicit integration gating, AC → tests → code (DEC-1674), run-tests contract (#2529) named as the load-bearing CI↔tests interface.
  2. Card-driven artifact-status advance. When /acp accepts a card that touches a Service, write the artifact-status transition (chorus:status) through the Athena API as part of the commit. Without this, the artifact lifecycle drifts behind reality and Athena's "graph is whole" claim fails.
  3. /werk/build + /werk/deploy as pre-/acp act jobs. The "try before you buy" gap. Designed in ci-pipeline doc, shipped only post-/acp today. This is the single highest-impact change to Werk-the-product experience.
  4. card_id across the launchd boundary. Plist env override on the kickstart, OR pipeline-side manifest read. Until this lands, post-/acp events fire as role=system and cross-step correlation by card is unbuildable. (Same gap Build+Deploy names as P1 #1.)
  5. Werk-domain instances in Athena. The four Domain instances and their hosted Service instances need to be authored under the recursive-Domain model — today the graph has SubDomain instances mis-typed per Athena's audit. Migration sequenced behind Athena's type-migration card.

Not in scope

  • Build + Deploy (Silas, sibling subproduct) — the artifact-production-to-running-state vertical. Werk consumes; doesn't own.
  • Athena schema itself — Werk conforms to Athena v2, doesn't redefine. The recursive-Domain model and SHACL shapes are upstream decisions.
  • The card system — Vikunja + cards CLI + the card lifecycle states are Wren's. Werk consumes the card-pipeline state-machine; doesn't define it.
  • Gathering's CI — lives in jeff-bridwell-personal-site repo; smoke check is its problem (ADR-026 addendum).
  • Quality at runtime — observability of the running system is Borg's substrate, not Werk's.
