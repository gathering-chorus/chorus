Chorus UI Architecture — the design-system + presentation layer (#3415)

Kade · 2026-06-14 · Designing-step artifact · instance of the product-design template · governs #3413 (werk pages), #3412 (/flow) and every future page

Thesis. A page is model + view. Today both are hand-fused per page, so 60 pages are one structure drawn 60 divergent ways. Split them: structure comes from the model (owl-api), visual comes from ONE design system every page renders into. Same model → same renderer → consistent pages, by construction. This is the generate-from-model bet — already made on the backend — applied to presentation.

1 · Identity

  • Name: Chorus UI Architecture / Design System
  • Owner: Kade (presentation layer); shared with Wren on the owl-api/forge generation path
  • Value-stream step: Designing — it is the presentation-layer instance of the api-driven value-stream template, the way Athena is the structural instance
  • Status: building — token base + specimen live (system.css + ui-system.html); generation path + component library are the open legs

2 · Promise

One legible, consistent, printable presentation surface across all of Chorus — and a capability any system Chorus builds inherits. Stop authoring pages; start generating them from the model into a shared design system. A page looks right because it can't look wrong: it has no private styles, only system tokens and components.

3 · The problem (from the de-facto audit)

An audit of ~60 served pages found:

  • Two internally-consistent themes — editorial-light (Georgia, #1a1a1a on white, blue accent — design docs) and operational-dark (system-ui, #e6e8eb on #0f1115, saturated accents — cockpits). The split is intentional and correct.
  • Zero shared CSS — every page is 100% inline <style>; the same component (card, badge, table) is re-styled 10+ times.
  • ~25 near-duplicate color values — five ways to write white, three grays.
  • Real semantic token systems already in use — value-stream stage colors and role colors, re-declared per page instead of named once.
Diagnosis. Strong thematic discipline, zero infrastructure discipline. The visual intent is clear; the implementation is fragmented. We don't need 60 designs — we need one system, two themes.

4 · Atomic Design mapping (Brad Frost), with the chorus twist

Standard atomic design hand-builds all five tiers. In Chorus the Pages tier is generated from the model, not authored — so the split is: atoms→templates = the design system (hand-crafted once); pages = model × template (forged).

5 · Two themes, one token base

One set of semantic tokens; two themes set their values. Pages opt in with <body class="theme-light"> or theme-dark. Editorial-light is the default and the print target; operational-dark is the live-cockpit mood. No page re-declares a palette.

6 · Print is first-class

Jeff prints; the /share skill renders Chrome-headless PDFs whose quality depends on page print CSS. So: @media print is baked into system.css (forces the ink-friendly light palette, hides nav + action bar, avoids mid-component page breaks), and the content-actions organism (Save-as-PDF / Share / Reflect) is standard page chrome on every page.

7 · Generation path — AX=JX

owl-api/forge already lists "page" as an output alongside routes + MCP + skill. The page is model × template, rendered into the design system. The same model data a role gets via MCP is what the page shows — AX (agent experience) = JX (Jeff experience), same source, no divergent fallback. Static output only — no runtime template engine (the #3361 deploy lesson: a runtime dep doesn't survive chorus-api's node_modules churn; the design system is static CSS + build/forge-time composition).

8 · Inherited capability

Chorus is a system that builds systems. This design system is part of what the factory imparts to anything it builds. Per the factory/product runtime-independence invariant, it is inherited generatively at build time — the CSS + templates + generator are forged/copied INTO the built system, which then runs standalone — never a runtime dependency back on Chorus. That is why system.css carries zero chorus-specific hardcoding: it drops into Gathering or any future product as-is.

9 · As-Is / To-Be / Path to close

  • As-is: 60 hand-authored pages, inline styles, no reuse, two ad-hoc themes.
  • To-be: one system.css (atoms→organisms) + ~5 templates; pages generated from the model into it; print + content-actions standard; inheritable into built systems.
  • Path to close: (1) tokens — done v1; (2) component library to organisms; (3) the ~5 templates; (4) forge emits pages into the templates; (5) migrate existing pages to link system.css (cheapest immediate win); (6) re-home strays (e.g. chorus-standards.html, #3405).
