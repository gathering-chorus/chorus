At a Glance — the werk pipeline (v8 model, 2026-05-23)

Current model — 2026-06-04 (what shipped). The pipeline is atomic verbs run in sequence — no orchestrator. Each verb (werk-pull/commit/push/merge/build/deploy/demo/accept) is self-contained (own flock, typed refusals, spine emit, inherited trace). The “sequence” is a dumb fail-fast list (werk-mcp.sh — no branching/state), and act, if used, is only a trigger (staged but currently unloaded), never the orchestrator. To-be deltas vs the v8 text below: (1) act is trigger, not orchestrator (see the reframed “Act as trigger” section); (2) werk-acp is retired — not needed, since werk-accept is atomic (finalize-only, #3175); you just run the verbs in order ending with werk-accept; (3) werk-demo = run the test suite against the deployed instance and gate accept on demo.test.passed — today it builds+deploys+waits-for-humans-to-browse but does not run tests (the “agents grading agents” stamp-mill). The v8 narrative below is retained for lineage.

The /werk pipeline is three verbs, keyed by card#: pull (get the work), demo (prove it live), acp (make it permanent). GitHub wraps the environment, which wraps the DSL (the verbs); the DSL runs on act, the local engine. There is exactly ONE environment, and it is production (live) — no preview, no staging. gh = source of truth (main) + per-card process holder; card# = the key. Full visual: werk-pipeline-model.html.

As-Is (2026-06-04) — atomic verbs shipped; run as a flat sequence

What's true now: the decomposed atomic verbs are shipped and live — werk-pull/commit/push/merge/build/deploy/demo/accept — and run as a flat fail-fast sequence (werk-mcp.sh), not a monolith and not orchestrated. Remaining gaps: (1) werk-mcp.sh hard-stops at the prod step — the from-main build path already exists (chorus-deploy builds from canonical=main, the manual proxy run by hand all night); the gap is wiring it into step-7 so prod builds the merged main, not the card's werk (don't write new deploy logic, bind the existing path); (2) werk-demo builds+deploys but does not run the test suite against the deployed instance; (3) act is staged but unloaded (no auto-trigger on merge); (4) werk-acp still exists (to be retired). To-Be — the flat set of atomic verbs, run in order (no orchestrator)

The pipeline IS the verbs, run in sequence — nothing orchestrates. The encoding is werk-mcp.sh (a dumb fail-fast list); act, if used, only triggers it on merge. Each verb is atomic (own flock, typed refusals, inherited trace, spine emit) and the run stops on the first refusal. The flat sequence:

  1. werk-pull — worktree off origin/main + register {card, trace, branch}
  2. werk-commit — rebase onto origin/main, commit (gates run inside the verb)
  3. werk-push — push the branch (force-with-lease on re-run)
  4. werk-build — build the card's code for the demo
  5. werk-deploy — deploy to the role's slot + env-up (the live test surface)
  6. werk-demo — run the test suite against the deployed instance; emit demo.test.passed/failed (gates accept)
  7. werk-merge — resolve the open PR by HEAD oid, squash to main, content-verify
  8. werk-build + werk-deploy from main — the prod build+deploy
  9. werk-accept — finalize (cards-done, close branch+werk); atomic, no merge

No werk-acp — the sequence simply ends with the atomic werk-accept. The diagram below is the current flat atomic-verb sequence.

Status — what's shipped

pull v2 (werk-pull, #3045) shipped 2026-05-23 — one ~294-line zero-dep Rust binary (only the git + cards + gh CLIs), proven against real git/gh/cards; gh registration is a commit-status (chorus/pull/<card>), not a push (pull does not push — the branch is empty at pull). demo v2 (#3046, Wren) in progress. The remaining verbs are carded to the ADR-032 contract — commit #3056, accept #3057, build #3061, deploy #3062, shared-trace #3063 — composed by the acp orchestrator #3064. ADR-033 ratifies the pipeline model they build to (the two diagrams above). There is no preview instance — build + deploy are sub-steps of demo (and acp) into the one production env.

Promise

Every change that reaches main is measured by the same gates regardless of who pushed it. Five hooks fire pre-merge. Five named code-review checks record card-level acceptance. One scheduled GitHub Actions run on main (~daily) is the post-merge witness. One reactive alert wakes the on-call when CI goes red.

Honest scope: four of the five pre-merge hooks block unconditionally; the fifth (pre-commit) is --no-verify-skippable for the carded KNOWN_FAILS pattern, and the allowlist enforcing that convention shipped (#2497, Done). With required-status-checks empty (post-#2600 cost-stop), an accidental main-push lands red and stays red until the next nightly run — up to 24h. The schedule-only retreat is a deliberate cost trade ($15/mo vs $225 projected); the gap is mitigated by pre-merge maturity, not by CI itself.

Pre-Merge Enforcement (Five Surfaces)

Each surface catches a distinct attack vector on the path from "engineer edits code" to "main has the change." The five aren't enumerated by convenience — they're derived from where wrong-shape commits can enter the system.

#Attack vector caughtHookRefusesCard
1Wrong-branch commit (engineer commits to a branch that isn't theirs)git-queue.sh branch-checkCross-role branch contamination#2580
2Raw push (bypassing the queue)pre-push hookgit push without the _GIT_QUEUE_PUSH marker, wrong-role branch#2598
3Dangerous git from agent (Claude Bash tool runs raw git mutations)chorus-hook-shim PreToolUseRaw git push/rebase/cherry-pick/reset --hard#2598
4Cross-worktree contamination (running git ops on the canonical clone while another role is mid-build)worktree-contamination guardgit checkout/pull/reset/switch on the canonical clone when role-state shows another role active#2625
5Bad commit content (typecheck/lint/test failures, secrets, oversize)pre-commit hooktsc errors, jest/cargo failures, lint violations, dup-strings, cog-complexity over 12, secrets, oversize binaries(built up over many cards)

All five emit JSONL audit events to platform/logs/chorus.log.

Surface 5 is the only one bypassable: --no-verify skips it for the KNOWN_FAILS pattern (carded test failures from other roles, traced via #NNNN in the commit message). The allowlist enforcing that convention shipped — #2497 (Done).

Pre-Commit Detail (Surface 5)

platform/hooks/pre-commit runs as part of every git-queue.sh commit. Each check fires only when staged files trigger its scope.

  • Typecheck — tsc --noEmit on changed TypeScript packages
  • Tests — jest on changed JS/TS packages, cargo test --lib --bins on changed Rust crates (hermetic only)
  • Secrets scan — refuses commits writing API keys / .env content / credentials
  • Catalog oversize — refuses oversize binaries to data/catalog/
  • Principle direct-edit — refuses edits to data/principles/ outside the canonical write path
  • Sonarjs error tier (Check 4.4, #2603 + #2627) — blocks on no-duplicate-string (threshold 5) or cognitive-complexity (threshold 12) in staged TypeScript. Magic-comment override // cog-override: <reason> exempts a function and emits a cog.override.used audit event
  • MCP tool description shape — staged MCP tool definitions match the description-shape contract
  • Doc-coherence ratchet — when CLAUDE.md fragments change, doc references stay coherent

Quality Rules (Convention: No Warn Tier)

Active rules at error (block at Check 4.4):

  • sonarjs/no-duplicate-string, threshold 5 — catches the agent-inlining pattern (same literal duplicated 5+ times)
  • sonarjs/cognitive-complexity, threshold 12 — catches the function-too-tangled pattern; PM lowers the threshold (12 → 11 → 10) when the codebase is visibly ready

Convention (not currently self-enforced): every lint either blocks or doesn't fire — no warn tier. Warnings are never handled, so adding a rule at warn level adds noise. This is a discipline, not a pattern with an enforcement surface; nothing in the pre-commit hooks today blocks a contributor from adding a warn-tier rule. A self-audit check would change that — currently a gap.

Install discipline: when wiring a new quality rule, ship the rule + fix-as-we-go. Don't pre-fix every existing violation as part of the install — that's how rule-install cards become unbounded. Fixes happen on touched files at commit time.

Retirement Gates (Forward-Only Structural Assertion)

When a surface is retired (HTTP endpoint, helper function, schema column, code path), the deletion ships with a small bats file containing grep-assertions: "no production reference to the retired surface remains anywhere in the codebase."

Without the gate, the test suite shrinks alongside the surface — a future contributor sees the empty space, types the helper back in, and no test catches it. The retirement gate is structural memory of why the surface was retired, encoded as test.

Live examples (verified 2026-04-30):

  • platform/tests/role-state-card-decoupled.bats (#2467) — 7 @test blocks asserting no skill source passes card= to role-state CLI; no CLAUDE.md fragment uses building card=<id> syntax
  • platform/tests/pulse-rolestate-retired.bats (#2632) — 5 assertions: no setRoleState, no getRoleState, no /api/role-state route, no role_state table CREATE, no Role state describe block in pulse store tests

Pattern earns its keep when the retired thing has structural pull (a future engineer might reach for it). Skip the gate when the surface was an obvious one-off no one would re-introduce.

Affordance-Layer Refusal

When a deprecated input shape needs to disappear (e.g., card= field on role-state writer), refuse it at every surface that could reach the affordance, not just one. Pattern from #2467 + #2629:

  1. Writer parser — refuses the input, exits with helpful error naming the corrective action
  2. HTTP API — refuses POST bodies with the deprecated field
  3. Bats gate — asserts no skill source / fixture / helper passes the deprecated input
  4. Schema — drops the column entirely so the storage path is closed

Each surface catches a different attack vector: writer rejects current callers; HTTP API rejects future REST clients; bats prevents new instances in skill markdown; schema drop closes the storage path entirely. Single invariant, four enforcement surfaces. Belt-and-suspenders is reinforcement here — they catch different things.

Act as orchestrator trigger — deferred (2026-06-04)

Superseded — there is no orchestrator. This section framed act as the pipeline orchestrator; as of 2026-06-04 that's retired. The verbs are atomic and self-contained (own flock, typed refusals, spine emit); the “pipeline” is a dumb sequence of verb calls — see werk-mcp.sh, a flat fail-fast list of step calls with no branching or state. act is built but staged-not-loaded: the com.chorus.building-pipeline plist is unloaded and building-pipeline.yml is workflow_dispatch-only (it does not auto-fire). Its eventual role is a trigger (fire the sequence on merge), never an orchestrator that owns logic. The three contracts below are design targets, not operative behavior — they were never wired (trace propagation breaks at the launchd boundary). Retained as the trigger spec for when act is activated.

The former orchestration framing, for reference: act running building-pipeline.yml under the com.chorus.building-pipeline LaunchAgent, kickstarted post-merge. Three contracts were drawn to make it legible to consumers:

Dispatch contract

What kicks act: launchctl kickstart gui/$UID/com.chorus.building-pipeline, fired by chorus_acp's release-trigger step on every accepted card. Post-#2933 the workflow trigger is on: [workflow_dispatch] — act dispatches the event locally; GHA stays quiet.

Env carried in: CHORUS_TRACE_ID + CHORUS_CARD_ID propagate via the commit's Chorus-Trace-Id / Chorus-Card-Id git trailers (#2931). Build scripts inherit by reading trailers at entry; chorus-log's env-bridge (#2857) tags every downstream build.* / deploy.* event with the same trace.

Signal back: Job-conclusion state (success / failure / cancelled) is observable via launchctl print exit code and the build.completed / build.failed spine events. No async webhook; observers query the spine.

Write-back: On completion or failure, act writes ~/.chorus/tmp/build-result-<card_id>.json with schema { card_id, sha, status, steps_completed, failed_step, cdhash, service }. This file is the synchronous result artifact — consumers that need a structured answer without a spine query (e.g. post-deploy verification scripts) read here first.

Step contract

Three named jobs run in sequence; each has a typed-refusal taxonomy mirroring the #2941 contract on MCP tools.

StepOwnerRefusal taxonomyRecovery
chorus-werk-syncVCSff-only-fail | dirty-tree | detached-headchorus-werk-sync repair (re-attach HEAD, fast-forward, align tree)
chorus-buildbuild+deploycargo-fail | codesign-fail | cdhash-drift | warn-baseline-breachfix code, re-trigger via workflow_dispatch
chorus-deploybuild+deployservice-not-found | bootstrap-fail | probe-fail | rollback-neededchorus-deploy --rollback (restore dist.prev + kickstart)

Sequential-fail-fast is implicit: a step's failure halts the chain, the next step never fires. The job-level concern (act's perspective) is the union of step refusals.

Deploy step contract: there is no preview instance — there is one environment (production). demo deploys werk code into it (live, not yet in main) so Jeff sees the feature running before acp makes the source permanent. The refusal probe-fail encodes the observability check: if the surface isn't observable post-deploy, the step refuses. acp re-deploys from main once the source lands.

Observability contract

Every act-emitted event carries the trace_id from the commit trailer, joining the ACP arc to the build/deploy arc as one chain queryable via chorus_logs_for_trace. On failure, the typed refusal (above) emits as build.failed or deploy.failed with result=fail + error= + structured step= + reason= fields (the #2944 ERR/EXIT trap pattern). Idempotent re-run path (#2943): re-invoke chorus_acp on an accepted card with an orphan branch triggers the cleanup-only path, skipping commit/push/PR.

Four step characteristics — every step in the pipeline must satisfy all four for the fitness function below to be authorable. Current status from the build+deploy domain:

Fitness function (reframed 2026-06-04 — no orchestrator): there are no pipeline.step.* / pipeline.completed events — those belonged to the retired orchestrator. The observability contract is the flat verb sequence's OWN spine events: each atomic verb emits <verb>.started / .completed / a typed-refusal on the inherited card trace (the #3163/#3167 emit work). For a card, the complete per-card trace is the ordered run of verb events — commit.completed → push.completed → merge.completed + content.verified → deploy.completed (with cdhash=binary.running confirming deployed==running). A missing verb event = a pipeline gap, queryable via chorus_logs_for_trace(trace_id) on the card's inherited trace. The verbs emit; Borg produces the shape and the gap-detection (#3215). The hung-pipeline window applies per verb, not to a nonexistent terminal pipeline.completed.
