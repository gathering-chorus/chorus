CI Pipeline — Service Design

2026-04-25 — Kade. Layer 3 of the three-layer quality system. 13 required checks, branch-protection-enforced on main, alert on red.

Every change that lands on main builds and passes the team's quality bar from a fresh runner — independent of what's cached on any role's laptop. Jeff can ask "is main shippable?" and the answer is yes by construction. When CI drifts, locks fall out of sync, ratchet baselines accumulate silent debt, in-flight work breaks other roles' PRs, and Jeff becomes the test suite.

Overview

CI is layer 3 of the three-layer quality system defined in ADR-026 §a. Pre-commit (Layer 1) is honest-mistake feedback, fast and skippable. Role gates (Layer 2) are card-acceptance — recorded on the card. CI (this) is authoritative on main — branch-protection-enforced, not skippable. The threat CI specifically closes: --no-verify plus local-vs-CI environment divergence.

One workflow at .github/workflows/quality.yml on push-to-main and PR-to-main. 13 required jobs today; mcp-roundtrip deferred to #2495 pending ESM/CJS reconciliation. Lock-file policy (ADR-026 §c): per-package package-lock.json committed, npm ci not npm install, Dependabot for centralized updates. Red main fires ci-main-red.yml to Silas with daily cooldown.

The Three Layers

Each layer answers a different question with a different threat model. Redundancy is intentional.

Layer 1 — Pre-commit
Local, skippable
"Did the dev mean to commit this?" Fast feedback. platform/hooks/pre-commit.
Layer 2 — Role gates
Card-acceptance
"Is this card team-acceptable?" Five skills, recorded on card. Owned per concern.
Layer 3 — CI
Authoritative on main
"Does main build cleanly from scratch?" Branch-protection-enforced. Not skippable.

Sub-Domain Interaction Model

CI doesn't act directly — it's invoked by GitHub on commit/PR events and produces status checks consumed by branch protection and the red-main alert. Like Quality, the service is a verdict producer, not an enforcer; branch protection is the enforcer.

The core pattern: CI is a verdict producer, branch protection is the enforcer, alerts are the safety net. The safety net catches the case where verdict + enforcement both pass but main goes red anyway (admin-bypass push, post-merge regression, flake).

Why dependencies matter to CI: the layer model is a shape derived from principles (CI is the safety net for what slipped through local), practices (CI proves shippability), policies (when a check bites: pre-commit vs gate vs CI), and decisions (why specific checks bite where they do). Observed today 2026-04-25: without a KNOWN_FAILS policy, role A's pre-existing test debt blocks role B's commit; B uses --no-verify with a trace, but the convention isn't machine-checkable, so future drift is invisible.

Branch protection REAL

Classic protection on main in gathering-chorus org. 13 required status checks. "Do not allow bypassing" unchecked → admins can push direct to main. Org-level Rulesets duplicates this; collapses under #2498 (Silas-owned, pulled).

Lock-file policy + Dependabot REAL

ADR-026 §c. Per-package package-lock.json committed (root + 7 active TS packages), Cargo locks committed. CI runs npm ci not npm install — drift between local and CI is a red flag. .github/dependabot.yml opens weekly Monday lock-update PRs across 11 ecosystem-directory pairs. Initial scan: 23 PRs at once (#2-#24). Steady-state: 2-5/week.

Reactive alert (ci-main-red) REAL

Polls GitHub Actions API every 10 minutes for the most recent run on main. If status=completed && conclusion=failure, fires nudge to Silas with daily cooldown. No GH_TOKEN → check returns ok and skips silently. Read-only check; never auto-reverts (ADR-026 §d).

ESLint ratchet REAL

Per-rule baseline at platform/state/eslint-baseline.json. Pre-commit and CI both run the same check. New rule firings require either a fix or npm run lint:baseline to adopt. Baseline updates today are manual; #2496 looks at automating or signaling drift.

Required-checks contract PARTIAL

Integration point between workflow and merge-protection guarantee. Listed manually in Settings → Branches → Edit main rule → Status checks. Rename a job → check runs but doesn't gate. Remove a job without removing from list → all PRs un-mergeable. #2500 drift detector closes this loop.

Red-main runbook (Silas-owned)

When ci-main-red fires, the triage tree is:

  • Recent merge + unrelated to in-flight → revert the merge commit, file fix card on original author. Forward progress preserved by the revert.
  • Forward-fix is quick (≤10 min, root cause obvious) → push the fix direct to main, watch CI re-run green. No revert needed.
  • Not (1) or (2) → file swat card with failing job name, mark main as known-red in #shared-observability, decide with team whether to revert or freeze merges.

Daily cooldown means triage runs once per incident, not per check.

Surfaces

  • GitHub Actions UI — github.com/gathering-chorus/chorus/actions — primary read surface, full run history.
  • PR Checks tab — verdict per job for the head commit; what branch protection consults.
  • Branch protection settings — manual mirror of quality.yml job names.
  • Org Rulesets — duplicate enforcement layer (collapses under #2498).
  • Dependabot PRs — gh pr list --label deps — weekly lock-update flow.
  • ci-main-red nudge — Silas's terminal + messaging API, daily cooldown.
  • Pipelines-domain page — localhost:3000/gathering-docs/domain-detail.html?id=pipelines-domain — this design as contract entity.

Consumers

  • Branch protection — consumes verdict-set, blocks merge if any required check is red.
  • ci-main-red.yml alert — consumes verdict on main post-merge, routes red to Silas.
  • Roles (PR authors) — consume their PR's verdict; fix red on their PR before /gate-code.
  • Dependabot — produces lock-update PRs; consumes CI verdict on those PRs.
  • /gate-code skill — consumes pre-commit + recent CI state to decide gate verdict.
  • /demo skill — consumes gate-chain state; today assumes user-facing demo, infra adapt via #2499.
