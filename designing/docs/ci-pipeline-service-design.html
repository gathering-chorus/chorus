The pipeline

The CI/CD pipeline is the werk verbs run in sequence — there is no orchestrator that owns logic. Each verb is a self-contained atomic unit (own flock, typed refusals, spine emit, inherited card trace). The driver is act running the ONE workflow file, .github/workflows/werk.yml (#3236 retired werk-mcp.sh; #3193 folded werk-land.yml back in): act sequences the verbs as legible steps — every step named for its verb — and stops on the first refusal; gh mirrors each step as a per-verb commit status. The single MCP trigger is chorus_werk; agents never shell to act directly. There is exactly one environment, production — no preview, no staging; the per-card werk slot is the demo variant, and gh holds main (source of truth) keyed by card#.

ONE file, two go-gated jobs with the human GO between them (#3279 stop-and-resume; #3193 one-file collapse). The wait between them costs nothing — it is a stopped pipeline, not a held connection.

Job werk — prove (go ≠ true; presents and STOPS):

  1. commit — werk-commit: rebase onto origin/main (conflict HOLDS for --continue/--abort, #3304) and commit; checks inside the verb
  2. push — werk-push (--force-with-lease on re-run; branch-ownership inside)
  3. build — werk-build for the card's werk
  4. test — hermetic suites (cargo hermetic + jest), ADVISORY until werk-test #3190: result recorded (demo.test_result + chorus/test/<card> gh status), never blocks
  5. review — werk-review's structured floor (#3193), ADVISORY: objective findings (unchecked AC, src-without-test, removed pub symbols) witnessed as review.floor + chorus/review/<card> gh status; the cold-eyes agent half runs session-side in demo prework and records review.verdict through the binary
  6. deploy — werk-deploy --target werk: the per-card slot, main untouched
  7. env-up — the variant comes up; reachability posted
  8. demo — werk-demo presents the cockpit (verbs · 5 gates · tests · review · variant) and stops. Refuses to present un-gated (#3284). The verdict is the human's — go / no / more.

← the GO (DEC-048) — and GO = ACCEPT (#3311): Jeff's go at the demo IS the acceptance; there is no second accept moment →

Job land — on GO (chorus_werk {go:true} dispatches the same werk.yml with --input go=true):

  1. merge — werk-merge: resolve the open PR by HEAD oid, squash to main, content-verify (idempotent on already-landed work, #3336)
  2. sync — canonical ff-sync to origin/main so script-only changes go live (SYNC_OK gates the “live” claim; werk-sync recover is the recovery verb, #3300)
  3. deploy — werk-deploy --target canonical: prod is structurally a build of MAIN (#3222), with structural unit discovery (no crate registry) and the detached self-deploy for chorus-mcp (#3320/#3323 — the continuation survives its own daemon restart)
  4. accept — werk-accept runs under the accepter named on the GO: records the authority-gated decision and finalizes (Done, teardown, card.accepted)

Each verb's completion surfaces as a commit-status check on the card's PR (chorus/commit · build · test · review · deploy · env-up · demo · merge · accept), derived from the artifact it produced — integrity, not mere existence — so all-green means every verb genuinely ran. Rollback is all-or-nothing per verb; logs are witness-only, never transactional.

Two axes in this doc. Delivery — how a change moves pull → demo → land (the sequence above). CI enforcement & quality — keeping main green (hooks, gates, scheduled CI, drift), covered in the enforcement sections below. Two layers of one pipeline, not competing states. The git layer (worktree, commit, push, branches) lives in the Version Control design; build + deploy in Build + Deploy (ownership: VC — Kade; build/deploy — Silas; gates — all roles).

Atomic verb execution (ADR-037)

Decided 2026-06-08 — ADR-037 (roles/silas/adr/ADR-037-atomic-verb-execution.md). The full model lives in the Version Control design + ADR-037; this section frames it for the pipeline — one decision, referenced, not restated.

The pipeline is the seven verbs run in sequence (pull → commit → push → build → deploy → merge → accept), stopping on the first refusal — there is no orchestrator. ADR-037 adds an --atomic mode to every verb: run one step outside the sequence with inputs given explicitly, instead of deriving them from the prior step's card+werk state. This is the operator/recovery door — “just rebuild/redeploy/merge this one thing” — that closes the hand-command-around-the-flow class behind merged≠live and the partial-deploy incidents.

Two axes, not conflated: --atomic on all seven (free on pull/commit/build — local, reversible); explicit human approval only on werk-deploy + werk-merge (the irreversible prod/main mutations), each emitting a spine event recording {who, what, when}. One gate (DEC-048 — the human authorizes prod), two doors: demo→go in-flow, or approval on --atomic out-of-flow.

D#5 (the constraint, Kade): --atomic MUST reuse the verb's pure core, never fork it — both doors bottom out in the same function (the #3285 pure-core/flow-adapter factoring), or it is a 7th competing path. One implementation, two entry points (chorus:principle-no-competing-implementations).

Promise

Every change that reaches main is measured by the same gates regardless of who pushed it. Five hooks fire pre-merge. Five named code-review checks record card-level acceptance. One scheduled GitHub Actions run on main (~daily) is the post-merge witness. One reactive alert wakes the on-call when CI goes red.

Honest scope: four of the five pre-merge hooks block unconditionally; the fifth (pre-commit) is --no-verify-skippable for the carded KNOWN_FAILS pattern, and the allowlist enforcing that convention shipped (#2497, Done). With required-status-checks empty (post-#2600 cost-stop), an accidental main-push lands red and stays red until the next nightly run — up to 24h. The schedule-only retreat is a deliberate cost trade ($15/mo vs $225 projected); the gap is mitigated by pre-merge maturity, not by CI itself.

Pre-Merge Enforcement (Five Surfaces)

Each surface catches a distinct attack vector on the path from "engineer edits code" to "main has the change." The five aren't enumerated by convenience — they're derived from where wrong-shape commits can enter the system.

#Attack vector caughtHookRefusesCard
1Wrong-branch commit (engineer commits to a branch that isn't theirs)werk-commit branch-check (inside the verb)Cross-role branch contamination#2580
2Raw push (bypassing the queue)pre-push hookgit push , wrong-role branch#2598
3Dangerous git from agent (Claude Bash tool runs raw git mutations)chorus-hook-shim PreToolUseRaw git push/rebase/cherry-pick/reset --hard#2598
4Canonical mutation (writing to the read-only canonical instead of the card werk)canonical_write_guard + infra_guardrailsEdit/Write to canonical from a role session; raw mutating git anywhere (#2913 ephemeral-werk model made the old per-role contamination guard obsolete)#2913
5Bad commit content (typecheck/lint/test failures, secrets, oversize)pre-commit hooktsc errors, jest/cargo failures, lint violations, dup-strings, cog-complexity over 12, secrets, oversize binaries(built up over many cards)

All five emit JSONL audit events to platform/logs/chorus.log.

Surface 5 is the only one bypassable: --no-verify skips it for the KNOWN_FAILS pattern (carded test failures from other roles, traced via #NNNN in the commit message). The allowlist enforcing that convention shipped — #2497 (Done).

Pre-Commit Detail (Surface 5)

platform/hooks/pre-commit runs inside every werk-commit. Each check fires only when staged files trigger its scope.

  • Typecheck — tsc --noEmit on changed TypeScript packages
  • Tests — jest on changed JS/TS packages, cargo test --lib --bins on changed Rust crates (hermetic only)
  • Secrets scan — refuses commits writing API keys / .env content / credentials
  • Catalog oversize — refuses oversize binaries to data/catalog/
  • Principle direct-edit — refuses edits to data/principles/ outside the canonical write path
  • Sonarjs error tier (Check 4.4, #2603 + #2627) — blocks on no-duplicate-string (threshold 5) or cognitive-complexity (threshold 12) in staged TypeScript. Magic-comment override // cog-override: <reason> exempts a function and emits a cog.override.used audit event
  • MCP tool description shape — staged MCP tool definitions match the description-shape contract
  • Doc-coherence ratchet — when CLAUDE.md fragments change, doc references stay coherent

Quality Rules (Convention: No Warn Tier)

Active rules at error (block at Check 4.4):

  • sonarjs/no-duplicate-string, threshold 5 — catches the agent-inlining pattern (same literal duplicated 5+ times)
  • sonarjs/cognitive-complexity, threshold 12 — catches the function-too-tangled pattern; PM lowers the threshold (12 → 11 → 10) when the codebase is visibly ready

Convention (not currently self-enforced): every lint either blocks or doesn't fire — no warn tier. Warnings are never handled, so adding a rule at warn level adds noise. This is a discipline, not a pattern with an enforcement surface; nothing in the pre-commit hooks today blocks a contributor from adding a warn-tier rule. A self-audit check would change that — currently a gap.

Install discipline: when wiring a new quality rule, ship the rule + fix-as-we-go. Don't pre-fix every existing violation as part of the install — that's how rule-install cards become unbounded. Fixes happen on touched files at commit time.

Retirement Gates (Forward-Only Structural Assertion)

When a surface is retired (HTTP endpoint, helper function, schema column, code path), the deletion ships with a small bats file containing grep-assertions: "no production reference to the retired surface remains anywhere in the codebase."

Without the gate, the test suite shrinks alongside the surface — a future contributor sees the empty space, types the helper back in, and no test catches it. The retirement gate is structural memory of why the surface was retired, encoded as test.

Live examples (verified 2026-04-30):

  • platform/tests/role-state-card-decoupled.bats (#2467) — 7 @test blocks asserting no skill source passes card= to role-state CLI; no CLAUDE.md fragment uses building card=<id> syntax
  • platform/tests/pulse-rolestate-retired.bats (#2632) — 5 assertions: no setRoleState, no getRoleState, no /api/role-state route, no role_state table CREATE, no Role state describe block in pulse store tests

Pattern earns its keep when the retired thing has structural pull (a future engineer might reach for it). Skip the gate when the surface was an obvious one-off no one would re-introduce.

Affordance-Layer Refusal

When a deprecated input shape needs to disappear (e.g., card= field on role-state writer), refuse it at every surface that could reach the affordance, not just one. Pattern from #2467 + #2629:

  1. Writer parser — refuses the input, exits with helpful error naming the corrective action
  2. HTTP API — refuses POST bodies with the deprecated field
  3. Bats gate — asserts no skill source / fixture / helper passes the deprecated input
  4. Schema — drops the column entirely so the storage path is closed

Each surface catches a different attack vector: writer rejects current callers; HTTP API rejects future REST clients; bats prevents new instances in skill markdown; schema drop closes the storage path entirely. Single invariant, four enforcement surfaces. Belt-and-suspenders is reinforcement here — they catch different things.

What this shape buys: cost ~$15/mo (down from ~$225 projected pre-cost-stop); five pre-merge gates refuse contamination at distinct vectors; per-PR validation runs locally; the scheduled run is the post-merge witness.

What it costs: a regression on main is discoverable up to 24h after merge, so the trust burden sits on pre-merge maturity. If pre-commit drifts (a flaky test bypassed via --no-verify without a card trace), the daily run is the only catch — and the trust audit that would detect such drift is itself open work (#2500).

Open enforcement work

  • A nightly drift lane (mutation / perf / integration / CVE / doc-coherence) surfacing slow signals as non-blocking cards.
  • A sensor consumer registry — every CI / alert / drift signal routes to a named consumer; no orphan signals.
  • A run-tests contract — the load-bearing interface between CI and the tests substrate.
  • A GHA org-billing alert to prevent recurring quota wedges; a KNOWN_FAILS allowlist closing --no-verify drift; a ratchet drift signal stopping baseline accretion.

What stays: scheduled-only main re-run; no per-PR matrix unless pre-merge maturity proves insufficient and Jeff reinstates required checks.

Observability

There is no orchestrator, so there are no pipeline.step.* / pipeline.completed events. The pipeline's observability is the flat verb sequence's own spine events: each verb emits <verb>.started / .completed / a typed refusal on the card's inherited trace. The complete per-card trace is the ordered run — commit.completed → push.completed → merge.completed + content.verified → deploy.completed (with cdhash=binary.running confirming deployed==running). A missing verb event is a pipeline gap, queryable via chorus_logs_for_trace(trace_id) on the inherited trace. The verbs emit; Borg produces the shape and the gap-detection. The hung-pipeline window applies per verb, not to a terminal pipeline.completed that does not exist.

act is LIVE as the driver: ONE workflow file, werk.yml, with go-gated prove/land jobs (werk-land.yml deleted, #3193), dispatched only through the chorus_werk MCP verb. act sequences and makes each verb a legible line in gh; it owns no logic. The witness layer is per-verb: werk.started/werk.landed bracket the run in ops/logs/werk.jsonl; each verb writes its own jsonl witness + spine events on the shared trace.
