What changed since 2026-05-04 rewrite

The 2026-05-04 architecture (pulse owns delivery, messages.db canonical, chorus.log audit-only, spine-tick-poller retires) stands. Three incident classes since the rewrite forced this revision; none of them changed the architecture, but each surfaced a load-bearing dependency or failure mode the original design treated as a black box.

1. The TCC trap closed. macOS binds Accessibility/AppleEvents permission to a binary's signature hash. Every cargo build --release changed the hash, which silently revoked the keystroke grant — 159 nudges failed in one day before anyone noticed why. The fix was making build-signed.sh use a keychain identity so the hash is stable across builds. The principle the design now carries: building chorus-inject IS deploying chorus-inject — the dev loop touches a production-permission surface.
2. The audit log can disappear. Nine days of spine events went missing in late April because the writer had failed silently inside the working tree (a worktree checkout truncated the log). The 5-4 design treats chorus.log as audit-only, so this didn't take delivery down — but anything that reads the audit projection (Borg, post-hoc forensics, the augment-envelope fold) saw a hole. Fixed by moving the file out of the working tree and adding a heartbeat probe that pages when the writer goes quiet.
3. Half a retirement is a leak. The 5-4 design called for retiring the sender-side write to chorus.log (pulse would own audit). Only the helpers got retired; the actual sender-side write still fires, and no consumer marks rows acknowledged. messages.db now has 10,528 rows stuck in pending with no consumer to drain them. "Fire-and-forget" becomes "fire-and-leak" until Phase 1 ships. The two cards in Phase 1 close this.

Four failure modes the original design didn't name also surfaced. Each gets a section below.

What nudge is

A nudge is a one-way message from a sender to a receiver, surfaced as a keystroke into the receiver's terminal. Two recipient classes:

  • Role → Role. Wren nudges Silas. Goes to Silas's Claude Code session via macOS UI scripting; Silas reads the message in his terminal as if Jeff had typed it.
  • Role → Jeff. Posted to Bridge API at :3470; surfaces in Jeff's Clearing browser. Different transport. Out of scope for this doc.

The contract is persistent (every nudge is recorded) and best-effort delivery (the keystroke may fail; persistence is guaranteed, delivery is retried).

Known problems

1. Two parallel writes maintained by convention. Senders write to messages.db AND emit to chorus.log. Pairing depends on every sender path going through nudge.rs. A future sender that skips nudge.rs gets silent divergence.
2. 2-second polling lag. spine-tick-poller polls chorus.log on a 2s loop. Average delivery latency is 1s, worst-case 2s + osascript time. Six full-file reads per second across three pollers.
3. Derived state duplicates implicit state. "Emitted minus surfaced" is a fold over events to compute "what's pending delivery." That state could live as a column on the canonical row.
4. Clearing doesn't show role-to-role nudges. Jeff watches a sender fire a nudge in his terminal; receiver gets the keystroke; Clearing browser stays empty. Real visibility gap.
5. chorus.log is schemaless. Log format drifts without ceremony. Two files have defensive comments documenting the "first kv pair gets packed into the JSON value" quirk that bit consumers when emit format changed. DB schema is governed via migrations; log shape is not.

Failure-mode taxonomy

The 5-4 design treated chorus-inject as a black box that either succeeded or returned an error string. Reality is finer: failures fall into permanent and transient classes, and the poller / pulse-worker should treat them differently. Permanent failures must not retry (retrying TCC-denied just burns cycles); transient ones may.

FailurePermanent?What's happeningHow you see itHow it gets resolved
TCC deniedYes (until manual re-grant)macOS revoked osascript's Accessibility permission, usually because a binary rebuild changed the signature hashstderr says not allowed to send keystrokes; system error code 1002Re-grant System Events for osascript in System Settings → Privacy. Stable signing prevents recurrence.
No window foundYes (until terminal opens)The target role's terminal isn't open, or the window title doesn't match the role-name patternchorus-inject's window count returns 0Open the role's terminal. Or fix window-naming so the match works.
Window ambiguousYes (until extras close)More than one window matches the role pattern — common after session rebootschorus-inject's window count returns >1Close stale windows. Make the disambiguation rule explicit.
Encoding errorYesA non-BMP codepoint (any character ≥ U+10000, including most emoji) passes through the AppleScript escape function unhandled and gets mangled by the keystroke interpreterReceiver sees the message with the prefix rendered as "aa…" instead of the emojiExtend the escape to handle non-BMP, or refuse non-BMP at the wrapper with a clear error.
Focus-gate missSometimesThe frontmost-app precheck refused injection because the target window wasn't frontmost at the moment the inject firedEvent includes not_frontmost:<app> reasonInvestigate per case — sometimes a real focus race, sometimes a window-naming drift wearing focus-miss clothing.
Inject-nonzero (catch-all)Maybechorus-inject returned non-zero for a reason not yet classifiedrc=1 with raw stderrRead stderr, classify, promote to one of the typed reasons over time.

This taxonomy is what the typed-error card surfaces back to the poller: the contract becomes "emit nudge.surface.failed with one of these typed reasons" instead of "return an OK-string and hope."

Error observability contract

The original design didn't say which failures are visible to whom. Today the codebase has multiple let _ discards on critical paths — the curl that persists to messages.db, the chorus_log emit, the captured command output. When those fail, nothing fires. The whole system depends on observability working, so silent observability is the worst class of failure.

The contract going forward:

  • Spine emit failure — the nudge.emitted event itself fails to write. Treat as fatal; caller exits non-zero with stderr. The whole system reads from this event.
  • Persist failure — Bridge API POST fails. Logged but not fatal until Phase 1 ships. The caller still exits zero if the spine emit succeeded.
  • Inject failure — emits nudge.surface.failed with the typed reason from the taxonomy above. The caller (poller or pulse worker) does not exit non-zero — crashing the daemon is worse than recording the failure.
  • Internal observability errors — chorus_log itself failing inside a hook. Fatal to the call site. We cannot afford silent gaps in observability.

No let _ on any of these paths. Every failure produces either a spine event or a non-zero exit code. Silence is not a state.

Fold convergence contract

The augment-envelope fold today computes pending = emitted minus surfaced. Permanent delivery failures emit surface.failed rather than surfaced, so the fold never converges — failed nudges accumulate as forever-pending and the envelope grows monotonically.

Revised: pending = emitted minus (surfaced or surface.failed). A permanent failure counts as resolved-by-failure for the fold's purpose. The augment envelope shows "pending" only for nudges still in the delivery pipeline.

Separately, a stuck-nudge metric — count of emitted events whose only resolution over the last 24h is surface.failed — is the right input for the fitness probe. The metric distinguishes a healthy "delivery worked, marked failed for legitimate reason" world from a sick "deliveries are silently failing" world.

Observability — what you can ask of the system

"Did that nudge land?" should be one query, not five. The design specifies the field schema, the correlation primitive, and the named queries the team uses to answer questions about nudge behavior. This is part of the contract, not an add-on.

Correlation primitive

Every nudge gets a trace_id at the sender (UUIDv7 — sortable, time-ordered). Every event on that nudge's lifecycle stamps it. Every messages.db row stores it. Every retry attempt carries it. That's how one Loki query returns one nudge's entire story in chronological order.

Event field schema

EventRequired fields
nudge.emittedtrace_id, sender_role, receiver_role, messages_db_row_id, message_len, message_hash (sha256 first 8), ts
nudge.surfacedtrace_id, receiver_role, attempt (1-indexed), cdhash (of chorus-inject), latency_ms (emitted → surfaced), ts
nudge.surface.failedtrace_id, receiver_role, attempt, reason (one of: tcc-denied, no-window-found, window-ambiguous, encoding-error, focus-gate-miss, inject-nonzero), stderr_excerpt (first 200 chars), cdhash, permanent (bool), ts
nudge.health.*trace_id (when applicable), condition, detail, ts

No call site emits a nudge event without these fields. The chorus_log helper validates the shape before write; missing required field is a non-zero exit at the call site, not a silent log line.

Spine ↔ messages.db join

The trace_id is the join key. Every messages.db row carries it as a column; every spine event carries it. To answer "where is this nudge in the lifecycle right now," you can either query the row (delivery_status, last_delivery_error) or the events (nudge.emitted with no terminal event yet) — they should agree, and a divergence is itself a signal.

Named queries

The team operates on these queries. They're stable; the design names them so investigations don't re-derive them.

What nudge_health emits

The pulse worker emits nudge.health.* events on internal conditions the system needs to surface but that aren't per-nudge: queue_depth on every tick (gauge), worker_restarted on process start (with reason if known), migration_completed on schema change. These are the "is the nervous system itself healthy" feed; the per-nudge events answer "did this one land."

What this nails down for #2727's AC. Every event the worker emits carries the schema above. Every messages.db row carries trace_id as a column. The chorus_log helper rejects events missing required fields. The named queries above run green against the worker's output. Without these, "delivery worker shipped" is half a deliverable.

OS-chain durability

The 5-4 design treats chorus-inject → osascript → keystroke as one black box. Three layers actually matter, and each has its own failure shape:

  1. Signature stability. Every release build of chorus-inject or chorus-hook-shim is a production deploy. Any sensor that builds these in place will break TCC for the team for hours, minimum, until someone notices and re-grants. The keychain-identity signing fix removes the recurring break, but the underlying constraint stays: do not rebuild these binaries in their deploy location without going through build-signed.sh.
  2. Window-naming convention. chorus-inject finds the target by matching the role name AND the substring "claude" in window titles. The convention is implicit — no enforcement. A session reboot that creates a window titled "wren - zsh" without "claude" makes inject fail with no-window-found. Convention should be explicit in role bootstrap and runtime-checked.
  3. Encoding boundary. The escape function handles em-dashes and smart quotes. Non-BMP codepoints (most emoji) pass through verbatim and get mangled by AppleScript's keystroke interpreter. The fix is either (a) escape non-BMP explicitly, (b) refuse non-BMP at the wrapper with a clear error, or (c) document the limitation as a content restriction.
