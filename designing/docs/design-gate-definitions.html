Card Completion Pipeline

Design — 2026-04-08 · Practices domain · Kade + Silas input

Problem

"Done" is one gate today. In practice it's five different checks owned by three different roles. Cards get stuck in "not done yet" with no clarity on which stage failed or who unblocks it. Roles push Jeff for ACP before the work is actually complete — sometimes when it merely compiles.

A card is a unit of work moving through a pipeline. Each stage has rules (what "pass" means) and a gate (who signs off). No stage can be skipped. No promotion without sign-off. Jeff touches the card twice: entry and exit.

Stages

1. Pull Wren writes AC · Kade pulls card
Card enters the pipeline.
  • AC written and attached to card
  • Card moved to WIP
  • Applicable stages tagged
2. Build Kade
Implementation. TDD cycle: AC → tests (red) → code → tests (green).
3. Product Wren
"Did we build what was asked?"
  • Every AC item demonstrably met
  • Demo script runs, Jeff sees the outcome
  • User-facing docs / help text updated if applicable
  • Card description matches what was actually built
gate:product-pass Fires at: feature-complete
4. Code Kade
"Is the code solid?"
  • Tests written before code (TDD)
  • Tests green — unit + integration
  • Matches existing code patterns
  • No unlogged tech debt introduced
  • Build clean, no warnings promoted
gate:code-pass Fires at: code-complete
5. Quality Kade
"Does it meet the standard?"
  • Hooks pass (write scrubber, infra guardrails, ICD gate)
  • No regression in adjacent domains
  • Observability: logs structured, metrics emitting
  • Error handling tested at boundaries
gate:quality-pass Fires at: code-complete
6. Architecture Silas
"Does it fit the system?"
  • Architecture review — fits the domain map
  • No cross-domain violations
  • Consistent with ICD contracts
  • Namespace / URI conventions followed
  • No unreviewed structural changes
gate:arch-pass Fires at: code-complete
7. Ops Silas
"Can we run it safely?"
  • Deploys via app-state.sh without manual steps
  • Health checks pass post-deploy
  • Logs flowing to Loki, queryable
  • Rollback path verified
  • No deploy-freeze violation
gate:ops-pass Fires at: deploy time
8. Accept Jeff
All applicable gates passed. ACP runs. Card exits the pipeline.
  • All required gate labels present
  • Jeff accepts — commit + push

Behavior Specifications

Feature: Card completion pipeline
As Jeff
I want cards to pass through a staged pipeline before acceptance
So that "done" means every applicable domain has signed off
Pipeline ordering
Scenario: Stages must pass in order
  Given a feature card is in the Build stage
  When Kade attempts to apply "gate:code-pass"
  But "gate:product-pass" has not been applied
  Then the label is rejected
  And the message "Product stage must pass first" is shown
Product stage
Scenario: Wren verifies AC and promotes
  Given a card has completed the Build stage
  And all AC items are demonstrably met
  And the demo script runs successfully
  When Wren applies "gate:product-pass"
  Then the card is promoted to the Code stage
Code stage
Scenario: Kade verifies code solidity and promotes
  Given a card has "gate:product-pass"
  And tests were written before code
  And all tests are green
  And code matches existing patterns
  When Kade applies "gate:code-pass"
  Then the card is promoted to the Quality stage
Quality stage
Scenario: Kade verifies quality standards and promotes
  Given a card has "gate:code-pass"
  And all hooks pass
  And no regression in adjacent domains
  And observability is in place
  When Kade applies "gate:quality-pass"
  Then the card is promoted to the Architecture stage
Architecture stage
Scenario: Silas verifies system fit and promotes
  Given a card has "gate:quality-pass"
  And the change fits the domain map
  And no cross-domain violations exist
  When Silas applies "gate:arch-pass"
  Then the card is promoted to the Ops stage
Ops stage
Scenario: Silas verifies deploy safety and promotes
  Given a card has "gate:arch-pass"
  And deploy runs via app-state.sh
  And health checks pass
  And logs flow to Loki
  When Silas applies "gate:ops-pass"
  Then the card is eligible for Accept
Stage owner enforcement
Scenario: Only the stage owner can apply the gate label
  Given a card is at the Architecture stage
  When Kade attempts to apply "gate:arch-pass"
  Then the label is rejected
  And the message "Only Silas can sign off on Architecture" is shown
ACP blocked on incomplete pipeline
Scenario: Cannot accept a card with missing stages
  Given a feature card with "gate:product-pass" and "gate:code-pass"
  But "gate:quality-pass" has not been applied
  When Jeff runs ACP
  Then ACP is blocked
  And the message "Pipeline incomplete — missing: quality, arch, ops" is shown
Skipped stages for card type
Scenario: Doc-only card skips non-applicable stages
  Given a doc-only card tagged with applicable stages: [product]
  When Wren applies "gate:product-pass"
  And Jeff runs ACP
  Then ACP succeeds
  And code, quality, arch, and ops stages are not required
Clean pipeline exit
Scenario: All stages passed — card exits pipeline
  Given a feature card with all five gate labels applied
  When Jeff runs ACP
  Then the card is accepted, committed, and pushed
  And Jeff touched the card exactly twice: pull and accept
No premature ACP pressure
Scenario: Roles do not prompt Jeff for ACP until pipeline completes
  Given a card has passed the Code stage but not later stages
  When Kade completes the Quality stage
  Then Kade nudges Silas for Architecture review
  And Kade does not nudge Jeff for ACP
Pipeline promotion nudges
Scenario: Completing a stage nudges the next stage owner
  Given Wren applies "gate:product-pass"
  Then Kade is nudged to begin the Code stage
  And the card's pipeline status is visible on the board

Open Questions

2. Code + Quality collapse for small cards? — Resolved: separate definitions, single-pass labeling allowed for small cards (Wren, 2026-04-08)

3. Board visibility? — Resolved: card labels (Silas, 2026-04-08)

4. Sign-off mechanism? — Resolved: gate:*-pass labels (Silas, 2026-04-08)

5. Quality vs Ops: one or two? — Resolved: separate stages, distinct timing (Silas, 2026-04-08)

6. Ownership? — Resolved: Kade = code + quality, Silas = arch + ops (Jeff, 2026-04-08)

Skill Architecture — Four Skills, Not One

Each role owns their gate skill. No routing logic. Automated checks run first, 1-2 manual confirms max. On pass, nudge the next owner. On fail, stop. Pilot mode: observe before enforce.

Gate Skill BDD

Feature: Gate skill — role-scoped pipeline checks
As a role completing my pipeline stage
I run my gate skill to verify my checks pass
So the next stage owner is nudged automatically and Jeff only touches the card at ACP
Automated checks run first
Scenario: Gate runs automated checks before manual confirms
  Given Silas runs /gate-arch 1814
  Then namespace check runs (exit code)
  And ICD consistency query runs (exit code)
  And domain boundary scan runs (exit code)
  And only after all automated checks pass does the manual confirm appear
Automated failure blocks manual confirms
Scenario: Failed automated check skips manual step
  Given Silas runs /gate-arch 1814
  And the namespace check fails
  Then the gate reports FAIL with the failing item
  And the manual "structural fit?" confirm is not shown
  And no nudge is sent forward
All pass nudges next owner
Scenario: Passing all checks nudges the next stage owner
  Given Kade runs /gate-quality 1814
  And all automated checks pass
  And Kade confirms "no new debt"
  Then gate reports PASS
  And Silas is nudged "gate:quality passed on #1814 — run /gate-arch"
  And a spine event is emitted: gate.quality.passed
Manual confirm rejected
Scenario: Role rejects a manual confirm
  Given Silas runs /gate-arch 1814
  And all automated checks pass
  But Silas answers "no" to "structural fit?"
  Then gate reports FAIL with reason
  And no nudge is sent forward
Wrong role runs gate
Scenario: Role runs a gate they don't own
  Given Kade runs /gate-arch 1814
  Then the skill says "Arch gate is owned by Silas"
  And no checks run
Non-applicable card type
Scenario: Doc-only card skips code gates
  Given a doc-only card
  When Kade runs /gate-code on it
  Then the skill reports SKIP (not applicable for this card type)
  And no checks run, no nudge sent
Ops gate fires at deploy, not code-complete
Scenario: Arch passes but ops waits for deploy
  Given Silas runs /gate-arch 1814 and it passes
  Then the nudge says "arch passed — run /gate-ops after deploy"
  And /gate-ops is not auto-triggered
  Because ops checks require a running deploy to verify
Final gate signals Jeff
Scenario: Last applicable gate passes
  Given Silas runs /gate-ops 1814 and it passes
  And all prior gates already passed
  Then Jeff is nudged "Pipeline complete on #1814 — all gates passed, ready for /acp"
  And no role nudges Jeff before all applicable gates pass
Pilot mode — observe, not enforce
Scenario: Role skips their gate
  Given a card reaches ACP without gate spine events
  Then the gap is visible in the pipeline log
  But ACP is not blocked
  Because pilot mode observes compliance before enforcing it

Resolved: Skill Open Questions

1. How does /gate-product relate to /demo? — Resolved: /gate-product wraps /demo, not replaces. Demo is the show, gate-product is the checklist. (Kade, 2026-04-08)

2. Gate result persistence? — Resolved: both card comments (visibility) and spine events (queryable). Comments are cheap. (Kade, 2026-04-08)

3. Retroactive or forward-only? — Resolved: new cards only. Retroactive would be noisy and the data isn't there. (Kade, 2026-04-08)
