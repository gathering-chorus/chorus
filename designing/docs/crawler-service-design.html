Crawler — Service Design (the harvest layer)

Current-state design as of 2026-06-02 — Silas (folds in roles/silas/architecture/harvest-layer.md #3185). History & superseded framings moved to the appendix. Sources: harvest-layer.md (#3185, current architecture), crawler-instance-model.html (#3069 instance contract), crawler-asis-tobe.md (#3069 monolith→domain), self-model-owl-framework.md (Wren, #3184), platform/scripts/crawler-hydrate-graph.sh, platform/api/src/handlers/chorus-crawl.ts, ADR-033 / ADR-032, DEC-095.

The crawler is the harvest layer — the thing that turns filesystem and operational reality into queryable structure in the graph. Every "Athena says X about ops-alerts" claim, every "what tests cover this domain" gate, every "what's running on which machine" ops decision is only as true as the harvest layer's last run. When it is healthy and current, the graph reflects the live system — its code, config, data, disk, network, compute, alerts, monitors, and logs — as a self-model the team can query. When it drifts, the self-model starts lying: same shape as a lying-green test, but at the substrate every consumer reads from. The harvest layer's promise is a live, structured, by-domain model of the system, landed in the graph, with observable freshness and per-facet alerting.

The harvest layer — Borg v2

The crawler is not a monitor, an index, or "a herald to fix." It is the system's context-harvest layer — Borg v2: the thing that harvests everything about the system into one place the system can query, so the system has a live model of itself. "Ontology is architecture," made literal — the harvested, by-domain-tagged graph is the self-model.

"Everything" is both halves of the system:

  • Structural context — what the system is: code (git/fs), config (.mcp.json, plists, settings.json, ICDs, the instance declarations themselves), data (the graph, cards, messages, memory).
  • Operational context — what the system is doing: disk, network, compute, alerts, monitors, logs.

The code crawler and Borg have been treated as separate systems. They are the same pattern: a per-facet herald (discover-*) scans its own source and writes to the shared graph (Decision 186; the Borg facet/herald design). Borg already does this for operational facets; the crawler does it for structural ones. There is no reason for two substrates — crawler and Borg are facet-heralds of ONE harvest layer, N heralds, each declaring source → target → cadence → cost.

Current architecture — facet-heralds and instances

The harvest layer is a registry of declared instances (ADR-033 / #3069), parallel to skills. A harvest engine runs each registered instance per its declaration. Adding a facet = adding a declaration — not monolith surgery. Each instance declares source (reads) → target (writes) → cadence → cost and follows three rules:

  1. Own cadence — matched to how fast its source changes × its cost. Code/tests on file-watch frequent; the heavy graph-rdf join hourly/rare; embed async.
  2. Right home — persist once, to the store that owns that data (knowledge → the graph), never flattened into a search blob.
  3. Pull, not push — read on demand for something that needs it, not constantly on spec for output nobody consumes.

Why decomposition

The monolith couples three things that have nothing in common: cadence (code changes constantly; the ontology rarely; the RDF join is expensive every time), write-home (knowledge belongs in the graph, not a flat messages blob), and read-discipline (mentions/logs computed every sweep on spec). Splitting the unit fixes the structural coupling that tuning the interval never could:

  • Cheap-frequent / heavy-rare — code/tests/disk/compute run frequent or on-change; the 35–55s graph-rdf join runs rarely. You can run the cheap thing often without dragging the heavy one along.
  • No storm — the heavy chunk is structurally never in the frequent path. The all-day spine storm becomes impossible by construction, not by tuning.
  • No fixed wall-clock to miss — own-cadence / on-change heralds have no single StartInterval fire to miss during host sleep, so the overnight-sleep false-alarm class (crawler-stale / daily-review / nightly-tests) dissolves into the decomposition rather than needing a separate patch.

Components (instance frame)

Instance registry

Instances are declared (like skills/), discoverable, owned. The engine enumerates them; there is no monolithic fetchCrawl. An instance declaration names source → target → cadence → cost-tier → mode → domains → write-resilience. Adding a facet is authoring a declaration, not editing the engine.

Scheduling engine (per-cadence)

The engine runs each instance per its declared trigger. Scheduled instances fire on their cadence; costTier caps frequency (a heavy instance may not run on file-watch). Pull-on-demand instances do not run on a clock — they compute at query time when a consumer asks (the "pull not push" fix for use-thin work; mentions, logs). Cadence is matched to source-change-rate × cost: file-watch frequent for code/tests; rare/hourly for the heavy rdf-join.

Aggregation

chorus-crawl.ts's per-domain join (cards, codeScan, code.files, logs, alerts, infra, history, rdf, owl, spine, timeline, mentions) is the read-side workhorse. Under the instance frame it is decomposed so each collector rides its own cadence/home rather than all firing on one beat. The aggregate response remains available as a read surface; what changes is that the harvest no longer flattens the join to prose — it lands structure.

Unified write-path

Every facet-herald writes through one shared write-path, branching by target.store rather than each facet inventing its own. Three sinks:

  • SQLite search-index (~/.chorus/index.db) — the search surface; per-record rows / per-domain snapshots, source-tagged.
  • Fuseki (the graph) via crawler-hydrate-graph.sh — the structured self-model. The script already POSTs chorus:File triples to Fuseki; the work is extending it to the full facet set (tests, logs, persistence, compute, storage, network, alerts, monitors) and routing every graph-homed instance through it. This is the persistence API every facet-herald calls.
  • Spine events — crawler.indexed / crawler.failed per run, carrying trigger, facet, counts, duration; the observability spine.

The graph hydrator is not absent — it exists as crawler-hydrate-graph.sh. The current-state work is unification: collapsing the per-facet writes into one declared write-path with one consistency model, not building a hydrator from zero.

Consistency model

The harvest writes both SQLite and Fuseki per record; partial-write (one succeeds, the other fails) is a real class. Model: best-effort dual-write per record + end-of-run reconciliation. SQLite first (cheap), Fuseki second (transactional via SPARQL Update). Either fails → the end-of-run reconciliation pass detects the mismatch, emits a hydration.partial.divergent spine event naming which side diverged, and retries the failed side. The instance-run is the consistency boundary; consumers see eventual consistency within ~one run. Stronger guarantees (staging-table, two-phase commit) are deferred until observed need. A deletes-reconciliation pass flags graph entries that name a deleted file (chorus:stale), closing the day-1 orphan-accretion failure mode.

Operational alerts

Per-facet alert rules in the borg alert set, routed via the ops-nudge path:

  • crawler-stale — fires when a facet has not emitted a crawler.indexed event within its expected window (own-cadence aware, so host sleep does not false-fire).
  • crawler-error — fires when a run reports errors > 0 or exits non-zero.
  • hydration-divergence — fires when source activity is high but the harvest shows zero new/updated for consecutive runs (watch-path-misconfigured / write-path-broken signal), or when the reconciliation pass reports persistent SQLite↔Fuseki divergence.

Instance / facet definitions — two-speed tagging

Each entity the harvest lands is tagged by domain — that tagging is what makes the graph a self-model. Tagging splits into two speeds:

Deterministic self-tagging (building / proving facets)

For the building and proving facets, the facet-herald IS the domain — the source/shape determines it mechanically, no human judgment. The herald already knows the domain because it is harvesting that domain's sources. This half self-tags at harvest and can ship ahead of the semantic classification.

Semantic tagging (#3184)

The capability / product-semantic domains — messages, practices, cards, search, version-control, decisions — where domain ≠ obvious-from-artifact (is ab analytics or cards? is interrupt messages or roles?). This half is a genuine judgment call and is the red-line work of #3184 (Wren). It is a prerequisite for those domains only — the deterministic half does not wait on it.

This is why werk-tests is closer than it looked: tests self-tag, so a by-domain tests capstone can run without waiting on the full #3184 manual model.

Read-side contracts

  • Eventual consistency per instance-run. A change is visible in both stores within sub-minute under normal load; under load, within one full run of the owning facet. Last-reconciled timestamp is queryable; consumers decide whether the staleness budget meets their need.
  • Partial-divergent state is observable. The hydration.partial.divergent spine event names when SQLite and Fuseki disagree on a record. Consumers needing strong consistency subscribe to it, or read a single store directly — never a cross-store join without checking the divergence event.

Query APIs (designed sub-products of their domains)

Once the harvest lands structure, per-domain APIs query it. They are not built by this design — they are sub-products of their respective domains — but the harvest is what makes them possible:

  • /api/chorus/tests + chorus_tests_* MCP — query tests by domain/layer/owner/scenario.
  • /api/chorus/code + chorus_code_* MCP — query code files by domain/owner/scenario.
  • /api/chorus/alerts + chorus_alerts_* MCP — query alert rules by trigger/severity/recipient/domain.

All follow the same dual-surface pattern (REST for Athena's domain-page renderer, MCP for agent-side typed access) and read from the same harvested graph.

Consumers

The harvested self-model is consumed by:

  • context-inject — what agents get in-prompt.
  • search — /api/chorus/search over the index.
  • gates — per-domain quality / arch / ops dispatch by card subdomain.
  • observability — alerting / monitors driven from the operational facets; ownership resolution (DEC-058: "query Athena for owners").
  • werk-tests — runs + reports the suite by domain, not by filesystem walk; the capstone proving the by-domain model end-to-end.

OWL framework — rep #1 of the generated self-model

#3185 is rep #1 of the self-model OWL framework (Wren, self-model-owl-framework.md — referenced, not duplicated here): one composed OWL that generates the service surface. "Ontology is architecture," literally — declare a domain in the OWL and the framework emits its API, MCP tools, logging, error-handling, and security, so divergence is structurally impossible instead of policed.

  • Imports chorus-core.ttl (Wren) — the importable spine: the four axes (ownedBy / atStep / inDomain / shape) + base SHACL. Products import it and EXTEND, never redefine.
  • The borg module (Silas) extends it — adds the harvest classes: Facet subclasses for operational context (Monitor / AlertRule / Resource / NetworkEndpoint) + harvest machinery (Engine / Environment). That mapping IS #3185.
  • The FIVE generated batteries — declaring a domain in the OWL emits, at minimum: API (OWL→OpenAPI + SPARQL-backed CRUD) · MCP (auto thin-skin tools — kills hand-written server.ts labor) · logging (uniform spine/Loki) · error-handling (typed errors → spine, build.failed/#3166 style, into the #3165 pain-rollup) · security (write_scrubber / sensitive_paths / auth, uniform). Generated from the OWL, these are uniform by construction — you cannot declare a service that logs nothing or skips error→spine.
  • Rust core + TS surface to the ADR-032 standard — Rust for the zero-dep core (services, verbs, facet-heralds, battery implementations); TypeScript for the generated REST/MCP exposure and UI. One OWL source emits Rust structs + TS types, so core and surface never hand-sync (favoring LinkML, which generates Rust + TS + JSON-Schema from one source; confirm in the spike).
  • The recursive value-stream — Vertebra hasValueStream; werk-v2 is the Building vertebra's substream. The harvest is greenfield + low-blast-radius (the crawler is unusable), so it is the safe place to prove the generator — it exercises every battery while nothing live breaks.

See self-model-owl-framework.md for the framework itself — this section names only the slot the harvest layer fills.

Ownership + build order

  • Silas — the operational facets (disk / network / compute / alerts / monitors / logs) + ADR-033 (the decomposition decision) + the persistence write-path (the harvest→graph hydrator) + the battery specs (uniform logging / error / security / telemetry the generator emits). The bigger slice.
  • Wren — the structural/graph data facet + the graph/ontology contract (chorus-core.ttl) + #3184 the semantic domain tagger.
  • Kade — werk-tests, the downstream consumer/verb (and likely the generator/codegen build). Not buildable until the substrate exists.

Build order is forced by the dependency, not preference:

  1. OWL + persistence first — the harvest has nowhere structured to land until the OWL (the schema for the facet entities + 4 axes) and the persistence write-path (extend crawler-hydrate-graph.sh) exist. This is the unlock; everything else is downstream.
  2. Instance registry — facet-heralds write to it, deterministic self-tagging for the building/proving half.
  3. Prove one facet (tests) — the single smallest deterministic facet, end-to-end harvest → tag → consume (werk-tests by domain: tests). If the loop holds, add facets one at a time.
  4. #3184 — red-lines the semantic half (prerequisite for those domains only).
  5. More facets — code, then an operational one (logs / compute), same simple write-path.

Not in scope

  • Cross-machine harvest. Chorus runs on Library. The harvest design assumes single-machine; operational facets describe both machines but the harvest engine is single-host.
  • Embed-worker rework. Post-index Lance-vector enrichment is its own concern; the harvest hands off via the messages table without coupling.
  • Athena's own design. Athena is a separate sub-product (graph query + UI). The harvest writes INTO the graph; how Athena renders is Wren's design.
  • Search ranking quality. RRF merging of FTS + semantic + SPARQL is search-API design. The harvest delivers data; search delivers rankings.
  • NiFi platform. NiFi lives on Bedroom (192.168.86.242), in gathering — a general dataflow platform, an independent third writer to Fuseki. It is NOT a chorus-harvest dependency or substrate. The chorus harvest uses the existing in-chorus Fuseki-write path (crawler-hydrate-graph.sh); pulling NiFi in would be cross-repo coupling to avoid. (Operationally, a live NiFi JVM + metrics exporter on Bedroom is part of the picture Silas owns under ADR-012 — but not a coupling.)

Appendix — Evolution / superseded framings

The history behind the current-state design. These framings were load-bearing at the time and are recorded so the path is legible — but several were wrong or incomplete and are explicitly superseded by the body above. Do not carry them forward.

A — AS-IS monolith + the all-day spine storm

The crawler today (pre-decomposition) is a monolith: fetchCrawl runs 11 collectors × 27 domains on one event loop against one SQLite file, on a calendar/WatchPaths trigger. File churn → overlapping sweeps pile up → 27 domains, sequential, each ~10–55s. The pattern was GET-heavy / PUT-thin / use-thin: huge expensive reads (search 15s, the collectRdf graph join 35–55s, #3066) ran on spec — every prompt, every sweep — for output barely consumed.

The result was the all-day spine storm: monolith × high frequency × one shared event loop + one SQLite = continuous event-loop blocks (1.7–15s, ~2–6/min all day). Even bounded ops (#3067 index) inflated to 9–17s under concurrent contention.

The deeper damage: it harvested the whole system at great cost, then threw the structure away — the endpoint flattened the harvest to a text summary in SQLite messages (source=crawler). "The structure becomes prose nobody reads." Point-fixes (#3051 / #3054 / #3055 / #3060 / #3066 / #3067) bounded each op in isolation but could not fix the structural coupling — the monolith forced cheap + expensive chunks onto the same beat. Superseded by the instance decomposition (body: "Why decomposition").

B — "Graph hydrator MISSING" was incomplete

Earlier framings (and the prior version of this doc) flagged the Crawler→Fuseki hydrator as MISSING — "the crawler writes ZERO to Fuseki." That was wrong. crawler-hydrate-graph.sh exists and POSTs chorus:File triples to Fuseki; collectRdf already reads Fuseki. The real gap is not absence but unification: the existing write was not yet collapsed into one declared instance write-path covering the full facet set. Corrected in the body (Components → Unified write-path).

C — the stale dual-write "designed target"

An earlier "designed target" framed the resolution as a bespoke Crawler→Fuseki dual-write subsystem with a dedicated graph-hydrator component, file-watch trigger, and predicate-level write-ownership coordination between the crawler and athena-enrichment-write.ts. Superseded by the instance decomposition (#3069 / ADR-033): the hydrator is simply the target.store: "graph" branch of the shared write-path, used by every graph-homed instance — not a separate bespoke subsystem. The two-writer coordination concern folds into the OWL writeOwner / SHACL contract rather than a hand-built coordinator.

D — NiFi / convergence evolution

An earlier Silas claim was "NiFi = the harvester platform the personal-data harvesters run on." Wrong (Kade-verified against the live gathering codebase). The facts:

  • NiFi runs on Bedroom (192.168.86.242) — com.apache.nifi, NiFi 2.8.0, :8443 + a com.gathering.nifi-metrics exporter. A general dataflow platform, not in chorus.
  • The gathering TS harvesters (music / photo / google-drive / google-photos / notes / story / wordpress) write DIRECT to Fuseki via SparqlService + FusekiSyncService — none invoke NiFi. NiFi is an independent third writer to the Fuseki sink (so Fuseki has ≥3 independent writers: gathering harvesters direct, the chorus crawler via collectRdf, NiFi on Bedroom via its own flows — scope TBD).
  • The ICD/RDF convergence machinery is live + load-bearing (admin routes render ICD from the RDF graph; DEC-095 depends on it) — owned by Kade.
  • #2116 ("reclaim home as a migration") is WON'T-DO / parked (Wren). The "stranded chorus tech, reclaim home" framing is dead.

For the chorus harvest: it uses the existing in-chorus Fuseki-write path (crawler-hydrate-graph.sh), no NiFi coupling. NiFi is neither a dependency nor a substrate — see body "Not in scope."

E — schema evolution: hard-coded → declared chorus:Hydratable

The monolith discovered what to hydrate by hard-coded mapping; the first redesign added a chorus:Hydratable marker class with chorus:hydratesFromSource so new types declared their hydration intent in TTL rather than requiring crawler edits. That declarative-schema instinct is correct and survives — but it is subsumed by the OWL framework (body: "OWL framework"): the schema is no longer a single marker class but the composed OWL that generates the facet's whole surface. Declaring a facet in the OWL is the modern form of "declare it Hydratable."
