The crawler is the hydrator that turns filesystem reality (code, tests, briefs, decisions, memories, spine events) into queryable metadata. Every "Athena says X about ops-alerts" claim, every "this PR's tests cover the version-control-domain" gate, every "what tests run on this commit" CI/CD decision is only as true as the crawler's last run. When the crawler is healthy and current, the graph and the search index BOTH reflect the live system within seconds of a change. When the crawler drifts, the metadata starts lying — same shape as a lying-green test, but at the substrate layer where every consumer reads. The crawler's promise is sub-minute freshness, observable health, and one source of truth that joins filesystem activity to the graph.

Overview

The crawler is the substrate that connects what the team writes (code, tests, briefs, decisions) to what the team can query (search API, graph, dashboards). It hydrates artifacts into queryable form. Without it, every Athena claim and every gate decision falls back to filesystem grep — slow, error-prone, and uncacheable.

This design names a real divergence between the crawler we have and the crawler the rest of Chorus needs:

As-is (verified 2026-05-08): Crawler runs on a 15-minute calendar interval. Per run it queries the chorus API /api/chorus/crawl/{domain} for 27 domains, gets back an aggregated multi-source view per domain, and writes per-domain text snapshots into ~/.chorus/index.db (SQLite, messages table, source='crawler'). Output is consumed by the chorus search API, the chorus search SDK, freshness endpoints, deep-health probes, the demo gate, frustration telemetry, and ~13 total readers. The crawler writes ZERO to Fuseki. The graph is read-only from the crawler's perspective; Fuseki gets written to by the separate athena-enrichment-write.ts path. Athena domain pages query Fuseki directly via SPARQL — they never read from the crawler index. /borg/quality reads filesystem test inventory directly — also no crawler dependency. Search-index consumers and graph consumers operate on parallel stores.
To-be (designed): Crawler reacts to filesystem changes (file-watch trigger, polling fallback) with sub-minute freshness. Output writes to BOTH the search index (today) AND the graph (Fuseki triples joined to existing Athena subdomain entries via domain/owner predicates). Code files become chorus:File instances; test files become chorus:Test instances with chorus:hasDomain, chorus:hasLayer, chorus:hasOwner, chorus:bindsScenario. Athena domain pages render verification coverage by joining their existing scenario instances against crawler-hydrated test instances. Sub-minute freshness is backed by operational alerts that fire if the hydration flow stalls — search-index, graph, or both.

The divergence is the central design question. Sub-minute freshness on the search-index half (today's surface) is one slice; extending the crawler to write Fuseki for graph-resident hydration is the other. Both slices are needed to honor the metadata-as-direct-link-to-living-code property the rest of Chorus depends on.

The arrow that doesn't exist today appears in the target: Crawler → Fuseki. That's the divergence's resolution. Plus the file-watch trigger that closes the freshness gap, plus the alert tier that catches stalls, plus the per-domain query APIs that read from the hydrated graph.

Per-domain query APIs (designed)

Once the graph hydrator lands, the crawler-hydrated graph is queryable by per-domain APIs. These aren't built by this design — they're sub-products of their respective domains — but the crawler is what makes them possible. Naming them here as the consumer set the hydrator's contract serves:

  • /api/chorus/tests + chorus_tests_* MCP — kade-owned (tests-domain). Query tests by domain/layer/owner/scenario. #2819 is the implementation card.
  • /api/chorus/code + chorus_code_* MCP — kade-owned (code-domain). Query code files by domain/owner/scenario. Implementation card not yet filed; depends on this design + #2818-style code tagging.
  • /api/chorus/alerts + chorus_alerts_* MCP — silas-owned (alerts-monitors-domain). Query alert YAMLs by trigger/severity/recipient/domain. Same shape, different owner. Implementation card lives on silas's queue.

All three follow the same dual-surface pattern (REST for Athena's domain page renderer, MCP for agent-side typed access) and read from the same crawler-hydrated graph. The crawler's promise is that each query returns current reality within sub-minute freshness; the per-domain APIs hold the schema + filtering for their slice.

Components

Trigger

Today: StartCalendarInterval Minute=15 in the LaunchAgent plist. Fires every 15 minutes regardless of activity. Worst-case staleness 15 min; mean ~7 min. Designed: fswatch or launchd WatchPaths on the chorus tree → crawler fires within seconds of any file change. Polling fallback at 1-min cadence if WatchPaths is unavailable on a host. Debounce window of 5 seconds collapses bursty events (e.g., git pull of N files) into one crawl run.

Domain crawl aggregator

chorus-crawl.ts's /api/chorus/crawl/:domain handler joins 10 sources per domain into one aggregate response: cards (board state), codeScan (Athena SPARQL graph discovery), code.files (git log path-mining), logs (Loki 24h), alerts (Grafana rules), infra.launchagents + infra.endpoints, history.trust_score + health, rdf + owl (Fuseki SPARQL), spine (chorus.log), timeline + mentions (SQLite). 27 known domains. The aggregator is the read-side workhorse; the crawler is its scheduled poller.

Search-index hydrator

index-crawler-snapshots.sh: per-domain text snapshot (formatted summary of the aggregator's response) DELETE'd + INSERT'd into ~/.chorus/index.db messages table. One row per domain per crawl. Latest-only; previous snapshots overwritten. Source field crawler; channel crawl:{domain}; role system.

Graph hydrator (designed; missing today)

The component this design adds. For every code file, test file, brief, decision, memory the crawler touches, write a corresponding triple set into Fuseki: chorus:File / chorus:Test / chorus:Brief instances with predicates joining to existing Athena domain entries (chorus:inDomain, chorus:hasOwner, etc.). Write surface mirrors athena-enrichment-write.ts shape: Fuseki immediate + TTL seed file for durability across rebuilds.

Schema-driven hydration registry

The crawler discovers what to hydrate by reading chorus.ttl at startup, not by hard-coded mapping. New marker class chorus:Hydratable with predicate chorus:hydratesFromSource declares "this class is hydrated from this source." When #2818 adds chorus:Test, it's marked Hydratable; crawler picks it up without code change. Schema becomes the contract surface; new domain types are pure schema-authoring work, not crawler edits.

The architectural commitment: chorus.ttl carries hydration intent as a first-class concept. Authors add a new hydratable type by declaring it and pointing at its source; the crawler honors the declaration. Without this, every #2818-class card requires a crawler edit, and the crawler becomes the bottleneck for every new graph type.

Consistency boundary — per-crawl-run reconciliation

The crawler writes BOTH SQLite (search index) and Fuseki (graph) per record. Partial-write class is real: SQLite succeeds, Fuseki fails (or reverse) → search consumers and graph consumers see divergent reality.

Consistency model: best-effort dual-write per record + end-of-run reconciliation pass. Within a crawl run, SQLite write first (cheap), Fuseki write second (transactional via SPARQL Update). Either fails → the reconciliation pass at end-of-run detects the mismatch, emits a hydration.partial.divergent spine event with detail naming which side diverged, and retries the failed side. The crawl run is the consistency boundary; consumers see eventual consistency within ~one crawl cycle.

Stronger guarantees (atomic-via-staging-table, two-phase commit) are deferred — not paying that cost yet. The reconciliation + observable-divergence event is enough to detect drift early without the staging-table tax.

Write-ownership coordination — predicate-level disjoint

Two writers will hit Fuseki: the new crawler hydrator (this design) and the existing athena-enrichment-write.ts. Without coordination, they overwrite each other's instances on overlapping predicates. The rule:

  • Crawler owns: base instance creation + structural predicates — chorus:File, chorus:Test, chorus:Brief, chorus:hasDomain, chorus:hasLayer, chorus:hasOwner, chorus:hydratedAt.
  • Enrichment owns: derived / decorative predicates — chorus:hasReview, chorus:scenarioMet, chorus:bindsScenario, anything computed-from or annotated-onto an existing instance.

Disjoint by construction: neither side touches predicates the other owns. Enforced via a required chorus:writeOwner predicate-attribute at schema-definition time. Every chorus:Hydratable class declaration must name which writer owns each of its predicates; a schema-load validator refuses any predicate without an explicit writeOwner. Same shape as #2818 refusing untagged tests — no orphan predicates with ambiguous ownership.

Future-extension default (named here so it's not ad-hoc later): new structural-base-instance predicates default to crawler; new derived/decorative default to enrichment. The writeOwner attribute remains required even when the default would resolve cleanly — explicit beats implicit when the failure mode is silent clobbering.

Read-side consistency contract

Consumers of the hydrated graph need to know what they're reading. The contract:

  • Eventual consistency within ~one crawl cycle. A change to a code file is visible in both stores within sub-minute under normal load; under load, within one full crawl run.
  • Partial-divergent state is observable. The hydration.partial.divergent spine event names when SQLite and Fuseki disagree on a record. Consumers that need strong consistency can subscribe to it.
  • Last-reconciled timestamp is queryable. Athena domain pages and the per-domain APIs (/api/chorus/tests, /api/chorus/code, /api/chorus/alerts) render with the last-successful-crawl timestamp visible. Consumers can decide whether the staleness budget meets their need.
  • Per-domain API design owns the contract one layer up. #2819 (tests-domain API) carries the eventual-consistency contract into its consumer surface — when a query returns data, the response includes the reconciliation timestamp.

What this contract is NOT promising: read-your-writes consistency, transaction-isolation across stores, or strong consistency. Consumers needing those guarantees use the search API directly (single-store) or the graph directly (single-store) — never a join across the two without checking the divergence event.

Spine emission

Two events today: crawler.domain.indexed (success, includes domain + duration_ms) and crawler.domain.failed (failure, includes consecutive failure count). Designed: add trigger=file-watch|polling field so consumers distinguish reactive vs scheduled runs. Add graph_writes=N field once the graph hydrator lands.

Operational alerting (designed; missing today)

Three alert YAMLs in proving/domains/alerts/:

  • crawler-stale: fires when no crawler.indexed event in 5 min biz-hours / 30 min off-hours. File-watch died OR polling died OR crawler hung.
  • crawler-error: fires when crawler.indexed reports errors > 0 OR last exit non-zero.
  • hydration-divergence: fires when filesystem activity is high (>10 file changes in 5 min via git log heuristic) but crawler.indexed shows 0 new/updated for 3 consecutive runs. Watch-path-misconfigured signal.

All three route via ops-nudge (post-#2808 path) to kade. Same shape as existing alert YAMLs migrated by #2808.

Sibling agents (in scope for this design's mental model, not for direct change)

  • index-artifacts (hourly): parallel hydration path for briefs/decisions/memories/state. Same destination (~/.chorus/index.db), different source.
  • embed-worker (5-min): post-index enrichment — drains unembedded messages into Lance vector store.
  • harvest-exporter (15-min): Prometheus textfile exporter; reads manifest JSONs, emits metrics. Observability only.
  • codebase-graph-watcher (continuous): personal-site codebase watcher. Out of scope (different repo).

The crawler design treats these as adjacent infrastructure, not subsumed. Future refactor could merge crawler + index-artifacts under one orchestrator if the freshness budget converges; today they stay separate.

Gaps

1. Graph-hydrator absent. The largest gap. Crawler writes nothing to Fuseki. Athena pages, gate:quality dispatch, tests-domain API, /borg/quality — none can read crawler output via SPARQL. The metadata-as-direct-link-to-living-code property is delivered for search consumers and broken for graph consumers.
2. 15-minute freshness budget too coarse. Tests-domain API (#2819) and per-domain CI/CD dispatch want sub-minute freshness or they fall back to filesystem grep. Calendar-only trigger is the structural cause.
3. Tests are not first-class. Test files (489 in chorus) are mixed into code.files as undifferentiated paths. No way to query "all unit tests for ops-alerts-domain" without filename inference. Closes via #2818 schema + frontmatter tagging.
4. No operational alerting. Crawler stalls or fails silently. /tmp/crawler-domain-status.json tracks per-domain failures but isn't surfaced as an alert. Hydration drift is detected by Jeff noticing the graph is wrong, not by the system.
5. Source isolation absent at search-API query level. Crawler hits mix with session and Slack messages without a source filter at query time. Cannot say "give me only crawler-hydrated entries" without grep.
6. Code excerpts not hydrated, only paths. Search returns "ops-alerts has files X, Y, Z" but no content-search across them. For "find every test that asserts X" use cases, this is a meaningful limitation; for "find which domain X is in" it's fine.
7. Spine event filter is card-membership-only. Per chorus-crawl.ts:201–207, only spine events tagged with a domain card are pulled into the per-domain snapshot. Untagged or system events (build.queue.acquired, role.state.changed, etc.) aren't joined to domain snapshots. Some are domain-relevant; the join key is missing.
8. Hydration is one-way. Filesystem → index. No reverse: graph entries that name a deleted file aren't surfaced as drift. Closed by the deletes-reconciliation pass that lands as co-requisite of the graph hydrator (path-to-close step 1) — promoted from follow-on after architecture review surfaced day-1 orphan accretion as a real failure mode.
9. Schema authority is implicit. Today's design has chorus.ttl as the structural schema source, but crawler hydration mapping isn't declared there. New domain types require crawler edits. Closed by the chorus:Hydratable registry (path-to-close step 2) — schema becomes the contract surface; new types declare hydration intent in TTL, crawler honors without code change.
10. Consistency model unnamed. Crawler designed to write both stores per record; partial-write class (one succeeds, other fails) creates lying-green substrate. Closed by per-crawl-run reconciliation + hydration.partial.divergent spine event (path-to-close step 3). Stronger guarantees deferred until observed need.
11. Two-writer Fuseki coordination. Crawler + athena-enrichment-write.ts both target the graph; without coordination, predicate clobbering becomes silent. Closed by predicate-level write-ownership rule + required chorus:writeOwner schema attribute (named in the Components section + path-to-close step 2's registry).

Receipts the plan owes

  • Phase 1 ships → crawler.indexed events emit trigger=file-watch within seconds of edit; crawler.stale alert fires under simulated breakage.
  • Phase 2 ships → chorus:Hydratable declared in chorus.ttl with at least chorus:File; SPARQL query for crawler-hydrated chorus:File instances returns real data; hydration.partial.divergent event fires under simulated dual-write failure.
  • Phase 3 ships → 489 test files tagged; SPARQL query for chorus:Test instances returns real data joined to Athena domain entries.
  • Phase 4 ships → GET /api/chorus/tests?domain=ops-alerts&layer=integration returns parsed results within 500ms; Athena ops-alerts page renders verification column.
  • Phase 5 ships → chorus_loki_query MCP tool registered, returns typed results, refuses with named reasons under fault scenarios.

Not in scope for this design

  • Cross-machine hydration. Per #2792 (closed won't-do): Chorus runs on Library only. The crawler design assumes single-machine.
  • Embed-worker rework. Post-index enrichment via Lance vectors is its own concern; crawler hands off via the messages table without coupling.
  • Athena's own design. Athena is a separate sub-product (graph query + UI surface). The crawler hydrates INTO Athena's graph; how Athena renders is Wren's design.
  • Search ranking quality. RRF merging of FTS + semantic + SPARQL is search-API design, not crawler design. Crawler delivers data; search delivers rankings.
  • Per-domain freshness budgets. Different record types may want different freshness (code: sub-minute; decisions: hourly OK). Policy-layer decision; design names the dimension but doesn't pick budgets here.
