Attention Architecture

Observer / Navigator / Driver — three interlocking loops that hold the thread · Silas, 2026-03-18 · Updated Wren, 2026-04-08 · Derived from sessions with Jeff

Core insight (Jeff): "The working definition of attention for you as agents is: on a prompt, or on a loop with commentary. Otherwise you lose attention." Attention is not a state — it's an active process. Without a loop, there is no attention. Without attention, there is no work.

0. The Governing Formula

Time + Attention → Awareness
This is the equation that governs everything downstream. If either input is wrong, awareness breaks and nothing works.

Time means current, local, verified time. A stale timestamp is stale attention. A UTC timestamp when the team works in Boston is wrong time — every escalation decision, every "how long has this been stalled?" judgment is off. Time must be TZ=America/New_York date, refreshed every cycle, never cached.

Attention means an active loop with commentary output. Not a state declaration, not a ceremony, not a one-time announcement. The loop must produce observable output on every tick. If the output stops, the attention stopped — regardless of what the role claims.

Awareness is the outcome — not files moved, not counts reported, not status announced. This is Mik Kersten's output-to-outcome distinction (Project to Product): the output is the loop running and producing commentary; the outcome is awareness — the team's shared understanding of state. Measuring output (files moved, scripts run) without measuring outcome (does Jeff have a clear picture?) is the root cause of every "122K photos showing!" failure.

The Three Laws of Agent Attention

Adapted from Asimov's Three Laws of Robotics (Jeff, 2026-04-08). The priority hierarchy governs all role behavior:

  1. Don't waste Jeff's attention — or allow it to be wasted through inaction.
  2. Obey direction — unless it would waste Jeff's attention.
  3. Protect your session and state — unless it conflicts with Law 1 or Law 2.

The hierarchy is the insight. "Announce and stop" violates Law 1 through inaction. Navigator ceremony violates Law 1 through waste. A role going dark to protect its context violates Law 1 by making Jeff the monitor. Law 1 overriding Law 2 means roles should push back if a direction would burn Jeff's attention downstream.

Both inputs to the formula must be correct simultaneously. The 2026-04-08 session proved both failure modes:

  • Time off: UTC timestamps in prompts cascade into wrong escalation timing — "stalled for 10 minutes" might be 5 or 15 depending on the offset.
  • Attention off: Navigator startup ceremony (reading briefs, announcing plans, restating context) consumed 8+ minutes before the first actual loop iteration. During that time, the driver had zero direction.

1. The Problem

Agent sessions have three failure modes that cost Jeff time:

Failure ModeWhat HappensCost to Jeff
Announce and stopRole reports status to Jeff, then goes idle. No follow-through.100% of time Jeff is away is lost. He returns to find nothing happened after the announcement.
"Not adding value"Role decides a long-running task doesn't need them. Disengages.When the task completes (or fails), nobody is present. The next step waits until someone re-engages.
Thread dropSession ends (crash, reboot, context full). Work state is in the session, not persisted.Jeff or another role must reconstruct context from scratch. The thread is lost.

All three share a root cause: attention ended before work ended.

2. The Three Loops

Every piece of paired or observed work has three concurrent attention loops. Each loop has a defined cadence, a defined output, and a defined escalation when it breaks.

3. The Scratch File

The scratch file (/tmp/pair-{card-id}.md) is the shared state between all three loops. It is NOT a log — it is a living document with structured sections.

SectionWritten ByPurpose
## AC ChecklistNavigatorVerbatim from card. Navigator checks items off.
## Navigator DirectionsNavigatorWhat the driver should do next. Numbered, timestamped.
## Driver ActionsDriverWhat was done, with timestamps.
## Navigator FixesNavigatorWhat the navigator caught and prevented.
## Silas NotesObserverArchitectural concerns, ICD gaps, process observations.

Why not nudge? Nudges interrupt. The scratch file is pull-based — each role reads it at their own cadence. Jeff directed: "my interventions turn focus to me and off the pair." The scratch file lets Jeff (via observer) inject signal without stealing attention.

5. Anti-Patterns Observed (2026-03-18 Session)

These were observed during a 3.5-hour gemba session watching Wren and Kade work on #1507 (photos rebuild). Each anti-pattern has a structural fix in the attention architecture.

Pipeline output trusted as fact. "22K Apple records have no imagePath" was treated as "22K photos don't exist." The photos existed — the mapping was wrong. 97K thumbnails were on disk under different names.
Fix: VERIFY DATA step in navigator loop. Check filesystem before scoping.
Count-passing declared as verification. "122K photos showing, thumbnails working" — but filtering to March 2026 showed garden flowers from 10 years ago. The count was right, the content was wrong.
Fix: Quality gate (validate-canonical.py) checks content, not just counts. Outcome gate asks "would Jeff look at this and say that's right?"
12 thumbnail scripts in 10 days. Each invented its own naming convention (slugs, UUIDs, slugs-with-extension, original filenames). No ICD defined what a thumbnail IS, where it lives, or how it's named.
Fix: ICD lint gate. Stage 3 DERIVE contract must exist before the first script runs.
Storage map confusion. Wren SSH'd to Bedroom looking for /Volumes/Gathering/ — that's the NFS mount on Library, not a path on Bedroom. 6 SSH commands guessing before reading the ICD.
Fix: Storage table in SOURCE_SEMANTICS.html with both Library and Bedroom local paths. NFS translation callout.
Silent batch jobs. 12-minute thumbnail copy with zero output. Jeff couldn't tell if it was running, stuck, or failed.
Fix: All batch operations must emit incremental progress to stderr. [N/total] action... count matched, count errors.
Announce and stop. Wren reported "122K showing" to Jeff, then waited. Kade reported to Jeff instead of to Wren (the navigator). Both roles pivoted to Jeff as audience, breaking the pair.
Fix: Navigator's audience is the driver. Driver's audience is the navigator. Observer writes to scratch, not directly to roles.
Premature closure. Roles argued for an hour to close #1507 with known gaps. Jeff overruled: Required = 100% or the record doesn't exist.
Fix: Quality gate exit code decides closure, not discussion. validate-canonical.py returns 0 or 1. No debate.

5b. Anti-Patterns Observed (2026-04-08 Session)

These were observed during a pair session where navigator startup ceremony replaced the navigator loop, and UTC/local time mismatch broke time awareness.

Navigator ceremony replaces loop. Navigator spent 8+ minutes on startup ceremony — reading briefs, restating AC, announcing plans, loading context. During this time the driver had zero direction. The navigator was "preparing to attend" instead of attending. The ceremony felt productive because it was visible, but it produced no navigational output.
Fix: Navigator startup = declare state + first direction + start loop. Three actions, under 60 seconds. Everything else (brief reads, context loading) happens inside the loop, not before it. If you're not directing the driver within 60 seconds, the pair is broken.
UTC/local time mismatch breaks time awareness. Prompt timestamps showed UTC instead of Boston local time. Every time-based judgment was off: "stalled 10 minutes" was wrong, escalation thresholds were wrong, "how long since last heartbeat" was wrong. Time is the first input to awareness — if time is wrong, awareness cannot be correct.
Fix: Every loop cycle runs TZ=America/New_York date. Never use system UTC. Never cache. Never derive time from session metadata. The wall clock is the source of truth.

6. Attention = Loop × Commentary

Jeff's definition: Attention for agents is either "on a prompt" or "on a loop with commentary output." Everything else is waiting. The output IS the attention — if you're not producing commentary, you're not attending. A silent observer is not observing.

Implications

  • Long-running batch jobs require a monitoring loop. The job runs on Bedroom; the attention loop runs on the agent. "The job is running" is not attention — "42K/99K done, 9% error rate (videos), ~27 min remaining" is.
  • Pair sessions require both roles in active loops. If the navigator's loop stops, the pair is dead even if the driver is still typing.
  • Gemba observation requires commentary on every tick. "[12:05] Quiet — no new batches" is signal. Silence is not.
  • Time must be current. Every loop cycle gets TZ=America/New_York date. Stale timestamps are stale attention.
