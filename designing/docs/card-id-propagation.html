card_id propagation contract (#2838)

Silas · 2026-05-09 · substrate design. Companion to trace_id (#2839); rides the same envelope, different mint/inheritance rules. Implementation lands per-surface in follow-on cards.

Headline

card_id is the work-binding field. It travels with card-scoped flows (a /pull, /acp, /demo, gate run, build triggered by a card commit) and is silent on system events (daemon heartbeats, health probes, canonical-sync, alert-fires not bound to a card). Card-scoped events MUST carry it; system events MUST NOT. The split is what closes the gate-chain-via-card-comments gap structurally — gates emit comments today because spine doesn't bind to card. With card_id on every event of a card-scoped flow, spine and board state join cleanly without a comment-as-channel workaround.

Scope rule (the load-bearing primitive)

card_id is optional in the envelope and required on a closed list of event-type prefixes. Adding it everywhere is wrong (heartbeat / health / sync / alert events have no card to bind to). Adding it only ad hoc is wrong (today's failure mode — gate emits drop the field, board state diverges from spine). The contract names the closed list; spine-events.json declares per-event-type whether card_id is required.

MUST carry card_id (card-scoped events)

  • card.* — card.pulled, card.demo.started, card.accepted, card.rejected, card.unpulled
  • gate.* — gate.product.passed, gate.code.passed, gate.quality.passed, gate.arch.passed, gate.ops.passed (and *.failed counterparts)
  • demo.* — demo.validate.completed, demo.preflight.completed, demo.stakes.completed, demo.signal.completed
  • chorus_acp.*, chorus_commit.*, chorus_pull_card.*, chorus_unpull_card.*, chorus_cards_done.* — every transactional MCP that derives a card from the board
  • build.*, deploy.* — only when the build/deploy is triggered by a card-scoped commit (see cycle stitching)
  • flow.* terminal events — when flow_kind ∈ {acp, pull, demo, gate, build}

MUST NOT carry card_id (system events)

  • library.health.*, canonical.sync.*, canonical.synced, canonical.repaired — infrastructure
  • alert.* when not bound to a card-scoped flow
  • nudge.*, mcp.nudge.* — nudges aren't card-scoped (they're role-to-role)
  • pulse.*, observer.* heartbeats and fan-outs not in a card flow
  • chorus-hooks-supervisor.* daemon lifecycle events
  • tool_call events from agent sessions outside an active card

MAY carry card_id (context-dependent)

  • guard.scrub.blocked, icd.gate.refused, doc.coherence.warned — hook events: carry card_id when the triggering tool call originated inside a card-scoped session, omit otherwise
  • commit.landed, binary.deployed — carry card_id when derivable from the commit trailer (see cycle stitching), omit when the commit had no card
  • crawler.graph.*, hydration.partial.divergent — substrate-data events: usually no card, but if a card-scoped command triggered the crawl, propagate via env

The propagation primitive — same shape as trace_id

card_id rides through the same envelope/closure/env mechanism as trace_id (#2839). The difference is in where it originates: trace_id is minted at flow entry; card_id is bound at flow entry if the flow is card-scoped, otherwise absent.

// MCP entry handler (TS):
const trace_id = mintTraceIdV7();
const card_id = await deriveCardIdFromBoard(role); // or args.card_id
const emit = createSpineEmitter(trace_id, defaultSpineEmitter(), card_id);
// every emit() inside this call carries trace_id + card_id (when present)

SpineEmitter extension (#2845 follow-on): grow createSpineEmitter from (traceId) to (traceId, base, cardId?). Same closure pattern: when card_id is set, every emit attaches it to fields; when absent, no card_id key in the JSON. Falsy-card_id early-return preserves the system-event MUST NOT contract — system handlers construct an emitter with no card_id and the field is structurally absent, not "card_id: null".

Per-surface handoff

MCP entry handlers (TS)

Card-scoped MCPs derive card_id at entry (board lookup or argv) and pass to createSpineEmitter. System-scope MCPs (chorus_nudge_message) construct emitter without card_id; field stays absent in their step events. This is the structural enforcement of the MUST/MUST NOT split.

Bash CLIs

card_id rides on env (CHORUS_CARD_ID) set by the MCP's execFile options when the MCP is card-scoped. chorus-log reads $CHORUS_CARD_ID and adds it to every emit IF the event-type is in the MUST-carry list (chorus-log subcommand or wrapper logic). For top-level bash entry (skill bash, cron), the entry script exports CHORUS_CARD_ID when invoked from a card-scoped skill (/demo, /gate-*); otherwise leaves unset.

chorus-log change: dual to the trace_id change in #2839 — if $CHORUS_CARD_ID set AND the event-type prefix is in the MUST-carry list (card.*, gate.*, demo.*, build.*, deploy.*, chorus_*), prepend card_id=$CHORUS_CARD_ID. Otherwise leave unset. The classifier table is small and stable; ship as a const list in the bash wrapper.

Rust hooks

Shim reads $CHORUS_CARD_ID from the invoking shell's env. Hook events follow the MAY-carry rule: include card_id when the env is set (active card-scoped session), omit when absent (file save outside a card, ad-hoc tool call). Tag flow_kind=hook-standalone when card_id absent so query surfaces can distinguish.

External boundaries

Same as trace_id: the MCP brackets external calls (gh, git, npm) with start/completed events that carry card_id. The interior is opaque.

spine-events.json schema change

card_id is optional in the envelope (envelope = global per-event default), but per-event-type schema declares it required for the closed list above.

// designing/schemas/spine-events.json

// envelope (global)
"envelope": {
  "card_id": {
    "type": "integer",
    "minimum": 1,
    "required": false,
    "description": "Work-binding id. Required on card-scoped events (card.*, gate.*, demo.*, build.*, deploy.*, chorus_*); MUST NOT appear on system events (library.*, canonical.*, nudge.*, pulse.*, daemon lifecycle)."
  }
}

// per-event-type override (for events in the MUST-carry list)
"events": {
  "card.pulled": {
    "fields": {
      "card_id": { "required": true, "type": "integer", "minimum": 1 }
    }
  },
  "gate.product.passed": { "fields": { "card_id": { "required": true } } },
  // ...
}

Validator: refuses an event in the MUST-carry list whose payload lacks card_id; refuses an event in the MUST-NOT list whose payload contains card_id. Soft-warn during migration, hardens once every surface conforms.

Multi-card flows

A /clearing session, a building-pipeline batch, or a card_done that closes N cards at once produces N events — one per card_id. Never a card_id list field. If the flow naturally fans out, the spine reflects that fan-out. Query surfaces filter by card_id and find every event for that card; a list-field would force every consumer to handle two shapes.

The flow itself stays one trace_id (one transaction); card_id varies per emitted event within it.

Cross-cycle stitching (/acp → building-pipeline)

Same channels as trace_id, dual trailer:

  1. Commit trailer: /acp writes Chorus-Card-Id: <card_id> alongside Chorus-Trace-Id:. Building-pipeline reads git log -1 --format='%(trailers:key=Chorus-Card-Id,valueonly)' on the commit it's about to build. If present, it sets CHORUS_CARD_ID for the build's child processes; build/deploy events inherit. If absent (commits made outside /acp), build/deploy events emit without card_id — that's correct behavior for non-card-scoped commits.
  2. launchd plist environment: scheduled crawler runs are not card-scoped; their trace exists, their card_id is absent. Don't fabricate one.
  3. Webhook X-Chorus-Card-Id header: future GitHub-Actions / external-CI handoff, dual to X-Chorus-Trace-Id.

The headline outcome: one /acp + the building-pipeline it triggers + the deploy events it produces all share one card_id (and one trace_id). Loki query {job=~".+"} |~ ""card_id":<N>" returns the full cycle for that card.

What this contract does NOT do

  • Does not require card_id on every emit. The MUST-NOT list is load-bearing — system events stay clean.
  • Does not introduce a card_id list field. Multi-card flows fan out into N events.
  • Does not retrofit historical events. Forward-only; pre-contract events stay un-bound and queryable only via temporal joins.
  • Does not subsume subdomain or caused_by binding. Tier-2 per #2832 audit; separate later contracts.
  • Does not redefine trace_id. trace_id (#2839) is the transaction-scope id; card_id is the work-binding id. Different mint rules, same envelope plumbing.
