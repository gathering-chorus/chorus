Properties — Domain Design — config-as-data, modeled in Athena

Silas · 2026-06-15 · Borg domain (Proving/Reflecting) · DRAFT (HTML, reworkable) · the config-driven substrate
The promise. Changing how Chorus behaves stops being a code change. Today config is registered in code — a threshold, a route, an allowlist, an ownership rule — so every change is commit→build→test→deploy→restart, on a flaky deploy. After: config is data in the model, read by generic engines; a change is a model write, live, no deploy. This is what makes Chorus configuration-driven — and it's the prerequisite for "model anything → build anything" (a client tunes via data, not your source).

1 · As-is — why "barely touched"

The properties domain holds 1 file (the #3426 crawl). It's essentially un-built — which is exactly why Chorus is "woefully not config-driven." The config that should live here is scattered, hardcoded: the deploy unit allowlist, nudge routing, the hasMapsTo rules, requiresAuth, the boston() boundary, ports/hosts in launchagents. Each is a property-on-a-node, written as code. Greenfield, not a painful migration — the work is to model it right and stop re-accreting config into code.

2 · The model — one uniform attachment

Config isn't a subsystem — it's modeled structure in Athena. A Property class, attached to any structural node via one uniform edge hasProperty. Same primitives, same machinery as the rest of the model — no bespoke config system (no-competing-implementations).

This generalizes what we already do ad-hoc. hasMapsTo (the L2 domain rules, #3426) is a Property on a Domain. The L1 type rules (code/test/doc/data/config/workspace — see below) are a Property too. requiresAuth (#3414) is a Property on a shape. Deploy units, nudge routes, the render boundary — all properties-on-nodes, smeared across code. The model doesn't invent config; it names and unifies it under one edge.

Worked example — the crawler's own classification rules ARE properties

The cleanest proof (and the reason the "make L1 clear" discussion belongs here): the L1 (type) and L2 (domain) tag rules are both properties, not code.

  • L2 — domain rules = hasMapsTo, a Property on each Domain (path-prefix → that domain). Already built (#3426).
  • L1 — type rules = a Property at system scope: the strata taxonomy {code · test · config · doc · data · workspace} + the extension/path → type rules (path overrides extension). Read by the type-herald.

The crawler engine reads both from the model and applies them — every file gets {type (L1), domain (L2)}, both from declared properties, zero hardcoded classifiers. That retires the scattered discover-code / discover-tests (the config-in-code anti-pattern) into one property-driven engine. So "make L1 clear" needs no separate L1 system — L1 is just another property the same engine reads.

3 · Resolution — the config cascade

Because the model is a hierarchy (product ⊃ domain ⊃ service; step ⊃ child stream), properties resolve by specificity: a node asks "what's my effective X?" and it resolves service → domain → product → value-stream — nearest wins, inherit otherwise. The same recursive tree we modeled, now carrying config that cascades down it.

4 · How it rides the machinery (no new substrate)

  • owl-api generates the config API (read + write) from the Property model — exactly as it generates the domain API. Reads are open; writes secured model-driven (#3414).
  • The DAL reads it on-demand → a config change is a model write, live, no restart (the prize: read-on-demand, not loaded-once). Startup-load only where churn is rare.
  • A config/infra herald hydrates the existing hardcodes (launchagents, ports, code constants) into the model — turning today's implicit config explicit-and-modeled. (Same herald family as the crawler facets; doubles as the toolchain/infra discovery.)

5 · Discipline

  • Discriminate — only varying config (by domain/instance/machine/client) becomes a Property; invariant logic stays code. Avoid the inner-platform trap (config so flexible it's a worse language).
  • Scope is intrinsic — the scope dimension (global / machine=Library·Bedroom / role / instance) is first-class; the properties domain's own definition already names "across Library and Bedroom."
  • Coherence-first — the model must be the trustworthy config source before we lean on it (clean IRIs #3430, the 2-axis tag). Otherwise we've moved the mess from code into an incoherent graph.
  • No re-accretion — a grep-gate blocks new hardcoded config landing in code (the ADR-041 attrition pattern); existing hardcodes lift on-touch.

6 · Precedent (the pattern is proven)

  • #3317 — deploy allowlist → structural discovery (a hardcoded registry replaced by model/structure-driven). Config-driven, shipped.
  • #3426 — file→domain rules in tree.json + a generic engine (= properties-on-domains, special case).
  • #3414 — auth derived from a model annotation; no code change to re-secure.
  • The program = generalize that one move: every hardcoded registry → a Property the engine reads.

7 · Sequencing

  1. Model Property + hasProperty in Athena (the four attachment points + scope + the cascade resolver).
  2. owl-api generates the read/write surface; DAL reads on-demand.
  3. Hydrate the first real properties (start with the ones that bite: deploy units, nudge routes, alert thresholds) — herald-discovered or authored.
  4. Migrate config-in-code → properties by attrition; grep-gate blocks new accretion.
