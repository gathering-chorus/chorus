Cards â€” Service Design

2026-05-01 â€” Wren (#2643). Refreshed 2026-05-02 â€” consumer-first restructure (Silas walk), versioned-output bridge for #2676. Implementation: #2652 accepted. Open cards: #2671 fail-loud sweep, #2672 Vikunja-write-refusal hook, #2673 cards-service Athena populate, #2676 validation single-source + dual-emit bridge.

A card mutation lands when, and only when, the role's request reaches the canonical adapter (the cards CLI) and the adapter (a) validates the input against a single canonical schema, (b) refuses unknown values at the writing surface, (c) writes to Vikunja through one client, and (d) emits exactly one spine event reflecting the actual state change. Direct writes to Vikunja are refused by hook. Spine events never lie about whether work happened. Validation has one source. Consumer migrations are bounded by a deprecation window with a hard retirement date â€” no follow-on-forever.

Overview

Cards is the kanban substrate for team coordination. Every unit of work (feature, fix, audit, follow-on) lives as a Vikunja card with structured labels (priority, owner, type, origin, domain, subdomain, subproduct). The substrate is API-with-validation: one canonical schema, transports derived from it, consumers migrating across a versioned bridge. Standard deprecation practice, not novel architecture.

Gaps in the cardsâ†”Loom contract

L1. No citation validation at card-write time. cards add --desc "..." accepts any free-text description. A card can cite DEC-9999 that doesn't exist; cards-service does not query Loom to verify.
L2. No back-reference from Loom artifact to driving card. chorus_principles_create MCP tool accepts optional dcSource but no cardId. handlers/loom-decisions.ts response has no relatedCard field. Query "which card filed this decision?" is unanswerable.
L3. No propagation when Loom artifact renames or retires. Decision DEC-NNN gets superseded; cards still cite the old ID with no flag.
L4. No production gate. Anyone can call chorus_principles_create regardless of subdomain ownership. TEAM_PROTOCOL.md is descriptive, not enforced.
L5. No citation audit script. No scheduled scan detecting "card #N cites DEC-X, but DEC-X does not exist." lint-fragments.sh R5 checks CLAUDE.md fragments, not card descriptions.

Components (designed)

Canonical zod schema + dual-emit bridge (#2676)

One file: directing/products/cards/src/schema.ts. Per-verb input shape with required fields, enums, refinements (subdomain âˆˆ Athena live query, subproduct âˆˆ closed list {athena, loom, werk, borg, convergence, clearing}). Source of truth for sdk + MCP + HTTP + CLI output schema + spec-doc generation. No hand-redeclared validation anywhere downstream.

Dual-emit bridge (consumer migration mechanism):

  • CLI emits both old prose output AND new --json structured output during the bridge window.
  • Old prose format carries a deprecation marker tagged with consumer attribution (X-Source header from The Clearing, calling-script name from skills, role from MCP/HTTP transports).
  • Telemetry visible in Borg per-consumer; team sees who's still on old format and at what rate.
  • Hard retirement date in schema metadata: 2026-06-15 (~6 weeks soak).
  • After retirement date, old prose format errors out; consumers migrate or break loud. No card cascade, no follow-on-forever, no per-consumer cards.

cards CLI canonical adapter

Verb router at directing/products/cards/src/cli.ts. Bash, MCP, HTTP all spawn cards <verb> --json with DEPLOY_ROLE injected. Verbs: add, move, done, tag, set, view, list, check, comment, swat, sequence-tag, bulk-move, snapshot, audit. Exit codes: 0 success/no-op | 2 validation | 3 persistence | 4 network | 1 unhandled.

sdk.ts business logic + truthful spine emit (#2671)

Validation derives from canonical schema (no hand-declared shape). Spine events fire only when state actually changes. Idempotent operations emit card.item.tag.idempotent (or no event); state-changing operations emit card.item.tagged. The "spine reflects what actually happened" rule is enforced at every emit site.

BoardClient Vikunja transport (#2671)

client.ts returns typed results that distinguish did-the-work from already-in-target-state. AlreadyTaggedError typed; 409 surfaced as a result type, not silently absorbed. Author field never defaults to 'unknown' â€” substrate corruption signal preserved by throwing on missing author. Per-role tokens loaded by config.ts:loadEnv.

Vikunja-write-refusal hook (#2672)

PreToolUse Bash hook in chorus-hooks detects bypass patterns: curl.*localhost:3456, fetch.*localhost:3456, gh api.*vikunja. Refuses with explicit message: "Direct Vikunja write blocked â€” use cards CLI." Allow-listed for the cards CLI binary itself. Bypass override CHORUS_VIKUNJA_DIRECT_WRITE=1 for one-off migration; bypass MUST emit a spine event for audit. Canonical-by-enforcement, not canonical-by-convention.

DEPLOY_ROLE-required identity

config.ts:detectRole() reads DEPLOY_ROLE env. Values: wren | silas | kade | jeff | automation. cwd-parsing fallback retired â€” refuse if unset (#2671). Role identity is required, not guessed.
