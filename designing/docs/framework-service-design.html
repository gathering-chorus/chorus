Framework — Service Design

2026-04-07 — Silas + Wren (#2300). Source: chorus.ttl v0.2.0, library-topology.ttl, domain-registry.md.

The contracts layer. Framework defines what data looks like (ICDs), what services do (service designs), and what the current truth is (domain pages) — all enforced by BDD gates. Framework reads from the OWL/RDF semantic layer above it to answer: "What services does Photos depend on? Who owns them? What gates protect them?" One SPARQL query, not four separate tools.

Semantic Layer (OWL/RDF)

Framework reads from the OWL/RDF semantic layer — it does not contain it. The semantic layer sits above Framework in the reference model, giving signals meaning that Framework tooling acts on.

Three Ontologies, Bridged BRIDGE

borg:urn:borg:ontology/
Infrastructure: Service, Host, dependsOn, hostedOn. 8 service instances.
chorus:jeffbridwell.com/chorus#
Coordination: Role, Gate, Vertebra, Tool, Artifact. Spine + trust metrics.
jb:jeffbridwell.com/ontology#
Domain data: Photo, Track, Story, Person, Note, Book, Seed. 14 classes, 15M+ triples.

framework.ttl uses owl:equivalentClass and owl:equivalentProperty to bridge all three without duplicating. Loaded in Fuseki as urn:chorus:framework (220 triples). Framework tooling — ICDs, domain pages, the domain crawler — queries this graph to discover what exists.

Domain Crawler Integration

How It Connects LIVE

The domain crawler (#1956) already queries 7 sources per domain. The framework ontology makes the crawler's implicit model explicit:

Crawler does todayQueries Fuseki counts, checks service health, reads ICD, scans harvest state — per domain
Framework addsThe relationships are in the graph, not in crawler code. Which services serve Photos? Query fw:servesDomain. Who owns them? Query fw:ownedBy.

Future: crawler reads framework.ttl to discover domains and their service dependencies instead of hardcoding them.

Gate Enforcement

Gates in the Graph ENFORCED

Chorus already has 4 gates (Direction, Design, Build, Proving) enforced by Rust hooks. The framework maps which domains and services are protected by which gates.

GateFrom → ToGatekeeperProtects
Direction GateDirecting → DesigningJeffSeeds domain (intake quality)
Design GateDesigning → BuildingSilasChorus, Infrastructure domains
Build GateBuilding → ProvingKadePhotos, Stories domains (data quality)
Proving GateProving → DoneJeffPhotos, Stories domains (acceptance)

Gate enforcement is mechanical (Rust hooks), not behavioral. The framework makes enforcement queryable: "Show me all domains with no gate protection" is a SPARQL query, not a manual audit.

Example Query

AC6: Domain → Services → Gates → Roles NEW

One query that traces the full chain for any domain:

PREFIX fw: <https://jeffbridwell.com/framework#>
PREFIX chorus: <https://jeffbridwell.com/chorus#>
PREFIX borg: <urn:borg:ontology/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?domain ?service ?gate ?gatekeeper ?owner
WHERE {
  ?domain a fw:Domain ;
           rdfs:label ?domainLabel .
  ?service fw:servesDomain ?domain ;
            rdfs:label ?serviceLabel .
  OPTIONAL { ?domain fw:protectedBy ?gate .
             ?gate chorus:gatekeeper ?gatekeeper . }
  OPTIONAL { ?service fw:ownedBy ?owner . }
  FILTER(?domainLabel = "Photos")
}
ORDER BY ?serviceLabel

Returns: Photos → Fuseki (Silas), Gathering App (Kade), NiFi (Silas) + Build Gate (Kade), Proving Gate (Jeff).

Gaps

Resolved: framework.ttl loaded into Fuseki as urn:chorus:framework (220 triples). jb-ontology.ttl loaded as urn:jb:ontology (47 triples, 14 classes). Duplicate urn:framework:bridge graph dropped.
Gap: Domain crawler still hardcodes domain-service relationships. Needs to read from fw:servesDomain instead.
Gap: API instances are minimal (5 endpoints). Full coverage requires mapping all 147 app endpoints into the graph.
Gap: Gate→domain protection is hand-mapped. Should be derived from hook telemetry — which hooks fire on which file paths tells you which gates protect which domains.
Gap: library-topology.ttl still shows Docker lifecycle for Grafana, Loki, Vikunja — all are native LaunchAgents now.
