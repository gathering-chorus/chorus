Page Anatomy — the pattern every generated page follows

A page is a rendering of its class's generated API plus its schema. The anatomy below is derivable from the shape: identity fields come from required properties, folds from edges and folds in the API, links from object-property ranges. One anatomy, N classes — the presentation-layer twin of "one generator, zero forked code paths."
1 · Breadcrumb — the walk back up

Always renders the containment path to the value stream: value stream → step → product → domain → service. Each crumb is a real model edge (atStep, hasDomain, hosts). Navigation starts from the value stream (Jeff's CMDB ruling) — so every page can walk home.

2 · Identity header — name + kind

label as the title, the class as the kind suffix ("— Domain"). Display labels may differ from IRI-derived names (lowercase keys); pages show labels, links carry names.

3 · Promise block — the one-paragraph answer

comment (Domain), vision (Product), overview (Service). The first thing a human reads; the field every shape requires (minCount 1).

4 · Stats bar — owner, step, status

ownedBy (role dot), atStep (step badge, colored by the five-step palette), status (the exploring|building|operating|retiring enum). Same three on every class — they're the properties every shape carries.

5 · Folds — one per edge or fold route, rendering ALWAYS

Each object property or generated fold route becomes a fold: hasDomain, contains, dependsOn, hasValueStream, hasSurface, hasDesignDoc. Honest-folds rule: a fold renders even when empty — "0 children — edges not yet minted" is information; a hidden fold is a lie of omission. Every fold carries a source tag naming its API route, so live vs authored vs hydrated is always visible.

6 · Layer labels — authored above, hydrated below

Where a class carries both provenance layers (Service: design + operational), the page draws the line visibly. Authored facts (asIs, toBe, vision) never mix with hydrated facts (runtimeForm, healthAt, cdhash) — the same authored-vs-hydrated annotation that drives #3372's authz derivation drives the page's visual grouping. One annotation, two consumers.

7 · Edges are links — no dead text

Any value that is a model entity renders as a link to that entity's page. If it can't be a link, either the edge isn't real or the page is missing — both are model findings, not styling choices. This is what makes the flow walkable and the model self-evident.

Conformance — how this gets teeth

Because pages are generated from the same shapes as the APIs, anatomy conformance is structural: the generator emits the anatomy; a hand-built page that drifts is visible against the schema. Until pages are generator-emitted, this document is the review standard for every fan-out leg's page (#3351 Product/Service, Silas's three, Kade's four).
