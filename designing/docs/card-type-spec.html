Card Type Specification

The behavioral trigger for gates, signals, and flow metrics • #1909 • Wren • 2026-03-31

I create a card and the first thing I choose is what kind of work it is. That choice drives everything: which gates fire, what level of research is required, how urgently the signal propagates, and how the work shows up in flow metrics.

Five Types

TypeWhen to useExample
newBuilding something that doesn't exist. New page, new endpoint, new service, new artifact.#1877 Doc-catalog live service
enhanceImproving something that works. Better UX, more data, additional filters, performance.#1882 Doc-catalog filters
fixSomething is broken. Bug, regression, data loss, incorrect behavior.#1868 Disk check false 96%
choreMaintenance. Cleanup, migration, dedup, documentation, tech debt.#1866 Remove stale Docker references
swatCrisis. Something is down, data is being lost, Jeff can't work.#1869 Seeds invisible to roles

Key insight: Fix cards get the most gates — git log, Loki logs, synthesis, TDD, demo. This is because fixes are where the team keeps repeating mistakes (seed pipeline: 18 commits, still broken). New features get the least friction — trust the builder in their own domain.

Flow Metrics

Card type enables tracking:

  • Fix:Feature ratio — high fix ratio = quality problem
  • Chore load — too many chores = tech debt accumulating
  • SWAT frequency — frequent crises = systemic instability
  • Enhancement velocity — how fast we improve what exists
  • New feature rate — how fast we expand capability

CLI Usage

cards add "title" --type fix --owner silas --priority P1 --domain chorus

cards add "title" --type new --owner kade --priority P2 --domain photos

Type is required on creation. No default — forces the choice.
