What validation corrected in pass-2 (these are the real findings)

  1. acp does NOT compose the atomics — it reinvents them. chorus_acp is inline TypeScript in server.ts (executeAcp(), ~9 steps) using git-queue.sh + gh + cards directly. The rust werk-commit/werk-push/werk-accept binaries exist but acp bypasses them. Two divergent implementations of the same operations, in two languages.
  2. demo and acp have different architectures. demo (rust) subprocesses the rust atomics; acp (TS) reimplements inline. Same concept, opposite codepaths.
  3. Logging is fragmented across 3 sinks, not one spine. (a) per-verb ops/logs/werk-*.jsonl witness files — local, NOT promtail-tailed; (b) the real spine ~/.chorus/chorus.log via chorus-log — only demo + accept write it; the atomics don't; (c) MCP-server events → stderr → /tmp/chorus-api.log, separate again. The atomic verbs' events never reach the spine.
  4. Confirmed: chorus_build/deploy/env_up are the only tools that shell the rust binaries. chorus_commit/pull_card/acp are all inline TS. No act anywhere.

5 · Target to-be

The design we agreed — now with a bigger gap than pass-2 implied: it requires unifying two implementations, not just rewiring one.
  • One implementation of each atomic — the rust core. acp and demo both call those cores; delete the inline-TS commit/push/accept in server.ts.
  • Orchestrations compose cores in-process (library), MCP skin only at the boundary. No subprocess/MCP in the hallway, no TS reimplementation.
  • One spine. Every verb — atomic and orchestration — writes ~/.chorus/chorus.log via chorus-log. Retire the per-verb witness files (or tail them into Loki). Stop routing MCP-server events to a separate stderr log.
  • gh once at the edge (the one failure point we don't own); base-currency step 0 on acp.
  • act is the missing piece that makes "compose cores" real — built layer vs. pattern is still the open fork.
