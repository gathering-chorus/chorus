Everything below in GENERATED green is the proposed derivation — a paper proof of what the generator would emit, not a running slice. The honest line: this is a derivation worked by hand to prove the shape; building the generator is the next card.

The domain OWL isn't a section in the spec. It is the spec — the one authoritative, machine-readable definition of what a domain is. The page, the API, the persistence schema, the crawler's tags, the MCP tools would all derive from it. Author the OWL once; everything else is generated from it — so it cannot drift. That is the bet this slice tests.

Not a return to RDF-as-runtime. Athena v2 deliberately left v1's Fuseki/RDF store for a JSON tree + SQLite — and v2 is not TTL (chorus-core.ttl itself says it is "not hand-authored… regenerated from v2"). OWL here is an authoring/spec format that compiles to that JSON/SQLite runtime — the model is written once in OWL, then compiled down; the system does not run on a triplestore. SQLite now; Fuseki/RDF stays the deferred, additive store, never the source.

1 · The inversion

Today an Athena domain page is the spec — it carries actor diagrams (who interacts) and BDD (how it behaves). Both describe the domain; neither models it. The two structural sections are conspicuously absent: the data model (what it is) and persistence (where its data lives). This slice flips the relationship.

Today — page is the spec

Actors + BDD authored by hand. OWL absent, persistence absent. Each surface (page, API, store, crawler) authored separately → they drift. Behavior without structure.

This slice — OWL is the spec

One OWL source. Page, API, persistence, crawler tags, MCP all render/generate from it. Actors + BDD attach as behavior on top of structure. Drift becomes structurally impossible — once the generator exists and every surface derives from the one source (the target state, not today).

"Ontology is architecture," literally: the OWL is the contract the surfaces are generated from (chorus-core.ttl: "The OWL is the contract the API is generated from.").

2 · The one source — the alert spec SPEC

The borg module imports the shared spine (chorus-core) and extends it — it never redefines the spine classes or the four axes. The whole spec for the facet is this:

# borg module — imports the spine, extends it (never redefines)
<https://jeffbridwell.com/chorus/borg> a owl:Ontology ;
    owl:imports <https://jeffbridwell.com/chorus/core> .   # the 4 axes + spine classes

chorus:Facet     a owl:Class ;
    rdfs:comment "A harvested aspect of the system. The crawler tags artifacts into Facet subclasses." .
chorus:AlertRule a owl:Class ; rdfs:subClassOf chorus:Facet ;
    rdfs:label "Alert Rule" ;
    rdfs:comment "A declarative alert: a scheduled check + an action when it fails." .

# ── datatype properties (the fields — lifted from the real artifact) ──
chorus:alertName    a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .
chorus:description  a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .
chorus:severity     a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .  # critical|warning|info
chorus:schedule     a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .  # cron
chorus:checkScript  a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .
chorus:actionScript a owl:DatatypeProperty ; rdfs:domain chorus:AlertRule ; rdfs:range xsd:string .

# ── three of the four chorus-core axes place this facet (shape is the 4th, below) ──
chorus:inDomain a owl:ObjectProperty .   # → chorus:alerts-monitors
chorus:ownedBy  a owl:ObjectProperty ; rdfs:range chorus:Role .  # → chorus:silas (an AgentRole ⊂ Role)
chorus:atStep   a owl:ObjectProperty .   # → chorus:Proving (Phase)
# shape = facet (the 4th axis) — carried by AlertRule ⊂ Facet, not re-declared here

# ── a domain PROPERTY (not an axis) — the action's notify target ──
chorus:notifies a owl:ObjectProperty ; rdfs:range chorus:Role .  # → chorus:silas (AgentRole ⊂ Role)

The four chorus-core axes are inDomain · ownedBy · atStep · shape. Three are declared above; shape is the 4th — it's facet here, carried by AlertRule ⊂ Facet rather than restated. notifies is not an axis — it's a domain property of AlertRule. Both ownedBy and notifies range over chorus:Role, and silas is an AgentRole (a subclass of Role) — consistent, not two vocabularies.

No SHACL, no validation rules — per the model-first call, this is the data model only. Validation is a later, additive layer. "Validated" here means just: it parses, it imports the spine, the axes are present.

The real artifact it models

This is a live file — proving/domains/alerts/chorus-mcp-down.yml — the thing the crawler tags into an AlertRule instance:

name: chorus-mcp-down
description: chorus-api /mcp endpoint not responding to MCP initialize handshake
severity: critical
schedule: "*/5 * * * *"
check:  | curl -s ... POST http://localhost:3341/mcp ... → ok | exit 1
action: | ops-nudge silas "chorus-mcp-down: /mcp not responding ..."
# → becomes one instance, every field tracing to a property above:
chorus:alert-chorus-mcp-down a chorus:AlertRule ;
    chorus:alertName "chorus-mcp-down" ;
    chorus:severity  "critical" ;
    chorus:schedule  "*/5 * * * *" ;
    chorus:inDomain  chorus:alerts-monitors ;   # axis
    chorus:ownedBy   chorus:silas ;             # axis
    chorus:atStep    chorus:Proving ;           # axis
    chorus:notifies  chorus:silas .

3 · Everything derives GENERATED (proposed)

From that one OWL, the generator would emit five surfaces. Each line below is hand-derived to show what it would emit — every line traces back to a declaration above. That's the point of the paper proof: in the generated design nothing would be hand-authored — the surfaces fall out of the spec. (The generator that does this is the unbuilt bet; what follows is the derivation, not generator output.)

① Persistence — SQLite schema

derives from: the datatype properties + the three axes

CREATE TABLE alert_rule (
  id          TEXT PRIMARY KEY,   -- IRI local name
  name        TEXT NOT NULL,      -- chorus:alertName
  description TEXT,               -- chorus:description
  severity    TEXT,               -- chorus:severity
  schedule    TEXT,               -- chorus:schedule
  in_domain   TEXT,               -- chorus:inDomain   (axis, indexed)
  owned_by    TEXT,               -- chorus:ownedBy    (axis, indexed)
  at_step     TEXT                -- chorus:atStep     (axis, indexed)
);  -- SQLite now; Fuseki/RDF is the deferred, additive store

② Crawler self-tag rule — deterministic

derives from: the class + its domain placement (the herald IS the domain — no judgment)

glob  proving/domains/alerts/*.yml   →   chorus:AlertRule
  name      ← yaml.name           severity ← yaml.severity
  schedule  ← yaml.schedule       description ← yaml.description
  inDomain  ← chorus:alerts-monitors   (location decides — deterministic)
  ownedBy   ← chorus:silas              (from the domain's ownedBy)
  atStep    ← chorus:Proving

This is the deterministic half of two-speed tagging: the artifact's location + shape decide the tag, mechanically. (The semantic half — capability domains where the domain isn't obvious — is the separate #3184 judgment, not needed here.)

③ Query API — generated REST

derives from: the class (the resource) + the indexed axes/properties (the filters)

GET /api/chorus/alerts                                  → all AlertRule
GET /api/chorus/alerts?domain=alerts-monitors&severity=critical
GET /api/chorus/alerts?owner=silas
GET /api/chorus/alerts/{id}                             → one AlertRule
# OWL → OpenAPI + SQL-backed CRUD. Filters are exactly the indexed axes.

④ MCP tools — generated thin-skin

derives from: the same class + filters, exposed for agents

chorus_alerts_list(domain?, severity?, owner?)
chorus_alerts_get(id)
# would be auto-generated — removing the hand-written server.ts labor for this domain (once the generator exists)

⑤ Page render — the Athena domain view

derives from: the API (above); actors + BDD attach as behavior

The alerts-monitors domain page queries /api/chorus/alerts?domain=alerts-monitors and renders its rules. The actor (who gets notified — silas via ops-nudge) and the BDD (given the check fails, when the schedule fires, then notify the owner) attach on top of the structural spec — behavior layered over structure, not instead of it.

6 · Why this is the right first slice

  • Deterministic tagging — an alert's domain is decided by its location (proving/domains/alerts/), no semantic judgment. Zero tagging risk.
  • Greenfield + low blast — the crawler half is already being rebuilt; proving the generator here breaks nothing live.
  • My lane, already real — alerts-monitors is owned by Silas, at Proving, status operating; the artifacts and the page already exist. We're modeling something true, not inventing.
  • Exercises every surface — persistence, crawler, API, MCP, page — so a green slice here proves the whole pattern before it scales to the seven products.

The bet in one line (the real-vs-proposed split is up top): write the alert OWL, build the generator, emit its five surfaces, harvest the real artifacts into them, and query the result. If that round-trips, the whole-model thesis holds — and we scale the same move across the seven products.
