Definition of Done

What "done" means on this team • Wren • 2026-03-30 • Born from 108 decisions and a lot of hard lessons

Done means Jeff can use it, not that the builder finished working on it. The card is done when the experience described in the Experience section is real — not when the code compiles, the tests pass, or the PR merges.

Why This Exists

We've shipped cards that weren't done. Tests passed but the pipeline was broken (#1770 — three bugs in one day). Code was deployed but the endpoint wasn't in the API table (#1869 — seeds invisible to roles). Features worked locally but Jeff couldn't find them (#1867 — 63 HTML artifacts with no catalog). Cards marked Done with 15 of 18 AC items complete.

"Done" is not a feeling. It's a checklist. If any item below is unchecked, the card is not done.

The Checklist

Before Building (WIP Entry)

  • Experience section exists — 2-5 sentences in Jeff's voice. What he sees, feels, gets. Hook-enforced on WIP entry.
  • AC derives from experience — every AC item traceable to the experience statement.
  • Design artifact linked — if a service design, sequence diagram, or assessment exists for this domain, it's referenced in the card.
  • Blast radius documented — files and domains this card touches. Hook-enforced.

During Building

  • Tests written first — TDD (DEC-1674). Tests describe Jeff's experience, not implementation. Red before green. Hook blocks Done without test runs.
  • All AC items complete — not 15 of 18. All of them. If scope grew, split the card. No partial acceptance.
  • No new lint warnings — ceiling is a ratchet (DEC-085). Only moves down.
  • Design artifact updated — if the implementation diverged from the design, update the design. Don't leave a stale blueprint.

Before Demo (Proving Gate — DEC-048)

  • Deployed — code is running, not just committed. app-state.sh deploy for TS changes.
  • Smoke check passes — smoke-check.sh --all exits zero. Hook-enforced.
  • Happy path verified — the builder has manually confirmed the feature works as described in the Experience section.
  • Visual validation — for front-end cards, /lc screenshot before sign-off. Tests passing ≠ renders correctly.

Demo

  • Demo to Jeff — no self-service Done on code cards. Jeff sees every demo. Hook-enforced.
  • Show, don't tell — open the page, hit the endpoint, run the command. Jeff sees the thing working, not a summary of it.
  • Demo brief generated — auto-generated by /demo skill. Proves demo happened.

After Demo (Acceptance)

  • Jeff or Wren accepts — /acp. The builder cannot accept their own work.
  • Spine event emitted — card.accepted logged.
  • State files updated — projects.md, backlog.md if affected.
  • Committed and pushed — via git-queue.sh.

The Final Test

  • The experience is real — re-read the Experience section. Is that what Jeff now sees? If not, the card is not done.

What Done Is Not

"Tests pass." Seed pipeline has 106 tests. Photo delivery is broken. Tests passing is necessary but not sufficient. The system must work end-to-end as Jeff experiences it.
"Code is committed." Committed ≠ deployed. Deployed ≠ working. Working ≠ findable. #1869: seeds endpoint worked but wasn't in the API table — roles couldn't see it.
"15 of 18 AC items complete." No partial acceptance. All AC or split the card. Every time we accept at 85%, the remaining 15% becomes invisible tech debt.
"I told Jeff about it." Telling is not showing. A demo is Jeff seeing the thing work, not hearing about it. /demo → show → pause → Jeff reacts.
"The endpoint returns 200." 200 OK with empty data is not done. The data Jeff expects must be there. The seed endpoint returned 200 while photos were silently dropped.

Incidents That Shaped This Definition

#1770 (2026-03-28): Seed pipeline "fixed" three times in one day. Each fix passed unit tests but failed live. Bug 1: URL regex. Bug 2: hashtag stripping. Bug 3: photo content detection. Bug 4: Cloudflare tunnel stale. → Led to: "unit tests necessary but not sufficient" and end-to-end verification requirement.
#1869 (2026-03-30): Seeds endpoint built on Chorus API but not added to endpoint table. Roles couldn't check seeds from terminal. Jeff discovered photos weren't arriving — the team couldn't even see the problem. → Led to: "observability is part of done" — if the team can't verify it, it's not done.
#1867 (2026-03-30): 63 HTML artifacts across 5 directories. Doc-catalog only read from one. Jeff asked for diagrams all morning and couldn't find them. → Led to: "findable is part of done" — if Jeff can't get to it, it doesn't exist.
#1836 (2026-03-29–30): Gemba scripts built and working but card still in WIP. PM didn't know it was done. → Led to: "if the card isn't moved, it isn't done" — status must reflect reality.
The Lobotomy (2026-03-26): Grep hook broke all 3 roles for 2 days. Nobody reported it. → Led to: circuit breaker (#1858) and "the system must detect failures before Jeff does."
Seed trust deficit (recurring): Jeff sends something, it doesn't arrive, he discovers it, he tells us, we card it. Every time. → Led to: "Jeff is not the test suite" and synthetic monitoring requirement.
