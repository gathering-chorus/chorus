What it is

Chorus is the coordination layer for one human (Jeff) and three AI roles (Wren, Silas, Kade) building two products simultaneously — Gathering (personal infrastructure) and Chorus itself (the coordination protocol). Everything is text, everything is versioned, everything is searchable, everything emits structured events.

The protocol IS the product. Coordination artifacts (decisions, briefs, kanban state, spine events, gate verdicts) are first-class versioned outputs, not overhead. Built on the canonical-model lineage from Jeff's Staples ESB work (US Patent 9,552,400 B2): RDF/OWL workflow ontologies + SPARQL-driven gate evaluation.

The engineering workflow: five MCPs

Engineering happens through five typed MCP transactions, in sequence:

chorus_pull_card → chorus_demo → chorus_acp → chorus_build → chorus_deploy

Each is an atomic transaction with discrete step events emitted to the spine, typed refusal taxonomy, and skill-side invocation that delegates entirely to the MCP. Slash-skills (/pull, /demo, /acp, /build, /deploy) collapse to one MCP call each — no model-walked steps, no markdown drift, no agent-curiosity skipping.

Status as of 2026-05-10:

  • chorus_pull_card, chorus_acp — shipped; /pull and /acp skills trimmed to MCP-call-only (#2751, #2865)
  • chorus_demo — designed, transitional substrate shipped (#2864 — show-gate.sh + accept_gate spine read), full MCP next
  • chorus_build, chorus_deploy — designed, not shipped. See designing/docs/build-and-deploy-service-design.html.

The MCP definitions live in platform/api/src/mcp/server.ts. The skill bodies live in ~/.claude/skills/<name>/SKILL.md (also in chorus/skills/<name>/).

Each MCP emits <name>.<step>.{started,completed,refused} spine events tagged with card_id (#2838) and trace_id (#2839, #2845). Cross-stream stitching via the trace_id; card-scoped queries via the card_id.

Each role:

  • Reads its session-context envelope from chorus-api on session start
  • Emits work to spine; reads peers' work via spine queries

The human (Jeff) is one navigator with three drivers, each with domain expertise. Roles exchange work directly through MCP tools and structured nudges — Jeff is not the relay.

Substrate primitives

Kanban board (cards CLI)

platform/scripts/cards is the CLI; backed by Vikunja (port 3456). Status: Next | Later | WIP | Done | Blocked | Harvesting | SWAT | Won't Do. Single board, role + domain + chunk + sequence labels. WIP limit 3 per role. Cards have AC (acceptance criteria) and an Experience section; both required to enter WIP.

cards add "title" --owner <role> --priority P1 --domain chorus --type new --origin reflective --desc "..."
cards view <id>
cards move <id> WIP
cards comment <id> "..."
cards demo <id>     # logs demo started
cards done <id>     # builder cannot self-accept code cards (DEC-048)

All board mutations go through cards. Direct Vikunja API calls bypass the typed taxonomy and are blocked by hooks.

Spine (Loki)

Operational telemetry flows through chorus-log to Loki under structured labels. Convention: domain.noun.verb_past (e.g. card.demo.started, chorus_acp.commit.completed). Every event carries role, card_id (when card-scoped per #2838), trace_id (when transaction-bound per #2839).

Schema: designing/schemas/spine-events.json.

Read via the typed log MCPs (#2840):

  • chorus_logs_for_card({ card_id }) — every event card-bound to N
  • chorus_logs_for_trace({ trace_id }) — full causal chain of one transaction
  • chorus_logs_recent_errors({ time_window }) — last hour of errors by role
  • chorus_logs_query({ query }) — raw LogQL for the cases the above don't fit

Briefs

Roles exchange work through markdown briefs in the recipient's briefs/ directory. Format: question/request, context, constraints, response needed. No meetings by default. The /clearing surface (Bridge) is the synchronous channel when async briefs are too slow.

Decisions + ADRs

roles/wren/decisions.md is the numbered decision log with rationale. ~110+ DECs as of this writing. architect/adr/ holds Architecture Decision Records (ADR-NNN). Both are searchable via /chorus search.

Fitness functions

Read-only consumer-shaped scripts that ask "is X fit?" and exit non-zero on flags. Same shape across the family:

Script Asks
commit-health Are commits landing cleanly on main?
spine-health Is the spine coherent (declared events firing, no orphans, weather:state-change ratio)?
quality-health Has quality discipline drifted (gate-skip rate, untraced --no-verify, stale skips, runtime outliers)?
building-pipeline-health Is the post-acp release chain firing as designed (#2870)?
graph-health Is the canonical-model graph well-formed (Product → SubProduct → Domain layout, OWL/SHACL state)?

Each runs on demand, no args, exits 1 if any concern. None emit spine events themselves — they read what's already there.

Gates + hooks

Five named gates (gate-product, gate-code, gate-quality, gate-arch, gate-ops) carry quality discipline at card-level. Pre-2026-05 these were comment-driven (a peer role posts gate:<name>-pass on the card). Post-#2864, gate verdicts move toward spine events read at acceptance time, with comments retained as transitional fallback.

Structural enforcement runs in chorus-hooks (Rust) at platform/services/chorus-hooks/src/:

  • PreToolUse hooks block: accept_gate (no self-accept on code cards, demo evidence required), tdd_gate (tests-before-code), canonical_write_guard (canonical is read-only during sessions), memory_gate (synthesis required before edit), icd_pre_read (read ICD before harvester writes)
  • PostToolUse hooks observe + emit: demo_provenance (writes brief, emits card.demo.started), interaction_pattern (detects pattern shifts), observer (digest tool calls)

Hooks deploy to ~/.chorus/bin/ via signed build (build-signed.sh); cdhash binding is stable across rebuilds (#2734) so macOS TCC AppleEvents grants survive.

Worktree model (chorus-werk)

Canonical /chorus/ sits on main and is read-only during sessions. Each role works in /chorus-werk/<role>/, a per-role git worktree branched from main with its own HEAD. Edits during a session land in the role's werk; canonical refreshes only at lock-guarded sync moments.

Substrate scripts:

  • chorus-werk init|repoint|remove|status|pull|close <role> [card-id] — werk lifecycle
  • chorus-werk-sync — repair primitive (manual recovery for canonical drift)
  • git-queue.sh commit|push — atomic add+commit / verified push under lock; auto-resolves known auto-generated file conflicts

The split keeps role sessions isolated from each other and from canonical's mutable git state. See designing/docs/version-control-service-design.html.

CLAUDE.md generator

Each role's CLAUDE.md is generated from ~110 shared fragments by claudemd-gen.sh. Edit the fragments, regenerate — never edit CLAUDE.md directly. Fragments live at designing/claudemd/shared/ and designing/claudemd/roles/<role>/. The protocol version is single-sourced from designing/claudemd/PROTOCOL_VERSION (currently v1.15).

Search / context

SQLite FTS5 index at ~/.chorus/index.db over Slack history, Claude session transcripts, briefs, decisions, ADRs, activity logs, card comments. ~57K+ messages. Searchable via /chorus search <term> skill or chorus-api's /api/chorus/search endpoint. Search hierarchy (DEC-074): chorus first → codebase graph → filesystem.

Pulse

Live team-state worker (platform/pulse/) emits a snapshot to /tmp/pulse-latest.json periodically. Includes role andon state (building / blocked / waiting / observing / idle), top-of-board cards, alerts firing, andon-shift events. Bridge UI subscribes; session-start envelopes embed pulse state.

Spine event taxonomy

Events declared in designing/schemas/spine-events.json. Names follow <domain>.<noun>.<verb_past>:

Family Examples
Card lifecycle card.pulled, card.demo.started, card.accepted, card.unpulled, card.stale.detected
MCP transactions chorus_pull_card.{validate,werk-preflight,cards-move,...}.{started,completed,refused}
Demo flow demo.preflight.{started,passed,failed}, demo.show.completed, demo.bypass.detected
Gates gate.product.passed, gate.code.passed, ... (transitioning to chorus_demo.gates.* via #2864)
Build/deploy build.completed, build.failed, deploy.completed, deploy.failed, binary.deployed
Observability library.health.*, canonical.sync.*, alert.fired, nudge.{requested,emitted,delivered}
Sessions session.context.built, session.boot.envelope.delivered, interaction.pattern.detected

Event volume can be heavy — spine-health classifies events as state-change vs weather (heartbeats, probes) and flags when weather dominates the top-N.

Three quality layers (ADR-026)

1. Pre-commit hooks      — "will this commit obviously break something?"   (local, fast, skippable via --no-verify)
2. Role gates            — "is this card team-acceptable?"                  (card-level done; recorded on card / spine)
3. CI                    — "does main build cleanly from scratch?"         (branch-protected; --no-verify is overridden)

Each owns a different question with a different threat model. Lock files (package-lock.json, Cargo.lock) are committed; CI uses npm ci against the locks.

Working with the code

Read first

Before changing anything, read the design that names it:

  • designing/docs/<service>-service-design.html — the canonical design for each surface
  • designing/decisions/dec-NNN.md — the rationale-rich decision logs
  • designing/adr/adr-NNN.md — architecture decisions
  • roles/<role>/CLAUDE.md — the role's operating rules (generated; edit fragments under designing/claudemd/)

Run

Many services are LaunchAgents that auto-start. Manual operation:

  • app-state.sh status — show all services
  • app-state.sh deploy <crate> — rebuild + restart (TypeScript)
  • build-signed.sh chorus-hooks — rebuild Rust hooks with signed identity (cdhash stable for TCC)
  • chorus-werk init <role> — create per-role worktree (one-time per role)

Test

  • npm test in platform/api/, directing/clearing/, directing/products/cards/ — TypeScript unit + integration
  • cargo test in platform/services/chorus-hooks/ — Rust hooks
  • bash platform/tests/<test-name>.test.sh — bash-level invariant tests
  • bash proving/scripts/nightly-suites.sh — full nightly run; emits canonical === Results: N passed, M failed ===

Debug

  • chorus_logs_recent_errors({ time_window: '1h' }) — last hour of error-level events
  • chorus_logs_for_trace({ trace_id }) — full causal chain of one transaction
  • Loki UI at localhost:3100 (Grafana)
  • Pulse: cat /tmp/pulse-latest.json — current team state snapshot

Patent lineage

Chorus implements concepts from US Patent 9,552,400 B2 (Bridwell): RDF/OWL workflow ontologies with SPARQL-driven gate evaluation. The patent describes typed artifacts flowing through quality gates with machine-readable state transitions. Chorus is the working implementation, with humans + AI agents as the workflow actors and Loki/Vikunja/Fuseki as the substrates.

The 2011 Staples ESB work — domain/integration architecture using the same canonical-model patterns — is the direct ancestor.
