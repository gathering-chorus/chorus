Tests — Service Design

Kade · 2026-06-17 · #3473 · DRAFT for Jeff review before further build. Companion to quality-service-design.html, werk-review-service-design.html, properties-domain-design.html.

One-line frame: Tests is a domain; werk is the subproduct that implements it. This design is how werk implements the tests domain — one declared, governed model of tests that the gate, werk-test, the pyramid UI, and agents (via generated MCP) all read as a single source of truth. The model is config-as-data: the test classification axes are declared properties, governed in the OWL, projected into the API + MCP by owl-api.

1. Why this exists — the sprawl

Tests is a base element: everything in the system depends on it, directly or indirectly. It is one of the few true hubs — high out-degree (a test :covers any of 30+ domains) and high in-degree (Proving, Borg, the gates, demo, ship-readiness all rest on "is it tested?"). Incoherence in a hub propagates everywhere. Yet today the hub has no service design and is governed by accident:

  • Two classification engines that disagree. The live /borg/quality pyramid + every per-domain test count read a filesystem scanner (quality-summary.ts) that infers a test's layer from its filename (regex). Separately, the graph carries a declared classification (discover-tests.ts + #3442's testType property). #2098 retired the graph as the UI source — so the live surface runs on filename inference, which we proved drifts (the *-unit.test.ts files that actually touch real fs and are integration).
  • Two pyramids — a stale hand-authored quality-service.html with hardcoded counts, and the live scanner-fed /borg/quality.
  • Triplicated gate definitions — gate-registry.md, design-gate-definitions.html, and five gate-*/SKILL.md files.
  • Real-but-undeclared taxonomy — the layer assignment is regex/filename inference in code; test-categorization.html is a human convention the code does not actually read.

The fix is not a 16th surface. It is to give the hub one governed model and make every surface a projection of it.

2. The model — a Test and its three axes

Core entity: chorus:Test, file-grained, making the test artifact first-class. Today's confusion comes from fusing two axes into one (*.integration.test.ts means both "integration layer" AND "needs a live stack"). The model separates three orthogonal axes, each a declared, governed property:

1. type / layer — what kind. Vocab: unit · integration · api · ui · perf · security · e2e · bdd. The pyramid axis. (Declared today by #3442's testType.)
2. hermeticity — where it can run: hermetic (no live deps) vs needs-stack (live Fuseki/app/db). werk-test gates on this: hermetic blocks always; needs-stack blocks only when the env is up. Independent of type — a security test may be either.
3. value — is it worth keeping (#3324 taxonomy): valid · asserts-retired-behavior · passes-by-definition · integration-not-gated.

Plus identity/relations and the dynamic side:

:Test
  :testFile     "platform/api/tests/server-unit.test.ts"
  :framework    jest | cargo | bats | cucumber
  :hasProperty  [ :propertyKey "testType"     ; :propertyValue "integration" ]
  :hasProperty  [ :propertyKey "hermeticity"  ; :propertyValue "needs-stack" ]
  :hasProperty  [ :propertyKey "testValue"    ; :propertyValue "valid" ]
  :covers       <SubDomain>          # what it tests — ties Test into the domain graph

:TestRun
  :runOf  <Test> ; :result pass|fail|skip ; :round/:patch_id ; :at … ; :context werk|nightly|pre-commit
# knownFail = a Test with :knownFail [ card ; reason ; ttl ] ; flake = a Test whose TestRuns disagree

3. The approach — declared, governed PROPERTIES (config-as-data)

The three axes are not bespoke fields — they are properties (per properties-domain-design.html, #3433/#3435), which is what makes the whole thing generate correctly. The critical discipline (Jeff): the governance must live in the OWL, because owl-api generates the API and MCP from the model. Constraints-in-OWL → the generated interface enforces them by construction; constraints-in-code → a dumb CRUD surface that re-creates the scatter.

So each axis is a registered PropertyKey in the properties OWL, carrying its own SHACL:

:testType a chorus:PropertyKey ;
    sh:in ( "unit" "integration" "api" "ui" "perf" "security" "e2e" "bdd" ) ;
    chorus:propertyValueType "string" ;
    sh:targetClass chorus:Test ;
    chorus:owner chorus:tests-domain .

Read vs write (corrected per Wren, 2026-06-17): owl-api is read-only — it generates the read API + MCP from the model. Writes go through the DAL (chorus-model), the single write authority, and that is where SHACL is enforced. So "the generated interface enforces it" precisely means: the DAL rejects writes that violate the shape. What the DAL enforces today is mixed:

SHACLEnforced by the DAL today?Bearing on tests
sh:in (enum)YES — out-of-enum write rejected (shape-violation: not in sh:in)The testType vocab migration works now.
sh:minCount (required)YES — fail-closed (completeness floor, #3468)Required axes can be mandated.
sh:datatype / valueTypeNO — every value written as an untyped string; no type checkGap. Bites thresholds-as-int, toggles-as-bool.
sh:class on edges (target type)PARTIAL — target must exist; target's type not checkedGap on the hub edge: :covers could point at any node, not just a SubDomain.

Consequences for the design:

  • Migrate #3442's in-code VALID_TYPES into the OWL as :testType sh:in (…) — the DAL then enforces the vocab; the gate's hardcoded enum is deleted. Precision: only the enum moves. The gate's contradiction check ("declared unit but content signals integration") is behavior, not a constraint — it stays in the gate. Clean split: vocab→OWL, drift-detection→gate.
  • Conformance tests prove the model's own enforcement (the elegant dogfood — Wren): the tests domain's first tests assert its SHACL is enforced. sh:in + sh:minCount tests are GREEN (real enforcement); sh:datatype + edge-target-type tests are RED — and that RED is good: it documents the gap and drives building the enforcement.
  • The properties browser UI (#3472) reads the same declared set via the generated read MCP — and doubles as the taxonomy health surface (orphan keys, drift, expiring toggles).

This is not a config sweep. Properties usage is curated — only config that varies, is deploy-painful to change, and is operationally valuable — and demand-driven. The test axes qualify (they're the hub's classification); the static long tail stays in code.

Peer consumer (Jeff + Wren): bootstrap the top-level CHORUS environment (paths, ports, the :33xx registry, bin locations) from properties — the env is another base-element config-as-data consumer. Discipline (the bootstrap trap): this must be generative, not runtime — properties are the source, but the env is emitted at build/setup time, because the properties store (owl-api/Fuseki) itself needs the env (ports/paths) to run. Runtime-resolving the env from properties is a circular dependency (can't read properties to find the store you need the env to reach). A minimal static seed reaches the store; everything else generates from properties at setup. The factory-independence test applies: turn chorus off — does the environment still bootstrap?

4. Convergence — one source of truth

The declared classification becomes the single source; the scanner's filename inference is retired (reverses #2098 — a numbered decision, not a quiet refactor). The pyramid UI, per-domain counts, the gates, and werk-test all read the declared model. The scanner was the old implementation that bypassed the domain; the new path is werk implementing against the domain.

5. Cross-domain dependencies — the integration crux

Because tests is a hub, its edges into the rest of the ontology are the load-bearing part, and the place coherence is won or lost at 30+ domains.

EdgeDirectionDiscipline
:Test :covers :SubDomaintests → every domainThe defining edge. Must reference the canonical SubDomain vocabulary (real IRIs, populated #3437 containment), so coverage scales 2n, not n². Coverage / blind-spots / fitness all derive from it.
blast-radius Test → code → SubDomaintests → code domainHow werk-test selects "the right tests for this change." Must be build/author-time, not a runtime coupling (factory-independence: a built product runs with chorus off).
testType :hasProperty (Property model)tests → properties domainReuses #3433. Tests depends on the properties model.
tests :implementedBy werktests → werkImplementation dep — werk-test/gates are werk reading the domain.
TestRun → Borg / gates / demoeverything → testsThe in-degree: Proving, the gates, the demo prover, ship-readiness all consume test state. This is why the hub must be accurate.

Coherence rule: tests covers domains; domains do not own tests (one direction, avoids bidirectional drift). One bad property name or dangling :covers on this hub is wrong in every generated surface — which is why standards (Silas) matter more here than for any leaf domain.

6. werk-test — the verb that reads the domain

werk-test is not greenfield: ci-pipeline-service-design.html already names it as "flip the advisory test step to a gate." This design specifies it as the gate that reads the tests domain, staged:

  1. Declared test-type gate — DONE (#3442, today): every changed test declares its type; pre-commit blocks undeclared/contradicting.
  2. Trustworthy nightly — the nightly stops crying false reds (env-break vs regression distinguishable).
  3. Flake quarantine (#2530) — flakes can't poison a blocking gate.
  4. Blast-radius selection — run the right tests for the card, via :covers + the code→domain edge.
  5. Flip to blocking — and the enforcement state is itself a property: :gate-test :hasProperty [ :propertyKey "enforcement" ; :propertyValue "advisory" ], flipped via a model write (lifecycle-bound toggle) when 2–4 are met. No code change to promote the gate.

7. Projections — every surface is a view of the model

  • Pyramid (/borg/quality) = Tests grouped by type, per repo.
  • Coverage = Tests per :covers SubDomain.
  • Fitness / blind-spots = SubDomains missing a risk-type (security/perf) — #3442's surface, made loud.
  • Properties browser (#3472) = the governed property set, browsable + sortable.
  • werk-test = the gate, reading the same model.
