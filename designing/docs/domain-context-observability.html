Purpose

Observability is the system's ability to expose its internal state through external signals. It spans two repos and two mechanisms:

  • shared-observability/ — canonical deployment source. Prometheus scrape targets, Grafana dashboards, Loki log aggregation, Alertmanager routing. Docker Compose stack, 7 services.
  • chorus/alerting/ — 7 shell-based alert rules running as cron jobs. Different mechanism, complementary coverage. These check application-level health (app down, tunnel down, hook server dead, seed write failures).

Together they answer: Is the system healthy? If not, what's broken and who should fix it?

Forcing Functions

Deep health (5min cron): 13 checks fire every 5 minutes. Subprocess liveness, log freshness, HTTP endpoints, session index timestamps, standards surface staleness. Failures nudge Silas with diagnostic context.
Alert runner (60s cron): Shell-based rules check application health continuously. Auto-heal where possible (tunnel restart, stale SSH kill). Fire to Bridge so Jeff sees it without checking.
Prometheus (15s scrape): Metrics collection is continuous and automatic. Alert rules evaluate every cycle. No opt-in required.
Ops awareness hook: Fires on every PostToolUse (Bash/Read). If the API is down or alerts fired, the active role sees it immediately. 60-second cooldown prevents spam.
