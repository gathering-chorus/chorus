This plan sequences the migration of three bare skills (/pull, /demo, /acp) into eight act-dispatched jobs (/werk/pull, /werk/build, /werk/deploy, /werk/demo, /werk/commit, /werk/push, /werk/approve, /werk/acp-orchestrator) without breaking the team's ability to ship cards mid-cutover. Closes the "can't try before you buy" gap (Jeff sees the feature running in a preview before /acp), the "receipts don't reconcile" gap (card_id propagates across the launchd boundary), and the "approve is buried in a transaction" gap (Jeff's approval becomes a first-class named state, not four different invocation paths).

The /acp decomposition — why it's the right call

Today /acp is a single atomic MCP call (chorus_acp) that does: commit → push → PR open → PR merge → cards-done → werk-close → release-trigger. The atomicity was earned through #2750 and #2782 and is real — partial-success states would re-introduce the failure modes those cards closed.

But the atomicity being defended is atomicity-as-single-MCP-call, which is substrate convenience, not customer value. Atomicity of the mechanical sequence is preserved at the orchestrator level either way — chorus_acp keeps its rollback semantics; the orchestrator just calls smaller MCPs in sequence with the same rollback contract.

What the decomposition buys:

  • /werk/approve becomes a first-class state. Today "Jeff approves" happens four different ways: typing "approve" in chat (UserPromptSubmit hook responds); clicking accept in some UI surface; typing cards add ... --owner X with DEPLOY_ROLE=jeff; or saying /acp. Making approve a named primitive disambiguates all of them and creates one approval-token surface that /werk/acp's orchestrator checks before running mechanical steps.
  • /werk/commit and /werk/push can be invoked alone. Useful when substrate is degraded — this morning's messaging-API outage + Vikunja-auth failure + orphan-branches accumulation chain would have benefited from "commit my work without trying to do the whole arc."
  • Step-level fitness function becomes authorable. Today chorus_acp emits one outer-bracket event arc; if commit succeeds but push fails, the spine tells you, but the role's experience is "the whole thing failed at step N." Decomposed steps give per-step started/completed pairs the chorus-health canary can walk for parity.
  • Maps to how production CI/CD actually works. PR review → approve → merge. Approve is a state, not a transaction step.

Cost honestly named: three new skills (/werk/commit, /werk/push, /werk/approve) plus two new MCPs (chorus_push, chorus_approve — chorus_commit already exists). /werk/acp becomes an orchestrator that calls them in sequence with rollback semantics on failure. More surface than the wrapper-pattern, but each surface is small and the mental model is cleaner.

Cutover sequence — eight phases

Phase 0 — Prerequisites (Silas + Kade, parallel)

Cards: P1, P2, P3 (Silas — build-and-deploy); P4, P5, P6, P7, P8 (Kade — werk scaffolding).

Fitness function: launchctl kickstart with CHORUS_CARD_ID env passes the var; act jobs emit events with that card_id. chorus:pipelines renders eight Service instances. chorus_push MCP refuses with typed errors on push-fail; chorus_approve MCP refuses with auth-fail when DEPLOY_ROLE != jeff.

Rollback / off-switch: additive only.

Phase 1 — /werk/build (Kade)

Card: C1. Add build job to werk-pipeline.yml; create /werk/build skill; act job invokes chorus-build <crate> against two-slot model.

Cutover semantics: additive. No replacement of any existing skill.

Fitness function: 10 consecutive /werk/build invocations across 3 roles produce 10 werk.build.completed events all carrying card_id.

Rollback: remove the build job + skill. Off-switch: roles fall back to manual cargo build.

Phase 2 — /werk/deploy (Kade)

Card: C2. Adds deploy job + skill; probe-fail refusal if surface not observable.

Cutover semantics: additive.

Fitness function: 10 consecutive /werk/build + /werk/deploy pairs result in 10 running preview instances responding ok on health probe; probe-fail correctly triggers on a synthetic broken build.

Rollback: remove job + skill. Off-switch: documented fallback to manual app-state.sh deploy.

Phase 3 — /werk/commit (Kade)

Card: C3. Adds commit job that wraps chorus_commit MCP; emits werk.commit.* event pairs.

Cutover semantics: additive. Roles can keep calling chorus_commit MCP directly during cutover; /werk/commit is the new event-emitting wrapper.

Fitness function: for every chorus_commit call across 3 roles for 1 week, both commit.completed AND werk.commit.completed events emit with same card_id.

Rollback: remove job + skill. Off-switch: direct chorus_commit MCP invocation remains available.

Phase 4 — /werk/push (Kade)

Card: C4. Adds push job that wraps the new chorus_push MCP (P7). Supports --var merge=false for code-review-before-merge.

Cutover semantics: additive. chorus_acp keeps its bundled push behavior; /werk/push is the standalone surface.

Fitness function: 5 /werk/push invocations with merge=true land on origin and merge cleanly; 3 with merge=false land on origin but stay un-merged with a PR opened.

Rollback: remove job + skill. Off-switch: /acp's chorus_acp still bundles push so the orchestrator path is unaffected.

Phase 5 — /werk/approve (Kade)

Card: C5. Adds approve job that wraps the new chorus_approve MCP (P8). DEPLOY_ROLE=jeff only.

Cutover semantics: additive but with a coordination requirement — once C5 lands, /werk/acp's orchestrator can begin checking for the approval-token; until C8 (orchestrator cutover), the marker exists but isn't gating anything.

Fitness function: 5 /werk/approve calls by Jeff produce 5 card.approved spine events; 5 attempts by Kade (DEPLOY_ROLE=kade) all refused with auth-fail; 1 attempt against a card with unchecked AC refused with ac-incomplete; 1 attempt against a card with gate:product-fail refused with gate-fail.

Rollback: remove job + skill. Off-switch: orchestrator (C8) skips approval-token check via env flag during rollback window.

Phase 6 — /werk/pull (Kade)

Card: C6. Adds pull job wrapping chorus_pull_card. Rewrites /pull skill to dispatch via act.

Cutover semantics: wrapper. Atomic transaction preserved.

Fitness function: for every /pull invocation for 1 week, both card.pulled AND werk.pull.completed events emit with same card_id and trace correlation.

Rollback: revert /pull skill to thin MCP wrapper. Off-switch: CHORUS_WERK_DISPATCH=off env makes skill call chorus_pull_card directly.

Phase 7 — /werk/demo with shadow-mode parallel-run (Kade + Wren + Silas)

Cards: C7a (shadow-mode build), C7b (cutover-flip after 10 clean shadow runs).

C7a: build the /werk/demo act job; wire it to run alongside today's /demo skill in shadow mode — both execute, both emit events, both produce a stakes brief, but only the old skill's output is shown to Jeff. Compare events for parity.

C7b: once shadow-mode produces identical event streams + identical stakes briefs for 10 consecutive demos, flip the skill to invoke /werk/demo as primary; old skill stays as fallback.

Cutover semantics: parallel-run then flag-flip. Longest and most cautious of the eight.

Fitness function: 10 consecutive demos across 3 roles emit complete werk.demo.* event arc with same card_id; stakes brief content matches old skill byte-for-byte (modulo timestamps).

Rollback: WERK_DEMO_DISPATCH=off falls back to inline orchestration.

Phase 8 — /werk/acp orchestrator (Kade, with Silas review on release-trigger)

Card: C8. Rewrites /acp skill to invoke an orchestrator act job that walks: /werk/commit → /werk/push → check approval-token (must have card.approved from Jeff) → cards-done → werk-close → release-trigger. Each step has typed refusal; failures roll forward state idempotently per #2943's re-run pattern.

Cutover semantics: the big switch. Before C8: chorus_acp is the atomic MCP that does the whole arc. After C8: chorus_acp's responsibilities decompose into the orchestrator + the smaller MCPs. The orchestrator preserves the atomic-rollback contract at a higher level.

Fitness function: 10 consecutive /acp invocations produce the event arc werk.acp.started → werk.commit.* → werk.push.* → card.approved → cards.done → werk.close → release.triggered → werk.acp.completed, all with same card_id and trace_id; chorus_logs_for_trace returns the complete arc. Failure-injection test: kill the push step mid-arc; verify orchestrator retries via #2943 idempotent path and produces a clean second-attempt arc.

Rollback: revert /acp skill to direct chorus_acp MCP. Off-switch: CHORUS_WERK_ACP_ORCHESTRATOR=off.

Open question for this phase: does chorus_acp MCP retire after C8, or stay as the inner-most atomic implementation that the orchestrator's fallback path calls? Recommend keeping it — the orchestrator-vs-monolith decision becomes a runtime config, not a code deletion.

Risks and open questions

  • OQ1 — RESOLVED. /werk/acp is an orchestrator over decomposed steps. Jeff signed off on the decomposition 2026-05-17 evening.
  • OQ2 — act runner reliability. If act crashes mid-job, what's the recovery path? Today the post-merge pipeline crashes are caught by Silas's chorus-health; the pre-merge lane needs similar instrumentation. Likely a Phase 1 sub-card.
  • OQ3 — multi-role concurrent /werk/build. Two-slot model is per-card; verify chorus_pull_card refuses concurrent pulls of the same card. Verify before C1.
  • OQ4 — chorus_acp retirement after C8. Recommend keeping as the inner-most atomic implementation behind a runtime flag — orchestrator-vs-monolith becomes config not code deletion. Confirm with Silas + Wren.
  • OQ5 — /werk/approve's interaction with the existing bouncer (#2964). The bouncer handles cards-add Jeff-approval today via the pickup-file substrate. /werk/approve is the per-card-acceptance Jeff-approval. They're different states but use the same DEPLOY_ROLE=jeff discipline. Wren should review whether the two should share an MCP (chorus_approve generalizes) or stay as two paths (one for cards-add, one for card-accept).
  • R1 — /demo shadow-mode complexity. Highest-risk cutover. 10-demo parity threshold mitigation.
  • R2 — claudemd-gen regeneration on skill changes. Each new /werk/* skill triggers per-role CLAUDE.md regeneration. Confirm regen path handles 8 new skills gracefully.
  • R3 — backwards compatibility with existing /pull, /demo, /acp invocations. Cards in flight mid-cutover. Mitigation: wrapper pattern preserves skill names; dispatch moves underneath.
  • R4 — approval-token state location. Where does the card.approved marker live so /werk/acp can check it? Three options: spine event only (queryable but no fast lookup); card metadata field (fast lookup, requires schema update); cards comment with structured prefix (cheapest, follows existing demo:preflight-pass pattern). Recommend cards comment with prefix approved-by-jeff: card=<id> ts=<iso> trace=<id>.
