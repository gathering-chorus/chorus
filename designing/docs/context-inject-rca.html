RCA — context-inject returns 0 (chorus search dropped)

2026-06-02 · 4 parallel Explore threads: indexing/freshness · search handler · inject query · live empirical · #3184/#3171 lineage

Problem

The context-inject's chorus search frequently returns 0 results and is dropped, so the inject surfaces little/no knowledge. Hypothesis on the table: "it's coded, but the data isn't returning properly."

Verdict

Root cause: the inject builds an over-constrained query. The search ANDs every keyword; the inject sends ~6 space-joined keywords (often including typos), so the AND intersects to 0 or a handful. The index is healthy and the data returns fine — this is a query-construction problem in the hook, not a data/index/endpoint failure.

What each thread found

1 · Indexing / freshness — not the cause

FTS is synchronous on insert — a DB trigger (messages_ai) writes messages_fts inside the INSERT transaction (sub-ms queryable). The 15-min reindex worker only covers file sources (briefs/decisions/docs), not claude messages, and it's alive/healthy (the better-sqlite3 ABI crash is fixed via absolute node path). So a row is queryable ~instantly once written. Freshness lag does not explain misses.

2 · Search handler — AND semantics, healthy otherwise

fts-query.ts:18 builds the match as "tok1" "tok2" … — space-separated quoted tokens = FTS5 implicit AND. mode=relevance is ORDER BY only (bm25 + source authority), it does not filter. domain= is a type-hint, not in the WHERE. Two robustness gaps (not the main cause): the worker ftsSearch call at chorus-search.ts:185 has no try/catch (a worker rejection propagates uncaught rather than returning 0 cleanly), and FTS errors fail-closed to [] (correct, by design — avoids a 3.6s LIKE scan).

3 · Inject query — the over-constraint

extract_keywords emits up to 6 keywords (significance/length-sorted per #3171) and build_search_url joins them with spaces → the search ANDs all 6. For "…products and domains may have child instances of products and doamins" it produced instances products domains doamins child know — the typo doamins matches nothing, so the AND is 0 before the real terms even matter. #3171 fixed which words survive the cap; it didn't lower the cap, so the AND still over-constrains.

4 · Empirical — confirms ANDing

(table above) Pattern is unambiguous: 1–2 keywords → 100; 4 → 8; 6-with-typo → 0; old content found; mode irrelevant. ANDing of multi-keyword queries is the dominant effect.

The "drop despite re-run returns data" paradox — a separate, narrow issue

In the one cycle we examined, the dropped chorus call returned 1 row on re-run — but that row was Jeff's own just-submitted prompt (logged 153 ms before the query). That's a tiny ingestion-timing edge (the message wasn't written to the messages table yet — not an index lag), and it only ever matches the prompt you just typed (rarely useful to inject). It is not the general 0-result cause; the AND over-constraint is.

Fix (in order of leverage)

1. Lower the keyword cap / OR the terms (the real fix, in the hook). Send the top 2–3 most-significant keywords (extend #3171's significance sort with a lower truncate), or join with OR instead of space. Drop non-dictionary tokens/typos. This directly stops the AND collapse. context_inject.rs extract_keywords/build_search_url.
2. Log the raw search response, not just the drop flag. Today the observability records candidates:[]/drop but not the raw payload — so 0-returned vs dropped-after-return can't be told from logs. Log the raw results.
3. Robustness: try/catch the worker ftsSearch (chorus-search.ts:185) so a worker rejection returns a clean empty + logs, not an uncaught propagation.
Not needed: no index rebuild, no reindex-worker fix, no relevance-mode change, no endpoint fix — the data layer is healthy.
