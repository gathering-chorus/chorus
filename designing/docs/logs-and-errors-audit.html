Logs + Errors + Execution-Flow Instrumentation — Audit

2026-05-08 — Silas (#2832). Audit-only card. Output: this doc + target shape + migration risks per emitter class. Implementation is downstream cards.

Premise. The proposed MCP-Loki-query agent surface (typed log/error access for agents) only earns its keep if upstream emit data is structured, consistent, severity-correct, and instrumented for end-to-end execution flow. Today none of those hold uniformly. This audit enumerates the gap, proposes a target shape, and names migration risks per emitter class.

Flow map — execution-flow gaps

Walked four named end-to-end flows. None are fully reconstructable from spine events alone.

/acp end-to-end (11 hops)

chorus_acp emits at 4 of 11 hops: invoked, board-lookup, card.accepted, completed. Push, PR-create, PR-merge, cards-done are silent on success. Error path emits chorus_acp.refused with stringly reason. No trace_id threaded from skill invocation through completion.

Building-pipeline (8 hops)

launchd → act → workflow steps → CLIs. Per-crate build.completed + deploy.completed emit, but launchd trigger and act invocation are completely silent. build.artifact.hashed and binary.deployed are designed in #2774 but NOT emitted in current code. No entry event when pipeline starts; no trace_id correlating crate-by-crate steps.

Gate chain (8 hops)

Gates emit via card comments (gate:product-pass, gate:code-pass, etc.), NOT spine events. SKILL.md mentions gate.X.passed spine events but the gate skills don't emit them. Two parallel audit trails (comments vs spine); Loki can't see gate outcomes from spine alone.

Demo flow (10 hops)

demo.validate.completed, demo.preflight.completed, demo.stakes.completed, demo.signal.completed, card.demo.started all emit. But: smoke-check.sh runs without spine event from /demo wrapper. Bridge POST is completely dark. Nudge events have UUIDv7 trace_id (post-#2804) but it's not threaded back to the rest of the demo flow — two trace_id regimes coexist.

Cross-flow correlation

Today, you cannot trace a single trace_id across flows. /acp doesn't share trace_id with the building-pipeline it triggers. The gate chain that precedes /acp doesn't share trace_id with /acp itself. "Show me everything that happened for card #2832" requires manual log archaeology + timestamp correlation, not a single trace query.

Pattern classes (cross-language)

Good citizens (replicate this pattern)

  • Bash: chorus-log calls with role + structured fields (alert-delivery-test.sh, deep-health.sh)
  • TypeScript: MCP handler logEvent + typed refusal (mcp/server.ts:435-502); Pulse delivery-worker classifyInjectResult (delivery-worker.ts:57-185)
  • Rust: infra_guardrails (infra_guardrails.rs:91,224-237) — tracing + chorus_log dual emit with decision + pattern fields

Common shape: structured event name following domain.noun.verb_past; role attribution; structured fields; typed reason on failure; severity matches action-required reality.

Stringly stderr (largest class)

Plain text echo/console.log/println without structure. ~94 bash + 142 TS + 12 Rust files. Each emits a string the consumer must parse to extract role, event type, severity. Loki can index but not query semantically. Migration cost: per-call rewrite, but mechanical (echo → chorus-log call with structured fields).

Silent failures / swallowers

Bash: 2>/dev/null || true on critical operations (73 scripts). TS: .catch(() => null) in async chains (~12 sites). Rust: let _ = result for best-effort operations. The hardest class to migrate — each site needs case-by-case judgment on whether the failure is genuinely OK to drop or was hiding a real problem. Some are correct (kickstart of non-launchd service warns and continues). Some hide bugs (curl timeouts to chorus-api swallowed during MCP investigation today).

Thrown without taxonomy

79 TS sites + most Rust gates. throw new Error("reason in string") or HookResponse::deny(&reason_string). The reason is structured information packed into a stringly field. Callers must regex-parse to extract the reason class. Migration: introduce { ok: false, reason: "code", detail: "human" } Result-like patterns; existing typed taxonomies (chorus_acp, chorus_commit) are the precedent.

Severity miscategorized

  • Alert YAMLs declare critical/warning at top, lose distinction in bash emit (5 of 5 alert YAMLs affected)
  • TS: console.error for non-blocking async failures (~6 sites in api/server.ts)
  • TS: console.log for FATAL uncaught exception
  • Rust: log_first_gate uses info! for deny decisions

Effect: Loki severity filters and alert-routing rules see noise where there should be signal, and silence where there should be action.

Target shape (proposed)

Every emit (log or error) must satisfy eight invariants:

  1. Routed via canonical helper — chorus-log (bash), logEvent / structured logger (TS), chorus_log + tracing (Rust). No direct stderr or stringly stdout for production paths.
  2. Schema-compliant — event name matches spine-events.json; required fields present; no field-name drift (commit not sha; cdhash not codesign_hash).
  3. trace_id threaded — UUIDv7 minted at flow entry; propagated through every hop; emitted on every event. No flow runs without one.
  4. card_id as first-class field (when applicable) — every emit happening in a card-scoped flow carries the card_id. Closes the gate-chain-via-card-comments gap structurally: today gates emit comments because spine doesn't bind to card; with card_id on every event, spine and board state join cleanly. Per Wren #2832 PM feedback — the highest-leverage addition.
  5. Role attributed — every emit names the role (or system for system actions). No anonymous events.
  6. Severity classified per action-required reality — INFO (state transition), WARN (degradation, retry-able), ERROR (action required). Mapping documented; not aspirational.
  7. Flow-step named — event name encodes which step in which flow (e.g., building.deploy.kickstart_completed not just deploy.completed).
  8. Errors typed at boundaries — { ok: false, reason: "code", detail: "msg", recoverable: bool }. Stringly errors only inside library internals; never crossing API/CLI/MCP boundaries.

Tier-2 additions (advanced agent UX)

Beyond the eight, two further fields make agent navigation across chains clean (per Wren #2832 PM feedback):

  1. subdomain binding — events carry their subdomain identifier (chorus:version-control-domain, etc.). Closes the "Logs (0) DERIVED" empty-section problem on Athena domain pages — pages can join directly to subdomain-tagged events instead of deriving from naming conventions.
  2. caused_by / parent_event_id — events declare which prior event caused them. Agents navigate causation chains explicitly rather than inferring from temporal proximity (lossy in both directions: missed events look unrelated; concurrent unrelated events look caused).

This is the contract MCP-Loki-query needs as input. With the eight, "show me what happened for card X" is a single card_id+trace_id query. With the tier-2 additions, agents can walk causation graphs and join to Athena pages without inference layers.

Recommended migration plan (priority order)

  1. Easy wins first — bash stringly → chorus-log, alert YAML severity preservation, Rust silent gates → chorus_log. Cost: low. Value: ~50% of audit-flagged sites become structured. Recommend: one card per surface (bash, alerts, Rust gates).
  2. Severity audit + correction — go through console.log/warn/error in TS and info!/warn!/error! in Rust; classify each per action-required reality; fix miscategorizations. Cost: per-call review. Value: alert-routing becomes reliable.
  3. TS thrown → typed taxonomy — extend chorus_acp pattern to other MCP handlers and HTTP route handlers. Author reason taxonomies per surface. Cost: medium. Value: typed errors at API boundaries unblock structured client error handling.
  4. trace_id threading (cross-cutting) — substrate work. Each flow boundary passes trace_id explicitly. This is a card-class of its own (probably 3-5 follow-on cards). Without it, MCP-Loki-query can index but not correlate.
  5. Silent swallower review — last because hardest. Each site needs case-by-case judgment. Likely a recurring quarterly audit rather than one card.

Out of scope (named honestly)

  • This card is audit-only. None of the migrations above are scoped here. Each becomes its own card.
  • The MCP-Loki-query agent surface itself is downstream — this audit is its prerequisite.
  • Standardizing on tracing vs log crate in Rust, or picking a TS structured logger library, are framework decisions out of scope.
  • Backfill of trace_id in historical data is not addressed.
