Convergence Architecture

ICD-Driven Integration for the AI Era · Jeff Bridwell + Silas · 2026-03-18

The contract defines the system. The graph holds the contracts. The agents execute against them. The gates prove correctness.

The Thesis

Every integration system — legacy or modern — encodes business decisions in a vendor-specific format. Those decisions are the real asset. The runtime is commodity. The vendor lock-in is artificial.

Convergence Architecture extracts those decisions into a platform-independent contract layer (the ICD), stores them in a knowledge graph, and generates execution code for any target platform. The contracts are the invariant. Everything else is derived.

The key insight: Integration is not a project. It's a capability. Legacy retirement, new source onboarding, schema evolution, cross-domain linking — they're all the same operation: update the ICD, regenerate, validate. The graph makes this continuous, not episodic.

The Architecture Stack

Layer 5: Attention — How agents maintain focus, quality, and continuity during long-running work. Observer/Navigator/Driver loops with scratch file coordination.
Layer 4: Quality — SHACL shapes and validation gates derived from the same ICD that generated the code. The contract IS the test. Required = 100% or reject.
Layer 3: Generation — ICD data-* attributes → harvester code, SHACL shapes, quality gates, deployment configs. Multi-target: choose your stack, regenerate.
Layer 2: Contract (ICD) — The core. Machine-readable, human-browsable integration contracts with severity tiers (Violation/Warning/Info), field-level mappings, transform rules, and Mermaid diagrams. Lives in HTML with data-* attributes. Version-controlled.
Layer 1: Graph — Knowledge graph (RDF/SPARQL) as the system of record for all ICDs. Supports blast radius queries, cross-domain impact analysis, evolution tracking, and continuous reconciliation.
Layer 0: Extraction — Agent-driven reading of legacy artifacts (vendor XML, SQL, configs) into structured ICD representations. Plugin architecture per vendor format.

Value Stream Orchestration

Once the graph knows every domain, every field mapping, every quality gate result, and every pipeline execution — orchestrating across domains is not a new layer. It's a query.

The value stream is already in the graph. The ICDs are the nodes. The field mappings are the edges. The severity tiers are the SLAs. The quality gates are the checkpoints.

Orchestration = graph query + event trigger. "When this domain's gate passes, trigger that domain's pipeline." No BPMN editor. No workflow designer. The graph IS the workflow definition. The ICDs define what each step produces and consumes. The quality gates define when a step is complete. The edges define what comes next.

This means:

  • Adding a step to a value stream = adding an ICD + mapping its outputs to the next domain's inputs. The graph connects them.
  • Removing a step = decommissioning an ICD. The graph shows blast radius — what downstream depends on it.
  • Monitoring a value stream = querying the graph for gate results across the chain. Green/yellow/red per domain, per field, per severity tier.
  • SLA enforcement = severity tiers on the edges. If Order → Shipment mapping has a Violation-tier field that fails, the value stream halts at that checkpoint. Warning-tier fields log and continue.

"Order → Shipment → Notification" is the same pattern as "Apple Photos → Canonical → Thumbnail → Browse." Different domain, same architecture. The convergence layer doesn't distinguish between enterprise order flow and personal photo pipeline. Both are: source → ICD → canonical → derive → validate → serve.

Code as Wiring, Not Writing

As integration systems mature, the code becomes less like authoring and more like wiring — connecting well-defined inputs to well-defined outputs through well-defined transforms. Event-driven architectures make each connection atomic: one event, one handler, one transform, one validation.

The pattern: If the ICD defines the fields, types, severity tiers, and transform rules — and the generated code follows a consistent pattern (read → transform → validate → write) — then the only novel code is in the callouts: the extension points where business logic lives that can't be derived from the contract alone.

The generated code is the skeleton — structurally complete, type-safe, gate-enforced. The callouts are the extension points where:

  • An AI agent adds business logic that the ICD describes in prose but can't express as a data attribute. "If order type is dropship, use ship date not order date" — the ICD's data-transform says this in English. The agent reads it and writes the callout implementation.
  • A human adds tribal knowledge that isn't in the legacy code at all. "We always exclude test accounts with IDs above 900000" — no code captures this. The human writes it into the callout. Now it's versioned and visible.
  • The agent evolves the callout when requirements change. The ICD updates, the skeleton regenerates, the callouts persist because they're in separate files. The wiring changes; the business logic stays.

The Atomic Event Pattern

ComponentGenerated from ICDCallout (agent or human)
Event trigger✓ — "when source changes" from graph edge—
Source read✓ — connection + query from data-source-path🔌 onBeforeTransform — enrich, filter, join
Field mapping✓ — from data-maps-to + data-transform-type🔌 onTransform — custom logic for complex transforms
Type casting✓ — from data-datatype—
Cardinality✓ — from data-cardinality (scalar vs array)—
Validation✓ — SHACL shapes from severity tiers🔌 onAfterValidate — edge case handling, cross-field rules
Write to canonical✓ — TTL/JSON output from data-output-pattern—
Downstream trigger✓ — graph edge to next domain🔌 onComplete — notifications, side effects, audit

The ratio shifts over time. Early in a domain's life, callouts are heavy — lots of custom logic. As the ICD matures and the data-transform-type vocabulary grows (regex, keyword-match, name-match, slugify, lookup, default, enum — all pattern-driven), more logic moves from callouts into the generated layer. The agent's job evolves from "write the callout" to "recognize the pattern and add it to the ICD."

The end state: Code is wiring. The ICD defines the circuit. The generator produces the board. The callouts are the handful of custom components that can't be off-the-shelf. The agent reads the ICD's prose descriptions and implements the callouts. The human reviews. The gate validates. The graph orchestrates.
