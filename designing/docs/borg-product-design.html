Reads after the value-stream. The value-stream-design is the spine (the reflect cycle + the domains); this product-design instances it — what Borg promises, who it serves, its structure, and the JX of its pages. Borg spans two steps — same eye, two focal lengths: Proving (prove one change, per-card, terminates) and Reflecting (watch the running system, continuous, loops back to Shaping). One self-judgment product at two cadences.
⚠ As-is vs to-be. A DESIGN doc — the to-be. The model-of-record (tree.json) today = 5 steps (Borg atStep Proving), a flat 12-domain set, alerts-monitors unsplit, no obs/ops/infra grouping. Deltas this design lands (to-be, not current): the Reflecting step + the span; ADR-043's monitor/alert split → 13; the 3-layer grouping; heralds→xray. Read "spans" / "13 in 3 layers" / "model of record" below as the to-be.

2 · Promise

The system sees itself — so no human has to be the eyes. Borg reflects on everything it can reach (its services, health, alerts, the gap between how it's drawn and how it runs) and surfaces the one thing that needs a person, calmly, to the person who owns it.

Borg's product isn't dashboards — it's your attention back. Every night Jeff is the first to notice a stall is a night Borg failed at its job (feedback_jeff_is_not_the_monitor). The freeze that hid for weeks (#3406) is the anti-promise made concrete.

3 · Value & audience

  • The roles — consume Borg to self-coordinate: notice a peer dark, a service flapping, a drift, before Jeff does. Borg is how the team becomes its own monitoring system.
  • Jeff — consumes outcomes only: a calm board, or the single thing that needs him. Never the 29-warning wall; never the role of monitor.

Every Borg page renders data from MCP the same way a role does (AX=JX) — no bespoke fetches — and conforms to the #3415 design system (system.css; never a one-off page). Entry is through the system drawing, not a standalone dashboard.

The landing — a verdict, not a readout

Borg opens with a judgment, not gauges: "is anything wrong that needs you — and if not, is the system doing its work?" Two states. Either calm ("all coherent, nothing needs you — go") or the one thing that does, alone, with nothing competing for the eye. You walk into a room and know instantly if it's calm or on fire; if calm, you leave — you don't stand reading gauges. The gauges are one drill down, not the front door. This is the antidote to the 29-warning wall.

The cycle, as a seasonal wheel

Not a progress bar (that's Werk's line) — a wheel. The borg-reflect cycle (plant→tend→read→surface→prune→feed) renders as the recurring shape it is; the page shows where attention is in the season, what's been surfaced, what's been pruned, what got composted back. The cadence is visible as cadence.

The drill pages

  • Monitor — the live signal: which services are alive (liveness), which jobs ran (freshness), saturation gauges, latency SLIs, error/crash-loop, synthetic flows. Each renders the ADR-043 contract.
  • Alert — what's firing / fired, severity, who it routed to (ownership), dedup state. Config-driven; reroute is a data change.
  • Health / Metrics / Fitness — the aggregate health verdict, the SLO burn, the flow-report (card cycle/step/error fitness) — composed from MCP + the imported-home observation pages.

8 · As-Is / To-Be / Path to close

As-is: a deployed stack (Prometheus/Grafana/Loki/Alertmanager/exporters/deep-health), ADR-043 landed (the domain model), #3405 landed (probe truthfulness — the cry-wolf killed). But: probes were bespoke + lied; no liveness/crash-loop signal (why #3406 hid); alerts hand-routed; no model-driven contract; no Borg page (the observation surfaces are scattered).

To-be: the borg-reflect cycle, model-driven; monitor/alert as ontology domains whose contracts generate; the verdict-first Borg page; absence/silence first-class.

Path to close (ordered): value-stream spine → liveness leg (the #3406 gap) → freshness/errors/saturation/latency/synthetic legs → config-driven alert routing → the Borg pages render it.

9 · Not in scope

  • The demo ceremony tooling (werk-demo orchestration, present→gather→verdict→accept) — that mechanism is Werk's. But the per-card proving judgment is in scope: it's the Proving half of Borg's span, carried by Borg's gate-arch/gate-ops lenses. Borg proves; Werk runs the ceremony.
  • Remediation / acting — in the Reflecting (observability) half Borg surfaces and the owning role prunes (observe-only). Note: Borg's operations layer does act (deploy/rollback/restart) — reflect-not-act is an observability property, not the whole product's.
  • Assimilate / mitose — the other two Borg modes (ICD legacy extraction, reproducing across products) are out of scope for this product-design, but not separate capabilities — they share this product's herald/xray as their first step (OBSERVE). Modeled in their own docs; named here so the boundary is "this doc covers reflect," not "Borg only reflects."
  • /flow — Directing, not Proving.

10 · Mitosis & the commercial bet

Borg is built to travel. Chorus is a factory; the systems it builds must run with chorus OFF — so Borg's capability ships generatively, never as a runtime dependency: the model generates the client system's own scrape configs, alert rules, SLO defs, dashboards, liveness emitters, deploy/rollback scripts — plus a standalone OSS stack it carries (Prometheus / Grafana / Loki / Alertmanager, the same ones we run natively). The factory generates; the artifact runs alone. The gate is a literal test: turn chorus off — does the client system still observe / alert / deploy itself? Any callback to chorus = a leaked dependency to cut.

Borg is the proof of the commercial bet. The hypothesis: if we can model value streams + steps + domains, we can quickly build almost anything. True in its strongest form — the model generates the commodity substrate (the ~80% every team rebuilds: structure + the ilities), and human craft goes to the differentiated behavior (the ~20% that is the product). Borg's ops/observability model is the generic, reusable part — every client system needs it and it generates uniformly. So finishing Borg's value-stream + domain model cleanly is the dogfood that proves the bet: if we can't make our own ops product a finished, coherent, queryable model, "model anything → build anything" is unproven. Borg = the exemplar; gathering = client #0, the first system the matured factory regenerates (the bootstrap: gathering births Chorus → Chorus rewrites Chorus → Chorus rewrites gathering — compiler self-hosting).

Adjacent future (parked): the observability data is the system's own-data corpus → an ML/training substrate (seeded by the analytics domain), plugged into the cycle's read/feed steps — once the telemetry is modeled, flowing, and trustworthy. Not before.
