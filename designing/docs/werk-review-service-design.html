werk-review — the cold-eyes gate

The machine reviews the code before Jeff does.

Kade · 2026-06-05 · #3193 · service design · shaped with Jeff · companion to the werk verb-sequence (pull→code→commit→push→build→test→review→demo→merge→accept→finalize)

werk-review is the machine's independent review of the diff, run before the human's review (demo). Two distinct provings, not one: review = independent-agent-eyes on diff-vs-AC; demo = Jeff's eyes on does it do what I wanted. Complementary — the cold eyes catch code problems so Jeff's attention is spent on intent, not bugs.

1 · Placement — after test, before demo

… build → test → werk-review → deploy-werk → env-up → demo → merge → …

Review needs the diff (it exists after commit/push) and gates before Jeff spends demo attention. Code is already green (test passed) when the cold eyes read it; if review fails, the builder fixes before the human is ever pulled in.

werk-review — the machine's eyes

A fresh agent reads the diff against the AC. Adversarial: its job is to find problems. Returns a verdict + specific findings. Catches code-level issues (AC not covered, aspirational-as-built, bugs, missing tests).

demo — Jeff's eyes (#3237)

Jeff judges intent against the live instance: go / no-go / more. The human gate. Catches "this isn't what I meant" — which no agent can.

2 · What "cold eyes" actually means load-bearing

A fresh review sub-agent given ONLY the diff + the card's AC — no build conversation, no rationale, no "I know I meant X." No build context = it cannot rationalize the way the builder can. And it is prompted adversarially: default-skeptical, its job is to find problems, not to bless.

What it hunts

  • AC items the diff doesn't actually cover — the #1 lie (a checked box with no code behind it).
  • Aspirational-as-built — comments/claims that say more than the code does (the exact class caught in Silas's #3242: "the generator emits" when the generator is unbuilt).
  • Bugs — logic the test didn't catch.
  • Missing tests — AC behavior with no covering test.
  • Scope creep — changes beyond the AC.

3 · Verdict & contract

Mirrors werk-demo's exit-code contract so act blocks on it the same way:

review.verdict { card_id, verdict: pass|fail, findings: [{file:line | "AC item N", issue}] }
  pass → exit 0 → act continues to demo
  fail → exit non-0 → act stops; builder fixes

Findings must be specific — file:line or "AC item N not covered". A verdict with no specific findings is not a pass — it's the failure mode (see below).

4 · The anti-ceremony guard

The risk: a review agent that rubber-stamps ("lgtm") is worse than nothing — it manufactures false confidence. This is the concern that kept werk-review parked while the pipeline was half-built. The defenses are the design:
  • Adversarial prompt — find problems, default-skeptical (the verify-pattern, not the bless-pattern).
  • Required specific findings — "lgtm" with no file:line is rejected as a non-review.
  • A structured floor it cannot skip — blast-radius (#3148) + AC-coverage check run regardless of the agent. Even a lazy agent run still does the objective minimum.
