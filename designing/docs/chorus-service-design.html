Chorus — Service Design

2026-04-11 — Silas (#1877). Updated 2026-04-18 — Wren. Today's landings: #2168 context injection envelope (pulse/spine/athena live per prompt, sub-100ms); role-state writer discipline (declared/inferred file split); #2173 Quality as its own sub-product; #2174 response quality for AX as distinct future-scope card.

Any role can ask "what happened last session?" or "what does the team know about photos?" and get a semantically rich answer without asking Jeff. When the index is stale or incomplete, every role's comprehension degrades silently. Chorus is the team's memory — if it's wrong, the team is wrong.

Overview

Chorus is the shared memory and coordination index. Five components: the SQLite message index (FTS), the LanceDB vector index (semantic), the HTTP API, the hooks service, and the ingestion pipelines that feed them. As of 2026-04-18: FTS works (512 MB, 306K+ messages and growing), vector index is populated (2.3 GB) and freshness is 11/12 sources fresh with 1 critical, hooks service now injects live Chorus context into every prompt envelope, and 8 data sources remain without ingestion pipelines. Response-quality tuning for agent experience is scoped as #2174 — this doc describes the mechanism; response-schema work is downstream.

Architecture Decisions

DecisionRationaleImpact
Ollama on Bedroom MacGPU acceleration, separates embedding load from appNetwork dependency — Bedroom sleep/wake causes transient failures
nomic-embed-text model768-dim, fast, good quality for retrievalLocked to Ollama availability
Hybrid FTS + vector searchRRF ranking combines keyword precision with semantic recallBoth indexes must be fresh for good results
SQLite FTS5 for messagesSimple, fast, no external dependencySingle-file DB, easy backup, hard to scale
LanceDB for vectorsEmbedded, columnar, good for batch operationsNo server process — rebuilt on app start, stale if app doesn't restart
Delta embedding on API callIncremental — only embed new messagesDepends on API being called. If nobody searches, vectors don't update.

Related Sub-Products

Chorus is the shared-memory substrate. Several sub-products depend on it:

  • Pulse — team-state assembler. Pure pass-through; reads role-state files, spine, board snapshot, freshness cache. Sub-100ms warm.
  • Loom — org substrate (principles, practices, policies, decisions, roles). Indexed into Chorus.
  • Clearing — multi-role browser chat. Reads pulse + role-state files; renders tiles with multi-WIP surfacing (#2168 AC-11).
  • Quality (new, 2026-04-18) — pyramid + coverage + gates + test-value policy. Horizontal discipline. Kade-owned, depends on gates-service + spine-service. Chorus-response-quality-for-AX (#2174) is the AX analog of Quality's test-value-policy.
