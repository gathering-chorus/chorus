Athena — Service Design

2026-04-11 — Wren (#1880). Sources: server.ts (Athena endpoints), chorus.ttl (ontology v0.3.0), 8 SPARQL query files, 5 UI pages, 29 integration tests.

Any role can query the operating model — who owns what, what depends on what, what breaks if something fails — from a single API. The graph is the implementation of how the team operates. When agent experience through this API is good, Jeff's experience is good. Athena is the CMDB that agents maintain by working, not by filling out forms.

Overview

Athena is the CMDB visibility layer for Chorus. Named after the system Jeff built at Staples (2010–2015), but fundamentally different: the original was designed for human data entry; this one is populated by agents as they build. The ontology (OWL/RDF in Fuseki) is the source of truth. The API makes it queryable. The UI is a view on the API. Gates and hooks enforce it.

Data Flow

Role edits chorus.ttl → curl PUT to Fuseki → urn:chorus:ontology graph → SPARQL queries → Athena API :3340 → UI pages :3000
Gates query API ← Ownership, blast radius ← consumes / ownedBy edges
Gap: The write path is manual — roles edit a Turtle file and curl it to Fuseki. This is the 2010 pattern inverted: humans used forms, agents use file editing. Neither is an API.

Ontology Loading

Current: Manual PARTIAL

Source: roles/silas/ontology/chorus.ttl → Fuseki GSP endpoint with admin auth.

StepCommandNotes
Loadcurl -X PUT localhost:3030/pods/data?graph=urn:chorus:ontology --data-binary @chorus.ttl -u admin:$PWReplaces entire graph
Verifycurl localhost:3340/api/athena/healthCheck triple count changed

Target: Automated DESIGN

TTL changes committed to git trigger reload via POST /api/athena/reload. The reload endpoint reads the TTL file, validates syntax, loads to Fuseki, and returns the new triple count. PostToolUse hook on git commit triggers the reload if chorus.ttl was in the changeset.

Gaps

Gap: No write API. Roles edit TTL by hand and curl to Fuseki. Every ontology change requires file editing, manual auth, and manual verification. No audit trail on mutations.
Gap: No automated ontology loading. TTL changes require a manual curl command with admin credentials. If a role forgets to reload, the API serves stale data.
Gap: 3 domain detail sections are placeholders (Active Cards, Code Inventory, Alerts/Monitors). Each needs a new API endpoint backed by indexed data.
Gap: No Agent Experience (AX) metrics in the model. "Does Kade have good tooling for this domain?" is a vibes question, not a queryable property. Need frictionIndicator, migrationReadiness as data properties.
Gap: Operating model data lives in prose (TEAM_PROTOCOL.md, team-architecture.md). Ownership tables, execution modes, service registries, decision flow — all in markdown, not in the graph. Gates can't query prose.
Gap: Test count assertions in integration tests are brittle. Adding a subdomain breaks 13 existing tests because counts are hardcoded. Tests should use relative assertions (count > 0, contains X) not absolute counts.
