Competing Implementations — As-Is vs To-Be Across Services

2026-04-21 — Silas. Input to #2437. Three services audited: Nudge, Spine, Chorus (product level). Same pattern as #2280 pulse: one concept, multiple implementations that drift.

One concept, one canonical surface. Every other representation is a derived projection, never a competing source of truth. When an agent wants "current state of X," there's one answer — not four, not filtered, not synthesized in middleware.

The pattern in plain language: every stateful signal in Chorus today has two or more places that claim to own it. A nudge exists in Bridge API + queue file + spine events. Role state exists in declared.json + inferred.json + spine + pulse + Clearing tile. Board state exists in Vikunja + board-wip-snapshot.json + pulse + Chorus index. These surfaces drift. Middleware filters one to match another. Agents form claims on whichever surface they happened to read. The remedy is structural: pick the canonical surface per concern, derive the rest.

1. Nudge

A single nudge invocation today writes to three persistence surfaces. Two of them are redundant with the spine; one of them (the queue file) is semantically a delivery-failure fallback but behaves like a parallel store when drained.

2. Spine

Spine today is a single file (chorus.log) with ad-hoc event shapes. No registry. No validation. Workflow events and telemetry events interleave. No retention policy — #2323 swat had to purge 24k search.query.executed rows the FTS was eating. The file is canonical in spirit but not in contract.

3. Chorus (product level)

At the product level, the competing-implementations pattern shows as the same signal living in multiple stores across multiple APIs with no clear canonical surface per concern. Role state lives in five places. Board state lives in four. Nudge state lives in three. Each API surfaces whichever store it grew up around. The detector this generates is: "does this signal have more than one write path?"

Green = three canonical stores, each owning one concern exclusively. Blue = derived projections that read from canonical, never write. Amber = APIs as thin adapters — they don't own state, they serve it. Every signal has exactly one write path.

What changes at the product level: every signal gets a canonical store designation. Stores that currently act as primary for their concern but shouldn't (messages.db for nudges, /tmp/* for role state) get demoted to derived projections or retired. APIs get re-scoped — bridge :3475 becomes "event publisher endpoint," chorus-api :3340 becomes "canonical-read + projection-serve," gathering-app :3000 becomes "UI routes + static assets" and stops holding state. The ontology-aware write-gate blocks new stores from claiming ownership that already has a canonical home.

What the detector does, given this map

With the canonical surfaces named, a write-gate can ask three questions against the ontology at emission time:

  1. New event type? Emission of an event name not in event-types.ttl → block or warn loudly.
  2. New state file? A write to /tmp/*.json that isn't a known derived-projection path → flag as a candidate competing implementation.
  3. Cross-store write? A handler that writes to more than one canonical store in the same operation → suspicious; audit whether one of the writes should be a derived projection instead.

None of these detections need to be perfect — they need to raise architect visibility before a new competing surface lands. The registry is the shared prior knowledge the detector enforces; the audit (#2437) populates the registry; the to-be diagrams here are the target the registry encodes.
