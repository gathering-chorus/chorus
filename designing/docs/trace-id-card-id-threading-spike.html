trace_id + card_id threading spike (#2841)

Silas · 2026-05-08 · spike for #2839 (trace_id contract) + #2838 (card_id contract). Investigation IS the deliverable.

Method

Used the just-shipped /acp #2835 as the live flow. Queried Loki across {job=~".+"} for any line referencing 2835 in the last 2h. For every emitted spine event in that flow, checked: did trace_id arrive? did card_id arrive? Then walked the result by hop boundary.

Headline finding

13 distinct trace_ids touched a single /acp transaction. No single correlation_id spans the run. Reconstructing "what happened on /acp #2835" today requires joining on card_id plus temporal proximity — which only works because every event in this flow happened to mention 2835. The audit's claim ("0 of 4 flows fully reconstructable") is now grounded: trace_id is per-emitter, not per-transaction. card_id is the de-facto correlation key, but it's missing on ~30% of in-flow events (notably nudge.* and build.* sub-events).

The two distinct trace_id concepts

The data shows two senses of "trace_id" already coexisting, neither matching what #2839 needs:

  1. Pino per-log-line trace_id (TS / platform-logs): minted at log time, never threaded between calls. Useful for finding lines from a single HTTP request; useless for "show me everything that happened on /acp #2835."
  2. state transition correlation_id (role-state via shim-wrapper.sh, see traces table): minted per role-state call as state-${ROLE}-${CARD}-${epoch}. Closer to a flow id, but scoped to one transition.

What's missing — and what #2839 must define — is a transaction trace_id minted at the entry point (the /acp call, the /pull call, the /demo call) that propagates through every downstream emission until the transaction completes.

Contract questions for #2839 and #2838

These are the questions the design cards have to answer. They surfaced from the threading exercise; without them the contract is guesswork.

For #2839 (trace_id propagation)

  1. Who mints, who carries. Does the entry-point skill mint trace_id and pass it to the MCP, or does the MCP mint and return? Recommendation from this spike: MCP mints on entry and returns it to the skill so subsequent observer queries can use it.
  2. Carrier shape across the bash/TS/Rust boundary. trace_id flows over which channel: argv (--trace-id=X), env (CHORUS_TRACE_ID), or process state? bash chorus-log accepts neither today. Pick one, retrofit chorus-log to forward it.
  3. Reconciling with Pino's per-line trace_id. Pino mints its own; if we add a flow trace_id, do we rename one (flow_id vs trace_id) or treat Pino's as the request-scope and ours as the flow-scope? Both have legitimate uses; the contract has to name them differently.
  4. What about gh / git / npm subprocess calls. External tools have no trace awareness. The contract has to say "the MCP brackets the call with chorus_acp.gh-merge.{started,completed} and the trace_id rides on the bracket events; the inside is opaque." Document that gap explicitly.
  5. Transaction boundaries. When does a trace_id stop being valid? After /acp returns? After werk-close? After the nudge fan-out? Right now there's no termination event; the contract needs flow.completed with trace_id as the close marker.

For #2838 (card_id propagation)

  1. Derived vs carried. chorus_acp derives card_id from HEAD branch. Other entry points (cards CLI, demo skill) carry it as argv. The contract has to specify: card_id is always passed through, but in some contexts it can be derived if absent (and the derivation must be logged).
  2. Why nudge / observer events drop card_id. nudge.* and observer.digest fire from chorus-api but emit without card_id even when the trigger was card-bound. The contract has to require: if the originating event had a card_id, every spawned event in the same flow inherits it.
  3. Multi-card transactions. /demo accepts multiple card IDs. Today it would emit 1 demo per card with each card_id field set. But what about a "batch /acp"? The contract has to say card_id is always singular per event; batch operations fan out into N events.

Implications for #2840 (MCP-Loki-query)

Until #2839 lands, the agent surface for "show me everything that happened on /acp #2835" must be card_id-keyed, not trace_id-keyed. trace_id is too noisy (13 distinct ids per flow). #2840 should expose:

  • flow_by_card(card_id, time_window) — the primary query, joins on card + temporal locality.
  • flow_by_trace(trace_id) — secondary, only useful once #2839 ships and trace_id is per-flow.
  • events_missing_card_id(time_window) — visibility into the gap (nudge/observer/build sub-events) so #2838's contract can be enforced over time.

What to NOT do based on this spike

  • Don't ship #2839 + #2838 as one card. They share a problem space but have different contracts (trace = transaction-scope, card = work-binding) and different propagation rules. Keep them sequenced.
  • Don't treat Pino's per-line trace_id as the answer. Renaming it to flow_id would silently break per-request trace correlation that already works.
  • Don't try to retrofit gh / git CLI to carry trace_id. Bracket them with started/completed events on the MCP side and accept the opaque interior.
