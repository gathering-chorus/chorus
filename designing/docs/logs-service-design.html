Logs — Service Design

2026-05-08 — Silas (#2848). Names logs as a first-class Chorus domain. Origin: Jeff (2026-05-08): "we are trying to normalize our interaction with logs." That's the through-line of this blueprint — one emit shape, one query surface, one trace_id concept, used the same way by every actor (agent, dashboard, alert, human, MCP). Today's substrate matches at ~80%; the remaining 20% is the normalization gap, named in Path-to-close.

Source of truth: Live actor list, BDD scenarios, contract entries, dependencies, gaps, and integrations live at:
  • /athena/domain-detail.html?id=logs-domain — pending creation; this doc is the spec until the subdomain entry lands.
  • designing/schemas/spine-events.json — canonical event type registry.
  • shared-observability/config/promtail/promtail-native.yaml — scrape configuration.
This doc narrates structure, contracts, and intent. Field-level event schema lives in spine-events.json; promtail scrape definitions live in the YAML. Both are referenced by name, not duplicated here.
Designed target state — normalized interaction. Every actor (agent, dashboard, alert, human, MCP) interacts with logs the same way. Four guarantees hold: (1) one emit shape — every script, daemon, and handler uses its language's canonical adapter (chorus-log / SpineEmitter / chorus-hooks); raw echo >&2 / println! / console.log for operational events is forbidden; the envelope is identical across languages; (2) one schema — every emission carries the seven required envelope fields; spine-events.json declares the type and payload; emits whose shape doesn't match are refused at write time; (3) one trace_id concept — transaction-scope, UUIDv7, minted at flow entry, propagated through every emit until the terminal event; one Loki filter returns the entire causal chain; (4) one query surface — agents call typed MCP tools with refusal taxonomy; dashboards and alerts query Loki directly; both shapes return the same events because emit is normalized upstream. Substrate ~80% built today; the remaining 20% is the normalization gap.
What "logs" is, distinct from neighbors. The blueprint claims this domain owns three things and explicitly disclaims four:
  • Owns: emit shape (envelope + payload), transport pipeline (Promtail → Loki), query/trace surface (Loki API + MCP-Loki-query).
  • Does NOT own: spine event type ontology (a separate concern with its own schema), metrics derivation (numerical time series, separate domain), Loki operational health (infrastructure), retention policy of derived artifacts (separate concern).
The boundary matters because every "should logs do X?" question reduces to one of these: if X touches emit/transport/query, it's logs; otherwise it's another domain consuming logs.

Contracts

Emit envelope (required on every emission)

{
  "event":     "<domain>.<noun>.<verb_past>",   // declared in spine-events.json
  "trace_id":  "<uuid-v7>",                       // transaction-scope correlation
  "role":      "silas|kade|wren|system",          // who emitted
  "ts":        "<ISO 8601 with timezone>",        // emit time
  "level":     "info|warn|error",                  // severity per outcome
  "source":    "<script | service | binary name>", // where it came from
  "payload":   { ... }                            // event-type-specific fields
}

Optional fields: card_id (when emit is card-scoped), span_id (per-log-line correlation, Pino's internal), parent_trace_id (when one flow spawns another), caused_by (reference to triggering event_id).

Event type contract

Every event name must appear in designing/schemas/spine-events.json. Schema entries declare:

{
  "vertebra":    "capturing|directing|operating|...",
  "legacy_name": "<old name or null>",
  "source":      "<canonical emit site>",
  "description": "<one-line semantic>",
  "fields":      { "<field-name>": "<description>", ... }
}

Naming convention: <domain>.<noun>.<verb_past> (e.g., card.accepted, build.completed, canonical.synced). Validator (Path-to-close item) refuses unknown event names at emit time.

Trace_id lifecycle

  • Mint at flow entry: MCP entry handler calls mintTraceIdV7() (UUIDv7); top-of-CLI bash mints via uuidgen | tr A-Z a-z.
  • Propagate via SpineEmitter closure (TS), $CHORUS_TRACE_ID env (bash and Rust subprocesses), X-Chorus-Trace-Id HTTP header (cross-service), Chorus-Trace-Id: commit trailer (cross-cycle: /acp → building-pipeline).
  • Terminate with one of three terminal events: flow.completed, flow.failed, flow.aborted. Each carries trace_id + flow_kind + duration_ms.

Full design: designing/docs/trace-id-propagation.html (#2839).

Card_id propagation

card_id is the work-binding field. Carried through the same envelope as trace_id. When the originating event has a card_id, every spawned event in the same flow inherits it. Multi-card flows fan out into N events, each with one card_id. (Full contract: #2838 design pending.)

Surfaces — adapter contracts

Bash — chorus-log

Caller shape: chorus-log <event> <role> key=val key=val

Mechanism: shim wrapper at platform/scripts/chorus-log dispatches via chorus-hook-shim binary (Rust). Reads $CHORUS_TRACE_ID from env if set; mints fresh otherwise. Writes JSON line to a daemon log file promtail scrapes.

Boundary: bash scripts that emit operational events MUST use chorus-log. Bash scripts that produce CLI output for human readers (validators, linters, query tools) emit via echo as before — those are user interfaces, not operational telemetry. The hook backstop (Path-to-close) enforces this distinction.

Refusal: shim refuses on missing event-name or unknown event (post-validator). Caller sees stderr with refusal reason; non-zero exit.

TypeScript — SpineEmitter + Pino

Caller shape: const emit = createSpineEmitter(traceId); emit('event.name', { card_id, ...fields });

Mechanism: closure constructed at MCP handler entry captures trace_id; every emit attaches it. Underlying Pino logger writes JSON to chorus-api stdout; daemon log file promtail scrapes.

Boundary: TypeScript handlers in chorus-api MUST construct a SpineEmitter at flow entry. Handlers do NOT call Pino directly for spine events. Pino's per-line trace_id renames to span_id to disambiguate.

Refusal: emit() rejects if event-name absent or trace_id type-invalid. Validator middleware (Path-to-close) refuses unknown events.

Rust — chorus-hooks daemon

Caller shape: emit_spine!("event.name", role=ROLE, trace_id=TRACE, payload)

Mechanism: daemon writes JSON via the standard log crate to its log file. chorus-hook-shim invocations write inline JSON for hook events. Reads $CHORUS_TRACE_ID from invoking shell's env.

Boundary: Rust binaries in chorus-hooks emit only via the daemon's logger or shim's inline path. No stderr/stdout side-channels for operational events.

Refusal: hook shim refuses if envelope incomplete; PreToolUse hooks refuse the tool call with the refusal reason in their response.

Aggregation — Promtail + Loki

Promtail scrapes JSON lines from ~/Library/Logs/Chorus/*.log. Scrape jobs in shared-observability/config/promtail/promtail-native.yaml:

Job labelSource filesPurpose
job=daemon-logschorus-hooks.log, chorus-hooks-supervisor.logHook daemon stdout/stderr + heartbeat
job=chorus-apichorus-api.logTS daemon emissions (Pino + SpineEmitter)
job=chorus-hooks(alias of daemon-logs in current config)Filtered view for hook-specific events
job=platform-logscards-cli.log, generic bash via chorus-logBash chorus-log emissions
job=platform-choruspulse.log, observer.log, nudge.logPulse worker + observer fan-out
job=platform-permission-promptstcc-prompts.logTCC / sandbox prompt telemetry

Loki at localhost:3102. Retention 3 days. Indexed on labels (job, level) plus full-text on the line body. No structured field index — a deliberate tradeoff: query is regex over the line, which is fast enough at current volume and avoids the cardinality explosion that fielded indexes hit on trace_ids.

Operational guarantees:

  • At-most-once delivery from emit to Loki (Promtail can drop on backpressure; not a queue).
  • Eventual consistency: emit-to-queryable lag ≤ 5s under normal load.
  • No emit-side persistence: log files are tail-and-rotate; the durable store is Loki + the source-of-truth (cards, git, files).
  • Retention 3 days (Loki). Queries against older events fail; "what happened a week ago" is answered from cards/git/spine-events.json, not from logs.

Query surface

Tier 1 — Loki HTTP API (canonical)

curl -G http://localhost:3102/loki/api/v1/query_range \
  --data-urlencode 'query={job=~".+"} |~ "<trace_id>"' \
  --data-urlencode 'start=<ns>' --data-urlencode 'end=<ns>' \
  --data-urlencode 'limit=1000'

Used by Grafana, alert YAMLs, and ad-hoc operator queries. Stable surface; never wrapped or hidden.

Tier 2 — MCP-Loki-query (typed agent surface, #2840)

chorus_logs_query({
  by: "trace_id" | "card_id" | "event_pattern",
  value: "<id-or-pattern>",
  time_window: "5m" | "1h" | "1d",
  limit: 1000
})
// → { events: [...], truncated: bool, cursor: string|null }

chorus_logs_flow({ trace_id })
// → { events: [...], terminal_event: "...", duration_ms, status }

chorus_logs_missing_field({ field: "card_id" | "trace_id", time_window })
// → { count, sample: [...] }   // visibility into adapter-coverage gaps

Refusal taxonomy: loki-unreachable | invalid-time-window | id-not-found | result-too-large.

Tier 3 — Grafana dashboards

At localhost:3100. Canonical panels: trace timeline, error rate by service, alert-fired-today, gate-pass rate. Dashboard JSON checked into dashboards/; Grafana provisions from there.

Three meta-invariants per ADR-028: I-1 spine-events.json is the single source of event-type truth, no surface caches; I-4 one source, multiple call shapes (bash + TS + Rust adapters all emit the same envelope); I-10 atomic per emission (each emit is independent; aggregation correlates after the fact, never as a synchronous fan-in).

As-Is — 2026-05-08

Substrate ~80% built. What works: three adapters, Promtail → Loki pipeline, Grafana dashboards, card_id-as-de-facto-correlation, alert YAMLs reading Loki. What doesn't:

  1. Trace_id contract not implemented. Spike found 13 trace_ids per /acp transaction. Contract designed (#2839); foundation card filed (#2845, SpineEmitter constructor); migration cohort pending. This is the biggest gap.
  2. 4–5 emitted events not in spine-events.json. Schema-drift class. Mechanical fix (#2843).
  3. No emit-time validator. ADR-028 req 6 unmet. Drift goes silent until manual audit.
  4. ~70% of bash scripts still echo to stderr for what would be operational events. Most are correct human-facing CLI output (#2835 honest re-scope); ~5 are real gaps that need migration case-by-case.
  5. chorus:logs-domain subdomain entry missing. ADR-028 req 7. Athena page doesn't render.
  6. MCP-Loki-query (typed agent surface) not shipped. Today agents use ad-hoc curl. #2840.
