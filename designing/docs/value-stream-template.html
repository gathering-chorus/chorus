The API-Driven Value Stream — Template · the reusable structure every product's value stream follows

Wren · 2026-06-13 · the generic pattern. Athena is its first instance (see athena-value-stream-design.html); werk is the Building-step instance.
What this is. The reusable shape for a value stream in Chorus — independent of which product realizes it. Author a model once; the generator forges its surfaces; the step structure organizes the work; one API spine serves both the agent (AX) and the human (JX). A new product's value stream is built by instantiating this template, not by inventing structure. That's the leverage: build the template (and the generator) once, materialize every stream.

1 · A value stream is an ordered step structure

A value stream is a sequence of steps, and each step can encapsulate a child value stream (hasValueStream) — the structure is fractal. The steps form a refinement pipeline: each step's output is the next step's input, which is exactly why the order is fixed.

flowchart LR
  A["step n"] -->|"handoff artifact
refined one stage"| B["step n+1"]
  A -.hasValueStream.-> C["child stream
(its own ordered steps)"]
Generic step structure: ordered steps (output→input), each optionally encapsulating a child stream. Instances name the steps (Chorus: Shaping→Designing→Directing→Building→Proving; werk: pull→…→accept; athena: shape→forge→seed→validate).

2 · The generative cascade — the model is the root

Within a stream, the structure is generated top-down, not assembled bottom-up. Design the model spine first; it generates everything below.

flowchart TD
  VS["ValueStream (root — design first)"] --> ST["ValueStreamStep (+ child stream)"]
  ST -->|realized by| P["Product (atStep)"]
  P -->|hasDomain| D["Domain"]
  D -->|hosts| SV["Service"]

Every Product/Domain/Service mandates atStep → the value-stream/step model is the thing every other class points at. It is the generative root.

3 · The forge mechanism — one model, every surface

A generator reads the OWL/SHACL model and forges the surfaces deterministically. The generator is the high-leverage asset: the marginal cost of a new class's surfaces drops to ~zero once the shape exists.

OWL + SHACL (the model)
      │   forge (generate)
      ▼
  ┌── routes (the API) ──┬── MCP tool ──┬── skill ──┬── page (HTML) ──┐
  │      read it          │  call it     │ foundational│  render it      │
  └───────────────────────┴──────────────┴─────────────┴─────────────────┘

Re-runs on every model change (it re-forges, it doesn't pour-once). Cross-cutting concerns (security, authz, logging) are born pre-wrapped at the generated seam — a generator layer, not per-endpoint goodwill (see the Harness Integration Pattern).

4 · AX = JX, one source

The generated API is the single spine. AX (agent experience) and JX (human experience) are both surfaces over it — never separate data paths.

  • AX = MCP tool + API + skills (the agent's foundational actions).
  • JX = the UI renders from that same API → matches AX on data by construction: one source, no drift.
The acceptance test for an instance: the human walks the page while an agent calls the tool — same data, because it is the same API.

5 · The OWL / SHACL shapes (generic)

Coherence is enforced at the write-door (DAL, fail-closed) + SHACL as the generator's contract — not by OWL alone (open-world: the schema sits below the data, describing what could exist, not gating what does).

ValueStream / ValueStreamStep — the NEW spine shapes (this design's output; not yet in the model repo, spec'd here in full):

chorus:inStream        a owl:ObjectProperty ;   # EXISTS — membership, child→parent
    rdfs:domain chorus:ValueStreamStep ; rdfs:range chorus:ValueStream .
chorus:hasValueStream  a owl:ObjectProperty ;   # NEW — encapsulation, step→child stream
    rdfs:domain chorus:ValueStreamStep ; rdfs:range chorus:ValueStream .
chorus:stepOrder       a owl:DatatypeProperty ;  # NEW — steps are ordered
    rdfs:domain chorus:ValueStreamStep ; rdfs:range xsd:integer .

chorus:ValueStreamShape a sh:NodeShape ; sh:targetClass chorus:ValueStream ;
    sh:property [ sh:path chorus:label ; sh:minCount 1 ; sh:datatype xsd:string ] ;
    sh:property [ sh:path [ sh:inversePath chorus:inStream ] ; sh:minCount 1 ] .   # ≥1 step
chorus:ValueStreamStepShape a sh:NodeShape ; sh:targetClass chorus:ValueStreamStep ;
    sh:property [ sh:path chorus:label     ; sh:minCount 1 ; sh:datatype xsd:string ] ;
    sh:property [ sh:path chorus:stepOrder ; sh:minCount 1 ; sh:maxCount 1 ] ;
    sh:property [ sh:path chorus:inStream  ; sh:minCount 1 ; sh:maxCount 1 ] ;     # exactly one parent
    sh:property [ sh:path chorus:hasValueStream ; sh:maxCount 1 ] .                # 0..1 child stream

Product / Domain / Service — existing structural classes (full SHACL is the source of record in chorus-v2-shapes.ttl; summarized here):

Product   # label·comment·ownedBy·vision·status(enum)·gaps·atStep·hasDesignDoc·hasDomain (+hasChild, hasSurface, consumes)
Domain    # label·comment·ownedBy·atStep·status·gaps·hasDesignDoc·contains(content)·hasChild(structural) ·inverse hasDomain→1 Product
Service   # design: overview·asIs·toBe·implementationPlan·pathToClose·gaps·notInScope·dependsOn
          # operational (Borg): runtimeForm·port·healthAt·deployVia·emitsEvent ·inverse hosts→1 Domain
# contains = CONTENT membership (records); hasChild = STRUCTURAL recursion (domain→domain). Never interchangeable.

6 · How to instantiate (the recipe)

  1. Author the model — the value stream's steps + the product/domain/service classes for it, in OWL+SHACL (rigorously; the model is the source of record).
  2. Forge — run the generator: it emits routes + MCP + skill + page from the model.
  3. Seed — populate instances through the write-door (DAL).
  4. Validate — tree-vs-graph conformance; the generated output matches the model.
  5. Demo — the human walks the page while an agent calls the tool (AX=JX, same data).

Each instance names its own step verbs and its forge engine. The pattern is fixed; the names and the model are what change.
