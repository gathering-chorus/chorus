Chorus — State of the System

Kade + Jeff | 2026-04-06 morning | For Jeff, Mark Nakib, and Deb Majumdar

"It makes me feel almost seasick." — Jeff, describing the experience of watching the system run.

What Chorus Is

Chorus is a coordination protocol for AI coding agents. Three AI roles (Wren/PM, Silas/Architect, Kade/Engineer) and one human (Jeff) work together using a shared harness of hooks, gates, cards, briefs, nudges, and spine events. The product is the protocol itself — Gathering (Jeff's personal app) is the test harness.

In Birgitta Bockeler's taxonomy (Harness Engineering, April 2026): Claude Code is the builder harness. Chorus is the user harness — feedforward guides (CLAUDE.md, decisions, skills, session context) and feedback sensors (hooks, gates, team-scan, spine events) that steer agent behavior.

What Works

  • Card velocity: 5-15 cards/day across three roles
  • Cross-role awareness: gemba observation, nudge delivery, activity logging
  • Decision discipline: 100+ locked decisions, behavioral memory across sessions
  • The architecture is ahead of most published AI agent coordination systems
  • 4,458 of 4,460 tests passing

What's Breaking

The core problem: we break things as much as we fix them

Board pagination bug led to bulk SQL cleanup led to 1,410 duplicate position rows. Nudge shell script became Rust binary became TCC permission failures became revert to osascript. Watchdog enabled, created false alerts, disabled. Each step rational. The sequence destructive.

The Human Experience

The system doesn't have a quiet mode

Three AI roles run concurrently. Every action emits spine events, triggers hooks, fires nudges. Jeff sits in the middle of all terminals. Nudges land mid-sentence. Cards move while he's thinking. Chat threads open during strategic conversations. There is no difference between "Jeff is steering" and "Jeff is reflecting" — the system treats both as full throughput.

The harness engineering is real. The execution quality means the human is still the primary sensor — and the sensor is overwhelmed.

Bockeler's closing insight: "A good harness should not necessarily aim to fully eliminate human input, but to direct it to where our input is most important." Right now Jeff's input goes to TCC popups, relay coordination, and re-nudging stalled roles. Not where it's most important.

The Real Questions

  1. What is Chorus v1? What's the minimum set that works reliably and demonstrates the value? Right now everything is v0.8 — functional enough to use, fragile enough to break weekly.
  2. Can the system run quietly? The roles need to be able to work without Jeff feeling every heartbeat. The system needs a mode where Jeff steers, not absorbs.
  3. How much coordination tooling serves the product vs. serves itself? Board, briefs, nudges earn their keep. Spine events, workflow manifests, standards surfaces — are they making the product better or making the process more observable at the cost of more things that can break?
  4. What can Mark and Deb actually touch? The protocol is portable. The wiring is not. Extracting Chorus from Jeff's machine is 3-4 config refactors, not an architecture change — but it's real work nobody has prioritized.
