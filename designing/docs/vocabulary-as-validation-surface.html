Vocabulary as the Validation Surface

A domain's vocabulary, declared as data, is the single source the system generates from and validates against. One surface, three jobs.

The one idea. In Chorus, a "vocabulary" is a domain's governed set of terms — its classes, its fields, its allowed values, its expected shape — declared as data in the OWL. That declaration is not documentation about the system. It is the system's contract: owl-api generates the API/MCP/monitors from it, and the same declaration is the surface every write is validated against. Change the vocabulary → the generated interface and the enforced rules move together, by construction.

The loop: Declare → Generate → Enforce

Every vocabulary does three jobs from one declaration. A classic data dictionary does only the first (it describes). Ours is executable in both downstream directions.

01 · DECLARE

Vocabulary as data

An owl:Class + its sh:NodeShape + governed registry rows. The dictionary: terms, fields, types, allowed values, expected grain.

→
02 · GENERATE

Surface from the model

owl-api reads the vocabulary and projects the API, MCP tools, and monitors. No hand-written routes — the surface is the vocabulary, rendered.

→
03 · ENFORCE

The validation surface

The same declaration is what every write/event is checked against. SHACL shapes, enum constraints, grain rules — validation projected, not hand-coded.

The validation-surface insight. Validation today is scattered — a regex here, an if there, a hook that re-implements a rule the model already knows. When the vocabulary is the surface, there is one place a rule can live: the declaration. allowedValue compiles to sh:in. edgeTargetClass compiles to sh:class. A required field is sh:minCount 1. An event's expected cardinality is a grain rule. The validator is generated from the dictionary, so the thing you read and the thing that's enforced can never drift apart.

The first two shipped today (#3489 / #3494). The third (EmitContract) is the vocabulary being authored now — and it's where the validation-surface idea pays off most, because the spine is the noisiest, least-governed data we have.

How we leverage it across Chorus

1 · Every surface generates from a vocabulary

The owl-api fan-out (#3494) generalizes: any domain that declares its vocabulary gets its API, MCP tools, and monitors projected — no per-domain hand-coding. The system's entire interface surface becomes a projection of the declared model. (This is the API-side of "the whole model renders as a conceptual map.")

2 · The spine-straighten — vocabulary as cleanup tool

The spine today is the counter-example: 4 storage patterns, 3 naming conventions, ~71 distinct events in werk alone, ~3.5% unparseable lines. No governing vocabulary → silent drift. The EmitContract declares one canonical vocabulary (<verb-or-domain>.<event>, with grain + required fields + plausible-ts), then we conform emitters vertebra-by-vertebra and retire the bare/non-verb/redundant events (blast-radius → ∅). werk is vertebra one (#3495). Incremental only — the spine feeds every gate, metric, gemba, and pulse; a big-bang blinds the system.

3 · It is the change-management loop, pointed at our own data

This is the deepest leverage. Drift in Chorus is not a drift problem — it's an absent change-management problem. The vocabulary closes the loop:

Change-mgmt legVocabulary mechanism
Declare canonical modelThe vocabulary is the canonical declaration (OWL + shapes + grain)
Propagate / reconcileConform emitters & surfaces to the vocabulary (generate-from-model; vertebra-by-vertebra)
VerifyThe validation surface — every write/event checked against the declaration; off-vocab = loud failure

N hand-built drift detectors collapse into one mechanism: declare it, generate from it, validate against it. That's consistency by construction, with the gate as a backstop rather than the strategy.

4 · One place to change a rule

Want to add an allowed test type, secure an endpoint, tighten an SLO, ban an event name? It's a model write — the generated interface and the enforced validation both move. No code change, no flaky deploy, no rule re-implemented in three places that quietly diverge.

What it looks like — the declaration

A vocabulary entry carrying all three jobs at once (PropertyKey shown; EmitContract grain-spec is the same shape):

# DECLARE — the dictionary entry (data, not code) chorus:pk-testType a chorus:PropertyKey ; chorus:keyName "testType" ; chorus:appliesToClass chorus:Test ; # where it's valid chorus:keyValueType "string" ; chorus:allowedValue "unit","integration","api","ui","e2e" . # the enum # GENERATE → owl-api projects the Property/PropertyKey CRUD surface # ENFORCE → allowedValue compiles to sh:in; a Property with # propertyKey "testType" + value "banana" is REJECTED at write, # by a validator nobody hand-wrote.

The discipline that keeps it honest. The vocabulary only stays the single surface if rules live nowhere else. Every hand-coded validation that duplicates a declarable rule is a future drift point. The work is continuous: as each domain's vocabulary lands, the scattered checks it subsumes get retired (with a retirement gate), so the declaration stays authoritative.

Why this is the right shape for Chorus

It's your Staples canonical-model / data-dictionary lineage — made executable. A data dictionary that doesn't just describe the data but builds the interface to it and guards every write against it. Three domains already prove it (config, domain-APIs, and now the spine). The leverage compounds: each vocabulary we declare removes a class of hand-coded surface and a class of silent drift at the same time.
