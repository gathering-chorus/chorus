Blocking Rules Audit

2026-05-10 · Wren · what stopped us today, by failure shape

Today every meaningful action hit a rule designed for a specific past failure. Several of the rules cross-fire on unrelated work, conflate distinct concepts, or enforce stale ceremony. The cumulative friction made simple changes infeasible and turned the substrate honest about its own brittleness. This catalog names every blocker, what it was designed for, what it actually blocked, and groups them by failure shape so the addressing pattern is per-class, not per-incident.

Failure shapes (group by pattern)

Group A — over-broad scope: a rule that should fire narrowly fires broadly because its detection is keyword/path/heuristic-based not semantic. Examples: #2 (cross-fires on peer-role fix cards), #5 (treats gate-pass comment as self-accept).
Group B — concept conflation: a rule treats two distinct acts as the same. Examples: #5 (gate-pass-comment vs cards-done), #8/#9 (rebase-resolution vs new-commit).
Group C — wrong altitude: a rule designed for a workflow happy-path fires during recovery. Examples: #8 (mid-rebase add can't go through queue), #4 (Jeff-authorized one-time edit needs special signal not just consent).
Group D — stale ceremony: a rule enforces discipline that downstream is hollow anyway. Examples: gate-chain comment ceremony (5 boilerplate gate-pass comments per card) blocking work the new substrate's spine-event check makes ceremonial.
Group E — blocking on someone else's flake: the rule fires on a real failure but the failure isn't in the role's scope to fix. Example: #6 (chorus-hooks test flake blocks all roles' /acp until someone else's domain owner fixes it).
Group F — proof-of-discipline cost: the rule forces a precondition action (run git log, query Loki, post a comment) that's genuinely useful sometimes but wallpaper most of the time. Examples: #2, #3.

Addressing approach by group

Group A — narrow detection: for memory_gate cross-fire, scope is_fix_card() to the role's own WIP card only, not any peer's. For self-gate detection, distinguish cards comment (writes a comment) from cards done / chorus_acp (mutates state). Specific Bash patterns instead of NLP heuristic.
Group B — split rules: DEC-048 self-accept-block applies to cards done + chorus_acp only. Gate-pass comments on own cards are explicitly allowed (single-PM single-architect team has no other passer; the alternative is "the gate never passes"). Rebase-resolution staging gets a recognized git-queue subcommand (git-queue resolve) that allows mid-rebase add without firing the new-commit branch-check.
Group C — recovery mode: recognize rebase-in-progress (.git/rebase-merge/ exists) and let raw git through for that scope. Recognize Jeff-authorized one-time-edits via an explicit env var or session marker; today the auto-mode classifier ignores his "yes" because it's text in chat, not a flag.
Group D — retire ceremony: when the new substrate (spine event chain via show-gate.sh, MCP step events) becomes the source of truth, the comment-gate chain becomes optional. The transitional fallback Wren added in accept_gate.rs already handles this; the cleanup is decommissioning the comment-shape once chorus_demo MCP ships and emits structured step events.
Group E — flake handoff protocol: any role whose /acp is blocked by an unrelated test failure can mark #[ignore] on the test in the role's werk + file the flake card for the domain owner. Pre-commit hook accepts #[ignore] as a valid bypass when accompanied by a // FLAKE-FOLLOWED-UP-IN-#NNNN comment. Today this would have unblocked /acp #2864 immediately.
Group F — proof-of-discipline gating: reduce surface to the cases where the precondition genuinely matters. log_first_gate should fire on bug-fix cards (where logs are likely root-cause-revealing), not architecture cards (where logs are just noise). memory_gate's git-history-check could just do the git log automatically and inject a tiny summary, instead of demanding the agent prove it ran the command.

What this would change for tomorrow

  • Self-gate-pass on Wren cards lands on first try (group A + B fix)
  • chorus_demo MCP build lands without cross-fire from peer fix cards (group A fix)
  • Test flakes from chorus-hooks don't block /acp across the team (group E fix)
  • Mid-rebase resolution proceeds through git-queue cleanly (group C fix)
  • Gate-chain ceremony retires once spine-event chain is enforced (group D fix)
  • Edit cycles aren't dominated by hook-prerequisite proofs (group F fix)

What this does NOT propose

  • Disabling any of the rules outright. Each catches a real anti-pattern.
  • Adding more agent-side discipline. The rules exist because that discipline failed.
  • Cards for each fix individually before triage. Group-level addressing first; per-card scoping after pattern alignment.
