Clearing — Value-Stream Design — the Directing-step instance; the team’s I/O layer

Wren · 2026-06-21 · INSTANCE of value-stream-template.html (read that for the generic pattern) · sibling of athena / borg / loom / werk · Directing step · DRAFT 0.1 (verb-family ideation, card #3545)
Scope. This is Clearing’s value stream — one instance of the api-driven value-stream template. Clearing is the team’s I/O layer: how the team and the running system talk and track work. Three faces — the board (input: what we choose to do), the comms (output: how it reaches a session), and the spine (record: what happened). Model of record per the product tree: “Clearing — interaction layer — messages, streams, the spine event log, and the kanban board.”
Clearing is a SUBSTRATE, not a line or a loop — it’s what the other streams plug into. Werk/Athena/Convergence are lines; Loom/Borg are loops. Clearing is neither: it’s the I/O fabric every other product borrows. Just as every product borrows Werk for its prove-and-land pipeline, every product borrows Clearing for its messages — the standard verbs nudge / surface / inject are Clearing’s, and the trace/emit every verb carries land in Clearing’s spine. It’s the seat where the team and the machine meet: your board choices and the products’ emits land in the same log.
⚠ As-is vs to-be. Clearing is the most built of the new streams — the board (Vikunja + the cards CLI), the spine (append-only ops/logs/*.jsonl + the messaging API), and the comms (the MCP nudge path, chorus-inject delivery) all run today. What’s to-be is the verb-family framing — naming the board/comms/spine operations as a coherent clearing-* family + a chorus_clearing orchestrator, and modeling them so the surfaces generate. The mechanisms exist; the model and the family naming don’t yet.

1 · The clearing-* verbs — three faces

Clearing isn’t a single cycle; it’s three coherent verb-sets over one substrate. Board (how we track) · Comms (how we talk) · Spine (the record).

VerbFaceWhat it doesFramework anchor
clearing-cardboardFile a work item onto the board.Kanban card
clearing-chunkboardGroup cards into themed buckets.swimlane / epic
clearing-tagboardApply the cross-axis labels (domain / type / origin / sequence).classification
clearing-prioritizeboardRank chunks by value — coarse, “which themes.”WSJF (Cost of Delay ÷ Job Size)

The Directing step hasValueStream this set (the template’s fractal recursion); a chorus_clearing orchestrator is the entry, though many verbs are also leaf-invoked (the cards CLI, the nudge MCP). Clearing is less “one trigger sequences the verbs” than Werk — it’s a library of I/O the other streams call.

prioritize ≠ sequence — different grains, different axes. clearing-prioritize ranks the NODES (chunks, by economic value — WSJF). clearing-sequence orders by the EDGES (cards, by dependency — a topological sort). They’re deliberately separate: a high-value card can be blocked behind a low-value prerequisite — value-ranking doesn’t respect edges; topo-sort does. (Same node/edge split as athena-model vs athena-wire, pointed at the board.) Mushing them is how a “priority list” turns into a backlog that ranks 270 cards and means nothing.

2 · The engine — the board-flow over the event-sourced spine

Clearing’s heart is a Kanban flow system whose system of record is an event-sourced log. Cards are pulled through workflow states under WIP limits; every board op and every comms op appends an immutable event to the spine; the board’s current state is a projection built by folding those events. The board you see is a read-model; the spine is the truth.

flowchart LR
  IN["INPUT — board: card / chunk / tag / prioritize / sequence"] --> ST["state: Next → WIP → Done (WIP-limited pull)"]
  ST --> EMIT["every op appends an immutable event"]
  PROD["every product's verbs — trace / emit"] --> EMIT
  EMIT --> SPINE[("SPINE — append-only event log (system of record)")]
  SPINE --> PROJ["projections: the board view · pulse · clearing-wip"]
  SPINE --> OUT["OUTPUT — comms: nudge / surface / inject / gather"]
  OUT -.delivers to.-> SESS["a role's session / Jeff"]
Clearing — input (board) + record (spine, event-sourced) + output (comms). The team’s board choices and the products’ emits land in ONE log; everything else (board view, pulse, metrics) is a projection over it.
  • The spine is the one primitive (Young’s CQRS / Kreps’ The Log): a totally-ordered, append-only, replayable event stream. Logs / metrics / traces are three lenses on one set of emissions — clearing-tail reads it raw; Borg aggregates it; the board renders it.
  • WIP-limited pull, not push (Anderson’s Kanban): capacity drives what starts — “stop starting, start finishing.” The Cumulative Flow Diagram is the diagnostic (cycle time, WIP, queue buildup).
  • Delivery is the weak spot (the load-bearing finding): clearing-surface — getting a message into a session — has been the cross-cutting flaky part, because every product borrows it. #3544 (deliver as an ops-nudge, not a self-echo) + #3548 (don’t re-fire on the land run) are the two fixes that hardened it. Persist is easy; deliver is the engineering.

5 · Sequencing — name the family, then model the surfaces

  1. Author the value-stream spine — value-stream-clearing + the three faces’ verbs as ValueStreamSteps, in the model. Land it (the mechanisms exist; this is the naming/modeling layer).
  2. Model the event grammar — re-seat spine-events.json as the events domain so emitters + registry rows generate together (kills the phantom class, athena-wire-before-make applied to events).
  3. Harden clearing-surface — the delivery half (#3544/#3548 landed; the run-state-finalize + the “from unknown” sender-label are the remaining polish). Delivery reliability is the whole-team payoff because every product borrows it.
  4. Build clearing-wip — enrich the existing WIP readout (pulse board.wip) with the live pipeline strip from the spine emits (Kade’s presentation lane reading the spine — the first tile projecting a trace).
  5. Generate the board view — the board render becomes a projection of the model + spine (no bespoke UI), the athena-make path applied to Clearing’s surfaces.

6 · Operating-model frames

  • Kanban (Anderson, from Toyota’s pull-signal) — how the board flows. WIP-limited pull, CFD diagnostics, “stop starting, start finishing.” = the board face.
  • WSJF (Reinertsen, Principles of Product Development Flow; SAFe) — prioritize. Cost of Delay ÷ Job Size — economic value ranking. = clearing-prioritize (the node-valuation axis).
  • Topological sort over a dependency DAG — sequence. Order by the edges, orthogonal to value. = clearing-sequence (the edge-ordering axis) + the coherence check.
  • Event Sourcing (Fowler; Young’s CQRS; Kreps’ The Log) — the spine. Append-only immutable log = system of record; state is a replayable projection; one log feeds many read-models. = the spine + clearing-tail.
Why “spine diet” and “how we talk” keep turning out to be the same problem: they’re the same store. Clearing is where the team and the running system meet — your messages and the products’ emits land in one event-sourced log. Fix the log’s grammar and you fix both the chat history and the observability feedstock at once.
