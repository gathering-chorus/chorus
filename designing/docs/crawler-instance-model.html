Crawlers as a domain of declared instances

The to-be model (#3069). Decompose the monolith into a registry of declared instances — like skills. Each instance: source → target → cadence → cost-tier. Wren, 2026-05-24.

This doc is the to-be only. The as-is is already documented — it rides on:
  • crawler-service-design.html — component anatomy, the four substrates, consumers, gaps (the older graph-hydrator wave, #2817/#2827).
  • crawler-dependency-map.html (#3071) — VERIFIED writers→outputs→consumers: 7 outputs / 4 stores.
  • data/athena/tree.json — chorus:service-crawler with produces/consumedBy + consumes/dependsOn on 6 domains.
This card does not redraw the as-is. It delivers the instance contract (AC2) + the collector→instance decomposition (AC3).

Why decompose — the structural argument

The monolith couples three things that have nothing in common: cadence (code metadata changes constantly; the ontology rarely; the 35–55s RDF cross-graph join is expensive every time), write-home (domain knowledge belongs in the graph, not a flat messages blob nobody reads), and read-discipline (mentions/logs are computed every sweep on spec, whether anyone asks or not). fetchCrawl runs all 11+ collectors × 27 domains on one cadence, on the shared event loop, against one SQLite file — which is the storm. Tuning the interval can't fix a structural coupling. Splitting the unit can.

A registry of instances also gives each facet one home and one owner, instead of scattering across the codebase with no domain model.

AC2 — the crawler-instance contract

An instance is a declaration, parallel to a skill. The engine reads declarations from a registry and runs each per its own terms — it does not hard-code a sweep.

{
  "name": "code-scan",
  "source":   { "type": "git/fs", "ref": "platform/**, roles/**" },
  "target":   { "store": "graph", "home": "chorus:knowledge" },   // RIGHT home, not a messages blob
  "cadence":  { "trigger": "file-watch", "minInterval": "60s" },  // own cadence
  "costTier": "cheap",                                            // governs how often it MAY run
  "mode":     "scheduled",                                        // scheduled | pull-on-demand
  "domains":  ["chorus:code", "chorus:tests"],                    // which Athena domains it serves
  "write":    { "busyTimeoutMs": 5000, "failVisible": true }      // write-resilience (the exit=5 class)
}

How the engine runs instances

  • Registry — instances are declared (like skills/), discoverable, owned. The engine enumerates them; there is no monolithic fetchCrawl.
  • Scheduled instances run on their declared trigger/cadence; costTier caps frequency (a heavy instance may not run on file-watch).
  • Pull-on-demand instances do not run on a clock at all — they compute at query time, when a consumer actually asks. This is the "pull not push" fix for use-thin work.
  • Write path — every instance's writes set busy_timeout and surface failures (no suppressed stderr). This closes the exit=5 / SQLITE_BUSY class traced 2026-05-24.
  • Supersedes the graph-hydrator framing (#2817/#2827): the hydrator becomes simply the target.store: "graph" branch of the write path, shared by every graph-homed instance — not a separate bespoke subsystem.

What the split buys

  • "Much more frequent" for the cheap slices (code-scan, cards) without ever dragging the 35–55s rdf-join into the frequent path — Jeff's ask, structurally.
  • No storm: the heavy instance runs rarely; the pull instances run only when asked; nothing fires all collectors at once on the shared loop.

How the point-fixes fold in

  • #3055 (off-spine): mentions becomes pull-on-demand — the heavy mention scan leaves the sweep entirely.
  • #3066 (collectRdf 35–55s): rdf-join is a rare-cadence heavy instance — never in the frequent path.
  • Index watermark: spine-index reads incrementally (watermark/tail), not whole-file (the #3067 lesson made structural).
  • Search-on-spec: the pull instances (mentions, logs) compute at query time, not pre-materialized every sweep.
  • busy_timeout write-resilience: baked into every instance's write path — closes the exit=5 / SQLITE_BUSY death class.
Must-handle constraints (carried verbatim from #3071's verified map — name in AC, don't discover mid-build):
  • Circular dep #2323: collectMentions reads messages_fts → writes a crawler row → reindex → read again. Making mentions pull-on-demand (compute, don't store) is the clean resolution — it stops writing the row back, breaking the cycle. This is the single most load-bearing reason the split must include a pull mode.
  • traces = clean cut: 108/min, 144K of 301K rows, zero readers. Cut or sample; do not make it an instance.
  • chorus_files = fold, not cut: no external reader, but it's the hydrator's own retry/dedup state. Fold it into the code-scan instance's internal state; do not retire standalone.
