Domain Model — As-Is vs To-Be

2026-05-09 · Wren · OWL + SHACL · context: #2851 / #2847 hierarchy cleanup

The current model has three peer subclasses of Domain (SubProduct, SubDomain, CollectionDomain) plus an overloaded chorus:hasDomain predicate that conflates containment and dependency. The proposed model collapses Domain to one recursive class, separates dependency from containment, and reduces SHACL surface to a single shape. SubProduct stays as a distinct type because product offerings aren't domains — they depend on them.

Layer note: chorus:CollectionDomain is removed entirely from Chorus. Photos, Music, Stories, Books, People, Property, etc. are Gathering's data domains, not Chorus's concern. They're modeled in Gathering's own ontology. Chorus only models how the team works (SubProducts depending on Domains); the Gathering data layer is downstream and doesn't need a Chorus-side type.

Shape — at a glance

As-Is today

Domain (abstract) ├── SubProduct — Loom, Werk, Athena, Borg, Clearing, Quality ├── CollectionDomain — Photos, Music, Stories, Books (Gathering, wrong layer) └── SubDomain — loom-principles, spine-service, cards-service, nudge, ... Edges: SubProduct --hasDomain--> SubDomain (used as containment in data) SubProduct --hasDomain--> Domain (per Document SHACL) Product --hasDomain--> Domain Product --hasSubProduct--> SubProduct

To-Be proposed

Domain (concrete, recursive) └── may contain other Domains (skill → nudge, ...) SubProduct (separate class) └── dependsOn → Domain (one-to-many; not containment) Product → SubProduct → Domain → Domain → ... CollectionDomain removed — Gathering's concern, not Chorus's. SubDomain goes away as a class. "Has children?" becomes a query, not a type.

OWL — To-Be

# One recursive concrete class. SubProduct stays separate.
# CollectionDomain removed — Gathering's data domains are not Chorus's concern;
# they live in Gathering's ontology, not chorus.ttl.
chorus:Domain a owl:Class ;
    rdfs:label "Domain" ;
    rdfs:comment "A bounded context with its own data, handlers, ontology, and routes. May contain other Domains. May be depended upon by SubProducts." .

chorus:SubProduct a owl:Class ;
    rdfs:label "Sub-Product" ;
    rdfs:comment "A named product offering — Loom, Werk, Athena, Borg, The Clearing, Quality. Depends on Domains; does not contain them." .

# Containment is on Domain — recursive. (Renamed from hasDomain to disambiguate.)
chorus:hasChild a owl:ObjectProperty ;
    rdfs:domain chorus:Domain ;
    rdfs:range  chorus:Domain ;
    rdfs:label "has child" ;
    rdfs:comment "Parent Domain holds this child Domain. Recursive. Used when a Domain is a collection (skill→nudge, photos→tag)." .

chorus:partOf a owl:ObjectProperty ;
    owl:inverseOf chorus:hasChild ;
    rdfs:domain chorus:Domain ;
    rdfs:range  chorus:Domain ;
    rdfs:label "part of" ;
    rdfs:comment "Inverse of hasChild for query convenience." .

# Dependency edge from SubProduct → Domain. New predicate to replace hasDomain in this slot.
chorus:dependsOn a owl:ObjectProperty ;
    rdfs:domain chorus:SubProduct ;
    rdfs:range  chorus:Domain ;
    rdfs:label "depends on" ;
    rdfs:comment "SubProduct relies on this Domain at runtime or by data. Many-to-many — multiple SubProducts may depend on the same Domain." .

# All facet predicates widen from SubDomain → Domain.
chorus:hasService a owl:ObjectProperty ;
    rdfs:domain chorus:Domain ;
    rdfs:range  chorus:Service .
# (...same widening for hasPage, hasPersistence, hasIntegration, hasPipeline,
#  hasLogSource, hasGap, hasActor, hasScenario, hasContract, hasCodeFile.)

SHACL — As-Is

Searched the entire ttl for shapes targeting the Domain hierarchy. There are none. The only SHACL references to SubDomain are sh:class constraints inside other shapes (Document, AuthBoundary) — they constrain the target of a property to be a SubDomain, not the SubDomain itself.

ShapeTouches Domain hierarchy?How
chorus:DocumentShapeindirectlysh:path chorus:hasDomain ; sh:class chorus:SubDomain — Document must tag a SubDomain
chorus:AuthBoundaryShapeindirectlyTwo sh:class chorus:SubDomain constraints on edges
(no shape targets Domain / SubDomain / SubProduct / CollectionDomain itself)—49 SubDomain instances are unvalidated structurally — owner, label, parent are all best-effort.

This is a finding on its own: the Domain hierarchy is the core of the model and has no SHACL well-formedness constraint. The to-be should add one.

Implications worth naming

  • Migration touches all 49 SubDomain instances. Re-type to Domain, optionally add hasChild/partOf where containment exists.
  • Migration touches all chorus:hasDomain edges from SubProducts. Each becomes either chorus:dependsOn (most) or chorus:hasChild (none, since SubProduct has no Domain children in this model).
  • 11 facet predicates widen from SubDomain to Domain. Existing data on SubDomain instances stays valid because every SubDomain becomes a Domain.
  • CollectionDomain class disappears from chorus.ttl entirely. Gathering's data domains (Photos, Music, etc.) move to Gathering's ontology — Chorus stops trying to know about them. Reduces chorus.ttl scope to "how the team works," not "what the team manages."
  • SHACL gains structural validation for the first time. Currently no shape constrains Domain instances — owner/label/comment are best-effort.
  • Document and AuthBoundary shapes need their sh:class retargeted. Three references in total — small surface.

Open questions

  • Keep chorus:hasDomain as the predicate name (just narrow its meaning to containment) or rename to chorus:hasChild? Rename is honest; keep is cheaper migration.
  • Is chorus:dependsOn typed (data dep / runtime dep / ontology dep) or one flat predicate? Recommend flat for v1, type later if needed.
  • Where does Product fit in the to-be? Product hasSubProduct SubProduct stays unchanged — but Product hasDomain Domain exists today and would also become Product dependsOn Domain? Or does Product not depend on Domains directly, only through its SubProducts?
