Build + Deploy — Service Design

2026-05-07 — Silas (#2775). Names build and deploy as two first-class domains separate from version control. Origin: Jeff (2026-05-07): "they are real domains silas — we must have them and they must work with invariant execution. it's not a nice to have." Today both are implicit — build-signed.sh, app-state.sh, ~/.chorus/bin/, the manifest-that-doesn't-exist — scattered scripts and conventions without a domain home, an owner, or a contract. This doc names the contract.

Designed target state. A version of the running system lands when, and only when, three things hold: (1) build is invariant — the same source commit produces the same artifact, every machine, every time; the artifact is named, hashed, and recorded in a manifest; (2) deploy is invariant — the same artifact reaches the same running state; schema is reloaded, cdhash is registered, kickstart fires; (3) /acp invokes both atomically — version-control merge → build → deploy is one transaction with one spine event, not three manual steps. Refusal taxonomy at each domain boundary is strict: build refuses on non-reproducible source state; deploy refuses on artifact-not-built or running-state-not-reached. Coordination state — board status, role-state, branch policy — never refuses build or deploy. Today's code does not yet match this contract — see As-Is below.

Overview

Build domain — owns the path from source to artifact. Inputs: a git ref + a build target name. Outputs: a versioned, hash-stable artifact in a known location, plus a manifest entry recording {commit, target, artifact_hash, build_time, machine}.

Deploy domain — owns the path from artifact to running state. Inputs: a manifest entry. Outputs: the artifact installed at its deploy location, the running process restarted (kickstart, schema reload, cache invalidation), plus a spine event recording {artifact_hash, deployed_at, role}.

Boundary — build is a pure function of source. Deploy is a pure function of artifact. The two domains never share state; they communicate only through the manifest. This is what makes invariance possible — each domain has one input class and one output class, and both are content-addressable.

Build domain

Purpose

Source → artifact. Reproducibly.

Owner

Silas (DEC-022 derivative; build is upstream of deploy, deploy is Silas, build follows the artifact-producing concern).

Inputs

Git commit SHA. Build target name (one of: chorus-hooks, chorus-inject, chorus-hook-shim, chorus-api, future targets).

Outputs

Artifact at a known path with a content-addressable name. Manifest entry. build.completed spine event with {commit, target, artifact_hash, machine}.

BDD scenarios

Scenario: same source produces same artifact
  • Given commit abc123 and target chorus-hooks
  • When the build runs on machine A
  • And the build runs on machine B
  • Then both produce artifacts with the same hash
  • And both write identical manifest entries (modulo machine and build_time)
Scenario: source drift refuses build
  • Given the working tree has uncommitted changes
  • When build-signed.sh chorus-hooks runs
  • Then it refuses with source-not-clean and names the dirty paths
  • And no artifact is written
  • And no manifest entry is recorded
Scenario: artifact already built is reused
  • Given an artifact for commit abc123 target chorus-hooks already exists in the manifest
  • When the build runs again for the same commit + target
  • Then the existing artifact is verified by hash
  • And no rebuild occurs
  • And the manifest is unchanged
Scenario: build emits one spine event per artifact
  • Given a successful build of commit abc123 target chorus-hooks
  • When the build completes
  • Then exactly one build.completed spine event is emitted with {commit, target, artifact_hash, machine}
  • And the event hash matches the artifact hash

Deploy domain

Purpose

Artifact → running state. Invariantly.

Owner

Silas (DEC-022).

Inputs

A manifest entry naming the artifact to deploy. The running system's current deploy state.

Outputs

Artifact installed at deploy location. Running process restarted with new artifact in place. deploy.completed spine event with {artifact_hash, target, deployed_at, role}.

BDD scenarios

Scenario: same artifact produces same running state
  • Given artifact chorus-hooks with hash H1 is in the manifest
  • When deploy runs on machine A
  • And deploy runs on machine B
  • Then both run paths point to a binary with hash H1
  • And both emit deploy.completed events with the same artifact hash
  • And the running process on both machines reports cdhash H1
Scenario: deploy refuses on artifact-not-built
  • Given the manifest has no entry for commit abc123 target chorus-hooks
  • When app-state.sh deploy chorus-hooks --commit abc123 runs
  • Then it refuses with artifact-not-built
  • And names the missing manifest entry
  • And no running state changes
Scenario: deploy invalidates MCP schema cache
  • Given a deploy of chorus-api with a new MCP schema
  • When deploy completes
  • Then the running chorus-api is kickstarted
  • And the MCP schema cache is marked stale
  • And clients connecting after deploy see the new schema
Scenario: deploy preserves cdhash-to-TCC binding
  • Given a deploy of a signed binary to ~/.chorus/bin/
  • When deploy completes
  • Then the binary's cdhash is unchanged from the artifact in the store
  • And TCC grants for AppleEvents (#2734) survive the deploy
Scenario: deploy emits one spine event per artifact
  • Given a successful deploy of artifact H1 for target chorus-hooks
  • When deploy completes
  • Then exactly one deploy.completed spine event is emitted with {artifact_hash, target, deployed_at, role}
Scenario: rollback uses prior manifest entry
  • Given the manifest has a prior deploy entry H0 for target chorus-hooks
  • When app-state.sh rollback chorus-hooks runs
  • Then deploy runs against H0 from the manifest
  • And the running process reports cdhash H0
  • And a deploy.completed event is emitted with H0

Build never reads run-path state. Deploy never reads source. Each domain's input is the prior domain's named output. This is what allows invariance to be checked at each boundary independently.

Surfaces

Write surfaces

  • Build: build-signed.sh <target> (CLI), called from /acp and CI.
  • Deploy: app-state.sh deploy <target>, app-state.sh rollback <target> (CLI), called from /acp.

Read surfaces

  • Manifest read: cat ~/.chorus/manifest.json or chorus-manifest <target> --commit <sha> (planned).
  • Spine: build.completed, deploy.completed events queryable via Loki / spine endpoint.

Path to close

  1. Manifest schema + writer. Define ~/.chorus/manifest.json shape. build-signed.sh writes one entry per build. (Card: future, sequenced from this design.)
  2. Build invariance check. A scheduled job (or pre-deploy gate) cross-compares manifest entries by {commit, target} across machines; any hash mismatch is a build.divergence spine event. (Card: future.)
  3. Deploy spine events. app-state.sh deploy emits deploy.completed with {artifact_hash, target, deployed_at, role}. Same for kickstart-after-source-change (#2780) and cdhash-after-rebuild (#2734). (Card: #2780 absorbs this.)
  4. /acp atomic. /acp invokes chorus_commit → chorus_build → chorus_deploy as one MCP call with typed refusal taxonomy. (Card: #2774, re-sequenced to follow this design.)
  5. Working-tree recovery. chorus-werk-sync detects detached canonical and either emits a recovery hint or repairs via plumbing. (Card: #2779.)

To-Be — designed target state

OSS shipping shape: build emits a versioned, hash-stable artifact; a lockfile (chorus.lock or similar) names the exact installed version per target; install fetches that exact artifact; deploy invokes install + kickstart.

Open question (the design surfaces, doesn't decide): committed binaries inside the repo, or lockfile-asserted with binaries fetched at install time?
  • Committed binaries. Repo IS source-of-truth. git pull is half of deploy. Repo size grows with every artifact rev.
  • Lockfile-asserted. Repo stays small. Install reaches an external store. Network dependency at deploy time. Casual-deletability of the install target preserved.
Today's evidence leans toward committed binaries (Jeff 2026-05-07: "code-and-binaries-together") but the trade-off matters across cross-machine and disk-budget concerns.

Gaps to close

P1. Manifest does not exist.
P1. Cross-machine artifact divergence is undetected.
P1. Deploy emits no spine event (#2780).
P1. /acp is not atomic (#2774).
P1. Working-tree recovery has no clean tool path (#2779).
P2. TCC cdhash binding outside repo (#2734) — design must say keep, collapse, or formalize.

Implementation Plan

Build (sequenced after design lands)

  • Manifest schema + writer.
  • Cross-machine divergence detection.
  • chorus_build MCP wrapper (typed refusal: source-not-clean | build-fail | hash-collision).

Deploy (sequenced after design lands)

  • deploy.completed spine event from app-state.sh deploy.
  • MCP schema reload integration (#2780).
  • chorus_deploy MCP wrapper (typed refusal: artifact-not-built | install-fail | kickstart-fail).
  • Rollback as named operation (today: implicit via git revert + redeploy).

/acp absorption

  • #2774 re-sequenced: /acp invokes chorus_commit → chorus_build → chorus_deploy as one MCP transaction.
