Build + Deploy — Service Design

Rewritten clean 2026-06-08 — Silas (#3285), conformed to the werk-verb contract (ADR-032). One current version; prior strata (2026-05-07 #2775, 2026-05-11 #2898/#2900, written around the retired chorus-build/chorus-deploy + building-pipeline.yml model) are in git history, not here.

The verb law (Jeff, 2026-06-08). Build and deploy have exactly two callable verbs, each a Rust binary: werk-build and werk-deploy. They sit in the canonical werk-verb set; the exact roster and the --atomic execution model are ADR-032 + ADR-037, not restated here (one open count question — whether werk-push is a distinct verb or part of werk-commit — is Jeff's call, and doesn't touch build/deploy). For build and deploy specifically the rule is fixed regardless: no other build or deploy surface exists.
  • One build verb. Every build — demo, test, prod — is werk-build. Nothing else builds.
  • One deploy verb. Every deploy — demo, test, prod — is werk-deploy; the target is an argument, not a different tool.
  • One engine. werk-build/werk-deploy are the engines. The MCP tools wrap them under those exact names; there is no parallel CLI or script.
  • One artifact, proven by reproducible build. What you demo is what ships — delivered by REBUILDING from merged main and PROVING the prod cdhash equals the demo'd cdhash (#3316; build is a pure function of source, so identical source ⇒ identical cdhash). Promote is rebuild+prove, never a copy of the demo binary.

Two short synchronous halves with the human between. Half A builds the card once and presents the running variant, then stops — nothing held. The human's GO runs Half B, which rebuilds from merged main and proves the prod cdhash equals the demo'd cdhash (#3316) — the demo'd identity ships, by proof rather than by copy. The wait between holds no connection (survives minutes, hours, a reboot).

Build domain

Source → artifact, reproducibly. Owner: Silas. Verb: werk-build (Rust).

  • Inputs: a git ref (the card's werk) + a crate/target.
  • Outputs: a hash-stable artifact in target/release/<name>, plus a manifest entry recording {commit, crate, identifier, cdhash, sha256, build_time, builder_host, builder_role} (fields aligned to the build.artifact.hashed spine event).
Invariant — build is a pure function of source. The same source commit produces the same artifact on every machine, every time. Build refuses on non-reproducible source state. It never consults board status, role-state, or branch policy — coordination state cannot refuse a build.

Deploy domain

Artifact → running state. Owner: Silas. Verb: werk-deploy (Rust), with --target = test | prod (and env-up/env-down for the per-role variant).

  • Inputs: a built artifact (a manifest entry) + a target.
  • Outputs: the artifact installed at its deploy location, the running process restarted (launchctl kickstart), cdhash registered; a binary.deployed / deploy.completed spine event recording {artifact_sha256, target, installed_cdhash, deployed_at, role}.
Invariant — deploy is a pure function of artifact = PROMOTE, never rebuild. The artifact the human saw in the demo is the one that reaches prod. werk-deploy --target prod takes the already-built artifact and installs it; it does not rebuild from main. Rebuilding would ship something never demo'd — the integration trap. Deploy refuses on artifact-not-built or running-state-not-reached.

Retired (do not reintroduce)

These were the pre-ADR-032 sprawl — competing names/engines for build and deploy. All collapse into werk-build/werk-deploy:
  • chorus-build / chorus-deploy (bash CLIs) — the deploy engine's logic moves into Rust werk-deploy; the scripts are deleted.
  • chorus_build / chorus_deploy / chorus_env_up (MCP tool names) — renamed to werk-build / werk-deploy / werk-deploy env-up.
  • building-pipeline.yml — replaced by werk.yml (Half A) + werk-land.yml (Half B).
  • chorus_werk_land as a distinct verb — its work is just werk-merge + werk-deploy --target prod + werk-accept.
  • werk-finalize — folds back into werk-accept (the board-Done + teardown step).
  • prod blind rebuild-from-main — the unproven rebuild is retired: prod still rebuilds from merged main (Jeff's ruling 2026-06-09: promote = rebuild + prove, NOT copy), but the rebuild is now PROVEN against the demo'd cdhash (#3316 emits deploy.cdhash.matched/diverged). deploy_canonical stays as the prod path; since #3317 it is all-native Rust (bash chorus-deploy absorbed and deleted).

Completeness Contract (ADR-028)

Build-domain and deploys-domain are governed-record, adapter-backed domains (the "data" is git refs / artifact files / running processes). One source per concern: werk-build is the only path source→artifact; werk-deploy is the only path artifact→running. Same engine behind every call shape (MCP, workflow step, hand-invoke all exec the one Rust binary). Atomic per record class — build separate from deploy, both separate from version-control. Live actors, BDD scenarios, and records live in the graph: build-domain and deploys-domain.

Boundary & related decisions

Boundary (confirmed clean, 2026-06-08 coherence audit). Build/deploy mechanics are this doc's authority — what the prod deploy does (promote = rebuild from merged main + prove cdhash == demo'd, #3316), cdhash/signing, install location, rollback. The CI/CD design sequences the build+deploy steps but defers their mechanics here; the Version Control design owns the git lifecycle. Where CI/CD and this doc disagreed on the prod step, this doc is authoritative; both now state promote = rebuild + prove (Jeff's ruling 2026-06-09; Kade reconciled the CI/CD framing).

  • ADR-032 — verb contract (zero-dep Rust, cdhash, all-or-nothing rollback): the blueprint werk-build/werk-deploy build to.
  • ADR-037 — --atomic execution + the human-authorizes-prod gate (one gate, two doors): the canonical home of the --atomic model this doc references rather than restates.
  • ADR-038 — Rust as the go-forward substrate language; names the bash chorus-deploy as the largest port-debt item, paid by Open-work step 2 here.

Cross-checked against the Version Control + CI/CD service designs in the 2026-06-08 two-lens (SA↔DE) coherence audit. The prod-build strategy contradiction and the deploy_canonical retired-vs-live mislisting were the findings folded back here; the promote-as-copy framing itself was corrected to rebuild+prove on 2026-06-09 (Jeff), proven live by #3316.
