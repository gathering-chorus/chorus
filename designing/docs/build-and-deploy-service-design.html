Build + Deploy — Service Design

2026-05-07 — Silas (#2775). Names build and deploy as two first-class domains separate from version control. Origin: Jeff (2026-05-07): "they are real domains silas — we must have them and they must work with invariant execution. it's not a nice to have." Today both are implicit — build-signed.sh, app-state.sh, ~/.chorus/bin/, the manifest-that-doesn't-exist — scattered scripts and conventions without a domain home, an owner, or a contract. This doc names the contract.

Designed target state. A version of the running system lands when, and only when, three things hold: (1) build is invariant — the same source commit produces the same artifact, every machine, every time; the artifact is named, hashed, and recorded in a manifest; (2) deploy is invariant — the same artifact reaches the same running state; schema is reloaded, cdhash is registered, kickstart fires; (3) /acp invokes all three domains atomically — version-control merge → build → deploy is one transaction with one spine event, not three manual steps. Refusal taxonomy at each domain boundary is strict: build refuses on non-reproducible source state; deploy refuses on artifact-not-built or running-state-not-reached. Coordination state — board status, role-state, branch policy — never refuses build or deploy. Today's code does not yet match this contract — see As-Is below.
What "atomic" means today vs target. Three distinct slices — only the first is shipped:
  • commit slice (shipped 2026-05-07, #2782): chorus_commit / chorus_acp run verify-after sequenced steps with typed refusal at each step. Not atomic across domains — atomic within version-control only.
  • build + deploy MCPs (designed, not yet shipped): chorus_build and chorus_deploy don't exist as MCPs. Today's build is build-signed.sh from CLI; deploy is app-state.sh deploy from CLI. /acp does not invoke them.
  • three-domain rollback semantics (designed, untouched): if build succeeds and deploy fails, the design must say what happens to the merged commit and the new artifact. Today: nothing; partial-success states leave the system incoherent. The atomic-across-three-domains contract is structurally hard and not addressed by any current card.
The "atomic" claim in the Promise block is the designed target. Today's /acp is the commit slice only.

Overview

Build domain — owns the path from source to artifact. Inputs: a git ref + a build crate name. Outputs: a versioned, hash-stable artifact in a known location, plus a manifest entry recording {commit, crate, identifier, cdhash, sha256, build_time, builder_host, builder_role} (fields aligned to build.artifact.hashed spine event).

Deploy domain — owns the path from artifact to running state. Inputs: a manifest entry. Outputs: the artifact installed at its deploy location, the running process restarted (kickstart, schema reload, cache invalidation), plus a spine event recording {artifact_sha256, target, installed_cdhash, deployed_at} (today's deploy.artifact.verified; designed deploy.completed with role + extended payload).

Boundary — build is a pure function of source. Deploy is a pure function of artifact. The two domains never share state; they communicate only through the manifest. This is what makes invariance possible — each domain has one input class and one output class, and both are content-addressable.

Build domain

Purpose

Source → artifact. Reproducibly.

Owner

Silas. Reasoning: build is upstream of deploy in the path-from-source-to-running-state. Deploy ownership is Silas's per DEC-022 (operational responsibility). Build sits in the same vertical — artifact production is infrastructure, not feature work, and the boundary between build and deploy is a single owner's call to draw and enforce. Splitting build/deploy across roles introduces the same coordination overhead the design is built to refuse. (Note: DEC-022 names ops/deploy explicitly; build-as-Silas is the design's choice to keep the artifact-production-to-running-state vertical under one owner, not a literal DEC-022 clause.)

Inputs

Git commit SHA. Crate name (one of: chorus-hooks, chorus-inject, chorus-api, future crates). One crate may produce multiple binary identifiers (e.g., chorus-hooks crate ships both chorus-hooks and chorus-hook-shim binaries).

Outputs

Signed binary at a known path. Manifest entry with fields aligned to build.artifact.hashed spine event: {commit, crate, identifier, cdhash, sha256, build_time, builder_host, builder_role}. cdhash is the binding identity (per #2734); sha256 is informational and drifts with cargo+codesign timestamp behavior.

Deploy domain

Purpose

Artifact → running state. Invariantly.

Owner

Silas (DEC-022).

Inputs

A manifest entry naming the artifact to deploy. The running system's current deploy state.

Outputs

Artifact installed at deploy location. Running process restarted with new artifact in place. deploy.artifact.verified spine event with {artifact_sha256, target, installed_cdhash, deployed_at} (matches the contract test event shipped 2026-05-07). Designed: chorus_build + chorus_deploy MCPs exist; /acp invokes commit→build→deploy as one atomic transaction with rollback semantics across all three domains. Three-domain rollback is structurally hard and not addressed by any current card.

Build never reads run-path state. Deploy never reads source. Each domain's input is the prior domain's named output. This is what allows invariance to be checked at each boundary independently.

Surfaces

Write surfaces

  • Build: build-signed.sh <target> (CLI), called from /acp and CI.
  • Deploy: app-state.sh deploy <target>, app-state.sh rollback <target> (CLI), called from /acp.

Read surfaces

  • Manifest read: cat ~/.chorus/manifest.json or chorus-manifest <target> --commit <sha> (planned).
  • Spine: build.completed, deploy.completed events queryable via Loki / spine endpoint.

To-Be — designed target state

OSS shipping shape: build emits a versioned, hash-stable artifact; a lockfile (chorus.lock or similar) names the exact installed version per target; install fetches that exact artifact; deploy invokes install + kickstart.

Open question (the design surfaces, doesn't decide): committed binaries inside the repo, or lockfile-asserted with binaries fetched at install time?
  • Committed binaries. Repo IS source-of-truth. git pull is half of deploy. Repo size grows with every artifact rev.
  • Lockfile-asserted. Repo stays small. Install reaches an external store. Network dependency at deploy time. Casual-deletability of the install target preserved.
Today's evidence leans toward committed binaries (Jeff 2026-05-07: "code-and-binaries-together") but the trade-off matters across cross-machine and disk-budget concerns.

Decision impacts version-control vertical. Committed binaries change chorus_commit's contract (paths grow, push/pull bandwidth, hash characteristics, possible git LFS). Lockfile-asserted introduces a new artifact-store fetch protocol crossing chorus_commit's boundary. Either way: Kade in the room when this lands.
