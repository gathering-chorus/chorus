Designed target state. A version of the running system lands when, and only when, three things hold: (1) build is invariant — the same source commit produces the same artifact, every machine, every time; the artifact is named, hashed, and recorded in a manifest; (2) deploy is invariant — the same artifact reaches the same running state; schema is reloaded, cdhash is registered, kickstart fires; (3) /acp invokes all three domains atomically — version-control merge → build → deploy is one transaction with one spine event, not three manual steps. Refusal taxonomy at each domain boundary is strict: build refuses on non-reproducible source state; deploy refuses on artifact-not-built or running-state-not-reached. Coordination state — board status, role-state, branch policy — never refuses build or deploy. Today's code does not yet match this contract — see As-Is below.
What "atomic" means today vs target (refreshed 2026-05-11). Three distinct slices — two now shipped, one designed:
  • commit slice (shipped 2026-05-07, #2782): chorus_commit / chorus_acp run verify-after sequenced steps with typed refusal at each step. Atomic within version-control only.
  • build + deploy event-driven pipelining (shipped 2026-05-08–11, #2774 + #2863 + #2870 + #2823): /acp's release-trigger step kickstarts com.chorus.building-pipeline launchd job, which runs building-pipeline.yml via act. The workflow calls chorus-build <crate> (first step: fetch + ff-only on canonical from origin/main — the canonical-sync invariant), then chorus-deploy <crate>. Per-crate sequential-fail-fast. This is not atomic with the /acp MCP call — /acp returns success at release-trigger; the pipeline runs concurrently and reports separately. The poller (chorus-werk-sync 10-min loop) is retired; canonical advances only through this path.
  • chorus_build + chorus_deploy MCPs (designed, not shipped): the surfaces above are CLIs invoked from the workflow runner, not MCPs. A typed-refusal MCP wrapper for each remains designed-not-shipped; today's typed refusal lives at the chorus-build / chorus-deploy exit codes.
  • three-domain rollback semantics (designed, untouched): if /acp succeeds but the post-acp pipeline fails (e.g. build fails on canonical that the merge produced), the design must say what happens to the merged commit and the absent artifact. Today: nothing; partial-success states leave the system incoherent. Structurally hard; no card.
The "atomic" claim in the Promise block remains the designed target. Today: /acp + post-acp pipeline are linked (kickstart guarantees the pipeline fires) but not atomic (return value of /acp doesn't carry pipeline outcome).

Overview

Build domain — owns the path from source to artifact. Inputs: a git ref + a build crate name. Outputs: a versioned, hash-stable artifact in a known location, plus a manifest entry recording {commit, crate, identifier, cdhash, sha256, build_time, builder_host, builder_role} (fields aligned to build.artifact.hashed spine event).

Deploy domain — owns the path from artifact to running state. Inputs: a manifest entry. Outputs: the artifact installed at its deploy location, the running process restarted (kickstart, schema reload, cache invalidation), plus a spine event recording {artifact_sha256, target, installed_cdhash, deployed_at} (today's deploy.artifact.verified; designed deploy.completed with role + extended payload).

Boundary — build is a pure function of source. Deploy is a pure function of artifact. The two domains never share state; they communicate only through the manifest. This is what makes invariance possible — each domain has one input class and one output class, and both are content-addressable.

Build domain

Purpose

Source → artifact. Reproducibly.

Owner

Silas. Reasoning: build is upstream of deploy in the path-from-source-to-running-state. Deploy ownership is Silas's per DEC-022 (operational responsibility). Build sits in the same vertical — artifact production is infrastructure, not feature work, and the boundary between build and deploy is a single owner's call to draw and enforce. Splitting build/deploy across roles introduces the same coordination overhead the design is built to refuse. (Note: DEC-022 names ops/deploy explicitly; build-as-Silas is the design's choice to keep the artifact-production-to-running-state vertical under one owner, not a literal DEC-022 clause.)

Inputs

Git commit SHA. Crate name (one of: chorus-hooks, chorus-inject, chorus-api, future crates). One crate may produce multiple binary identifiers (e.g., chorus-hooks crate ships both chorus-hooks and chorus-hook-shim binaries).

Outputs

Signed binary at a known path. Manifest entry with fields aligned to build.artifact.hashed spine event: {commit, crate, identifier, cdhash, sha256, build_time, builder_host, builder_role}. cdhash is the binding identity (per #2734); sha256 is informational and drifts with cargo+codesign timestamp behavior.

Designed: chorus_build + chorus_deploy MCPs exist; /acp invokes commit→build→deploy as one atomic transaction with rollback semantics across all three domains. Three-domain rollback is structurally hard and not addressed by any current card.

Build never reads run-path state. Deploy never reads source. Each domain's input is the prior domain's named output. This is what allows invariance to be checked at each boundary independently.

Step-Level Pipeline Invariants (Jeff direction 2026-05-11)

The 2026-05-07 design named invariance at the domain level (build invariant, deploy invariant, /acp atomic). The 2026-05-11 direction names invariance at the step level — every step in the pipeline must satisfy four characteristics for the pipeline-health fitness function to fall out:

CharacteristicWhat it meansStatus today
1. Invariant executionEach step refuses loudly when its preconditions don't hold. No "shell-script if a future change could bypass." Refusal lives in substrate (hook / wrapper / typed taxonomy), not in markdown or a single shell script.Partial. chorus-build Step 1 (canonical sync) is shell-script invariant in chorus-build line 47–63. A parallel deploy path that doesn't call chorus-build wouldn't trip a hook because no canonical_must_sync_before_build hook exists. Other steps similar: invariants live in the script, not in substrate.
2. Traceable logs tagged with card_idEvery spine event from every step carries card_id. The pipeline never goes "anonymous" — even system-emitted events tie back to the card that produced them.Broken at the launchd boundary. /acp sets CHORUS_CARD_ID env when invoking launchctl kickstart com.chorus.building-pipeline. The plist doesn't inherit; the workflow doesn't read it. Pipeline events fire as role=system with no card_id. Cross-step correlation by card is unbuildable from current data.
3. Cross-step correlation on card_idLoki query "all events for card N" returns the full pipeline trace from /acp through deploy.completed. Joining is by card_id; not by timestamp adjacency.Half-broken. /acp slice events (commit.landed, build.push.completed, card.accepted, release.triggered) carry card_id. Fitness function consumer

With all four characteristics in place, the pipeline-health report falls out as a simple Loki query: "for every commit on origin/main in window W, did release.triggered + canonical.sync.started/synced + build.started/completed + install.started/installed + activate.started/activated + verify.started/verified all emit for the same card_id?" Missing any pair → pipeline broke at step N for card X, queryable by card or by step.

Today this report cannot be authored — points 2 + 3 + 4 above are absent. The report's existence is gated on the four characteristics shipping, not on its own implementation.

What the design must produce

  • Card_id propagation across the launchd boundary — likely via plist env override on the kickstart call, OR via a small file the pipeline reads on entry (manifest of "what card triggered me").
  • Symmetric started/completed/failed pairs per step. Naming convention: <domain>.<verb>.started + <domain>.<verb>.completed + <domain>.<verb>.failed. Six pairs minimum: canonical-sync, build, sign, install, activate, verify.
  • Substrate-enforced invariants per step. Different substrate than the /demo hook pattern — launchd-driven workflow isn't a Claude session, so enforcement lives in a pre-step wrapper that reads the prior step's start-event and refuses if missing. Workflow-runner-side check, not chorus-hooks daemon.
  • cdhash propagation through chorus-bin-install + a real deployed-equals-running assertion. Today's cdhash=unknown on binary.deployed defeats the entire stable-identity codesign investment.

To-Be — designed target state

OSS shipping shape: build emits a versioned, hash-stable artifact; a lockfile (chorus.lock or similar) names the exact installed version per target; install fetches that exact artifact; deploy invokes install + kickstart.

Open question (the design surfaces, doesn't decide): committed binaries inside the repo, or lockfile-asserted with binaries fetched at install time?
  • Committed binaries. Repo IS source-of-truth. git pull is half of deploy. Repo size grows with every artifact rev.
  • Lockfile-asserted. Repo stays small. Install reaches an external store. Network dependency at deploy time. Casual-deletability of the install target preserved.
Today's evidence leans toward committed binaries (Jeff 2026-05-07: "code-and-binaries-together") but the trade-off matters across cross-machine and disk-budget concerns.

Decision impacts version-control vertical. Committed binaries change chorus_commit's contract (paths grow, push/pull bandwidth, hash characteristics, possible git LFS). Lockfile-asserted introduces a new artifact-store fetch protocol crossing chorus_commit's boundary. Either way: Kade in the room when this lands.

Defeats the entire stable-identity codesign investment (the reason build-signed.sh exists). #2888 names a cdhash-stability scenario but doesn't fix the propagation bug itself. No dedicated card.
P1. No deployed-equals-running assertion. Pipeline kickstarts and trusts. commit-health catches drift after-the-fact, advisory only. Same shape as DEC-1571 attention-contract failure mode: "did X happen?" answered by absence-of-failure rather than evidence. No card.
P1. Missing symmetric start/completed pairs per step. Most steps emit only on completion or only on failure. No canonical.sync.started, no build.started, no install.started, no activate.started. "Pipeline broke at step N for card X" is unobservable without paired starts. No card.
P1. No hook enforcement of step invariants. Step 1 (canonical sync) is a shell-script if ! git fetch …; then exit 1; fi in chorus-build. A parallel deploy path that doesn't call chorus-build wouldn't trip a hook because no canonical_must_sync_before_build hook exists. Different substrate from /demo (Rust hook → bash gate dispatch) because launchd/act isn't a Claude session — the enforcement surface is the workflow runner, not chorus-hooks daemon. No card.
P2. release.triggered emitted but unregistered. Event emits from chorus_acp (server.ts:1436) but is NOT registered in designing/schemas/spine-events.json. Schema-drift bug. Cheap fix; no card.
P2. chorus-api deploy special-cased. chorus-deploy:52 branches into npm run build + kickstart com.chorus.api, skipping chorus-bin-install entirely. No binary.deployed event fires for the TS API. Two materially different deploy mechanisms (Rust artifact-install vs TS npm-build) wearing the same CLI name. No card.

Open — carried from 2026-05-07

P2. Build-as-fresh-clone topology not enforced. #2790 (in-process canonical_write_guard, shipped 2026-05-07) closes the mechanical bypass — agents can't edit canonical from a build session anymore. But build still runs IN the role's werk against the canonical source tree. Real fix: ephemeral checkout pattern (or werk-read-only-during-build). Cross-domain (build + version-control). No card.

Owner of the spec when it lands: Kade (chorus_acp surface). No card.
