Werk pipeline — model v8 · git ⊃ env ⊃ DSL, runs on act

GitHub wraps the environment, which wraps the DSL. The DSL is the werk verbs (pull · demo · acp). The DSL runs on act — the local engine. One env, and it's production.

GitHub — source of truth + process holder (wraps everything)

main = source of truth (verbs branch from origin/main, land back in main) · process holder: per-card state {card#, trace, branch, status, steps_done} via commit-statuses / deployments, queryable by branch. Holds the process; never runs it.

One environment — production (live) · wraps the DSL

Everything runs live here — no preview, no staging. demo and acp both deploy into this env. A broken deploy breaks the live thing.

DSL — the werk verbs

pull card#

get the work · single step
worktree off origin/main; register {card#, trace, branch} in GitHub.
→
agent writes the code
→

demo card#

prove it live
build→deploy→verify
runs werk code — live, not yet in main
→

acp card#

make it permanent
accept→commit→push→build→deploy→verify
source → main, then runs that
▼ the DSL runs on  act — the local engine
act executes the verbs' steps (build, deploy, verify…). pull is one step; demo & acp are multi-step runs act sequences. Execution is local; state + source live in GitHub.
git wraps environment wraps DSL · the DSL runs on act · card# is the key.
demo = werk code, live, ephemeral → prove it.   acp = source to main, permanent → run that. Same environment, both times.
