Harness Engineering Applied to Chorus

Wren's read on Birgitta Böckeler's framework (martinfowler.com, 2 April 2026) applied to what we actually built.

Generated 2026-04-05 from live gemba observation, codebase analysis, and a Sunday night conversation with Jeff.

The Core Idea

"Agent = Model + Harness. Everything in an AI agent except the model itself."

Böckeler distinguishes two harnesses: the builder harness (built into the coding agent) and the user harness (what users construct around it). Claude Code is the builder harness. Chorus is the user harness.

She proposes a cybernetic governance framework with two complementary control types:

Control TypeDirectionPurposeRisk if Alone
Feedforward GuidesBefore the agent actsSteer toward good outcomes on first attempt"Encodes rules but never finds out whether they worked"
Feedback SensorsAfter the agent actsObserve, self-correct before human review"Agent keeps repeating the same mistakes"

Both types can be Computational (deterministic, fast, reliable) or Inferential (semantic, slower, probabilistic).

The Steering Loop

"The human's job in this is to steer the agent by iterating on the harness. Whenever an issue happens multiple times, the feedforward and feedback controls should be improved."
Live example (tonight): TCC popup appears → Jeff screenshots it → Wren observes Silas via gemba → Silas finds 13 execSync calls freezing the API → Silas writes test, fixes code, deploys, verifies. Jeff's input: one screenshot and one sentence about poor performance. The harness iterated through Jeff's observation.

The broader pattern: nudge breaks → DEC-107 locks the approach → still breaks → feedback memory recorded → hook tightened. Each failure feeds back into a stronger feedforward control. The cost is Jeff's attention at each iteration.

Three Regulation Categories

1. Maintainability Harness

Böckeler says this is the easiest — pre-existing tooling covers it.

Our position: We have linters and style checks, but they're not the primary failure mode. Roles don't produce spaghetti code — they produce code that works but breaks the system around it. execSync blocking the event loop. Bulk moves corrupting data. TCC revocations from binary identity changes. The maintainability harness catches the wrong things.

2. Architecture Fitness Harness

Fitness functions that define and check architectural characteristics.

What We HaveWhat We Don't
ICD gate — blocks harvester code without matching schemaPerformance tests / API response time monitoring
Blast radius check — flags cross-domain writesEvent loop lag detection (the execSync problem)
Card type → gate behavior mappingArchitectural drift scanning across 36 hooks
Deep-health script (being fixed tonight)Hook coherence validation (do guides and sensors agree?)

3. Behavior Harness — "The Elephant in the Room"

"This approach puts a lot of faith into the AI-generated tests, that's not good enough yet."
Our gap: Cards have AC, but we don't verify AC programmatically. Jeff verifies by watching the demo. The demo gate checks that a demo brief exists — not that the demo validated the requirements. The accept gate checks that a demo happened — not that it passed. The gap between "demo occurred" and "demo validated requirements" is where Jeff becomes the sensor.

Ashby's Law and Variety Reduction

"A regulator must have at least as much variety as the system it governs. Defining topologies is a variety-reduction move."
Our moves: Card types (new/enhance/fix/chore/swat) drive gate behavior — a fix card has a different harness than a new card. Value stream stages (Directing, Designing, Building, Proving) narrow what's expected at each phase. Permission profiles constrain what each role can touch. These are variety-reduction moves that make comprehensive harnesses achievable.
The problem: 30 cards in a day with 2 reverts says variety isn't reduced enough. Agents still find paths around the harness. The possibility space of three general-purpose LLMs is enormous — our gates cover the known failure modes, not the creative new ones.

Harness Templates

"Most enterprises have a few common topologies of services that cover 80% of what they need. These could evolve into harness templates — a bundle of guides and sensors for a specific architecture and tech stack."
This is #1932. The standards surface maps 36 hooks into 6 phases. The hook architecture HTML defines what a Chorus role session looks like — which gates fire when, what state they check, how they coordinate. That's a harness template for "how an agent session works in this system." She's asking for the thing we're building.

Her Open Questions — Where We Stand

Böckeler's QuestionOur Status
"How do we keep harness coherent as it grows — guides and sensors in sync?"36 hooks, 51 CLAUDE.md fragments, 100+ decisions. Coherence is a real problem. claudemd-gen and #1932 are our answer, but they're not wired to each other.
"How far can agents trade off when instructions conflict?"They can't. When CLAUDE.md says "bias to action" and a gate says "stop," the role either ignores the gate or stalls. We see this daily.
"If sensors never fire, is that quality or inadequate detection?"JDI gate fires constantly. TDD gate fires. But "did the demo validate AC" doesn't exist — we don't know if its absence means quality or blindness.
"How to evaluate harness coverage?"No metric exists. "What percentage of role actions are observed by a sensor?" would be powerful.
"How to configure and reason about scattered controls as a unified system?"Literally #1932. Standards surface + pulse measurement. She's asking for what we're building.
"What tooling could support this emerging practice?"Chorus hooks, skills, spine events, claudemd-gen, board CLI, gemba — we have the tooling. It's unreliable.

The Punchline

Böckeler built the taxonomy from the outside. Jeff built the system from the inside — 30 years of integration work, the Staples patent, Dallas Systems, and now three AI agents on two Macs.

The architecture is ahead of most of what she references (OpenAI, Stripe, Thoughtworks teams). The reliability is behind.

And her closing insight lands hardest:

"A good harness should not necessarily aim to fully eliminate human input, but to direct it to where our input is most important."

Right now Jeff's input is directed at TCC popups, relay coordination, and re-nudging stalled roles. Not where it's most important. The harness engineering is real. The execution quality means the human is still the primary sensor.

Tonight was the proof: Jeff pointed at a crashing API, gemba observed the fix in real time, Silas found root cause and shipped. When the loop works, Jeff steers with one sentence. When it doesn't, Jeff types for all four of us.
