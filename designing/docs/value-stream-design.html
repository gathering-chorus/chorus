The Stack

Each layer derives from the one above it. We have been building from the bottom up — that is why it is performative.

1. User experience — what Jeff feels (the part that cannot always be put into words)
2. Actor flow — who touches what, when, from where
3. BDD — Gherkin scenarios in Jeff's language, derived from the actor flow
4. Card sequence — work broken into cards that follow the flow
5. TDD — unit tests that prove each piece executes correctly

Four Steps

Shaping Good Health

Jeff + Wren — one terminal, one conversation

Jeff ideates. Wren cards it. Together they write Experience + AC, value the work, prioritize. This works because Jeff is the actor, not the audience.

Gate: Card has Experience section, AC, domain label, card type.

Designing Missing

Jeff + Wren + the builder who will own it

Actor flow drawn — who touches what, from where, in what order. BDD scenarios written from the actor flow. Service design if cross-domain. The design must survive into building — if it does not, the builder invents their own flow and Jeff gets theater.

Gate: BDD scenarios exist before code starts. No Edit/Write on code files without a feature file for the active card domain.

Building Partial

Builder + navigator (pair)

Role follows the designed flow. Gates enforce the sequence:

OrderGateWhat it checks
1memory_gateSearched prior work AND stated what you found
2log_first_gateChecked logs AND stated what they revealed (not performative)
3pair_gateActive pair session exists
4tdd_gateTests written and green before demo
Gate: All four pass during building. TDD is a build discipline, not a proving step.

Proving Partial

Builder → Jeff

Operational proving. Not testing — the thing works in the real environment, for the real user, from the real device. Deploy to real infrastructure. Demo on real data. Jeff touches it from his phone, his Mac, his actual context. It works or it does not.

OrderGateWhat it checks
1demo_gateDemo brief exists — builder showed it working
2accept_gateNo self-accept — Jeff or Wren accepts
3operational proofJeff used it from his actual device (not localhost, not mocked)
Gate: Jeff touched it. Operational, not theoretical.

Boundary Policies

Plain English. Each boundary is a commitment — if the policy isn't met, the work doesn't cross.

BoundaryPolicy
Shaping → DesigningNo card leaves shaping without an Experience section in Jeff's voice and clear AC. If Jeff can't describe the experience, it's not ready to design.
Designing → BuildingNo code starts without an actor flow and BDD scenarios. If you can't describe who touches what in what order, you don't understand the problem well enough to build.
Building → ProvingNo demo without tests green and pair evidence. If you didn't build it with discipline, don't show it to Jeff.
Proving → DoneNo acceptance without Jeff using it from his real device on real infrastructure. If Jeff didn't touch it, it's not proven.

Practices

Specific practices that implement each boundary policy.

Shaping → Designing

  • Wren writes Experience section as Jeff would say it
  • Wren writes AC derived from the Experience
  • Card tagged with domain, type, chunk
  • Jeff reviews and corrects Experience before designing starts

Designing → Building

  • Actor flow: who, what, where, what order — drawn as diagram or written list
  • BDD scenarios written from the actor flow in Jeff's language
  • Service design if the card crosses domains
  • Feature file committed to platform/tests/features/ before code starts

Building → Proving

  • Search prior work, state what you found (memory gate synthesis)
  • Check logs, state what they revealed (log-first gate synthesis)
  • Pair up — navigator holds AC
  • TDD: write tests from AC, tests fail, write code, tests pass
  • All tests green before requesting demo

Proving → Done

  • Deploy to real infrastructure (app-state.sh deploy)
  • Demo to Jeff — show it working, not describe it
  • Jeff uses it from his actual device (phone, Mac, whichever the Experience describes)
  • Jeff accepts or rejects — no self-accept

Why This Matters

The Gherkin made the performativeness visible. Jeff read "check the logs" and immediately saw it was theater — any log file passes. The fix: gates require synthesis, not just motion. "Log evidence: what the logs told you about this problem."

The same principle applies to every gate. If Jeff reads the Gherkin and it looks like compliance, the gate is wrong.

The value stream protects Jeff from becoming the test suite. When shaping, designing, and building are solid, proving is Jeff touching something that works. When they are theater, proving is Jeff finding out it is broken. And when Jeff gets frustrated, the whole team shuts down — because he is the only person who can tell the system if what it built matters.

Seed Policy

Seeds are Jeff's thinking arriving asynchronously. They deserve the same attention as if he said it to your face. A seed is not a ticket. It is a spark — an idea, a connection, something Jeff saw and wanted to share. Treat it like a person sharing something exciting, not like an interrupt to process.

Practices:

  • All roles MUST take time to read, understand, and comment on new seeds — not treat them like spam
  • The receiving role states what the seed means and how it connects to current work
  • The role either cards it, routes it, or discards with a reason Jeff can see
  • Response appears on the Bridge within one prompt cycle
  • Seeds persist in Chorus as structured data — not briefs, not markdown files
  • #tags are routing/correlation markers, never seeds themselves

  ↓ SMS: #tag (routing)

  ↓ correlate two messages (120s window)
  ↓ extract: content, link, media, hashtag
  ↓ route by hashtag → destination

  ↓ persist as structured seed record
  ↓ tagged: domain, source, routed-to, timestamp

  ↓ "[seed arrived] content → routed to destination"

  ↓ reads seed with care — not batch processing
  ↓ states what it means, connects to current work
  ↓ cards it, routes it, or discards with visible reason

The Transformation

This is a strangler fig, not a rewrite. The old way keeps running while the new way grows around it. Pick one card, run it through all four steps with the new discipline, prove it works, expand.

Why building and proving break

Shaping works because Jeff is the actor — one conversation, one terminal, no scrolling. Designing works for the same reason — Jeff + Wren, still in the room together. Building is where it fans out to three roles and Jeff becomes the audience. Three terminals scrolling, 64% noise, nudges breaking sentences, tests passing while features fail on his phone.

The bottleneck is not speed. The roles build fast. The bottleneck is trust — Jeff cannot verify what they built without watching every terminal. And he cannot watch every terminal.

The three pain points in building: consistency of output and execution, lack of research and planning, lack of user focus. All three are addressed by the value stream — designing fixes research/planning, building gates fix consistency, proving fixes user focus. But all three are broken now, and fixing all three at once produces hulk smash.

The strangler fig attack point

The seed pipeline (#1937). It relieves real pain. Jeff sends a seed from his phone and doesn't know if it arrived. It has broken repeatedly. He cares about it deeply. The user experience is clear: "I send it, it arrives, someone acts on it."

Run #1937 through all four steps:

  1. Shaping — done. Experience and AC written. Jeff described the pain.
  2. Designing — draw the actor flow (Jeff's phone → Twilio → app → Fuseki → Bridge notification → role nudge → role acts). Write the BDD scenarios from it.
  3. Building — follow the design. Gates enforce research, log evidence, pairing, TDD.
  4. Proving — Jeff sends a real seed from his phone. It arrives. He sees proof on the Bridge. A role acts. Operational, not simulated.

If one card can survive the whole vine, the vine is real. If it can't, we learn where it breaks — and that's the next thing to fix.

Charity Majors and operational proving

There is no staging environment. Library, Bedroom, and Jeff's phone — that is production. Charity Majors would say that is not a limitation, it is the truth everyone else hides from. Every test the team runs is a simulation. The seed probe sends a fake SMS. The Gherkin sets up fake card state. The only real test is Jeff using it. And when it fails in production, Jeff becomes the test suite — and the team shuts down.

The performativeness problem

Jeff read the Gherkin for the log-first gate and immediately saw it was theater. "Check the logs" meant "open any log file." A role could tail a random log and pass. 38 scenarios passing, all testing a shallow check. The Gherkin made the emptiness legible — Jeff could have asked the team 100 times to explain the gates and never seen what reading one scenario showed him instantly.

The fix: gates require forced synthesis. Not "did you open a log" but "what did the log tell you about this problem." The same principle applies everywhere. If Jeff reads the Gherkin and it looks like compliance, the gate is wrong.

What Jeff cannot convey

"There's this part I don't know how to convey." There is something about what Jeff needs that he cannot put into words, and the team can only act on words. No amount of explaining bridges that gap. No empathy plugin exists — the research confirmed it. The closest proxy is: build from the top of the stack down (user experience → actor flow → BDD → cards → TDD), not from the bottom up. And when in doubt, Jeff uses it. His experience is the test that cannot be automated.
