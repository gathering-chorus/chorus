Test Categorization convention

Drafted 2026-04-27 · Silas · Phase 0 of CI Harness Disconnect (#2524)
Every test file declares its category in its filename. The CI runner filters by category to fire the right tests at the right stage with the right tolerance for flake. One source of truth (the filename), one filter pattern per stage, no tags or runtime annotations to drift out of sync.

Why this exists

The harness-disconnect plan (Phase 1) narrows the new required-checks list to hermetic tests only. To implement that, CI needs to filter the test corpus to "hermetic" without scanning every file. This convention is the filter — the act of writing a test forces a category choice; the act of changing a test's category is a rename, visible in PR diff.

This document is the convention. The act of applying it (renaming every existing test) is the hermeticity audit (#2523) — same operation, because each test is renamed because its actual isolation profile was checked. Categorization without audit locks in lies; audit without categorization has nowhere to write its verdict.

Categories

CategoryPropertiesWhere it runsBlock?
hermetic No network, no filesystem outside tmp, no clock or randomness, no env-coupling, order-independent Pre-commit (changed scope) + PR (full) Block
contract Cheap, deterministic; verifies the boundary between two components (e.g. TS↔Rust); may use in-process stubs but no network PR (when contract layer exists, post-Phase 3) Block (Phase 3+)
integration Hits real services (Vikunja, Fuseki, chorus-api); reads/writes real filesystem; depends on env Nightly drift lane (#2527) + on RUN_INTEGRATION=1 Warn — files card on red, never blocks PR
smoke Tiny — log in, hit one endpoint per surface; runs against a live process Post-deploy (#2333), <30s budget Triggers auto-rollback, not PR block
perf Measures runtime/memory against a rolling baseline on fixed-spec runner Nightly drift lane (#2527) Warn — files card on regression, never blocks
mutation Mutation testing scoped to changed files (PR) or full repo (nightly) Nightly drift lane (#2527); PR scoped post-Phase 2 Warn — files card on score drop, never blocks
The default is hermetic. Bare *.test.ts means hermetic. Any non-hermetic test must declare its category in the filename. This makes the trust boundary visible in ls output — if every file in a directory is bare *.test.ts, the whole directory is gate-blocking-eligible. One non-hermetic file in the mix is loud.

Mutation isn't a test file kind — it's a runner mode applied to the existing hermetic + contract files.

Rust (cargo)

Cargo's existing convention does most of the work — unit tests live inline in mod tests in src/; integration-style tests live in tests/ at the crate root. We extend with filename prefixes inside tests/:

LocationCategoryExample
src/**/mod testshermetic (default)src/parser.rs with #[cfg(test)] mod tests
tests/hermetic_*.rshermetic (cross-module)tests/hermetic_pipeline.rs
tests/integration_*.rsintegrationtests/integration_vikunja.rs
tests/contract_*.rscontracttests/contract_ts_bridge.rs
tests/smoke_*.rssmoketests/smoke_health.rs
tests/perf_*.rsperftests/perf_throughput.rs

Inline mod tests is treated as hermetic by default. If a test in mod tests is non-hermetic, it should be moved to a categorized file in tests/ rather than tagged in place — keeping inline tests consistently hermetic preserves the affordance that cargo test --lib is always cheap and fast.

The hermeticity rules (canonical)

A test is hermetic if and only if it passes all five:

  1. No network. No HTTP calls, no socket opens, no DNS lookups. Mock or stub at the boundary.
  2. No filesystem outside tmp. No reading from repo paths, no writing to user-home, no dependence on file layout. tmpfile() / tempdir() only.
  3. No clock or randomness. No Date.now() / Math.random() / uuid() / SystemTime::now() without explicit seeding or injected fakes.
  4. No env-coupling. Independent of $HOME, $USER, macOS-specific paths, LaunchAgent state, network interface presence, locale.
  5. Order-independent. Passes when the suite runs shuffled. No test leaves state that another test reads. Detection mechanism (#2532, shipped 2026-04-27): CI runs jest with --randomize on every PR. Order-dependent tests surface as flake on at least one shuffled run. Cargo equivalent (cargo nextest --shuffle) is a follow-on once the workspace adopts nextest.

These five are the audit definition cited by #2525 DEC. Future PRs that propose adding a test to the hermetic set must verify all five; future PRs that demote a test from hermetic must show which rule it violated.

CI filter patterns

Pre-commit (changed scope, hermetic only)

jest --findRelatedTests --testPathPattern '\.test\.ts$' \
     --testPathIgnorePatterns '\.(integration|contract|smoke|perf)\.test\.ts$'

PR required-check (full hermetic)

jest --testPathPattern '\.test\.ts$' \
     --testPathIgnorePatterns '\.(integration|contract|smoke|perf)\.test\.ts$'

cargo test --lib --workspace
cargo test --test 'hermetic_*' --workspace

Nightly drift lane (#2527)

jest --testPathPattern '\.(integration|perf)\.test\.ts$'
cargo test --test 'integration_*' --test 'perf_*' --workspace
# Mutation runs as a separate stryker / cargo-mutants invocation against the hermetic set

Smoke (post-deploy, #2333)

jest --testPathPattern '\.smoke\.test\.ts$' --testTimeout=30000
cargo test --test 'smoke_*' --workspace

Migration approach

This convention is co-produced with the hermeticity audit (#2523), not bulk-renamed in advance:

  1. Read each test file.
  2. Run it against the five hermeticity rules (network sandbox, alt cwd, shuffled order).
  3. Pass → keep bare *.test.ts name (or rename if it currently has a misleading suffix).
  4. Fail → rename with the appropriate suffix (integration if it hits services, smoke if it's a deploy probe, etc.). File a remediation card if the test could be made hermetic with reasonable effort.
  5. Commit per-package or per-directory, not as one giant rename PR — keeps the audit trail readable.

Anti-pattern: bulk-renaming files based on guess (e.g. "everything in tests/integration/ gets .integration.test.ts") without running them against the rules. That re-creates the same lie in a new shape — file names that claim a category they don't actually meet.

Adding a new test

Pick the most restrictive category the test can meet:

  1. Default to hermetic. If you can write the test without violating any of the five rules, name it bare *.test.ts / put it in mod tests.
  2. If you need a service, the filesystem, or env → use the category that fits and accept the test runs in the slower lane.
  3. If you're tempted to disable the network for a test that conceptually needs it → that's a contract test, not a hermetic one. Use *.contract.test.ts with stubs at the boundary.

The convention's value comes from writers being honest. A test mislabeled hermetic eventually flakes the gate; a test mislabeled integration runs slower than necessary and clutters the nightly lane. Both are bugs against the convention.
