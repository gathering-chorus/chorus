Ultimately, Athena is a product-and-design pipeline. Not "a schema" or "an API rebuild." The structural model (Product / Domain / Service) is the product layer — vision, audience, value proposition, status, owner, gaps, value-stream placement. The narrative attributes (asIs / toBe / implementationPlan / pathToClose / notInScope / hasDesignDoc) are the design layer — how decisions get encoded structurally rather than living in scattered docs. The substrate is the pipeline: every product surface, every design doc, every service definition is an instance under the same contract (SHACL-gated), flowing through the same artifact lifecycle (exploring → building → operating → retiring), rendered by the same UI. That's why API-only writes + SHACL gate + invariant rendering matter: an instance can't advance to the next state until the prior state's invariant holds.

Two pipelines, one substrate. The card pipeline (idea → carded → AC → designed → implemented → demoed → accepted) is the team's work-tracking flow — it lives in the cards-service and emits spine events per transition. The artifact lifecycle (chorus:status: exploring → building → operating → retiring) is the state of the Product/Domain/Service itself, SHACL-gated, advancing in Athena's own write path. The two intersect: a card with status "demoed" can advance the artifact it's about from building to operating — but they are different pipelines, observable through different surfaces. This doc's machinery is the artifact-lifecycle pipeline; the card pipeline is referenced through cards-service-design and emits its own spine events.

What Athena is

The structural and query layer of Chorus — the canonical model of what exists in the team's product world (products, subproducts, domains, services) and the predicates that connect them. A subproduct of the Chorus product at the Designing value-stream step. Intent-set subdomains: domains, services. The pipeline framing above is what the rest of this doc operationalizes — the audit names where the pipeline is broken today; the RDF + edge handling name the contract that closes those gaps; the cookbook + ADR-028 alignment names the discipline that keeps it closed.

Athena is what makes the chorus-product-tree queryable. Every other subproduct's structural placement (Loom's principles, Werk's pipelines, Borg's logs) is an Athena instance. chorus-instance-explorer.html already proves the read-surface works — but it bypasses the chorus API entirely to query Fuseki direct because the API exposes 3 of 29 node types as thin nouns and zero traversal verbs (per reasoning-substrate-shift.html). Athena v2 is what fixes that.

OWL — AS-IS (from chorus.ttl)

Structural classes (chorus.ttl L226–243, L416):

  • chorus:Domain — abstract; three subclasses (SubProduct, SubDomain, CollectionDomain) all rdfs:subClassOf chorus:Domain.
  • chorus:SubProduct is a subclass of chorus:Domain — same structural oddity Loom doc flagged. SubProduct semantically is a parent of domains, not a kind of domain.
  • chorus:Service — declared as its own class (L416) but instantiated zero times; service-named nodes live as SubDomain.
  • chorus:Product — declared standalone, not a subclass.

Predicate-layer incoherences — same defects as Loom audit, still live

hasDomain declared twice with conflicting signatures — Document → SubDomain (tag-shape) and Product → Domain (containment). One IRI, two semantics. This is the OWL-layer source of the API problem: a traversal verb over hasDomain can't return one consistent type of thing.
consumes declared Product → Service, used SubDomain → SubDomain — the ontology has dependsOn explicitly for this and it's ignored in practice.
belongsTo used 34 times, never declared. Phantom predicate carrying real structure.
  • partOf — neither declared nor used. Phantom concept.
  • dependsOn, expresses — declared without rdfs:domain / rdfs:range.
  • framework.ttl runs a parallel vocabulary (fw:Domain owl:equivalentClass chorus:Domain, plus fw:ownedBy / fw:dependsOn with no equivalence bridge to the chorus: versions).

SHACL — AS-IS

Structural minimums constrained: ProductDomainShape, SubProductParentShape, SubProductDomainShape, SubDomainParentShape. Substance unconstrained: Domain (the abstract class), Service, and the Athena-content classes themselves have no shapes. The CatalogDocShape targets a class (chorus:CatalogDoc) that isn't declared in the OWL — a shape constraining a phantom.

Coherence findings — synthesis

1. The predicate layer is the bottleneck for the API. Every defect in the AS-IS list above directly blocks the API surface Athena v2 needs. A graph-traversal verb can't have predictable types when hasDomain means two things. A dependency-query verb can't return useful results when consumes is overloaded with containment. The API problem (3 nouns, 0 verbs) is downstream of the predicate problem.
2. Service is a missing layer, not just a missing class. Service nodes exist conceptually (cards-service, gates-service, etc.) but are typed SubDomain. Athena can't expose Service as a queryable entity until the type-migration runs.
3. The recursive-Domain refactor (domain-model-asis-tobe) is the unblock. Collapsing the three subclasses of Domain into one recursive class plus separating SubProduct as its own kind fixes both (1) and (2). It is the prerequisite for the API surface, not a parallel concern.
4. Two vocabularies, no bridge. chorus: and fw: model the same domains, classes, ownership twice with no equivalence axioms. Same finding as Loom audit; still open. Out of scope for this doc (graph-wide, not Athena-specific) but worth naming.

Why two tiers — schema posture

RDF and OWL are very powerful — open extension, multi-typing, inference, federated queryability. But the schema, in many ways, sits below the data — instead of around or above it. The ontology describes what triples could exist; it doesn't gate what does. Triples can be inserted with no schema awareness and remain valid — the open-world assumption treats absence-of-schema as absence-of-knowledge, not absence-of-permission. That permissiveness allows sloppiness that leads to bad data.

The AS-IS findings above are exactly this pattern: belongsTo used 34 times but never declared; hasDomain declared twice with conflicting signatures; Service nodes typed as SubDomain because no rule enforces type discipline. The schema didn't catch these because it doesn't try to — it describes, it doesn't gate.

The two-tier design responds to this directly:
  • Relational core — Product, Domain, Service, Role, ValueStreamStep, ValueStream. Schema sits around and above the data: SHACL-enforced at write, closed-world, required attributes verified, type discipline a hard gate. Relational rigor where predictability matters — typed API endpoints (/athena/product/:id) need shapes clients can rely on.
  • Graph overlays — authority, ownership-history, deploy-route, customer-happiness binding, future annotations not yet imagined. Schema sits below the data, RDF-native: open extension, new predicates land without core schema migration, queryable via /athena/edges and /athena/sparql.
Same RDF substrate; two enforcement postures, one per tier. The relational core stops sloppiness in the load-bearing places; the overlay layer keeps the graph's evolutionary openness where new dimensions need to land cheaply.

Structural layer — three classes, recursive containment

From domain-model-asis-tobe.html and Loom's "Structural layer" section, consolidated:

  • Product is recursive. A Product can contain Products via hasChild; a "subproduct" is just a Product that is the child of another Product. SubProduct as a separate class is removed — erases the "SubProduct rdfs:subClassOf Domain" structural oddity.
  • Domain is recursive. A Domain can contain Domains via hasChild; SubDomain as a separate class is removed — depth becomes a data decision (add hasChild edges), not a schema decision (invent a deeper class).
  • Service is a first-class structural layer. Real service nodes (cards-service, gates-service, spine-service, skills-service) get re-typed to Service. The chorus:Service OWL class exists today; what's missing is type migration of the mis-typed SubDomain instances.
  • One clean predicate set — hasChild (recursive containment, validated per-source-shape, never via global rdfs:range); hasDomain single-declared Product → Domain (Document→SubDomain tag-use migrates to taggedDomain); consumes : Product → Service (a Product depends on Services to function — cross-domain consumption); hosts : Domain → Service (the Domain that owns the Service's code and runtime — every Service has exactly one host Domain); contains for content (Domain → content-instance); atStep : Product → ValueStreamStep replaces both hasValueStreamStep and the instance-level primaryStep drift; belongsTo and partOf dropped.
  • Services attach at the Domain level primarily, the Product level secondarily. The cards-service / gates-service / spine-service / skills-service nodes today mis-typed as SubDomain each have an owning Domain — the code lives there. A Service is hosted by exactly one Domain (where its code lives) and consumed by any number of Products / other Domains (who depend on it). This resolves the AS-IS ambiguity where "where does cards-service belong" had no unique answer.
  • CollectionDomain is removed from Chorus per domain-model-asis-tobe — Gathering's data domains (Photos, Music, Stories, Books, People, Property) belong to Gathering's ontology, not Chorus's. Chorus models how the team works; the Gathering data layer is downstream.

Attribute schema — typed per class

The diagram shows the shape; the tables below give the schema. Each attribute is chorus:-prefixed, has a range (xsd: datatype or class reference), cardinality, and SHACL required-status. Attribute lists were lifted from loom-subproduct-design.html (Product, Domain, Service) + the existing Athena UI domain page (platform/api/public/athena/domain-detail.html + 20 service-design docs); refined here with types, cardinality, the literal-vs-relational distinction, and the inverse-property pattern.

Two kinds of attributes per class:

  • Stored — present as triples on the instance. Returned inline by structural endpoints (/athena/product/:id, etc.).
  • Derived — graph patterns evaluated at query time. Not stored as triples on the instance. Returned by dedicated traversal endpoints (e.g., /athena/domain/:id/decisions, /athena/service/:id/consumers).
Invariant rendering — all stored attributes are required. The UI page for a Product / Domain / Service must render the same sections every time, in the same order, regardless of which instance is being viewed. That requires every stored attribute to exist on every instance — no conditional "show only if defined" branches in the rendering. SHACL minCount 1 on every stored attribute. Multi-valued attributes (gaps, notInScope, etc.) require at least one entry; the entry can be a placeholder literal like "none identified yet", but the triple must exist. Genuinely-optional attributes (e.g., inParent on root Products that have no parent, consumes on Products without external service deps) are the explicit exceptions and stay optional with their semantics named.

Universal status enum across all three classes: {exploring, building, operating, retiring} (this is the artifact lifecycle — distinct from the card pipeline named in the top frame; the two are reconciled in the pipeline-framing section below).

Storage direction (resolved 2026-05-15): all containment edges store on the parent / containment side, matching the classDiagram. Product stores hasChild (child Products), hasDomain (owned Domains), consumes (cross-cutting Service deps). Domain stores hasChild (child Domains), hosts (owned Services), contains (content records — actors, scenarios, decisions, …, per ADR-028 req-1). Service stores its dependsOn / commitsTo / integratesWith / runsOn / persistsIn outbound edges. inParent / inProduct / hostedBy are dropped from the schema — child→parent views are derived via SPARQL sh:inversePath at query time.

RDF — OWL + SHACL TO-BE

The invariant: if every write goes through the API, the graph is whole — and the Athena UI renders every instance properly, no conditional branches. SHACL is the contract, the API is the only writer that honors it. Every required attribute populated, every required edge present, every retired predicate refused at the write boundary, every cardinality enforced — by construction, not by post-hoc audit. Because the graph is whole, the consumer-side guarantee follows: /athena/product/:id, /athena/domain/:id, /athena/service/:id always return complete instances, and the Athena UI pages (domain-detail.html, the service-design renderer, the product surface) render every section on every instance without "show if defined" branches or empty-state placeholders for required fields. The AS-IS sloppiness exists because writes happened through paths that didn't validate (direct triplestore writes, ad-hoc TTL files, harvesters with no shape contract) — which forced the UI to defensively branch on "is this field even there?" v2 closes those paths: the API is the only writer; SHACL is the only gate; the graph stays whole because it cannot become un-whole through any sanctioned write; the UI renders consistently because it can trust the API response.

Mechanics: storage direction follows the classDiagram (containment direction is canonical) — hasChild, hasDomain, hosts, contains, consumes, atStep stored on the subject side named in the diagram. Inverse views (parentOf, hostingDomain) computed at query time via sh:inversePath, not stored. The earlier inParent / inProduct / hostedBy direction is dropped — one storage direction per edge, no double-keyed truth. The cookbook req-1 predicate chorus:contains is the canonical Domain→Record / Service→Record membership edge (renderers query by it). Content records carry no back-pointer; external entities (cards, spine events) that reference a Service use chorus:atService instead of chorus:contains.

Edge handling on API calls

Edges are first-class resources, not embedded fields. Reads project them inline on the parent instance for convenience; writes always go through edge-typed endpoints so SHACL validates one predicate at a time and refusals point to the exact constraint that failed.

Read projection

GET /athena/product/:id returns the instance plus its outbound edges as JSON arrays keyed by predicate (hasChild, hasDomain, consumes, atStep, ownedBy, hasDesignDoc) — IRIs only, not embedded objects, to keep the payload bounded. Domain reads project hasChild, hosts, contains, integratesWith, runsOn, persistsIn plus the same common edges. Service reads project contains, dependsOn, commitsTo plus the common edges. Inverse views (parentOf for Products/Domains, hostingDomain for Services, consumers for Services) are computed at query time from SPARQL and projected on the appropriate side; they are not stored. Traversal endpoints (/children, /domains, /services, /host, /consumers) follow one predicate recursively or one-hop.

Atomic instance creation — composite POST

POST /athena/{product,domain,service} with the full required-attribute payload creates the instance atomically. The body includes both literal attributes and required edges; SHACL validates the whole record as a unit before any triple is written. Refusal returns 422 shape-violation with the failing constraint paths. Example shape for Domain create:

POST /athena/domain
{
  "label": "cards",
  "comment": "team board — work intake to acceptance",
  "ownedBy": "chorus:wren",
  "atStep": "chorus:value-stream-step-directing",
  "status": "operating",
  "gaps": ["board sometimes lags real WIP"],
  "hasDesignDoc": ["chorus:doc-cards-service-design"],
  "hasDomain_inverse": "chorus:athena",     // satisfies inverse-hasDomain minCount 1 atomically
  "contains": ["chorus:record-cards-actors", "chorus:record-cards-scenarios", ...]  // ≥1 satisfies cookbook req-1
}

Post-create, edges are mutated via the edge endpoints above (POST to add, DELETE to remove). The composite create endpoint exists only for the create-with-required-edges atomic guarantee; once an instance is whole, edges flow through their typed paths. Refusal is binary — if any required attribute or edge is missing, the entire instance is refused; no partial writes ever land in the graph.

Edge removal

DELETE /athena/{product,domain,service}/:id/{edge-path}/:target-id for each edge endpoint above. Refuses if removal would violate a SHACL minCount (e.g., deleting the last hasDesignDoc, the only ownedBy, the sole hosts edge for a Service). Refusal message names the shape + constraint, not just "validation failed."

Retired-predicate writes — explicit refusal

The API maintains a deny-list at the write boundary: chorus:inParent, chorus:inProduct, chorus:hostedBy, chorus:belongsTo, chorus:partOf, chorus:domain (back-pointer; superseded by chorus:contains), chorus:service (back-pointer; superseded by chorus:contains for records, chorus:atService for external references). Any write attempting these predicates — via the structured endpoints, via POST /athena/sparql (which is read-only by guard anyway), or via the underlying triplestore if proxied — returns 409 retired-predicate with the canonical-direction predicate to use instead (e.g., "write chorus:hosts on the Domain, not chorus:hostedBy on the Service"; "write chorus:contains on the Domain, not chorus:domain on the record"). This is how AS-IS data migrates forward without new writes degrading the schema.

/athena/edges — read-only predicate-pattern query

GET /athena/edges?subject=&predicate=&object= answers "which triples match this pattern" — direct equivalent of logs_query for the graph. Any combination of the three slots may be bound or wildcard. Read-only by guard; the typed mutation endpoints above are the only write surface for edges. This is the verb that closes the substrate-shift master-lever ("make the graph as interrogable as the logs already are").

Clients can't programmatically branch on "is this fixable by me or is it a bug?"
Bootstrap orderRole, ValueStream, ValueStreamStep, and a small set of seed Document instances must be populated before any Product / Domain / Service can validate (every Product requires ownedBy, atStep, hasDesignDoc). The seed migration card runs this in order: classes → bootstrap instances → Service type-migration → Domain instance backfill → Product instance backfill.Schema-first bootstrap has a chicken-and-egg if not sequenced — every required edge has to have an existing target before the source instance validates.
SHACL → OpenAPI generationSHACL is the canonical source; OpenAPI / JSON-Schema for the HTTP layer is generated from it (single source of truth for both the graph validator and the typed clients). No hand-maintained parallel contract.Two contracts drift. Today's TypeScript types for Athena responses don't match what the graph actually holds; that's how harvesters started writing fields the readers didn't expect.
SPARQL passthrough guardPOST /athena/sparql is read-only (no INSERT/DELETE); enforces a per-query timeout, a complexity limit (triple-pattern count / join depth), and a result-size cap. /athena/edges pattern queries flow through the same guard.Without a guard, the passthrough verb is a DoS surface and a bypass of the typed mutation contract. With the guard, it's only a read tool — the wholeness invariant survives the existence of an open query API.
Schema evolution disciplineAdding a required attribute to a Shape ships with the backfill migration; deprecating a predicate ships with the read-side replacement. Shape changes never land alone. shapes.ttl versions are tagged and the migration card references the version it migrates from / to.Schema-evolves-faster-than-data is exactly how AS-IS got belongsTo declared one way and used another. Coupling shape + migration prevents the gap.
Read-time consistencyStrong-read-after-write for the resource just written (the same client sees its own writes immediately). Derived views (SPARQL-computed inverse views like consumers, parentOf) are computed at query time, so they reflect the latest write by construction — no separate index to keep in sync.Today's subdomains_get reads can lag harvester writes; consumers see stale data and start treating "is the field there" as an open question. Computing inverses on the fly removes the eventual-consistency surface entirely.
Auth surface (read-free, write-gated)Reads (GET /athena/*) require no auth, per chorus-api convention. Writes require the call to carry DEPLOY_ROLE attribution (already enforced on git-queue and the cards pipeline); the API records that role as the writer on every spine event and on triple provenance. Per-class authority (who can mutate which subject type) follows the Authority overlay (#2927 AC4) — owner of the unit's domain is the authorized writer.AS-IS has no write-attribution at all; the harvesters write under whichever process happened to invoke them. v2 wires write-authority into the existing role-attribution substrate.
Comprehensive logging + error handlingEvery API request emits a structured log line (Loki, role + endpoint + subject + outcome + latency). Every refusal emits a spine event (athena.<class>.<op>.refused) with the typed reason and offending payload — no silent 4xx. Every internal error emits both a log line at error level and a spine event (athena.internal.error) with stack + request context; the response carries a correlation ID that resolves the spine event. No console.log, no swallowed exceptions, no unwrapped throw new Error(message) — every error path types its reason against the uniform taxonomy above.AS-IS today: harvester failures go to stdout and disappear; client 4xx have ad-hoc bodies; the operator can't reconstruct what a failed write attempted. Without comprehensive observability the wholeness invariant becomes unfalsifiable — you can't prove the graph is whole if you can't see what got refused.
Infer dependencies from the graphDependency edges that the structural graph already implies are derived, not stored. Examples: if Product P consumes Service S and Domain D hosts S (so D inverse-hosts → S), then "P depends on D" is a SPARQL traversal, not a separate dependsOn triple. Cross-domain dependencies between Domain A and Domain B emerge when Services in A consume Services in B — same inference, no hand-maintained integratesWith required. Stored dependsOn / integratesWith stay in the schema only for relationships the structural graph can't express (e.g., "Domain A's behavior depends on Domain B even though no shared Service"). GET /athena/{product,domain,service}/:id/depends returns the union of stored + inferred. GET /athena/blast-radius/:id runs this inference recursively.Hand-maintained dependency lists drift the moment a Service moves. The structural graph already encodes "who calls whom"; restating it as a separate dependency edge is double-storage that goes stale. Inferring keeps the source of truth in one place — the call graph — and removes a whole class of "we forgot to update X" failures.
Comprehensive test coverage — five layers(1) Unit — every SHACL shape ships with valid + invalid fixture corpora (one fixture per violation type: missing required, wrong range, cardinality, retired-predicate). (2) Integration — full write→read round trips per resource: POST then GET, POST then DELETE then GET, POST that violates a shape then verify graph unchanged. (3) API contract — every endpoint × every refusal-taxonomy reason; OpenAPI conformance check (response shape matches generated schema). (4) UI — domain-detail.html / service-design renderer / product surface against canned API responses; every required-attribute section renders, no empty-state placeholders for required fields, edge-array projections render correctly. (5) Data quality — scheduled SHACL validation pass over the live graph (not just write-gate) emits a spine event per violation found; catches AS-IS residue, harvester drift, schema-evolution gaps. SHACL tests run in CI before API tests; API tests run before UI tests; data-quality runs continuously against the live graph.Each layer catches a different failure class — unit alone passes against an empty graph; integration alone misses UI rendering bugs; UI alone misses data-quality drift over time. The wholeness invariant has five attack surfaces; each layer defends one.

Cross-cutting: every requirement above leans on the wholeness invariant, and every requirement above also protects it. Atomic creation prevents partial instances; provenance answers who-wrote-what; uniform errors keep clients honest; bootstrap order prevents the validator from being its own first victim; OpenAPI-from-SHACL prevents contract drift; the SPARQL guard prevents the open query API from being a bypass; schema-evolution discipline keeps the invariant across time; consistency rules keep it across requests; auth wires it to roles; tests prove it. Take any one out and "the graph stays whole" becomes "the graph mostly stays whole" — which is where AS-IS lives.

Implementation path — OWL+SHACL first, API second, data third

v1 went data → schema-by-discovery → API-by-priority. v2 inverts: schema → API → data. Same three layers, reversed dependency. v1's order let each layer be constrained by what already existed below it — sloppiness in the data became normal because the schema had to accommodate it. v2's order makes each layer constrain what gets added above it — the schema gates what the API can express; the API gates what data clients can write.
Fresh write, not rewrite. v2 lives under a new namespace (/athena/*, parallel to the AS-IS /api/chorus/subdomains/*) and is built SHACL-gated from line 1 — no path through the v2 code reaches the triplestore without validating against shapes.ttl. Rewriting the AS-IS endpoints in place would carry forward the un-validated write paths that produced today's sloppiness, because the retrofit would have to keep them working while validation gets bolted on, and the team would defer to "existing behavior" every time the gate refused. Greenfield avoids that trap. AS-IS endpoints stay live as legacy reads during migration; cutover is by namespace, not by endpoint — when v2's coverage is complete, the v1 namespace retires wholesale.

SessionStart gives instant "is this my domain / what should I be on" answers without an MCP call (cheap, costs context budget once per session). Tagging layer — operational interface between Athena and everything else

2907's framing: the tree's IRIs are not just data, they are the canonical tag enumeration for cards, code, and tests. Card #2931 (gated tonight) already tags domain:chorus, subproduct:borg, sequence:werk, type:enhance. That convention is running today; Athena's job is to be the source-of-truth for those tag values.
  • Card labels: domain:<iri-slug>, subproduct:<iri-slug>, product:<iri-slug>, service:<iri-slug>, sequence:<value-stream-step>. Validity = tag value resolves to an IRI in the tree.
  • Code tags: source files declare their owning Service / Domain via header comment or per-language convention (TS module-level chorus:domain JSDoc, Rust module attribute, shell-script preamble); the tree maps each Service to one or more pathPattern globs so a diff's file list resolves to affected Services automatically.
  • Test tags: tests declare the Service/Domain they exercise; CI can then "run only tests for the domains this card touches" — bounded blast-radius, no full-suite cost per card.
  • The queries this unlocks (from 2907, verbatim shape): blast-radius (diff → file tags → affected domains/services); domain-radius (domain tag → all code/tests/services with that tag); demo-question (peer reads card tags vs their owned-products list → bounded few-word answer); CI-routing (per-domain test execution).

Validators → Loom learning corpus

The data-quality scan defined in "Other requirements" (scheduled SHACL pass over the live graph) is not just a monitoring mechanism — per 2907, it is the learning substrate for Loom. Recurring incoherence patterns surface as candidate principles, policies, practices.

Each validator (SHACL shape, retired-predicate write attempt, orphan record, stale design doc, completeness regression) emits a structured spine event with: validator name, query, row count, affected entities, timestamp, card_id if known. These events feed:

  • Loki dashboards for trend visibility (per Silas's Borg surface).
  • Threshold alerts when violation rates spike or stay stale.
  • Auto-card-filing on structural-repair work when a violation cluster exceeds a configurable bar.
  • Demo-gate preconditions — a card touching a domain with active high-severity violations gets flagged at /gate-product.
  • Loom corpus — recurring incoherence patterns are candidate inputs for Loom's principle/policy/practice formation. The substrate stops being passively-described and starts being actively-learned-from.

This is the recursion 2907 named explicitly: "validators emit structured data → great learning substrate for Loom." Athena is not a closed structural model; it is the input to the team's normative-layer evolution. Drift seen by validators today becomes principle/policy authored tomorrow. The system grows substrate for its own evolution.
