What Athena is

The structural and query layer of Chorus — the canonical model of what exists in the team's product world (products, subproducts, domains, services) and the predicates that connect them. A subproduct of the Chorus product at the Designing value-stream step. Intent-set subdomains: domains, services.

Athena is what makes the chorus-product-tree queryable. Every other subproduct's structural placement (Loom's principles, Werk's pipelines, Borg's logs) is an Athena instance. chorus-instance-explorer.html already proves the read-surface works — but it bypasses the chorus API entirely to query Fuseki direct because the API exposes 3 of 29 node types as thin nouns and zero traversal verbs (per reasoning-substrate-shift.html). Athena v2 is what fixes that.

OWL — AS-IS (from chorus.ttl)

Structural classes (chorus.ttl L226–243, L416):

  • chorus:Domain — abstract; three subclasses (SubProduct, SubDomain, CollectionDomain) all rdfs:subClassOf chorus:Domain.
  • chorus:SubProduct is a subclass of chorus:Domain — same structural oddity Loom doc flagged. SubProduct semantically is a parent of domains, not a kind of domain.
  • chorus:Service — declared as its own class (L416) but instantiated zero times; service-named nodes live as SubDomain.
  • chorus:Product — declared standalone, not a subclass.

Predicate-layer incoherences — same defects as Loom audit, still live

hasDomain declared twice with conflicting signatures — Document → SubDomain (tag-shape) and Product → Domain (containment). One IRI, two semantics. This is the OWL-layer source of the API problem: a traversal verb over hasDomain can't return one consistent type of thing.
consumes declared Product → Service, used SubDomain → SubDomain — the ontology has dependsOn explicitly for this and it's ignored in practice.
belongsTo used 34 times, never declared. Phantom predicate carrying real structure.
  • partOf — neither declared nor used. Phantom concept.
  • dependsOn, expresses — declared without rdfs:domain / rdfs:range.
  • framework.ttl runs a parallel vocabulary (fw:Domain owl:equivalentClass chorus:Domain, plus fw:ownedBy / fw:dependsOn with no equivalence bridge to the chorus: versions).

SHACL — AS-IS

Structural minimums constrained: ProductDomainShape, SubProductParentShape, SubProductDomainShape, SubDomainParentShape. Substance unconstrained: Domain (the abstract class), Service, and the Athena-content classes themselves have no shapes. The CatalogDocShape targets a class (chorus:CatalogDoc) that isn't declared in the OWL — a shape constraining a phantom.

Coherence findings — synthesis

1. The predicate layer is the bottleneck for the API. Every defect in the AS-IS list above directly blocks the API surface Athena v2 needs. A graph-traversal verb can't have predictable types when hasDomain means two things. A dependency-query verb can't return useful results when consumes is overloaded with containment. The API problem (3 nouns, 0 verbs) is downstream of the predicate problem.
2. Service is a missing layer, not just a missing class. Service nodes exist conceptually (cards-service, gates-service, etc.) but are typed SubDomain. Athena can't expose Service as a queryable entity until the type-migration runs.
3. The recursive-Domain refactor (domain-model-asis-tobe) is the unblock. Collapsing the three subclasses of Domain into one recursive class plus separating SubProduct as its own kind fixes both (1) and (2). It is the prerequisite for the API surface, not a parallel concern.
4. Two vocabularies, no bridge. chorus: and fw: model the same domains, classes, ownership twice with no equivalence axioms. Same finding as Loom audit; still open. Out of scope for this doc (graph-wide, not Athena-specific) but worth naming.

OWL + SHACL Model — Athena TO-BE

This is the structural model the audit findings demand — not Athena-specific in the sense that it's the model every subproduct uses. The Loom doc already named this layer "the Athena product model" and noted it is "not Loom-specific." This doc is its canonical home; future subproduct-design docs reference back here for the structural classes rather than inlining them.

Why two tiers — schema posture

RDF and OWL are very powerful — open extension, multi-typing, inference, federated queryability. But the schema, in many ways, sits below the data — instead of around or above it. The ontology describes what triples could exist; it doesn't gate what does. Triples can be inserted with no schema awareness and remain valid — the open-world assumption treats absence-of-schema as absence-of-knowledge, not absence-of-permission. That permissiveness allows sloppiness that leads to bad data.

The AS-IS findings above are exactly this pattern: belongsTo used 34 times but never declared; hasDomain declared twice with conflicting signatures; Service nodes typed as SubDomain because no rule enforces type discipline. The schema didn't catch these because it doesn't try to — it describes, it doesn't gate.

The two-tier design responds to this directly:
  • Relational core — Product, Domain, Service, Role, ValueStreamStep, ValueStream. Schema sits around and above the data: SHACL-enforced at write, closed-world, required attributes verified, type discipline a hard gate. Relational rigor where predictability matters — typed API endpoints (/athena/product/:id) need shapes clients can rely on.
  • Graph overlays — authority, ownership-history, deploy-route, customer-happiness binding, future annotations not yet imagined. Schema sits below the data, RDF-native: open extension, new predicates land without core schema migration, queryable via /athena/edges and /athena/sparql.
Same RDF substrate; two enforcement postures, one per tier. The relational core stops sloppiness in the load-bearing places; the overlay layer keeps the graph's evolutionary openness where new dimensions need to land cheaply.

Structural layer — three classes, recursive containment

From domain-model-asis-tobe.html and Loom's "Structural layer" section, consolidated:

  • Product is recursive. A Product can contain Products via hasChild; a "subproduct" is just a Product that is the child of another Product. SubProduct as a separate class is removed — erases the "SubProduct rdfs:subClassOf Domain" structural oddity.
  • Domain is recursive. A Domain can contain Domains via hasChild; SubDomain as a separate class is removed — depth becomes a data decision (add hasChild edges), not a schema decision (invent a deeper class).
  • Service is a first-class structural layer. Real service nodes (cards-service, gates-service, spine-service, skills-service) get re-typed to Service. The chorus:Service OWL class exists today; what's missing is type migration of the mis-typed SubDomain instances.
  • One clean predicate set — hasChild (recursive containment, validated per-source-shape, never via global rdfs:range); hasDomain single-declared Product → Domain (Document→SubDomain tag-use migrates to taggedDomain); consumes : Product → Service (a Product depends on Services to function — cross-domain consumption); hosts : Domain → Service (the Domain that owns the Service's code and runtime — every Service has exactly one host Domain); contains for content (Domain → content-instance); atStep : Product → ValueStreamStep replaces both hasValueStreamStep and the instance-level primaryStep drift; belongsTo and partOf dropped.
  • Services attach at the Domain level primarily, the Product level secondarily. The cards-service / gates-service / spine-service / skills-service nodes today mis-typed as SubDomain each have an owning Domain — the code lives there. A Service is hosted by exactly one Domain (where its code lives) and consumed by any number of Products / other Domains (who depend on it). This resolves the AS-IS ambiguity where "where does cards-service belong" had no unique answer.
  • CollectionDomain is removed from Chorus per domain-model-asis-tobe — Gathering's data domains (Photos, Music, Stories, Books, People, Property) belong to Gathering's ontology, not Chorus's. Chorus models how the team works; the Gathering data layer is downstream.

Attribute schema — typed per class

The diagram shows the shape; the tables below give the schema. Each attribute is chorus:-prefixed, has a range (xsd: datatype or class reference), cardinality, and SHACL required-status. Attribute lists were lifted from loom-subproduct-design.html (Product, Domain, Service) + the existing Athena UI domain page (platform/api/public/athena/domain-detail.html + 20 service-design docs); refined here with types, cardinality, the literal-vs-relational distinction, and the inverse-property pattern.

Two kinds of attributes per class:

  • Stored — present as triples on the instance. Returned inline by structural endpoints (/athena/product/:id, etc.).
  • Derived — graph patterns evaluated at query time. Not stored as triples on the instance. Returned by dedicated traversal endpoints (e.g., /athena/domain/:id/decisions, /athena/service/:id/consumers).
Invariant rendering — all stored attributes are required. The UI page for a Product / Domain / Service must render the same sections every time, in the same order, regardless of which instance is being viewed. That requires every stored attribute to exist on every instance — no conditional "show only if defined" branches in the rendering. SHACL minCount 1 on every stored attribute. Multi-valued attributes (gaps, notInScope, etc.) require at least one entry; the entry can be a placeholder literal like "none identified yet", but the triple must exist. Genuinely-optional attributes (e.g., inParent on root Products that have no parent, consumes on Products without external service deps) are the explicit exceptions and stay optional with their semantics named.

Universal status enum across all three classes: {draft, active, sunsetting, retired}.

RDF — OWL + SHACL TO-BE

The invariant: if every write goes through the API, the graph is whole — and the Athena UI renders every instance properly, no conditional branches. SHACL is the contract, the API is the only writer that honors it. Every required attribute populated, every required edge present, every retired predicate refused at the write boundary, every cardinality enforced — by construction, not by post-hoc audit. Because the graph is whole, the consumer-side guarantee follows: /athena/product/:id, /athena/domain/:id, /athena/service/:id always return complete instances, and the Athena UI pages (domain-detail.html, the service-design renderer, the product surface) render every section on every instance without "show if defined" branches or empty-state placeholders for required fields. The AS-IS sloppiness exists because writes happened through paths that didn't validate (direct triplestore writes, ad-hoc TTL files, harvesters with no shape contract) — which forced the UI to defensively branch on "is this field even there?" v2 closes those paths: the API is the only writer; SHACL is the only gate; the graph stays whole because it cannot become un-whole through any sanctioned write; the UI renders consistently because it can trust the API response.

Mechanics: storage direction follows the classDiagram (containment direction is canonical) — hasChild, hasDomain, hosts, consumes, atStep stored on the subject side named in the diagram. Inverse views (parentOf, hostedIn) computed at query time, not stored. The earlier inParent / inProduct / hostedBy direction is dropped — one storage direction per edge, no double-keyed truth. Predicate-direction drift in the prose / class tables / diagrams above is acknowledged and queued as a follow-on cleanup; this section is canonical.

Edge handling on API calls

Edges are first-class resources, not embedded fields. Reads project them inline on the parent instance for convenience; writes always go through edge-typed endpoints so SHACL validates one predicate at a time and refusals point to the exact constraint that failed.

Read projection

GET /athena/product/:id returns the instance plus its outbound edges as JSON arrays keyed by predicate (hasChild, hasDomain, consumes, atStep, ownedBy, hasDesignDoc) — IRIs only, not embedded objects, to keep the payload bounded. Inverse views (parentOf, hostedIn for Services, consumers for Services) are computed at query time from SPARQL and projected on the appropriate side; they are not stored. Traversal endpoints (/children, /domains, /services, /host, /consumers) follow one predicate recursively or one-hop.

Edge removal

DELETE /athena/{product,domain,service}/:id/{edge-path}/:target-id for each edge endpoint above. Refuses if removal would violate a SHACL minCount (e.g., deleting the last hasDesignDoc, the only ownedBy, the sole hosts edge for a Service). Refusal message names the shape + constraint, not just "validation failed."

Retired-predicate writes — explicit refusal

The API maintains a deny-list at the write boundary: chorus:inParent, chorus:inProduct, chorus:hostedBy, chorus:belongsTo, chorus:partOf. Any write attempting these predicates — via the structured endpoints, via POST /athena/sparql (which is read-only by guard anyway), or via the underlying triplestore if proxied — returns 409 retired-predicate with the canonical-direction predicate to use instead (e.g., "write chorus:hosts on the Domain, not chorus:hostedBy on the Service"). This is how the AS-IS data gets migrated forward without the schema being downgraded by new writes during the transition.

/athena/edges — read-only predicate-pattern query

GET /athena/edges?subject=&predicate=&object= answers "which triples match this pattern" — direct equivalent of logs_query for the graph. Any combination of the three slots may be bound or wildcard. Read-only by guard; the typed mutation endpoints above are the only write surface for edges. This is the verb that closes the substrate-shift master-lever ("make the graph as interrogable as the logs already are").

Overlay model — first-class queryable annotations

The structural tree (Product / Domain / Service) names what exists. Overlays name relations across the tree that aren't containment. Each overlay is queryable independently of the structural graph and applies across leaves.

Implementation path — OWL+SHACL first, API second, data third

v1 went data → schema-by-discovery → API-by-priority. v2 inverts: schema → API → data. Same three layers, reversed dependency. v1's order let each layer be constrained by what already existed below it — sloppiness in the data became normal because the schema had to accommodate it. v2's order makes each layer constrain what gets added above it — the schema gates what the API can express; the API gates what data clients can write.

Three downstream cards derive from #2928, in order:

  1. OWL + SHACL rigor. Formally declare Product / Domain / Service / Role / ValueStreamStep / ValueStream as chorus:-prefixed classes; every attribute typed (xsd: range or class reference) with cardinality and SHACL required-status per the tables above; inverse properties declared (hostedBy ↔ hosts; consumers as computed view, not stored); status enum domains as SHACL constraints; the predicate-layer defects from the audit (belongsTo undeclared, hasDomain double-declared, consumes overloaded) closed. Output: a chorus.ttl + shapes.ttl pair that gates writes.
  2. API surface. Build the /athena/{product,domain,service} endpoints (and the edge / sparql / blast-radius / vertebra / principle-invoke / practice-invoke verbs) backed by the SHACL shapes from card 1. Reads return shape-shaped JSON; mutation paths validate against shapes at write. The API and the schema are the same contract, expressed twice — once in SHACL for the graph, once in OpenAPI/JSON-schema for the clients.
  3. Data seed / migration. Re-type the service-named SubDomain instances to Service; populate hosts edges from those re-types; backfill the typed attributes from existing markdown / HTML / TTL sources; link design-docs as hasDesignDoc instances (including this very file, the Loom subproduct-design doc, and domain-model-asis-tobe). If the API is right, this is mechanical: feed values through POST /athena/{product,domain,service} until everything validates. Validation refusals identify the AS-IS sloppiness still in the data and force decisions case-by-case.

Order matters because each layer constrains the next: schema gates API contract, API gates client writes, data flows through the gates. Reverse order (data first, schema later — the v1 pattern) is exactly how the AS-IS findings accumulated. The two-tier model (relational core + graph overlays) is the design that holds this discipline; the implementation order is what enforces it in practice.
