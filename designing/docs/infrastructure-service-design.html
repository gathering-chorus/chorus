Invariants: agent-state.sh is the single lifecycle execution surface for chorus services. Role sessions invoke via per-verb MCP tools (typed refusals, role authorization); Jeff invokes via direct shell. Both paths land on the same agent-state.sh implementation — no second mechanism. All lifecycle transitions emit a paired service.<verb>.{started,completed,failed} spine event with PID + cdhash so "did the daemon pick up the new binary?" is a Loki query, not a triangulation. Infrastructure is a Domain in its own right; SubProducts (including the observability subproduct) depend-on it. The doc does NOT position Infrastructure underneath any SubProduct.

Designed target state. A role session calls chorus_service_deploy({ crate: "chorus-hooks" }); the MCP enforces role-authorization, invokes agent-state.sh chorus-hooks deploy, gets a typed-refusal-or-success response, and the spine event chain (service.deploy.started → chorus-deploy.completed → kickstart → service.deploy.completed) renders as one Loki query keyed by trace_id. Cdhash divergence at any point surfaces as service.deploy.failed with reason=cdhash-divergence — the "deployed but not running" failure mode becomes observable, not silent.

Overview

Chorus today runs on two physical machines (Library + Bedroom) with 42 LaunchAgents on the Library side (6 core daemons operators interact with directly; 16 scheduled probes; 17 scheduled jobs; 2 file watchers; 1 build-pipeline trigger). Services communicate over LAN mesh and a Cloudflare tunnel for remote access. State lives on local volumes (Fuseki triplestore, SQLite indices, Loki log store) with no formal backup design yet. Observability is split across Loki (logs), Prometheus (metrics), and the spine event stream — each useful, none authoritative for cross-stream questions.

This doc names the eight Infrastructure sections in order — compute, network, storage, services, service-lifecycle, observability, security, environment. Service-lifecycle is the only "shipped + designed" section in this card; others are "named, gaps surfaced" — they identify what exists and what's missing without proposing fixes. Cards file when the pain surfaces, not preemptively.

Compute

Two physical machines: Library (192.168.86.36, primary — runs all chorus daemons, app, fuseki, observability stack) and Bedroom (192.168.86.242, secondary — photo harvester, thumbnail pipeline, dedicated GPU work). Library is the canonical chorus host; Bedroom-side services managed via SSH + Bedroom's own local agent-state.sh. Hard constraints C1-C7 documented in infrastructure-constraints.md.

Gaps: no formal compute capacity model; no migration plan if Library hardware fails; Bedroom-Library coordination is ad-hoc per task (#2107 etc.); no automated failover for chorus-api.

References gathering-side infrastructure-service-design.html for the broader compute model — chorus-side adapts that shape.

Network

LAN mesh between Library and Bedroom (direct 192.168.86.x). Cloudflare tunnel for remote Jeff access to localhost services (configured but not always active — gap surfaced 2026-05-15 alert). Per-service endpoint table maintained in TEAM_PROTOCOL.md (3340 chorus-api, 3000 app, 3030 fuseki, 3100 grafana, 3102 loki, 3470 bridge, 3475 messaging, 9090 prometheus). caddy-edge fronts 3000 with localhost-only TLS.

Gaps: tunnel uptime alert exists but has fired twice 2026-05-15 — recovery is manual. No formal endpoint registry separate from TEAM_PROTOCOL.md prose. mDNS-style service discovery absent.

Storage

Local volumes only. Fuseki triplestore (Library) holds the canonical OWL graph (2,562 triples 2026-05-15) plus instance data. SQLite indices for chorus search (~/.chorus/index.db) and cards / activity. Loki log store retains last ~30d. Manifest history under ~/.chorus/manifest/ retains prior cdhashes for rollback (#2791 shipped).

Gaps: no formal backup design — Time Machine on Library is the implicit answer. No volume-capacity alerts beyond the daily disk-health probe. No documented restore procedure.

MCP registration (per-verb tools)

Role sessions invoke lifecycle via per-verb MCP tools — mirrors chorus_acp / chorus_pull_card / chorus_commit pattern (one tool = one verb = one typed-refusal taxonomy). Jeff retains direct shell access; MCP layer is for role sessions where typed refusals and role authorization matter.

MCP toolWrapsTyped refusalsRole authorization
chorus_service_statusagent-state.sh statusservice-not-found | label-resolve-failRead — any role
chorus_service_startagent-state.sh startservice-not-found | already-running | bootstrap-failWrite — silas only (DEC-022)
chorus_service_stopagent-state.sh stopservice-not-found | already-stopped | bootout-failWrite — silas only
chorus_service_restartagent-state.sh restartservice-not-found | kickstart-fail | verify-failWrite — silas only
chorus_service_deployagent-state.sh deployservice-not-found | build-fail | kickstart-fail | cdhash-divergence | verify-timeoutWrite — owner-of-deploy-unit (#2927 distribution)
chorus_service_rollbackagent-state.sh rollbackservice-not-found | no-prior-cdhash | restore-fail | kickstart-fail | verify-failWrite — owner-of-deploy-unit

Each tool returns { service, verb, pre_pid, post_pid, pre_cdhash, post_cdhash, role } on success — same shape as the paired spine emit. Refusals carry the typed reason + a recovery hint. Tool descriptions name the full refusal taxonomy verbatim so the cardinal-six audit pattern (#2941) extends here on day one.

Authorization layer. Read verb (status) opens to all roles. Write verbs (start/stop/restart) gated to silas. Deploy/rollback follow per-unit domain ownership per #2927 — cards-sdk → wren, chorus-api → kade, chorus-hooks → silas. DEC-022 narrowed: LaunchAgent lifecycle is silas-owned by default; deploy/rollback delegate by unit.

Conventions — Chorus .sh house style

Every shell addition to agent-state.sh (and sibling scripts) follows the practices established in chorus-build, chorus-deploy, chorus-bin-install, deploy-daemon-card.sh. "Conventions" here = shell-script house style. Distinct from loom-practices (Wren-owned), which names framework-level disciplines (TDD / Actor-BDD / API-first). One layer narrower, written down so the next script doesn't drift.

What ships in this card

  1. agent-state.sh deploy verb. agent-state.sh <crate> deploy calls chorus-deploy <crate> then launchctl kickstart -k then verifies running PID's cdhash matches just-installed cdhash. Emits paired service.deploy.{started,completed,failed} with full payload.
  2. agent-state.sh rollback verb. Invokes chorus-deploy <crate> --rollback, kickstarts, verifies. Emits paired service.rollback.{started,completed,failed}.
  3. Cdhash verify in deploy + restart paths. codesign -d --verbose=4 parsed for CandidateCDHash line. Mismatch between pre/post cdhash on deploy = service.deploy.failed reason=cdhash-divergence.
  4. 6 spine events registered. service.deploy.{started,completed,failed} + service.rollback.{started,completed,failed} in designing/schemas/spine-events.json, vertebra=operating, source=chorus-events.
  5. 6 per-verb MCP tools. chorus_service_{status,start,stop,restart,deploy,rollback} registered on chorus-api MCP server. Each tool wraps the corresponding agent-state.sh verb, enforces the role-authorization rule, returns typed-refusal taxonomy.
  6. Lifecycle test file. proving/scripts/tests/test-agent-state-lifecycle.sh covers: status on running service, deploy + cdhash verify success, deploy + cdhash mismatch failure (synthetic), rollback to prior cdhash, refusal on unknown service.

Open questions (handled in implementation, surfaced for future cards)

  • Concurrent operators. Two roles invoking agent-state.sh com.chorus.api restart simultaneously — current behavior is last-write-wins (no lockfile). Same shape as deploy-daemon-card.sh's per-card slot isolation gap. Future card adds ~/.chorus/locks/<svc>.lock with stale-lock recovery.
  • start/stop/restart spine registration. The script emits service.{start,stop,restart}.{started,completed} but the schema only registers deploy + rollback per this card's AC4. Follow-on card registers the remaining 6 verbs so cardinal-six audit (#2941) reaches full coverage.

Observability

Three streams today: Loki (logs, port 3102, ~30d retention), Prometheus (metrics, port 9090, scraped from chorus-api + node-exporter + grafana), spine events (chorus-log → chorus-hooks daemon → SQLite + Loki shadow). Grafana dashboards (port 3100) read from all three. Frame: Loki = "what happened", Prometheus = "how much", spine = "what did the team intend".

Gaps: no single query that joins all three across a trace_id (#2931 work points toward this); cross-stream correlation lives in Jeff's head. Dashboard coverage uneven — some surfaces (cardinal-six, deploy pipeline) have dashboards, others (alert flow, harvester health) don't.

Security

Three primitives in place: macOS TCC (AppleEvents permissions for chorus-hooks injection, granted per-cdhash), codesign + cdhash (every chorus binary signed with local "Chorus Local Signing" identity; cdhash tracked in chorus-manifest), write-scrubber hook (Rust PreToolUse hook blocks credential writes to shared files). No formal threat model document; no audit of "what could an adversarial agent do" beyond the hook coverage.

Gaps: no documented threat model; TCC permission grants drift across cdhash churn (mitigated by #2734 atomic install but not eliminated); no rotation policy for the local signing identity; no audit of MCP tool surface for write-amplification paths.

Environment

Single environment today: production-on-Library. No dev/prod split. No staging concept. Werks (per-card ephemeral worktrees per #2913) provide build-time isolation but not runtime — they share the same Loki, Fuseki, LaunchAgent space as production.

Gaps: no path to test daemon changes without affecting production (chorus-api restart during a session affects every active MCP client — observed 2026-05-15). No staging chorus-api to dry-run MCP changes. $CHORUS_DEMO_BIN / $CHORUS_BIN two-slot model drafted in build-and-deploy-service-design.html Per-card slot isolation section addresses one dimension of this.
