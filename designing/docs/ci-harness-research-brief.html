CI Harness Architecture research brief

Drafted 2026-04-27 · Author: Silas (Chorus architect role) · Origin: research agent + Chorus synthesis
The CI/CD pipeline is a harness — a feedback loop for humans and agents — not a constraint engine. Each stage places sensors with a declared contract: what it watches for, when it fires, what its failure means. Block narrows as you move right (commit → main → production). Friction tolerance scales inversely with commit frequency. Drift detection runs async; in-loop correction stays fast.

Why this exists

Chorus runs at multi-times-per-day shipping cadence with three AI roles plus one human committing to main. The CI apparatus we built was over-rotated into rules-as-enforcement — branch protection, required-checks, ratchets that block commits — which produced gate-stacking, cards-as-protection, and architect-clean splits that consumed days of work without delivering card outcomes. This brief captures the architecture we want to converge on: sensors-as-feedback, with each test annotated by stage and contract.

The load-bearing distinction. A sensor gives the agent (and the human) data about what was produced. A rule forces an action when a sensor fires red. Sensors enable correction; rules enforce policy. Most "CI failures" we've fought were rules failing to be useful, not sensors failing to detect. The fix is to keep most sensors and remove most rules.

Day-1 required-checks list: the rules side of the architecture is governed by DEC-2525 — TS hermetic + Rust hermetic only at Phase 1 ship. Add/retire criteria + 48h shuffled-runs verification gate live in that DEC; this brief sets the architecture, the DEC sets the actual list.

Charity Majors's principle applies: don't add tests to compensate for missing observability. If production signal is good and rollback is fast, you can ship with less pre-merge testing — and you should.

Friction math

Rough rule from DORA elite-performer literature and Böckeler: friction tolerance scales inversely with commit frequency. For Chorus's shape (4 actors × ~10 commits/day, deploys via app-state.sh hot-reload):

  • Pre-commit budget: <10s, hard ceiling 30s. Past 30s, agents start --no-verify-ing and humans context-switch. Empirical breakpoint cited across the literature: humans tolerate ~7s before disengaging; agents tolerate longer but compound the cost across retries.
  • PR budget: <5 min for blocking signals, <15 min total. Past 15 min, PRs stack and reviewers batch — kills trunk-based flow.
  • Hooks that pay for themselves at high cadence: secret scanning (catastrophic to miss), formatter (zero-thought, prevents diff noise), type-check on changed files (catches the majority of broken-main causes for the cost of seconds).
  • Hooks that don't: full test suite pre-commit, mutation testing pre-commit, any inferential check pre-commit.

CI → CD path

What separates "CI green" from "confidently deploy from main N times/day" (Charity Majors / Honeycomb canon):

  1. Smoke test post-deploy — one synthetic transaction per surface, runs in <30s, rolls back automatically on fail.
  2. Observability over tests — structured logs + traces + a small set of SLOs.
  3. Feature flags — decouple deploy from release. New behavior dark-launched, flipped after observation.
  4. Automated rollback — the deploy pipeline must know how to undo itself without a human. Single highest-leverage CD investment.
  5. Canary or soak window — for fleet deploys it's a canary; for hot-reload single-host (Chorus's shape) it's a "soak window": hold the new process under synthetic + real traffic for N seconds before declaring success.

Minimum viable CD for Chorus's hot-reload context

  • app-state.sh deploy writes a deploy marker (git SHA + timestamp) to a known location.
  • Smoke test runs immediately after reload, hitting one endpoint per surface (chorus-api 3340, gathering 3000, clearing 3470, vikunja 3456, fuseki 3030).
  • If smoke fails OR error-rate over next 60s exceeds baseline, app-state.sh rollback reverts to previous SHA.
  • Deploy marker + smoke result emitted as structured log → Loki → dashboard.

That's the whole shape. No blue/green, no canary infrastructure, no fleet — just deploy-marker + smoke + rollback + observed window.

Böckeler's named meta-anti-patterns:

  • Feedback-only harness — agent that keeps repeating the same mistakes (no guides to encode learning).
  • Feedforward-only harness — agent that encodes rules but never finds out whether they worked (no sensors to detect drift).

Agent-specific concerns

What agents need from CI that humans don't:

  • Sensor messages written for LLM consumption. Custom linter messages that include instructions for self-correction. Human-friendly errors ("expected ;") are wasted on agents; tell them how to fix it.
  • Self-evaluation surface before human review. Agents generate faster than humans review; the harness must catch most issues pre-handoff. ("Humans on the loop," not in it.)
  • Determinism over speed. Flakes poison agents far worse than humans because the agent will retry, masking real signal. Quarantine flakes aggressively.
  • Loop-detection sensors. LangChain reported a substantial Terminal Bench 2.0 jump (52.8% → 66.5%) by adding only a self-verification + loop-detection sensor — no model change.

Working terms not yet canonical

"Story-numbered backstops" and "ratchet-as-rule" are working terms in Chorus, not industry-named patterns. The behaviors are widely described in the testing-smell literature; the names aren't. Surfaced here so future readers can map our vocabulary onto the established frameworks.
