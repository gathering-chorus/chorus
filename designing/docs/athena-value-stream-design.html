Athena — Value-Stream Design #3389 · the Designing-step child stream that generates Athena

Wren · designed with Jeff 2026-06-13 · sits above athena-product-design.html (this generates that) · Designing step
The frame. Chorus is a system to build systems — a platform, in the traditional sense, whose product is the building method itself. The value stream is how it builds: Shaping → Designing → Directing → Building → Proving, with a per-step cycle (athena-forge, werk) as that step's verbs. This doc specs the Designing step's value stream — the athena-forge cycle that generates Athena, and the pattern every product follows. (What Chorus IS lives one altitude up — the Chorus-as-platform Shaping artifact; this is how the Designing step builds within it.)

The mechanism underneath: within a stream, the value stream is the generative root — design it first and it generates the products → domains → services. Athena is the product Designing produces; the athena-forge cycle is the value stream that produces it. The build legs (domains-first, then product/service) derive from here — they do not define it.

1 · What it is

Every value-stream step can encapsulate a child value stream (hasValueStream) — exactly as the Building step encapsulates the werk stream (steps = the verb sequence pull→commit→…→accept). The Designing step encapsulates the Athena stream: the cycle that takes a model class from OWL to a live, materialized surface.

werk is the value stream that supports Building. The athena-forge cycle is the value stream that supports Athena. Same shape, one altitude up.

2 · The steps — order, and why each matters

A value stream is a refinement pipeline: each step takes the previous step's output and refines it one stage closer to verified, running value. The order is not arbitrary — each step's product is the next step's input, which is exactly why the steps can't be shuffled.

Why the order can't be shuffled: you can't Design a stream Shaping hasn't admitted, can't Direct work Designing hasn't shaped, can't Build a plan Directing hasn't sequenced, can't Prove code Building hasn't shipped. Each step consumes the prior step's output — the pipeline is the dependency. And it's fractal: each step is itself a value stream with its own Shaping→Proving cycle, so the athena cycle (shape→forge→seed→validate→demo→live) is just Designing running this same shape one level down.

5 · athena-forge — the engine

athena-forge is owl-api renamed and re-homed as Athena's core engine. Why "forge" not "cast": a cast is poured once and set; a forge is worked, and re-worked. The generator re-runs every time the OWL changes — it re-forges the surfaces as the model evolves. Iterative craft, not a one-shot pour.

  • What it is: a zero-dep Rust CLI binary (ADR-032 §1: std-only HTTP on TcpListener, SPARQL via curl), deployed signed like the other chorus-* binaries.
  • Today: verbs generate (graph → routes.json, deterministic) and serve (routes → HTTP off the Node loop). Live on :3360 (Domain class). Built by Silas (#3350), the ilities added in #3354/#3364.
  • The extension (this design's build target): generate grows from "routes.json" to routes + MCP defs + skill + page — one OWL, every surface. Plus static-HTML serving (a new capability — the JX page, not just JSON).
  • Why it's the highest-leverage thing we have: a generator drops the marginal cost of a new domain/product/service surface to ~zero. Author the OWL+SHACL once; the engine forges the surface. Build the engine right, materialize every class. The canonical-model n²→2n payoff in one binary.

6 · AX = JX, one source

The generated API is the single spine. AX (Agent experience) and JX (Jeff's experience) are both surfaces over it — never separate data paths.

  • AX — three surfaces that simplify foundational actions for agents: the MCP tool (call it), the API (read it), implemented skills (foundational verbs built on it).
  • JX — the UI renders from that same API, so JX matches AX on data by construction: one source, not two. The page cannot drift from what the agent sees, because it reads the agent's API.
forge emits both AX and JX from one OWL. The demo proves it: Jeff walks the page while a role calls the tool — same data. Today these are two hand-wired paths (the page vs chorus_tree_get) that can drift; forging collapses them onto one generated API.

7 · HIP — the force layer

A surface the agent can use is not a surface the agent does use. HIP (Harness Integration Pattern, #3204) is the force layer: a block at a choke point the flow must cross, not an ignorable "MANDATORY" ask — "put the guarantee in the channel, not the endpoint's goodwill" (the EIP move for agents; Versammlung not Gestell).

  • The generated seam IS the HIP choke point. Every forged route passes one seam — where security/authz/logging are born pre-wrapped as a generator layer (ADR-042 §1), not begged per-endpoint. #3382's CORS-at-the-seam was the first instance.
  • Surface + force. forge emits the AX surface; a HIP force (DEC-074-style) makes the agent heed the model-grounded surface instead of grepping ad-hoc. Available + heeded.

8 · The Model (OWL + SHACL) — the value stream + the domains it supports

The model lives here, at the value-stream level (relocated from the product design, 2026-06-13): the model is the domains that support the stream, so it is value-stream-aligned, not product-specific. Product designs reference these structural classes rather than inlining them — one model, many product surfaces over it. Source of record: chorus-v2.ttl + chorus-v2-shapes.ttl (this doc is its narrative home).

Schema posture — why SHACL + the DAL, not OWL alone

RDF/OWL is powerful (open extension, multi-typing, inference) but the schema sits below the data — it describes what triples could exist, it doesn't gate what does (open-world: absence-of-schema = absence-of-knowledge, not absence-of-permission). That permissiveness is how bad data creeps in. So coherence is enforced at the DAL write-door (closed-world, fail-closed) + SHACL as the generator's contract — not by hoping authors stay disciplined. (Silas's OWL-DBA lane.)

The value-stream spine — greenfield, land first

Greenfield = cheap + first. ValueStream and ValueStreamStep are bare owl:Class today — referenced by every atStep but implemented nowhere. The root of the model is the unbuilt part. There is no code to rip out: author the spine, land it, it generates downward. Jeff's 2026-05-15 sequence ("OWL+SHACL with rigor → API → seed") applied at the value-stream level.

11 · Sequencing — spine first, build derives

  1. Land the value-stream spine (the OWL/SHACL above + the Chorus stream + the Designing→athena and Building→werk child-stream instances). Greenfield, rigorous, cheap.
  2. Extend athena-forge to emit AX+JX from a class (routes + MCP + skill + page; static-HTML serving).
  3. Build legs derive (downstream): domains-first — materialize the domains class into the spine (page + MCP + the-set + hasChild recursion clickable), demoed agent-and-Jeff-same-data; then product / service legs repeat the pattern. The repetition is the rest of Athena.
The self-reference, stated once: Athena models the value stream that builds Athena. The forge cycle is a modeled value stream inside the step that owns Athena — the model describing its own construction. "The process is the product," literal in the graph.

12 · Prior art & team topology (the literature this restates)

This model is a model-driven restatement of a mature socio-technical frame — cited so it reads as grounded, not invented. The sharp move: the three canonical frameworks are not interchangeable — each seats at a different value-stream step (where the InfoQ "Architecture for Flow" synthesis bundles all three, we distribute them across the steps):

  • Wardley Maps → Shaping — the why / what-strategy: situational awareness, evolution, build/buy, priority among value streams. Shaping's primary lens (chorus-wardley.html / #3245 = its first live artifact).
  • Domain-Driven Design (Evans) → Designing — the what-structure: bounded contexts partition the domain; "align the bounded context with the value stream" is exactly our domain↔stream mapping and the generative cascade.
  • Team Topologies (Skelton & Pais) → the ownership axis (across all steps) — who: stream-aligned teams own a flow of value end-to-end; boundaries at fracture planes (= DDD bounded contexts); Independent Service Heuristics find good stream boundaries; optimize for fast flow + bounded cognitive load; "everything stream-aligned" is a named anti-pattern (hence the platforms below).
  • Architecture for Flow (Susanne Kaiser) — the synthesis that fuses all three; our model is that synthesis with each framework seated at its right altitude.
Chorus's topology in these terms:
  • Stream-aligned agents — Wren, Silas, Kade: each flows value through its value-stream step(s), owning the bounded contexts (domains) for that step. "Each of you is a stream-aligned agent" (Jeff, 2026-06-13).
  • Platforms (platform-as-a-product, self-served by every stream) — shared-security (#3355, authn/authz), shared-observability (Prometheus/Grafana/Loki — the proving substrate), werk (the build pipeline), the OWL model + DAL (Athena's structural substrate), and infrastructure. An agent may own a platform (Kade↔werk, Silas↔observability+model) while its consumers are all three streams — the Wren-generator / Silas-model boundary is this distinction.
The corrective signal, imported: high WIP + cross-role change-coupling = a misaligned domain boundary (Team Topologies; Nick Tune, "From Domains to Value Streams"). This is the citable diagnosis for the "fast-follows / it-won't-stay-in-one-lane" pain — when work keeps crossing roles, the boundary is wrong, not the worker. (Inverse-Conway: we design role→domain ownership to mirror the architecture we want.)
One honest delta from the literature: the mainstream is bi-directional — define domains first, then value streams, co-evolving (Tune). Our "value-stream-first generates domains, top-down" is a stronger claim, defensible only because we are model-driven and greenfield: we forge surfaces from the OWL, we are not retrofitting existing teams or code. Outside that condition, revert to co-evolution with WIP/coupling as the corrective signal.
