Processing dependencies tied to products (revised #3077)

The output→consumer→domain edges above roll up to the 7 Athena products — the level that answers "if this processing stops or moves, which product degrades?" This is the unit that gates the #3077 off-loop refactor: a step may move off the event loop, but each product's processing inputs must be preserved.

Data-dependency ≠ loop-cost. This map answers what breaks if an output stops (data dependency). That is a SEPARATE axis from what blocks the event loop (loop-cost = synchronous in-process work). Per #3077's code-verified classification: collectRdf is fully async (await fetch to Fuseki) — its 44s is Fuseki wall-clock latency, NOT a loop-block, so it's #3066's lane. The synchronous better-sqlite3/parse work (index reindex, embed COUNT/UPDATE, search FTS) is what blocks the loop. When #3077 moves the SYNC processing off the loop, it must preserve every product dependency in the table above — off-loading changes where/when processing runs, never whether a product still gets its input.

How this folds into Athena v2. The crawler is now a service entry in data/athena/tree.json (chorus:service-crawler) carrying produces (each output → owning domain) and consumedBy (each consumer → its domain). The consuming domains carry consumes/dependsOn edges pointing back. "Safe to turn off / decompose vs has unmapped runtime consumers" is now a tree query, not a guess.

Implications for #3069 / ADR-033

  • Decomposition must preserve the load-bearing consumers — above all context_inject's every-prompt reads (crawler rows + spine + LanceDB) and the 4 alerts. These were the "runtime dependencies we don't understand" — now mapped.
  • The graph-write (crawler-hydrate-graph.sh) is the to-be done right already — structure into the graph. Extend that pattern; retire the flat-summary-to-messages where the graph can hold it.
  • The traces 108/min is the first thing to question — highest write volume, no found consumer. Likely cut or sample, not decompose.
  • Safe vs needs-care: frequency throttle (WatchPaths, #3068) = safe, removes no data. Turning off any OUTPUT = needs care (this map shows which break). Turning off traces writes = likely safe (no consumer found) — verify.
