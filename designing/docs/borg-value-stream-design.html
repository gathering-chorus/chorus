1 · The borg-* cycle

Werk is a line: commit→…→accept, and it ends. Borg is a loop: it tends a living system that never graduates. The steps are the gardener's, not the builder's — you don't build the plant (that's Werk/sowing), you watch and tend it, and the system's own feedback tells you what's wrong. The skeleton is MAPE-K (the canonical autonomic self-managing loop); the soul is permaculture (tending a living system, the homeostasis/Ashby lineage) — which is why Borg reflects, it does not assimilate-itself.

…and around again, by season. The cadence is the product, not a finish line. The Reflecting step hasValueStream this cycle (name matches the step; the template's fractal recursion); orchestrated by a chorus_borg reflection loop (the chorus_werk / chorus_athena analog). The sibling Proving step hasValueStream the per-card ceremony (present→gather→verdict→accept); Borg spans both.

Three modes, one root — one shared first step. The borg corpus names three: reflect (observe itself — this doc), assimilate (extract a legacy system's knowledge to ICD and regenerate — the assimilation pattern), mitose (reproduce the team across products — borg-mitosis). They are not disjoint: each begins with the herald / xray (assimilation's Phase 1 OBSERVE is the Silver Surfer) — a system is seen into the graph first; only then does the mode diverge (reflect Surfaces, assimilate Regenerates, mitose reproduces). This doc models reflect, but the herald it shares is the front door to all three. Self-reflection stays observe-only (Surface; the role Prunes); assimilation is where Borg acts on the outside.

2 · borg-reflect — the engine

borg-reflect is the heart, the way build/deploy are werk's and forge is athena's. It is the continuous monitor→alert loop run over the model: every service/surface declared in the model is reflectable through the same contract, so adding a thing to watch is a model act, not a new bespoke probe (the #3405 lesson — a probe that lies is a probe hand-wired apart from the model). It runs on the substrate that already exists.

flowchart LR
  M["plant — monitor / SLO, declared in model"] --> T["tend — emit / scrape"]
  T --> S["spine + Prometheus (the signal substrate)"]
  S --> R["read — evaluate vs model (map vs mirror delta)"]
  R --> U["surface — route verdict to owner by ownership"]
  U --> P["owner prunes"]
  P --> F["feed — learning into knowledge"]
  F -.->|"next season"| M
borg-reflect — the autonomic loop, model-driven, reflect-not-act. The owner (not Borg) prunes; the loop closes by feeding the learning back.
  • Substrate, not new tools: the spine (events: liveness/freshness/latency/errors) + Prometheus (gauges: saturation) + Alertmanager (evaluation/routing) + Loki (logs) already run. Borg-reflect uses them correctly and retires the bespoke probe-and-nudge scripts.
  • Bounded by construction: the engine's own reads are bounded (the #3406 lesson — the watcher must not be the disease). Borg is the most boring, most reliable code in the system precisely because everything trusts it.
  • Symptom/SLO, not cause-noise: page on what impacts a role/Jeff; everything else is a surface, not a page (Majors). Silence ≠ healthy — a monitor that stops emitting is itself an absence alert (meta-monitoring; the #3406/#3405 lesson, first-class in the contract).

Crawlers ARE heralds — the Silver Surfer (Jeff; team metaphor of record). The name is deliberate: "heralds, a la Silver Surfer and Galactus." Galactus = the graph — it assimilates everything the heralds find into one unified structure (to borg a system = Galactus consuming a world). The Silver Surfer = the herald — the crawler / discover-* that flies ahead, scans the terrain of an existing system, and reports back what's there. So the crawler is not a separate Directing-step service outside Borg; it's the inbound face of heralds, the xray. To bring any system under Borg you first send the Surfer: discover its structure, feed Galactus — only then can you plant a monitor on what you found. That makes the xray the first step in applying Borg to an existing system, and it's read-only discovery (reads structure, never mutates the target) — so assimilation isn't a mode outside reflect-not-act, it's how a system enters the reflection cycle. Permaculture principle 1, observe & interact, at onboarding depth: survey the site before you garden it.

5 · Sequencing — spine first, legs derive

  1. Author the value-stream spine — value-stream-borg + the six cycle steps as ValueStreamSteps, and the monitor/alert domains, in the model. Land it (greenfield, no rip-out).
  2. borg-reflect leg one = liveness (ADR-043's first build, Jeff's "alive every N" directive): every persistent daemon emits service.alive; one absence-detector pages the owner after X misses. This is plant→tend→read→surface for one monitor type, proven end-to-end, and it closes the #3406-class gap.
  3. Then the legs repeat the contract: freshness (scheduled jobs) → errors/crash-loop → saturation → latency-SLI → synthetic. Each is one monitor type instanced through the same model→emit→evaluate→route path. The repetition is the rest of Borg.
  4. Alert domain as config-driven routing (#3407 generalized) — recipient = ownership lookup, no code change to reroute.
  5. Probe truthfulness (#3405, landed) folds in — bespoke probes retire as their signals move to the typed contracts.

6 · Operating-model frames & the two self-observations

Borg's cycle isn't invented — it instances three well-trodden frames, each the same loop from a different angle. Borrow their questions, recalibrate their benchmarks to our scale (our constraint is attention/coherence, not deploy frequency):

  • SRE — how it operates. SLOs + error budgets, alert on symptoms not causes, blameless postmortems, toil reduction = plant→tend→read→surface→prune→feed, one-to-one.
  • DORA — how we measure it. the four keys (deploy freq, lead time, change-fail rate, restore time) are computable from our own spine/cards/deploy data — fitness on our own telemetry.
  • Team Topologies — what Borg is. a platform-team product delivered X-as-a-Service to the stream-aligned roles (= AX=JX); Thinnest Viable Platform (reduce cognitive load, never add it); reliability is the contract.
Two kinds of self-observation — don't conflate them. Runtime health (monitor/alert/logs/metrics) asks is it running well? Structural coherence (the crawler / heralds — the xray) asks is it structured clearly; does the map match the territory? The crawler is the deeper one for a system-for-building-systems: coherence has two halves — the model is well-formed (SHACL/gates) AND true to the system; the crawler is the true half, and without it a well-formed model is a beautiful lie. Because mitosis copies the model forward into every system Chorus builds, model coherence is germ-line integrity — a drifted core breeds drifted offspring.
