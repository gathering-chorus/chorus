Commits — Service Design

2026-04-29 — Kade (#2586). Updated 2026-05-02 — v3, three-lens authoring: PM constraint Wren, architecture Silas. Today's landing: #2645 v3 amendment — substrate owns the working tree, inversion of control, Class B thin→thick, SWAT-as-card-type encoding (Wren), substrate-owns role-state-to-card attestation (Silas). Cards filed: #2661 v3-1 chorus_commit MCP tool, #2662 v3-2 skill migration, #2663 v3-3 block raw git mutations, #2665 v3-4 retire env-bypass, #2666 v3-5 SWAT-as-type, #2667 v3-6 retire bash scripts.

A commit lands when, and only when, the role calls the service's commit for its active card, and the service (a) verifies the role's branch matches its role-prefix and active-card, (b) runs the pre-commit checks against the staged paths and they pass, (c) pushes via its serialized internal queue, and (d) GitHub's branch-protection rebase-merges to main with required-checks green against the merged SHA. The agent's only signal is the response from the service: SHA-on-main, or refused-with-reason.

Why v3

v2.5 named three failure modes (A read-stale-content, B cross-role wrong-branch, C same-role wrong-card) and accepted (A) as residual risk because physical isolation (worktrees) was rejected. The morning of 2026-05-02 produced the (A) receipt: Silas's uncommitted ontology + design-doc work was overwritten on disk when Kade ran git checkout on the shared tree to gate-code wren/2649. ~30 min of architect work, recovered from /tmp fragments. The lesson is not "be more careful" — it's that invariant execution cannot rest on agent discipline. v2.5's accepted-residual was a norm with a hopeful name.

Architectural primitive (load-bearing)

The substrate owns the working tree. Everything else in this design — who owns branches, locks, hooks, push, role-state — follows from that single shift.

As-Is — v2.5 today (norms with hopeful names)

Agent has direct paths to the working tree (edit, checkout) and to git operations (commit, push). Bypass envs short-circuit pre-commit. Mode A contamination follows from any agent's git checkout rewriting shared .git/HEAD.

Agent ▸
edit files in shared /chorus working tree
Agent ▸
git checkout / switch / reset — unrestricted; rewrites shared .git/HEAD and overwrites peers' uncommitted work on disk (Mode A)
Agent ▸
set CHORUS_TEST_FORCE_FIX_CARD, --no-verify, etc. — bypass envs/flags short-circuit gates
Agent ▸
bash git-queue.sh commit → branch-check.sh → pre-commit hook (skippable) → flock → stage
Agent ▸
bash git-queue.sh push → rebase-on-conflict → push
GitHub ▸
actions + branch protection → rebase-merge to main
Failure surfaces visible: agent has direct paths into both working tree and git operations. Bypass envs/flags work. Mode A is structurally possible.

To-Be — v3 (substrate owns the working tree)

Agent has no git checkout/commit/push exposure. No env on the wire — chorus_commit accepts (role, paths, message) only. Card-id is derived from role-state, not passed. The substrate is the only entity that can move HEAD; Mode A contamination disappears structurally.

Agent ▸
edit files in /chorus (read + write content; cannot move HEAD or run git mutations)
Agent ▸
chorus_commit(role, paths, message) — single declarative call. No card-id arg. No flags. No envs.
Service ▸
derive card from role-state (no agent-passed card-id) → read card.type → apply ruleset (default or swat)
Service ▸
create / switch branch — only the service moves HEAD → stage paths → run pre-commit (internal, no --no-verify) → internal flock → push (rebase-on-conflict)
Service ▸
emit spine event chorus_commit.invoked / refused with reason
GitHub ▸
actions + branch protection → rebase-merge to main (required-checks green against merged SHA)
Service ▸
response to agent: SHA-on-main, or refused-with-reason. That is the agent's only signal.
Failure surfaces removed: no agent-side checkout, commit, push, env, or flag. Card-id derived. HEAD-mover is the substrate. Mode A is structurally impossible.

Class B implementation mode — thin → thick

The shift above (As-Is → To-Be) is a single architectural move named in ADR-028 terms. ADR-028's Class B "single canonical adapter" did not previously distinguish thin (shells out to git, integrity depends on agent discipline) from thick (substrate executes the contract internally). v2.5's git-queue.sh is thin; v3 graduates it to thick. Same shape as #2652 cards CLI (was thin shell-script, became canonical adapter all transports delegate to) and #2649 chorus-api (was vendor copies, became transport over gathering's public/). v3 is the same move applied to commits. Maps to chorus:AuthBoundary.checkType: 'service' vs 'ad-hoc' from the security model — ad-hoc boundaries are migration candidates exactly because their integrity depends on agent discipline.

No branch_create, no push, no set_env, no bypass. If the surface needs to grow, it grows by adding card-types the service recognizes — never by adding flags the agent passes.

SWAT / emergency — encoded as card type, not as flag

(Wren constraint.) Any "skip" or "emergency" flag exposed at the wire is a bypass-class door. v3 closes that door by making fast-path a property of the card the agent is committing for, not of the call.

A type:swat card declares its lighter-weight rules (e.g., allow no-test-evidence, allow same-branch-as-target). The service reads the card type at commit time and applies the matching rule set. The agent's call is the same — chorus_commit(role, paths, message) — but the card itself carries the relaxation. There is no agent-side flag to flip; the relaxation is a property of the work, not of the actor. Spine event chorus_commit.swat_path_used emits on every fast-path use, with reason and card id.

What this design does not do

  • It does not rename existing artifacts (git-queue.sh, hooks). They become internal.
  • It does not retire --no-verify from raw git — that's outside chorus's enforcement scope. It removes the path through chorus that exposes it.
  • It does not solve (E) stale-CI; that lives at the GitHub-settings layer.
  • It does not specify the wire protocol (HTTP vs MCP vs IPC) — implementation detail for cards under this design.
  • It does not canonicalize raw-git read surfaces (git log, git diff, git show, git blame); these stay direct. Reads are safe-by-construction (no HEAD or tree mutation). ADR-028 MUST 3 audit verdict: PARTIAL, gap explicitly accepted.

History

v2.5 (2026-05-01) — worktree retirement, mode A residual accepted

The v2 amendment drafted earlier on 2026-05-01 (internal chorus/.worktrees/<role>-<hash>/ + mandatory SessionStart cwd-enforcement) was rejected by Jeff. Protected primitive: role-directory IS session-start. Any mechanism that moves session-start away from /chorus/roles/<role>/ is in the rejected family.

  • All sibling worktrees retired (chorus-kade/, chorus-wren/, chorus-silas/, chorus-2526/).
  • worktree_contamination_guard hook retired (#2625 dispatcher hookup removed in #2640).
  • Three failure modes named: (A) read-stale-content accepted-residual; (B) cross-role caught by branch-check + pre-push; (C) same-role wrong-card sequenced as #2641.

v1 (2026-04-29) — original service-design surface

The commits domain — every git mechanic between "I have a change" and "main reflects it" — emerged as a stack of tactical fixes rather than a designed surface. v1 named the four classes of incident (cross-branch contamination ×3, required-checks asymmetry, git-queue branch-blindness, worktree grain unresolved) and proposed per-role worktrees + branch-check + GIT_* env scrub + lockstep enforcer as the To-Be moves.
