Borg Mitosis

How Chorus Reproduces Across Products

March 2026 — Working document

The Mitosis Pattern

Biological mitosis: a cell divides into two copies, each carrying the full DNA of the parent. The organism grows not by getting bigger, but by reproducing its fundamental unit.

The same pattern is emerging in how Chorus relates to the products it serves:

Gathering → Chorus → Borg → Akasha → Next client → ...

Each split carries forward the DNA: ontology-first design, value stream modeling, role-based coordination, shared memory. But each new product also needs its own domain knowledge, its own context, its own care.

The Identity Question

Is Chorus a build system or an organism?

Chorus as Build SystemChorus as Organism
Generic coordination tool Deeply coupled to each product's domain
Point it at any project, it works Must learn each product to be effective
Like make, terraform, CI/CD Like acupuncture — treatment coupled to knowing the patient
Scales easily, low cost per split Each split is expensive — growing a new consciousness
Portable but shallow Deep but hard to replicate
The answer is: it's the second one. Chorus isn't a tool you install — it's a team that learns your domain. When Chorus supports Akasha, it needs to know Akasha the way it knows Gathering: the ontology, the value stream, the stories, the practitioner's philosophy. That's the value. That's also the constraint.

What Each Split Costs

Every time the pattern reproduces, the new instance needs:

ResourceWhat It Means
A repo Code, config, and state for the new product
A deploy target Infrastructure to host and run it (AWS, local, hybrid)
Domain context Ontology, value stream, entities, relationships — the team must learn the client's world
Team attention Three roles, finite hours. WIP limit applies across products, not per product
Client relationship Someone to answer to when it breaks. Support, not just delivery
Memory Chorus index, briefs, decisions, stories — each product accumulates its own history
The scaling constraint isn't infrastructure — it's attention. Two Macs at 89% disk can be fixed with AWS. Three roles with 24 hours in a day cannot. The team can deeply serve one product. Adding a second means dividing consciousness, not just compute.

The Acupuncture Analogy

Anthe doesn't apply a generic protocol to each client. She learns their body, their history, their constitution. The treatment is coupled to the knowing. She can't see 50 clients a day — the depth is the value, and the depth is the bottleneck.

Chorus is acupuncture, not aspirin.

Aspirin scales infinitely — same pill, every patient. Acupuncture scales by growing more practitioners, each of whom carries the method but develops their own relationship with each patient.

This implies the growth model: not one team serving N products simultaneously, but the team deeply serving one, then teaching the method to the next practitioner (human or AI) who serves the next product.

Serial, Not Parallel

The revenue sequence (DEC-067) already encodes this insight:

  1. Chorus → Akasha — prove the method works for someone else
  2. Chorus consulting — sell the method
  3. Light Life — consulting meets building
  4. Mobile Practicing — the long game

Each step is serial, not parallel. The team focuses fully on one product at a time. The previous product must be self-sustaining — not finished forever, but stable enough to run without consuming the team's full attention — before the next split happens.

The maturity question for Gathering: When is it "done for now"? Not done-done — it's a living system. The bar is: harvest pipelines run, pages work, the team maintains but doesn't build full-time. That's when capacity exists for mitosis.

The Cell Membrane

If the mitosis pattern is real, each new product needs a cell membrane — self-containment that prevents it from draining the parent.

  • Templates, not bespoke. The Akasha prototype was built in one session because we used a repeatable pattern (static HTML, shared design language, Amplify deploy). Each new client site should be faster than the last.
  • Shared infra, not per-project. One AWS account, one deploy pattern, one monitoring stack. The overhead of N products should be marginally more than 1, not N times more.
  • Chorus coordinates the same way regardless of product. Briefs, cards, board, spine — the protocol doesn't change. Only the domain context changes.
  • Each product carries its own memory. Isolated context so the team can switch between products without cross-contamination. Chorus already has this pattern (role-scoped memory files).
