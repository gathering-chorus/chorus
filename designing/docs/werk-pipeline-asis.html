Why this needs the to-be
  • On-loop freeze: pull/acp/index run synchronously on the chorus-api event loop — heavy work blocks the coordination spine (the #3050 / #3051 / #3055 / #3067 freeze class).
  • Opaque monoliths: pull and acp are one tool each — a mid-acp failure is a single tool error, not a per-step state you can see or resume from.
  • State scattered: progress is split across the index, spine, and cards; gh holds only commits — no single queryable "how far did this card get?"
  • Skill-as-logic: demo's gate logic lives in markdown the model executes — the shortcut-prone pattern #2751/#2750 already hardened pull/acp away from, but demo still carries it.
  • No act, no verb binaries, no commit-status mapping — nothing local, testable, and per-verb to run or prove.
As-is = three opaque tools on the shared loop, state scattered, gh holds only commits.
To-be → werk-pipeline-model.html: decomposed verbs on act (local, off-loop), gh as source-of-truth + process-holder with one commit-status check per verb.
