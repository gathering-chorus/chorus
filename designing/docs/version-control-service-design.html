Version Control — Service Design

2026-05-03 — Wren + Kade pair (#2683). Replaces commits-service-design.html; subdomain renamed from commits-domain to version-control-domain the same day. Origin: chorus_commit MCP regression surfaced that the prior framing anchored on commit-side mechanics while the actual incidents were pull / checkout / working-tree races. The rewrite covers the full lifecycle.

Source of truth: Live actor list, BDD scenarios, contract entries, dependencies, gaps, and integrations live at /athena/domain-detail.html?id=version-control-domain. This doc narrates structure and intent — diagrams, rejected paths, implementation plan. It does not restate counts or content from the graph; references point at graph entities via the page above.
Designed target state. A version-control operation lands when, and only when, the role's request reaches git-queue.sh through the typed MCP surface (chorus_commit for write, chorus_commit_status for read), and the adapter (a) stages caller-named paths, (b) runs the pre-commit hook synchronously and surfaces stderr on failure, (c) commits and pushes under flock with bounded retry, and (d) emits one commit.landed spine event reflecting the actual SHA on main. Refusal taxonomy is strictly commit-safety: hook-fail, push-conflict. Coordination state — board status, role-state, branch-naming convention — never refuses Jeff's commit. Coordination is observed after the fact via spine, not enforced at the surface. The shared /chorus/.git/HEAD race during pull / checkout is an accepted residual risk addressed at the branching layer (micro-branches), not the substrate layer. Today's code does not yet match this contract — see As-Is below.

Overview

Version control is the substrate for how code in the chorus repo moves through its full git lifecycle: branch creation, commit, push, pull, rebase, merge, tag. The adapter is git wrapped by git-queue.sh — a shell-level flock-serialized surface for stage / commit / push. Gates run via standard git hooks (pre-commit, pre-push) at the adapter boundary; agents reach the surface through the typed chorus_commit MCP wrapper or the /acp slash command. The operating model is trunk-based development; the substrate exists to serve that discipline, not to invent a parallel coordination layer.

Jeff invokes commits via the /acp slash command, which routes through chorus_commit. Contract: the command performs commit + push and reports back what happened. Tail platform/logs/chorus.log for commit.landed events. Contract: one event per successful push, fields { sha, role, branch, timestamp }. Borg ingests for activity heatmaps; dashboards count emit-rate; Loki indexes for retrieval. Additional events chorus_commit.refused and chorus_commit.status_queried are emitted today and consumed by audit views.

Components (designed)

Adapter — git-queue.sh SHIPPED

The serialized commit surface. Wraps git stage / commit / push in a flock so concurrent role commits queue rather than race. Emits commit.landed on successful push. Returns non-zero on hook failure or persistent push conflict.

Hook layer — pre-commit, pre-push, branch-check, write-scrubber SHIPPED

Synchronous gates at the git boundary. pre-commit runs write-scrubber + branch-check on staged paths; pre-push validates outbound refs. All run inside git-queue.sh; failures surface stderr to the caller. Internal to the adapter — not separate actors.

MCP wrappers — chorus_commit, chorus_commit_status PARTIAL

Typed surfaces over git-queue.sh. Refusal taxonomy strict by design but not yet by code: today's chorus_commit still refuses on board state. The rewrite target removes the coordination-state reasons.

Trunk-based discipline (operating model, not code) NOT ADOPTED

Micro-branches, feature flags, vertical slicing, hard CI gate, force-rebase-before-merge, active-zones, merge train, fast rollback. The substrate doesn't enforce these — they're how the team uses the substrate. Adoption tracked separately as practice work.

As-Is — 2026-05-03 today

The typed write path ships and works for the commit half of the lifecycle: chorus_commit MCP routes through git-queue.sh, hooks run, push under flock, spine emits commit.landed. The refusal taxonomy in code still includes coordination-state reasons (board-unreachable, no-wip-card, multi-wip, branch-mismatch) — the v3 regression that triggered this rewrite. The pull half does not yet have a typed surface; roles fall back to direct git pull / git rebase. The chorus_commit handler has a known false-positive: returns push-conflict when the push actually succeeded but git-queue.sh exits non-zero on a warning path. The shared /chorus/.git/HEAD race during pull / checkout is unmediated by the surface.

Concurrent reads against shared mutable git state (#2706)

All three roles share one git repository at /chorus/, with shared metadata (HEAD, refs, index) in /chorus/.git/. The HEAD pointer and the index — the metadata that tells git which branch and commit you're on — are single files mutated by whichever role most recently ran a checkout, branch-switch, or pull. The working tree on disk (the actual .ts, .md, .rs files the roles read) is repainted by git to match HEAD on every checkout. Mid-session, a role can read a file at T1 and have its next read return content from a different branch, because a peer's checkout fired between T1 and T1+ε and rewrote both HEAD and the on-disk files. The race is silent: the reading role gets stale or wrong content with no error.

User-visible receipts (2026-05-03):

  • Dangling-commit recovery. Wren's commits during a build session landed on the wrong branch because HEAD bumped between her commit and her next checkout. Recovery required reading commits out of git reflog and cherry-picking them back. This was not a bug — the substrate was operating as designed under concurrent use.
  • Mid-demo cwd-bump. During Kade's demo, Wren's checkout to her branch repainted the working tree under Kade's session; his terminal was sitting in /chorus reading files that suddenly belonged to a different branch. Demo had to pause and reset.
  • The cascade. A run of substrate-fix cards (#2696, #2698, #2699, #2700, #2701, #2705) all shipped real fixes for downstream symptoms of this race. Each fix was correct in scope; together they were the substrate naming itself by repeated injury.

This is currently accepted-residual risk (status as of 2026-05-03). The team chose to absorb the race rather than pay the worktree-isolation cost. The per-role-worktree v1 convention was retired (#2640); a v2 redesign was rejected the same day; the protected primitive is "session-start cwd binds to /chorus/roles/<role>/." Watching has been spine-event-based: contamination incidents are recorded as metric data, not regressions.

Naming honestly: previous internal docs used "Mode-A" as shorthand for this state. The shorthand obscured what was actually happening (a concurrent-reader race against a shared mutable index). This document and the rest of the chorus tree use plain English.

Path to close (#2706)

Three candidate approaches. Each pays a different cost. The protected primitive — session-start cwd binds to /chorus/roles/<role>/ — survives in all three; any approach that moves session-start is in the rejected family.

Candidate A — Serialize working-tree mutation through the canonical adapter. Extend git-queue.sh to gate git checkout, git switch, and git branch under the same lockf that already serializes commit and push. Add the deny entries to the chorus-hooks guard so raw checkout routes through the queue. When a role wants to switch branches, it acquires the lock; concurrent peers wait. Cost-in-Jeff-coordination: zero (Jeff doesn't see lock contention). Cost-in-build: one new do_checkout in git-queue.sh (parallels do_pull / do_push), one regex addition to the guard, skill updates to route through the typed adapter (/pull and /acp currently call git checkout directly). Ongoing maintenance: same as today's adapter — bounded. Closes the race.

Candidate B — Read-side snapshot at session start. Each role's SessionStart hook copies the current working tree state into a role-private overlay at /tmp/chorus-session-<role>-<sha>/. Reads during the session resolve against the overlay. Writes still go through git-queue.sh against canonical /chorus/. Refresh on demand or per-card. Cost-in-Jeff-coordination: zero. Cost-in-build: SessionStart hook extension + a read-overlay layer that intercepts Read/Bash cat against repo paths. Ongoing maintenance: cache-coherence rules (when does the snapshot refresh?), disk overhead for three live overlays. Insulates reads completely; doesn't restrict peer writes.

Candidate C — Formal acceptance with named SLA and recovery contract. Don't change the substrate. Convert the implicit "accepted residual" status into an explicit contract that names (a) expected race-incident frequency, (b) standard recovery pattern (the git checkout <SHA> -- <file> + reflog-recovery memory we already documented), (c) what spine event makes a contamination incident "in scope" for follow-up. The current cost stops hiding under jargon and becomes a measurable SLA. Cost-in-Jeff-coordination: one decision (set the SLA). Cost-in-build: documentation + spine-event taxonomy update. Ongoing maintenance: the team eats the recovery cost forever; we just measure it instead of carrying it as silent debt.

Which trade to pick. A trades mid-session lock latency for substrate correctness. B trades disk + a small mental-model layer for read-side insulation without restricting peer writes. C trades zero substrate change for full ongoing recovery cost.

Recommendation: A. Builds on the typed-adapter pattern that's already shipping (do_commit / do_push / do_pull from #2682 / #2688 / #2701 / #2705). Closes the race rather than insulating around it. Lock cost is bounded (release within seconds for routine ops); the spike confirmed cold-start contention does not fire — session_init_gate.rs is a flag-file check with no git ops, and branch-switching only happens on bounded events (/pull, /acp, manual checkout). B as fallback if A's lock cost surfaces in practice as a cadence problem the team can't absorb. C if the team explicitly chooses to keep the residual rather than pay either build cost — but choosing C means committing to measure incident frequency, not just leave it implicit. Jeff picks before any follow-on card moves to WIP.

To-Be — designed target state

The typed surface extends across the full lifecycle. Pull and rebase get the same shape as commit: a typed wrapper over git-queue.sh with refusal taxonomy strictly limited to safety (rebase-conflict, flock-timeout) — no coordination state. chorus_commit's refusal taxonomy reduces to hook-fail and push-conflict only; the coordination-state reasons (no-wip-card, etc.) become readable via chorus_commit_status but never blocking. Trunk-based prerequisites land: micro-branches collapse the HEAD-race window from days to minutes; feature flags allow incomplete work on main; merge train serializes PR integration deliberately rather than as a flock side-effect. The false-positive push-conflict is fixed by distinguishing git-queue.sh's exit codes from push success. Spine carries the full lifecycle (event names below are design intent — not yet emitted): pull.fetched, rebase.attempted, rebase.aborted alongside commit.landed.

Gaps to close

chorus_commit refusal taxonomy still includes coordination-state reasons. Code at platform/api/src/mcp/server.ts refuses on board-unreachable, no-wip-card, multi-wip, branch-mismatch, path-not-found. Designed target: only hook-fail, push-conflict. Strip not yet shipped.
False-positive push-conflict in chorus_commit handler. Returns push-conflict with empty stderr while origin has the SHA. Cause: handler treats any non-zero exit from git-queue.sh as push-conflict regardless of whether push succeeded.
Pull / rebase has no typed surface. Roles fall back to direct git pull / git rebase. Lifecycle is half-typed (commit yes, pull no). Symmetry with commit-side is the right shape.
Working-tree HEAD race during pull / checkout — Problem 1. /chorus/.git/HEAD shared across role sessions; peer's git checkout rewrites it under another role's working tree. Race window equals branch lifespan. Receipts: 2026-04-29 Wren ~30min loss; 2026-05-02 Wren /reboot blocked by Kade mid-/acp staged tree. Mitigation lives at branching layer (micro-branches), not substrate layer (worktrees rejected).
Long-lived <role>/<card-id> branches widen the race window. Branch lifespan in hours to days. Trunk-based playbook says micro-branches collapse this. Adoption not started.
No feature flags — incomplete work cannot merge to main safely. Without flags, agents stay on long branches because main cannot tolerate WIP. Trunk-based prerequisite.
No merge train — concurrent green PRs can break each other on integration. flock serializes commits but not PR-merge integration. Standard fix is a merge queue.
No PreToolUse hook refusing direct git commit outside the surface. Bypass surface tempting; flock and spine emit can be skipped silently. Hook would refuse git commit Bash patterns outside git-queue.sh / chorus_commit.

Implementation Plan

The actors and BDD scenarios above are the contract. Every row in this plan is bound by them. A card changes the surface only if (a) the existing actors still describe who interacts, and (b) the existing scenarios still describe what happens — or the change updates the actor model / BDD first, on the page, with PM approval, before any code edit. Page-graph mutations on actors / scenarios precede the code that satisfies them. New scenarios are authored before they are implementable. Rejected scenarios retire before the code that violates them. The BDD is the regression line; the plan is its execution sequence. Anything that drifts from the page is the bug, not the page.

Honest cost. Twenty-one rows below across three card-classes — not the original eight. The strip-refusals "row" is five atomic edits; the chorus_pull "row" is seven; trunk-based adoption is four phase-rows, three of which are themselves arcs.
