Source of truth: Live actor list, BDD scenarios, contract entries, dependencies, gaps, and integrations live at /athena/domain-detail.html?id=version-control-domain. Promise. Each card gets an ephemeral git worktree, created from origin/main on /pull and removed on /acp. Canonical sits on main, read-only during sessions, refreshing only on lock-guarded sync. Inside the werk, normal git works. Code moves through its lifecycle as a sequence of atomic, self-contained Rust verb binaries — werk-commit → werk-push → werk-merge → werk-accept, with werk-build/werk-deploy/werk-demo for the prove half. Each verb owns its own flock, typed refusals, and spine emit; there is no orchestrator — the pipeline is the verbs run in order, stopping on the first refusal. The human authorizes what reaches prod and main.

The model

Version control is the substrate for how code in the chorus repo moves through its full git lifecycle — branch, commit, push, pull, rebase, merge, tag. The adapter is git, driven by atomic, zero-dependency Rust verb binaries — werk-pull, werk-commit, werk-push, werk-merge, werk-build, werk-deploy, and the finalize-only werk-accept. Each is self-contained: its own flock, typed refusals, spine emit, inherited trace. Gate logic lives inside the verbs, not in standalone shell hooks. Agents reach them through typed MCP tools or slash commands (/pull, /acp, /unpull). The operating model is trunk-based development; the substrate serves that discipline rather than inventing a parallel coordination layer.

Enforcement sits at exactly two shared boundaries: the push to origin (branch-ownership — <role>/* matches the pushing role) and canonical's two invariants (HEAD on main, working tree clean). Everything inside the werk is local and reversible.

Atomic verb execution & the prod gate

Each verb runs two ways. In the flow, it derives its inputs from the prior step's state — the werk, the branch, the artifact path. Standalone (--atomic), it takes those inputs explicitly, so a single step runs on its own. --atomic is the operator/recovery door: “just rebuild, redeploy, or merge this one thing,” which is what every recovery needs — and what, absent it, forces hand-run commands around the verbs, the direct source of merged≠live and partial-deploy failures.

Two axes, never conflated:

  • --atomic on all seven verbs. Free on pull/commit/build — local and reversible.
  • Explicit human approval on only werk-deploy and werk-merge — the two verbs that mutate prod and main irreversibly. Each approved run emits a spine event recording who, what, and when.

One gate, two doors. The human authorizes prod, reachable as demo→go inside the flow or an explicit approval on --atomic outside it. --atomic is not a bypass: approval kills “unauthorized,” the spine event kills “invisible.”

The load-bearing constraint (ADR-037 D#5): --atomic reuses the verb's pure core, never a fork — one implementation, two entry points. The full statement and rationale are in ADR-037; this doc references it rather than restating it.

The werk-commit refactor

The model lands first on the verb that hurts most. werk-commit today commits, then rebases onto main internally; on a real content conflict it aborts and restores the werk — with no way to resolve, because rebase, cherry-pick, and reset are all blocked at the guard boundary. The refactor folds three things into one change: factor werk-commit into a pure core plus a thin flow-adapter; expose commit --atomic (commit without the rebase); and add a real conflict-resolution path so a conflict is resolvable through the verb, not by hand. Shipped #3295: the pure-core/flow-adapter factoring, commit --atomic (commit-without-rebase), and the rebase-conflict refusal now points at the --atomic escape instead of a dead re-run. The guard-message correction — the deny-strings that still name a since-deleted tool — is a separate, smaller change to the hook (#3290), and ships on its own.

Substrate

The adapter is git via the atomic verb binaries above; agents reach them through typed MCP tools (one per verb) or the slash commands (/pull, /acp, /unpull). The standalone v1 shell adapter and its pre-commit/pre-push/branch-check hooks no longer exist; their gate logic moved inside the verb binaries.

Surfaces

Write

  • werk-pull — atomic /pull: worktree + card→WIP + role-state + card.pulled. Via MCP or /pull.
  • werk-commit → werk-push → werk-merge — the commit-to-main sequence, each its own verb (own flock, typed refusals, spine emit).
  • werk-accept — finalize-only: card Done + branch/werk teardown. Does not merge.
  • werk-build · werk-deploy — the prove half; werk-deploy mutates prod and carries the human-approval gate.
  • chorus-werk — worktree lifecycle (add / remove / status); werk-unpull — atomic inverse of /pull.
  • Slash commands: /pull, /acp, /unpull.

Read

  • git log / git status inside the werk — always available.
  • chorus.log tail (spine) — one event per lifecycle step, trace IDs propagated.

Components

Adapter — the werk verb binaries

Zero-dependency Rust binaries over git: werk-pull / werk-commit / werk-push / werk-merge / werk-build / werk-deploy / werk-accept. Each owns its own flock, typed refusals, spine emit, and inherited trace. The gate logic that once lived in standalone shell hooks (branch-check, pre-commit) runs inside the verbs.

Lifecycle — chorus-werk & werk-sync

chorus-werk add / remove / status around git worktree; git cherry patch-id on remove handles squash-merged delete; git worktree prune cleans stale admin entries. werk-sync is the lock-guarded canonical refresh (git pull --ff-only origin main), with repair / recover for a detached or corrupted canonical.

Guards

  • canonical-write-guard — refuses Edit/Write to canonical from a role session and redirects to the werk; permits the role's own canonical state files when no single werk is live.
  • infra_guardrails — refuses raw mutating git (commit/add/push/rebase/checkout) from role sessions, routing them through the verbs. (Its deny-messages still name a since-deleted tool — corrected by #3290.)
  • pre-push branch-ownership — an outbound ref must be <role>/* matching the pushing role.

Enforcement boundaries

Two shared boundaries, and only two: the push to origin (branch-ownership) and canonical's two invariants — HEAD stays on main and the working tree stays clean, so the next --ff-only sync can't break. Everything inside the werk is local and reversible; contention on the shared .git object DB / packed-refs is absorbed by bounded push-retry and git worktree prune on sync.

Trunk-based discipline (operating model, not code)

Micro-branches, feature flags, vertical slicing, fast rollback. The substrate doesn't enforce these — they're how the team uses it.

Loom contracts

No loom-decision, loom-policy, or loom-practice currently cites version-control-domain. If a "no force-push to main" policy formalizes, this section gains content. No fabricated citations until then.

Not in scope

Approaches rejected. These do not return without naming why the rejection no longer applies. Retirement dates and receipts in Appendix A / Appendix B.

  • Per-role sibling worktrees (chorus-kade/, chorus-wren/, chorus-silas/) — broke role-directory-is-session-start.
  • Internal worktrees with cwd enforcement (chorus/.worktrees/<role>-<hash>/) — same family as above.
  • Persistent per-role worktree (one stable dir per role with branch repoint) — branch-swap-in-place is the detached-HEAD generator. Any restoration must first name why detached-HEAD is no longer a concern.
  • Working-tree-lock service — adapter-side mutex on the working tree. Coordination cost moved into Jeff's commit path; trunk-based playbook handles this at the branching layer.
  • Broker layer between roles — agent-to-agent coordination via centralized broker before commits hit git. Jeff: "no broker."
  • Pre-commit as coordination broker — using hooks to enforce board state, role-state, or peer activity. Hooks stay commit-safety.
  • Class-A graph-backed migration of git history — git remains canonical store.

Appendix A — Ephemeral correction decision record (2026-05-14)

Diagnostic record that earned the rewrite shipped 2026-05-14 (#2913). Preserved for decision lineage; not heart-of-doc.

The receipt — 2026-05-13/14 session

Candidate D shipped (#2735) and the implementation was not the lightweight git worktree the candidate described. It was canonical plus three persistent per-role pseudo-worktrees plus a guard layer that intercepted normal git verbs inside a role's own isolated werk. Each guard was added for a real reason. Stacked, they inverted the property that makes worktrees worth adopting: a worktree is supposed to be a normal checkout where normal git works.

A single /acp on a three-commit fix, plus the follow-on to land a peer's orphaned consolidation, took a full evening. Every blocker fired on an operation entirely inside a role's own werk, touching no shared state:

  • Werk HEAD detached three times — recovery required hand-running git symbolic-ref / update-ref plumbing.
  • git rebase blocked by hook. The documented escape, DEPLOY_ROLE_PREPUSH_OVERRIDE=1, did not work for the Rust path (it was dead code there). Wren independently hit the same dead override.
  • git merge --abort blocked.
  • git commit-tree (plumbing) blocked as "direct git commit."
  • git-queue.sh commit refused "partial commit during merge" — git requires a pathless commit mid-merge; git-queue.sh requires path arguments. The two contracts cannot both be satisfied.
  • GitHub PR merge failed on a conflict that existed only because the branch's base commit (cb2503c9) and origin/main's squash of it (31c65471) had identical trees but disjoint history.

None of these are problems git worktree has. They were problems the wrapper had.

Root cause: branch-swap-in-place

The persistent-werk model kept one directory per role (chorus-werk/kade/) and swapped its branch with a repoint verb when the role moved to a new card. Branch-swap-in-place is the detached-HEAD generator: the worktree's HEAD file gets rewritten by plumbing because the porcelain path is hook-blocked, and any interruption leaves it detached. The canonical git worktree model never does this — git worktree add per task, git worktree remove when done. You never swap a branch inside a live directory, so the detached-HEAD failure class cannot occur.

Silas conditions on enforcement (folded into shipped components)

  1. Remove interior hooks outright would let a role run git checkout in canonical and detach canonical's HEAD. The N interior hooks collapse into one cwd-discriminating guard; they do not disappear.
  2. HEAD-stays-main alone is under-specified. A dirty canonical working tree breaks the next git pull --ff-only sync. canonical_write_guard narrows to two invariants: HEAD on main AND working tree clean.
  3. All worktrees share one .git/objects and one packed-refs. (a) Concurrent /acp can contend on packed-refs lock — bounded retry shipped (#2915). (b) Non-clean git worktree remove leaves stale admin entries — git worktree prune in sync shipped (#2913).

The already-merged check — three forms in one night (#2911)

  • Was: gh pr view <branch> --json state. Wrong question — gh resolves to the most recent PR for the branch name; a branch with a prior merged PR plus a new unmerged commit returns MERGED and short-circuits /acp to closure with the new work unshipped. (Wren's wren/2910: bed47114 unmerged, prior PR #233 marked the branch merged, /acp closed the card and dropped the commit.)
  • First fix: git merge-base --is-ancestor HEAD origin/main. Correct for new-commit; regresses squash-merge (#2177) — HEAD's SHAs are no longer ancestors of main even though their content is on main as the squash commit.
  • Final: git cherry origin/main HEAD. Patch-id answer: + = not upstream, - = upstream. Catches sha-ancestor merge and patch-id squash-merge.

2026-05-14 evening (#2923): even with the correct predicate, the fast-path skipped commit/push/PR when git cherry showed no + lines — but that signal is identical whether merged OR never committed. An uncommitted werk took the shortcut and marked the card Done with nothing shipped (#2915 self-acp). Fast-path removed entirely; normal path is idempotent on re-runs.

The squash-merged branch delete (#2913 follow-on)

The first /acp of #2913 itself left kade/2913 dangling with branch_closed:false. chorus-werk remove used git branch -d, an ancestry check — but after gh pr merge --squash the branch is NOT an ancestor of origin/main even though its commits ARE upstream by content. -d refused → exit 7 → branch never deleted. Same family. Fix: same predicate (git cherry); upstream by patch-id → git branch -D; + line is genuinely unmerged → keep the refusal so remove never silently drops it.

Appendix B — Shared-HEAD race decision record

The original driver. Structurally closed by per-worktree HEADs (#2735 → #2913). Preserved for lineage; not heart-of-doc.

Pre-resolution receipts

  • Dangling-commit recovery (2026-04-29). Wren's commits during a build session landed on the wrong branch because HEAD bumped between her commit and her next checkout. Recovery via git reflog + cherry-pick.
  • Mid-demo cwd-bump (2026-05-02). Wren's checkout to her branch repainted the working tree under Kade's session; his terminal in /chorus was reading files that suddenly belonged to a different branch. Demo paused and reset.
  • The cascade. #2696, #2698, #2699, #2700, #2701, #2705 — substrate-fix cards each correct in scope; together the substrate naming itself by repeated injury.
  • Cross-role contamination at acp (2026-05-05). Kade tried to ship #2733; refused because canonical's HEAD was sitting on wren/2731 from Wren's prior session (PR had merged hours earlier; HEAD never moved back to main). The 2026-05-05 incident is what earned reopening Candidate D after its 2026-05-01 rejection.

Four candidates evaluated

  • A — Serialize working-tree mutation through git-queue.sh. Extend the adapter to gate checkout/switch/branch under the same lock as commit/push. Closes race-frequency, not structural anti-pattern. SHIPPED #2710 as belt-and-suspenders (lock surface narrowed to canonical-on-main during sync; useful as typed surface for werk-side checkout).
  • B — Read-side snapshot at session start. Role-private overlay per session; reads resolve against overlay. Insulates reads; doesn't restrict peer writes. NOT PURSUED.
  • C — Formal acceptance with named SLA. Don't change the substrate; convert implicit accepted-residual into explicit measured SLA. NO LONGER A CANDIDATE — 2026-05-05 incident overtook "leave it implicit."
  • D — Per-role worktrees outside canonical. Each role works in /chorus-werk/<role>-<card>/ — ephemeral worktree with own HEAD file. Closes race + structural anti-pattern. SHIPPED #2735 (persistent shape) → #2913 (ephemeral correction). Distinct from rejected v1 (sibling-worktrees-inside-chorus) and v2 (internal-with-cwd-enforcement) by being outside canonical entirely; session-start cwd remains at /chorus/roles/<role>/ — only the write surface moves.

Jeff endorsed D + A 2026-05-05: "I do NOT want fucking sharing of your branches" + "anti-pattern for three engineers to share the same canonical work area."
