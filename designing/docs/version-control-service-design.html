Designed target state. A version-control operation lands when, and only when, the role's request reaches git-queue.sh through the typed MCP surface (chorus_commit for write, chorus_commit_status for read), and the adapter (a) stages caller-named paths, (b) runs the pre-commit hook synchronously and surfaces stderr on failure, (c) commits and pushes under flock with bounded retry, and (d) emits one commit.landed spine event reflecting the actual SHA on main. Refusal taxonomy is strictly commit-safety: hook-fail, push-conflict. Coordination state — board status, role-state, branch-naming convention — never refuses Jeff's commit. Coordination is observed after the fact via spine, not enforced at the surface. Per-card isolation (2026-05-14): each card gets an ephemeral git worktree at /CascadeProjects/chorus-werk/<role>-<card>/ — created from main on /pull, removed on /acp. Canonical /CascadeProjects/chorus/ always sits on main and is read-only during sessions; it refreshes only on lock-guarded sync. Normal git works inside the werk — no hooks on its interior. Enforcement sits at exactly two shared boundaries: the push to origin (pre-push branch-ownership) and canonical's HEAD (stays main). The shared /chorus/.git/HEAD race is closed structurally — separate HEADs, no shared write surface. Today's code does not yet match this contract — see As-Is below.

2026-05-14 — Candidate D shipped, and it isn't a worktree

Candidate D (#2735) landed: each role has a directory at /CascadeProjects/chorus-werk/<role>/ and canonical stays on main. The shared-HEAD race is gone. But the thing that shipped is not the lightweight git worktree the candidate described. It is canonical, plus three persistent per-role pseudo-worktrees, plus a guard layer (canonical-write-guard, git-queue.sh branch/checkout/commit serialization, chorus-werk lifecycle wrapper, pre-push hooks) that intercepts normal git verbs inside a role's own isolated werk. Each guard was added for a real reason. Stacked, they invert the exact property that makes worktrees worth adopting: a worktree is supposed to be a normal checkout where normal git works.

The receipt — 2026-05-13/14 session

A single /acp on a three-commit fix, plus the follow-on to land a peer's orphaned consolidation, took a full evening. Every blocker below fired on an operation that was entirely inside a role's own werk, touching no shared state — and every one is a our-machinery failure, not a git-worktree failure:

  • Werk HEAD detached three times — recovery required hand-running git symbolic-ref / update-ref plumbing.
  • git rebase blocked by hook. The documented escape, DEPLOY_ROLE_PREPUSH_OVERRIDE=1, does not actually work — the hook claims it does. (Wren independently hit the same dead override.)
  • git merge --abort blocked.
  • git commit-tree (plumbing) blocked as "direct git commit."
  • git-queue.sh commit refused "partial commit during merge" — git requires a pathless commit mid-merge; git-queue.sh requires path arguments. The two contracts cannot both be satisfied.
  • git-queue.sh push documents that it rebases onto origin/main internally; it did not.
  • GitHub PR merge failed on a conflict that existed only because the branch's base commit (cb2503c9) and origin/main's squash of it (31c65471) had identical trees but disjoint history — a conflict manufactured by the squash-merge-plus-persistent-werk interaction, not by any real divergence.

None of these are problems git worktree has. They are problems the wrapper has.

Root cause: the persistent werk, repointed across cards

Today's chorus-werk keeps one directory per role (chorus-werk/kade/) and swaps its branch with a repoint verb when the role moves to a new card. Branch-swap-in-place is the detached-HEAD generator: the worktree's HEAD file gets rewritten by plumbing because the porcelain path is hook-blocked, and any interruption leaves it detached. The canonical git worktree model never does this — it is git worktree add per task, git worktree remove when done. You never swap a branch inside a live directory, so the entire detached-HEAD failure class cannot occur.

The correction: ephemeral worktree per card

One structural decision drives everything else. Stop reusing one persistent per-role werk and repointing it. Create an ephemeral worktree per card; destroy it on /acp.

Where enforcement actually belongs

The guard layer is re-implementing isolation that a real worktree already provides. It does not vanish — it collapses: many hooks intercepting many git verbs become one cwd-aware guard plus the existing push boundary. Keep / cut — chorus-werk, /pull, /acp

The root change is chorus-werk itself: rewritten from a stateful init / repoint / pull / close lifecycle around one persistent dir to thin add / remove around ephemeral per-card worktrees. /pull and /acp are the two callers that get simpler once it lands. Order matters: chorus-werk first, then the callers adopt it — doing the callers first means rewriting them twice.

/pull

Keep: board card → WIP; create branch <role>/<card-id>; role-state building; create the worktree; node_modules symlink-from-canonical bootstrap (solves the primer's disk-duplication pitfall).

Cut: the chorus-werk init / repoint / pull three-verb dance → one git worktree add ../werk-<role>-<card> -b <role>/<card-id>. Routing branch creation through git-queue.sh checkout for "flock serialization" — creating a branch in your own new worktree touches zero shared state; no lock needed.

/acp

Keep: cardId derivation from branch (branch is source of truth); already-merged check via git cherry origin/main HEAD (patch-id idempotency — see 2026-05-14 fix note below); the actual transaction (commit → push → PR create → PR merge → cards-done → card.accepted); release-trigger (kickstart building-pipeline); typed refusals that name the failing step; intent-assertion (#2868) — cheap, keep it.

Cut: commit routed through git-queue.sh — inside a real isolated worktree, commit is just git commit; git-queue's flock only earns its place on the shared operation, the push. The werk-close contortion (#2753: "no --delete-branch because gh's deletion does an implicit git checkout main that collides with canonical") — that knot exists only because of shared HEAD; ephemeral worktree close is git worktree remove + git branch -d + remote-ref delete, no collision possible. The #2793 refuse-werk-on-main guard — guards a drift state that only happens because of repoint; gone with repoint.

Narrow, don't cut (Silas condition 1+2): canonical-write-guard does not vanish. It stops intercepting werk-interior edits — those run as plain git — but it narrows to the cwd-aware form: refuse git verbs whose cwd is canonical, and hold canonical's two invariants (HEAD on main, working tree clean). Editing freely inside the werk is correct; editing freely in canonical is the detached-HEAD bug the guard exists to prevent.

The already-merged check — 2026-05-14 fix (#2911)

Independent of the worktree restructure but landed the same session, because it is the same family of bug — a check asking the wrong question. chorus_acp's fast-path "is this already merged" check went through three forms in one night:

  • Was: gh pr view <branch> --json state. Wrong question — gh resolves to the most recent PR for the branch name, so a branch with a prior merged PR plus a new unmerged commit returns MERGED and short-circuits /acp to closure with the new work unshipped. (Wren's wren/2910: bed47114 unmerged, prior PR #233 marked the branch merged, /acp closed the card and dropped the commit.)
  • First fix: git merge-base --is-ancestor HEAD origin/main. Correct for the new-commit case, but regresses the squash-merge case (#2177): after gh pr merge --squash, HEAD's commit SHAs are no longer ancestors of main even though their content is on main as the squash commit.
  • Final: git cherry origin/main HEAD. Answers by patch-id: + = commit's patch not upstream, - = patch is upstream. No + lines → already merged. Catches sha-ancestor merge and patch-id squash-merge in one predicate.

Same session also fixed checkDemoEvidence (the cards-CLI demo gate) — it grepped chorus.log for "card_id":"<n>" (quoted string) but spine emits "card_id":<n> (JSON number) post-#2857, so every card.demo.started event was invisible to the gate. That fix was then superseded by Wren's #2910 consolidation: checkDemoEvidence now reads one canonical source — the demo:preflight-pass card comment — via client.comments(), the same signal the Rust accept-gate hook already used. One reader, one form, no drift.

Loose end — role-state under >1 card

The ephemeral model supports a role having more than one card in flight (each card its own worktree). But role-state is a single value. If a role has two live cards, role-state can only reflect one as active; the board shows both in WIP. This is a state-model question, not a folder-structure one — flagged here as an explicit loose end to settle before the >1-card path is exercised in anger. Board WIP discipline ("prefer to finish one card before pulling the next") remains a policy on the board; what changes is that the filesystem stops enforcing it by making >1 card painful.

Overview

Version control is the substrate for how code in the chorus repo moves through its full git lifecycle: branch creation, commit, push, pull, rebase, merge, tag. The adapter is git wrapped by git-queue.sh — a shell-level flock-serialized surface for stage / commit / push. Gates run via standard git hooks (pre-commit, pre-push) at the adapter boundary; agents reach the surface through the typed chorus_commit MCP wrapper or the /acp slash command. The operating model is trunk-based development; the substrate exists to serve that discipline, not to invent a parallel coordination layer.

Components (designed)

Adapter — git-queue.sh SHIPPED

The serialized commit surface. Wraps git stage / commit / push in a flock so concurrent role commits queue rather than race. Emits commit.landed on successful push. Returns non-zero on hook failure or persistent push conflict.

Hook layer — pre-commit, pre-push, branch-check, write-scrubber SHIPPED

Synchronous gates at the git boundary. pre-commit runs write-scrubber + branch-check on staged paths; pre-push validates outbound refs. All run inside git-queue.sh; failures surface stderr to the caller. Internal to the adapter — not separate actors.

MCP wrappers — chorus_commit, chorus_commit_status PARTIAL

Typed surfaces over git-queue.sh. Refusal taxonomy strict by design but not yet by code: today's chorus_commit still refuses on board state. The rewrite target removes the coordination-state reasons.

Trunk-based discipline (operating model, not code) NOT ADOPTED

Micro-branches, feature flags, vertical slicing, hard CI gate, force-rebase-before-merge, active-zones, merge train, fast rollback. The substrate doesn't enforce these — they're how the team uses the substrate. Adoption tracked separately as practice work.

As-Is — 2026-05-14

Candidate D shipped (#2735, 2026-05-13). But what shipped is not a git worktree. It is canonical + persistent per-role pseudo-worktrees + a guard layer that intercepts normal git inside each werk. Concurrent reads against shared mutable git state (#2706) — RESOLVED

Resolved. Candidate D shipped 2026-05-13 (#2735); the shared-HEAD race is closed. The A/B/C/D analysis below is preserved as the decision record, not open work. The race is gone — the new open problem is that D's implementation isn't a real worktree (see the 2026-05-14 section near the top and the To-Be below).

All three roles share one git repository at /chorus/, with shared metadata (HEAD, refs, index) in /chorus/.git/. The HEAD pointer and the index — the metadata that tells git which branch and commit you're on — are single files mutated by whichever role most recently ran a checkout, branch-switch, or pull. The working tree on disk (the actual .ts, .md, .rs files the roles read) is repainted by git to match HEAD on every checkout. Mid-session, a role can read a file at T1 and have its next read return content from a different branch, because a peer's checkout fired between T1 and T1+ε and rewrote both HEAD and the on-disk files. The race is silent: the reading role gets stale or wrong content with no error.

User-visible receipts:

  • Dangling-commit recovery (2026-04-29). Wren's commits during a build session landed on the wrong branch because HEAD bumped between her commit and her next checkout. Recovery required reading commits out of git reflog and cherry-picking them back. This was not a bug — the substrate was operating as designed under concurrent use.
  • Mid-demo cwd-bump (2026-05-02). During Kade's demo, Wren's checkout to her branch repainted the working tree under Kade's session; his terminal was sitting in /chorus reading files that suddenly belonged to a different branch. Demo had to pause and reset.
  • The cascade. A run of substrate-fix cards (#2696, #2698, #2699, #2700, #2701, #2705) all shipped real fixes for downstream symptoms of this race. Each fix was correct in scope; together they were the substrate naming itself by repeated injury.
  • Cross-role contamination at acp (2026-05-05). Kade tried to ship #2733 (TDD-gate fix) and was refused by the cross-role contamination guard because canonical's HEAD was sitting on wren/2731 from Wren's prior session. Wren's PR had merged hours earlier, but canonical's local HEAD never moved back to main — because canonical is being used as a working tree by all three roles, and whoever last ran a checkout leaves it sitting there. The branch-check guard caught the would-be cross-role commit at the gate, but the rotten state upstream is the shared canonical work area itself.

Was accepted-residual risk; reopened 2026-05-05. Through 2026-05-03 the team chose to absorb the race rather than pay the worktree-isolation cost. The per-role sibling-worktree v1 convention was retired (#2640) and a v2 redesign rejected the same day on the protected-primitive ground that "session-start cwd binds to /chorus/roles/<role>/." Spine-event watching was the chosen mitigation: incidents recorded as metric data, not regressions.

The 2026-05-05 incident provides the data Wren's hold-until-incident position required, and Jeff named the structural framing directly: "anti-pattern for three engineers to share the same canonical work area … happens on every fucking commit." Per-role worktrees outside canonical (/chorus-werk/<role>/, distinct from the v1 sibling-worktrees-inside-chorus shape and the v2 internal-with-cwd-enforcement shape) preserve the protected primitive: session-start anchor stays at /chorus/roles/<role>/ as the read surface for role state; only the write surface moves to the werk. Candidate D below names the path. Candidate A still ships, reframed as belt-and-suspenders for canonical's HEAD during sync (a much narrower lock surface than gating every role checkout).

Naming honestly: previous internal docs used "Mode-A" as shorthand for this state. The shorthand obscured what was actually happening (a concurrent-reader race against a shared mutable index). This document and the rest of the chorus tree use plain English.

Path to close (#2706)

Four candidate approaches. Each pays a different cost. The protected primitive — session-start cwd binds to /chorus/roles/<role>/ — survives in all four; any approach that moves session-start is in the rejected family. (D distinguishes session-start — where the role lands and reads role state — from the write surface where edits land. Earlier rejected proposals conflated these; D separates them explicitly.)

Candidate A — Serialize working-tree mutation through the canonical adapter. Extend git-queue.sh to gate git checkout, git switch, and git branch under the same lockf that already serializes commit and push. Add the deny entries to the chorus-hooks guard so raw checkout routes through the queue. When a role wants to switch branches, it acquires the lock; concurrent peers wait. Cost-in-Jeff-coordination: zero (Jeff doesn't see lock contention). Cost-in-build: one new do_checkout in git-queue.sh (parallels do_pull / do_push), one regex addition to the guard, skill updates to route through the typed adapter (/pull and /acp currently call git checkout directly). Ongoing maintenance: same as today's adapter — bounded. Closes the race-frequency problem; leaves the structural problem (three roles writing into one tree) intact.

Candidate B — Read-side snapshot at session start. Each role's SessionStart hook copies the current working tree state into a role-private overlay at /tmp/chorus-session-<role>-<sha>/. Reads during the session resolve against the overlay. Writes still go through git-queue.sh against canonical /chorus/. Refresh on demand or per-card. Cost-in-Jeff-coordination: zero. Cost-in-build: SessionStart hook extension + a read-overlay layer that intercepts Read/Bash cat against repo paths. Ongoing maintenance: cache-coherence rules (when does the snapshot refresh?), disk overhead for three live overlays. Insulates reads completely; doesn't restrict peer writes.

Candidate C — Formal acceptance with named SLA and recovery contract. Don't change the substrate. Convert the implicit "accepted residual" status into an explicit contract that names (a) expected race-incident frequency, (b) standard recovery pattern (the git checkout <SHA> -- <file> + reflog-recovery memory we already documented), (c) what spine event makes a contamination incident "in scope" for follow-up. The current cost stops hiding under jargon and becomes a measurable SLA. Cost-in-Jeff-coordination: one decision (set the SLA). Cost-in-build: documentation + spine-event taxonomy update. Ongoing maintenance: the team eats the recovery cost forever; we just measure it instead of carrying it as silent debt.

Candidate D — Per-role worktrees outside canonical. Each role works in /CascadeProjects/chorus-werk/<role>/ — a git worktree of canonical, branched from main, with its own HEAD file. Canonical /CascadeProjects/chorus/ always sits on main and is read-only during sessions. Edits during the session land in the role's werk; the SessionStart hook sets $<ROLE>_WERK and chorus-hooks redirects Edit/Write tool calls under canonical to the werk path. Canonical refreshes only via a lock-guarded sync hook (SessionEnd or scheduled tick) doing git pull --ff-only on main; no role checkout ever moves canonical's HEAD off main. Cost-in-Jeff-coordination: zero (Jeff sees fewer cross-role contamination errors, not more). Cost-in-build: chorus-werk bootstrap script, env-var set in SessionStart, Edit/Write redirect hook, sync hook, canonical-checkout deny in chorus-hooks, migration of existing in-flight branches. Ongoing maintenance: bounded — once installed, each role's werk is just a git worktree like any other. Closes both the race and the structural anti-pattern. Distinct from rejected v1 (sibling-worktrees-inside-chorus) and v2 (internal-with-cwd-enforcement) by being outside canonical entirely; session-start cwd remains at /chorus/roles/<role>/ for role-state reads — only the write surface moves.

Which trade to pick. A trades mid-session lock latency for race correctness; leaves the shared-tree structure. B trades disk + a small mental-model layer for read-side insulation; doesn't restrict peer writes. C trades zero substrate change for full ongoing recovery cost. D trades a one-time bootstrap + per-session env wiring for full structural isolation — separate HEADs, no shared write surface, canonical never moves off main during sessions.

Recommendation as of 2026-05-05: D primary, A belt-and-suspenders. The 2026-05-05 incident showed that Candidate A alone is insufficient under Jeff's bar — serializing checkout contention does not fix the anti-pattern Jeff named ("three engineers sharing the same canonical work area"). Per-role worktrees fix the structure: each role's HEAD lives in a separate file, peer checkouts cannot rewrite each other's working trees, the cross-role contamination guard's job becomes vestigial because the wrong-role-on-wrong-branch state cannot occur in canonical. Candidate A still ships (#2710 / #2711 / #2712) — but as a narrow lock around canonical-on-main during the sync window, not as the primary mitigation for every role checkout. B is reserved as fallback only if D's bootstrap cost surfaces unanticipated friction in practice. C is no longer a candidate: the 2026-05-05 incident overtakes "leave it implicit." Jeff endorsed this reframing 2026-05-05 ("I do NOT want fucking sharing of your branches" + "anti-pattern for three engineers to share the same canonical work area").

To-Be — designed target state

The worktree model is the headline. Each card gets an ephemeral worktree — git worktree add from main on /pull, git worktree remove on /acp. No persistent per-role dir, no repoint, no branch-swap-in-place — so the detached-HEAD failure class cannot occur. chorus-werk shrinks from a stateful init / repoint / pull / close lifecycle to thin add / remove. Enforcement collapses (it does not vanish — Silas gate, 2026-05-14) onto three points: a single cwd-aware git guard (allow in werk, refuse in canonical), the push to origin (pre-push branch-ownership, #2598), and canonical's two invariants (HEAD on main, working tree clean). The shared .git object DB still needs packed-refs bounded-retry and git worktree prune in the sync path. The full keep/cut and the enforcement table are in the 2026-05-14 section near the top; rows 30-39 of the Implementation Plan sequence it — chorus-werk rewritten first, callers adopt after, verification last.

Secondary, still open. The typed surface extends across the full lifecycle: chorus_pull gets the same typed shape as chorus_commit; the refusal taxonomy reduces to safety-only (hook-fail, push-conflict, rebase-conflict, flock-timeout), with coordination state readable via chorus_commit_status but never blocking; the false-positive push-conflict is fixed by distinguishing git-queue.sh's exit codes from push success. Spine carries the full lifecycle (design intent — not yet emitted): pull.fetched, rebase.attempted, rebase.aborted alongside commit.landed. Trunk-based prerequisites (micro-branches, feature flags, merge train) remain practice work, tracked separately.

Gaps to close

chorus_commit refusal taxonomy still includes coordination-state reasons. Code at platform/api/src/mcp/server.ts refuses on board-unreachable, no-wip-card, multi-wip, branch-mismatch, path-not-found. Designed target: only hook-fail, push-conflict. Strip not yet shipped.
False-positive push-conflict in chorus_commit handler. Returns push-conflict with empty stderr while origin has the SHA. Cause: handler treats any non-zero exit from git-queue.sh as push-conflict regardless of whether push succeeded.
Pull / rebase has no typed surface. Roles fall back to direct git pull / git rebase. Lifecycle is half-typed (commit yes, pull no). Symmetry with commit-side is the right shape.
Working-tree HEAD race during pull / checkout — Problem 1. /chorus/.git/HEAD shared across role sessions; peer's git checkout rewrites it under another role's working tree. Race window equals branch lifespan. Receipts: 2026-04-29 Wren ~30min loss; 2026-05-02 Wren /reboot blocked by Kade mid-/acp staged tree; 2026-05-05 cross-role contamination at acp — canonical sat on wren/2731 after PR merge; Kade's /acp for #2733 refused at the cross-role guard. Mitigation 2026-05-05: Candidate D (per-role worktrees outside canonical, #2735) at the substrate layer + Candidate A (do_checkout flock, #2710) as belt-and-suspenders. The earlier "branching layer only" framing is overtaken — micro-branches narrow the race window but do not address the structural shared-tree anti-pattern.
Long-lived <role>/<card-id> branches widen the race window. Branch lifespan in hours to days. Trunk-based playbook says micro-branches collapse this. Adoption not started.
No feature flags — incomplete work cannot merge to main safely. Without flags, agents stay on long branches because main cannot tolerate WIP. Trunk-based prerequisite.
No merge train — concurrent green PRs can break each other on integration. flock serializes commits but not PR-merge integration. Standard fix is a merge queue.
No PreToolUse hook refusing direct git commit outside the surface. Bypass surface tempting; flock and spine emit can be skipped silently. Hook would refuse git commit Bash patterns outside git-queue.sh / chorus_commit.

The actors and BDD scenarios above are the contract. Every row in this plan is bound by them. A card changes the surface only if (a) the existing actors still describe who interacts, and (b) the existing scenarios still describe what happens — or the change updates the actor model / BDD first, on the page, with PM approval, before any code edit.
