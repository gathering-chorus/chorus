Wardley Map — ICD as Product · Convergence Architecture · Borg Assimilation Pattern · Attention Architecture · Ingest Complexity · Architecture Patterns

Core Principle (Staples "Power Mapping"): Complete the ICD and source-to-canonical maps fully and to high quality before writing any code. If the contract is sloppy, the code inherits the slop. The map phase always feels slow until the build phase goes fast.
XSD Model — Almost Every Field Is Required: The ICD is a schema, not a wish list. If a field exists in the canonical type and at least one source provides it, that field is Required (minOccurs="1"). The document is rejected without it. There are only two tiers:
  • Required — a source exists that provides this field. If the record doesn't have it, the record is rejected. No soft warnings. No "we'll track it." Rejected.
  • Enrichment — no source exists yet for this field. It's a future capability, not a current gap. When a source is wired, the field moves to Required.

The old three-tier model (Violation/Warning/Info) created a soft middle where "Warning" meant "we expect it but won't reject without it." That's how data quality debt accumulates silently. The XSD model eliminates the middle: you either have the data or you don't ship the record.

Source coverage ≠ severity. LinkedIn email at ~30% coverage doesn't make email "optional" — it means 30% of LinkedIn records have it and 70% are rejected at the email field. The source limitation is visible in the ICD coverage %, but the schema doesn't compromise. If the source can't provide a required field, the record doesn't enter canonical. Fix the source or accept the smaller dataset.

Tier Change Protocol — When Required Feels Too Tight

The tension: too tight rejects records Jeff wants. Too loose breaks automation.

The XSD model will sometimes feel too strict — a dry run rejects 70% of records because a "Required" field has low source coverage. The response is not "make it optional." The response is a structured decision:

  1. Is this field truly required for the canonical type? Can a Person exist without email? If yes → move to Enrichment. If no → the source has a gap to fix.
  2. Is this field required for a specific pipeline operation? Email for dedup matching ≠ email for the Person record. Two schemas: one for the record shape, one for the pipeline operation. The record schema is lenient. The pipeline schema is strict.
  3. Does moving this field to Enrichment create a data compensation problem downstream? If the UI needs the field, making it Enrichment means the page has gaps. Better to reject the record than show an incomplete row that Jeff has to mentally compensate for.

The decision gets recorded in the ICD. Every tier change gets a comment in the cf-row: why it moved, who decided, what the coverage was. This is the audit trail. Without it, the next person who reads the ICD doesn't know if "Enrichment" means "truly optional" or "we gave up on getting this data."

Rule: Tier changes always flow back through the ICD (Phase 2), never through the code (Phase 3). If a developer changes the schema to make a field optional, that's a process violation — the ICD and the code are out of sync.

Reference Implementation — Copy, Don't Interpret

The problem: three roles reading the same spec, building three variants.

Each role understands the ICD principles and then improvises slightly when applying them to a new domain. Kade's harvester field names drift from Silas's schema field names. Silas's Social schemas don't structurally match his Photos schemas. Wren's Facebook ICD sections don't match her LinkedIn ICD sections. Every domain is a "slight variant" — and slight variants compound into incompatible implementations that break automation.

The fix: one reference implementation, copied structurally.

Rule: When building a new domain, diff your output against the reference. If the structure differs — not just field names, but section order, attribute names, function signatures — that's a bug, not a design decision. The only things that change between domains are field names and coverage percentages. Everything else is the template.

Anti-pattern: "I followed the same pattern" — followed conceptually, not structurally. The result looks similar but has different attribute names, different section ordering, different error handling. Each variation is a future debugging session when automation assumes consistency.

Session Learnings (2026-03-19) — Proven in Practice

1. Consumer ICDs have the same 5 mandatory sections as providers.

No exceptions. Purpose & Scope, Risks, Non-Functionals, Interface, Diagrams — every provider, every consumer. If a section has no data, it exists as an empty section. The structure is mandatory, the content varies. The lint enforces this on both sides. Jeff asked for this 10+ times before it was implemented — the team kept softening it with "optional" and "domain-specific." The rule is absolute.

2. RDF is the source of truth. HTML is a rendered view.

The ICD lives in Fuseki as an OWL ontology (5,148 triples, 7 domains). The HTML (SEMANTIC_MAPPER.html) is the authoring surface — edits happen there, then migrate to RDF via migrate-icd-tree.py --publish. The Convergence page renders FROM the RDF graph via the API. The HTML file is kept as a fallback view but is not the source of truth.

Athena pattern (Staples): ICD and mappings were ontologies, instances were RDF, system could diff versions. Implemented here: versioned named graphs (icd/v1, icd/v2, ...), semantic diff via --diff flag, /current alias.

3. The lint validates the graph, not the HTML.

icd-lint-sparql.py validates the RDF graph in Fuseki. icd-lint.py validates the HTML (legacy, still useful for authoring). icd-consistency-test.py validates agreement between HTML, graph, and TypeScript schemas. Three layers, one truth. The SPARQL lint is what CI should run.

4. No front-end compensation. Fix the data.

The page renders what the graph gives it. No || 'Unknown' fallbacks. No entity decoding in the template. No datatype guessing in the view. If the data is wrong — leading colons, HTML entities stored as literals, incorrect datatypes — fix the migration script and republish. The front end is a dumb pipe from API JSON to HTML. Jeff was explicit: "If it is data I don't want him compensating on the front end."

5. Versioned publish + semantic diff.

migrate-icd-tree.py --publish creates a versioned named graph (icd/v{N+1}) and updates icd/current. --diff shows semantic changes between versions: fields added, removed, severity changed. No more 1,500-line HTML diffs. The diff shows: + [People] birthday (Enrichment) or ~ [Stories] themes: Warning → Violation.

6. Tree structure preserves document fidelity.

The RDF uses intermediate nodes: icd:Section → icd:Paragraph (with label + order), icd:RiskItem (with ok/risk status + order), NF as named values (nfVolume, nfFreshness, nfLatency, nfAuth), mermaid source preserved exactly. This enables identical rendering from SPARQL — every piece of the HTML document is recoverable from the graph.

7. Canonical URI namespace.

All ICD entities live at https://jeffbridwell.com/icd/.... TTL files use @base <https://jeffbridwell.com/>. Stale graphs under old URI bases are dropped on publish. Pod data stays at http://localhost:3000/pods/... (separate namespace). ADR-018 documents the XSD severity model. #1539 established the canonical URI scheme.

8. Future: ICD Editor (Athena fully realized).

Current flow: edit HTML → migrate to RDF → render from graph. Future flow: edit via form UI → write directly to RDF → HTML is generated output. The editor is the Athena pattern fully realized — people enter data into structured forms, the system stores it as RDF, the view is rendered. No hand-editing HTML as if it were a database. Card #1538 (live validation, request-response) and the editor are the next evolution.
