2 · Promise

Athena is the structural and query layer of Chorus — the canonical model of what exists in the team's product world (products, domains, services) and the predicates that connect them. It is what makes the chorus product tree queryable: every other subproduct's structural placement (Loom's principles, Werk's pipelines, Borg's logs) is an Athena instance. Ultimately Athena is a product-and-design pipeline, not "a schema" — the structural model (Product / Domain / Service) is the product layer; the narrative attributes (asIs / toBe / pathToClose / notInScope / hasDesignDoc) are the design layer; the substrate is the pipeline: every instance under the same SHACL-gated contract, advancing through one artifact lifecycle (exploring → building → operating → retiring), rendered by the same UI.

3 · Value & audience

One source, multiple call shapes, no fallback path that produces a different answer. Consumers:

  • Gates — /gate-product verifies a card's Domains: label resolves to a real tree IRI; refuses if not.
  • SessionStart — per-role envelope injection: owned units, most-active and needs-work lists, flat ownership map, tree URL.
  • Blast-radius — chorus_blast_radius walks consumes + hosts + dependsOn recursively; "what does changing X affect" for gates and cards.
  • Role-nudge routing — "who owns this?" resolves via chorus_ownership_lookup, not pattern-matching role files.
  • /demo feedback — product-impact answers cite tree IRIs, not memory.
  • Jeff & the team — the rendered tree page; a shared visual artifact every demo can reference.
  • Tagging layer — tree IRIs are the canonical tag enumeration for cards (domain:<slug> etc.), code (per-file owning-Service tags), and tests (per-domain CI routing).

4 · Structure

Up: single inbound edge — chorus-product hasChild athena-product (recursive-Product model; a "subproduct" is just a Product that is the child of another Product). Down: Athena's substance is the union of its child Domains; each Domain hosts its Services (where the code lives); the Product consumes the Services it depends on.

  • Domains (per the chorus tree): domains, services, knowledge, domain-context. The domains domain's page renders THE SET — the generated catalog live.
  • Child products: none — Athena is a leaf Product.
  • Surfaces: MCP tools (chorus_tree_get / chorus_ownership_lookup / chorus_blast_radius); the athena pages (tree.html, domain-detail.html, the flow mock + anatomy — §6); the generated domain APIs (:3360).
  • Overlays (TO-BE): non-containment annotations — authority, ownership, deploy-route, customer-happiness — queryable as their own predicates (overlay_get / overlay_query). The AS-IS Athena has no overlay surface at all.

Why two tiers — schema posture

RDF and OWL are very powerful — open extension, multi-typing, inference, federated queryability. But the schema, in many ways, sits below the data — instead of around or above it. The ontology describes what triples could exist; it doesn't gate what does. Triples can be inserted with no schema awareness and remain valid — the open-world assumption treats absence-of-schema as absence-of-knowledge, not absence-of-permission. That permissiveness allows sloppiness that leads to bad data.

The AS-IS defects above are exactly this pattern: belongsTo used 34 times but never declared; hasDomain declared twice with conflicting signatures; Service nodes typed as SubDomain because no rule enforces type discipline. The schema didn't catch these because it doesn't try to — it describes, it doesn't gate.

The two-tier design responds to this directly:
  • Relational core — Product, Domain, Service, Role, ValueStreamStep, ValueStream. Schema sits around and above the data: SHACL-enforced at write, closed-world, required attributes verified, type discipline a hard gate. Relational rigor where predictability matters — typed API endpoints (/athena/product/:id) need shapes clients can rely on.
  • Graph overlays — authority, ownership-history, deploy-route, customer-happiness binding, future annotations not yet imagined. Schema sits below the data, RDF-native: open extension, new predicates land without core schema migration, queryable via /athena/edges and /athena/sparql.
Same RDF substrate; two enforcement postures, one per tier. The relational core stops sloppiness in the load-bearing places; the overlay layer keeps the graph's evolutionary openness where new dimensions need to land cheaply.

Structural layer — three classes, recursive containment

From domain-model-asis-tobe.html and Loom's "Structural layer" section, consolidated:

  • Product is recursive. A Product can contain Products via hasChild; a "subproduct" is just a Product that is the child of another Product. SubProduct as a separate class is removed — erases the "SubProduct rdfs:subClassOf Domain" structural oddity.
  • Domain is recursive. A Domain can contain Domains via hasChild; SubDomain as a separate class is removed — depth becomes a data decision (add hasChild edges), not a schema decision (invent a deeper class).
  • Service is a first-class structural layer. Real service nodes (cards-service, gates-service, spine-service, skills-service) get re-typed to Service. The chorus:Service OWL class exists today; what's missing is type migration of the mis-typed SubDomain instances.
  • One clean predicate set — hasChild (recursive containment, validated per-source-shape, never via global rdfs:range); hasDomain single-declared Product → Domain (Document→SubDomain tag-use migrates to taggedDomain); consumes : Product → Service (a Product depends on Services to function — cross-domain consumption); hosts : Domain → Service (the Domain that owns the Service's code and runtime — every Service has exactly one host Domain); contains for content (Domain → content-instance); atStep : Product → ValueStreamStep replaces both hasValueStreamStep and the instance-level primaryStep drift; belongsTo and partOf dropped.
  • Services attach at the Domain level primarily, the Product level secondarily. A Service is hosted by exactly one Domain (where its code lives) and consumed by any number of Products / other Domains (who depend on it). This resolves the AS-IS ambiguity where "where does cards-service belong" had no unique answer.
  • CollectionDomain is removed from Chorus per domain-model-asis-tobe — Gathering's data domains (Photos, Music, Stories, Books, People, Property) belong to Gathering's ontology, not Chorus's. Chorus models how the team works; the Gathering data layer is downstream.

Attribute schema — typed per class

The diagram shows the shape; the tables below give the schema. Each attribute is chorus:-prefixed, has a range (xsd: datatype or class reference), cardinality, and SHACL required-status. Attribute lists were lifted from loom-subproduct-design.html (Product, Domain, Service) + the existing Athena UI domain page (platform/api/public/athena/domain-detail.html + 20 service-design docs); refined here with types, cardinality, the literal-vs-relational distinction, and the inverse-property pattern.

Two kinds of attributes per class:

  • Stored — present as triples on the instance. Returned inline by structural endpoints (/athena/product/:id, etc.).
  • Derived — graph patterns evaluated at query time. Not stored as triples on the instance. Returned by dedicated traversal endpoints (e.g., /athena/domain/:id/decisions, /athena/service/:id/consumers).
Invariant rendering — all stored attributes are required. The UI page for a Product / Domain / Service must render the same sections every time, in the same order, regardless of which instance is being viewed. That requires every stored attribute to exist on every instance — no conditional "show only if defined" branches in the rendering. SHACL minCount 1 on every stored attribute. Multi-valued attributes (gaps, notInScope, etc.) require at least one entry; the entry can be a placeholder literal like "none identified yet", but the triple must exist. Genuinely-optional attributes (e.g., inParent on root Products that have no parent, consumes on Products without external service deps) are the explicit exceptions and stay optional with their semantics named.

Universal status enum across all three classes: {exploring, building, operating, retiring} (this is the artifact lifecycle — distinct from the card pipeline, which lives in the cards-service and emits its own spine events; the two intersect but are different pipelines).

Storage direction (resolved 2026-05-15): all containment edges store on the parent / containment side, matching the classDiagram. Product stores hasChild (child Products), hasDomain (owned Domains), consumes (cross-cutting Service deps). Domain stores hasChild (child Domains), hosts (owned Services), contains (content records — actors, scenarios, decisions, …, per ADR-028 req-1). Service stores its dependsOn / commitsTo / integratesWith / runsOn / persistsIn outbound edges. inParent / inProduct / hostedBy are dropped from the schema — child→parent views are derived via SPARQL sh:inversePath at query time.

RDF — OWL + SHACL TO-BE

The invariant: if every write goes through the API, the graph is whole — and the Athena UI renders every instance properly, no conditional branches. SHACL is the contract, the API is the only writer that honors it. Every required attribute populated, every required edge present, every retired predicate refused at the write boundary, every cardinality enforced — by construction, not by post-hoc audit. Because the graph is whole, the consumer-side guarantee follows: /athena/product/:id, /athena/domain/:id, /athena/service/:id always return complete instances, and the Athena UI pages (domain-detail.html, the service-design renderer, the product surface) render every section on every instance without "show if defined" branches or empty-state placeholders for required fields. The AS-IS sloppiness exists because writes happened through paths that didn't validate (direct triplestore writes, ad-hoc TTL files, harvesters with no shape contract) — which forced the UI to defensively branch on "is this field even there?" v2 closes those paths: the API is the only writer; SHACL is the only gate; the graph stays whole because it cannot become un-whole through any sanctioned write; the UI renders consistently because it can trust the API response.

Mechanics: storage direction follows the classDiagram (containment direction is canonical) — hasChild, hasDomain, hosts, contains, consumes, atStep stored on the subject side named in the diagram. Inverse views (parentOf, hostingDomain) computed at query time via sh:inversePath, not stored. The earlier inParent / inProduct / hostedBy direction is dropped — one storage direction per edge, no double-keyed truth. The cookbook req-1 predicate chorus:contains is the canonical Domain→Record / Service→Record membership edge (renderers query by it). Content records carry no back-pointer; external entities (cards, spine events) that reference a Service use chorus:atService instead of chorus:contains.

This is the contract the API enforces. Writes through POST /athena/{product,domain,service} run SHACL validation against shapes.ttl; refusals name the failing constraint. The retired predicates (inParent, inProduct, hostedBy, chorus:domain / chorus:service back-pointers) are write-blocked by the API regardless of what AS-IS data still carries them — the migration card backfills the canonical-direction predicate before retiring the old triples. Inverse-path SHACL on DomainShape and ServiceShape enforces that every Domain belongs to exactly one Product, every Service is hosted by exactly one Domain — without storing the inverse edge on the child side.

OWL — Value Stream & Steps TO-BE · #3378 design → #3351 build

Gap confirmed 2026-06-12: chorus:ValueStream and chorus:ValueStreamStep exist in chorus-v2.ttl as bare class declarations — every shape's mandatory atStep points at them, but they have no shape, no properties, no instances, and therefore no generated API. Jeff's rulings modeled here: "we must have value stream and value stream step" (05-15); "a step must be able to encapsulate a child value stream and steps — eg werk is a value stream for the step Building" (06-12). Two distinct edges, never conflated: inStream (membership — a step belongs UP to its parent stream; exists today) and hasValueStream (encapsulation — a step contains its child stream; NEW). Recursion is unbounded by construction. 
# ── Entities ─────────────────────────────────────────────────────────
chorus:ValueStream      a owl:Class .   # an ordered flow of steps
chorus:ValueStreamStep  a owl:Class .   # one stage; may encapsulate a child stream

# ── Properties ───────────────────────────────────────────────────────
chorus:inStream        a owl:ObjectProperty ;          # EXISTS (membership, child-side)
    rdfs:domain chorus:ValueStreamStep ;
    rdfs:range  chorus:ValueStream .                   # step → its ONE parent stream

chorus:hasValueStream  a owl:ObjectProperty ;          # NEW (encapsulation — Jeff 06-12)
    rdfs:domain chorus:ValueStreamStep ;
    rdfs:range  chorus:ValueStream .                   # step → the child stream it contains

chorus:stepOrder       a owl:DatatypeProperty ;        # NEW — steps are ordered
    rdfs:domain chorus:ValueStreamStep ;
    rdfs:range  xsd:integer .

# atStep (existing, subject ∈ {Product,Domain,Service}) ranges over ANY
# ValueStreamStep — top-level or substream; placement is recursion-agnostic.

# ── Shapes (the generator's input) ──────────────────────────────────

    sh:property [ sh:path chorus:label   ; sh:minCount 1 ; sh:datatype xsd:string ] ;
    sh:property [ sh:path chorus:comment ; sh:minCount 1 ; sh:datatype xsd:string ] ;

chorus:ValueStreamStepShape a sh:NodeShape ;

6 · Pages & Flow — the UX of the model #3378

Jeff's ruling (2026-06-12): "the pages and owl are all part of the athena product design." The UX is a flow of pages — landing (the value stream itself, five steps + seven products) → step → product → domain → service — where every page is a rendering of its class's generated API plus its schema, and every inter-page link follows a real model edge. Navigation starts from the value stream, not products (CMDB ruling, April). The domains domain's page renders THE SET — the generated catalog live (one taxonomy source with catalog/domains.md and the #2159 board hierarchy). A step encapsulates its child value stream (hasValueStream — werk is Building's stream, its steps the verb sequence).

Artifacts: the walkable mock (domain pages live on the generated API; product/service pages are static exemplars that ARE the #3351 page spec) and the page anatomy (breadcrumb → identity → promise → stats → honest-folds → provenance layer-labels → edges-are-links; the review standard for every fan-out leg's page until pages are generator-emitted).

7 · API Surface — Athena v2 TO-BE

Three resource namespaces — /athena/product, /athena/domain, /athena/service — one per structural class, plus edge / query / passthrough endpoints. The lever (per reasoning-substrate-shift.html): "make the graph as interrogable as the logs already are."

Mutation paths (POST/PATCH/DELETE on the same resources) ship in a later wave — read-side first. Overlay paths (/athena/overlay/*) land as their own resource namespace alongside the structural reads.

Edge handling (constraint, compressed): edges are first-class resources — reads project them inline as IRI arrays; all edge writes go through typed edge endpoints (POST /athena/.../children, /services, /records, …) so SHACL validates one predicate at a time and refusals name the exact constraint. Instance creation is an atomic composite POST (full required payload validated as a unit; partial creates refused 422 — no half-written instances). Retired predicates (inParent / inProduct / hostedBy / belongsTo / partOf / back-pointers) return 409 retired-predicate naming the canonical replacement. Edge removal refuses minCount violations. /athena/edges and /athena/sparql are read-only by guard.

8 · As-Is / To-Be / Path to close

AS-IS (what exists and runs today)

  • v1 — SPARQL/Fuseki (localhost:3030, urn:chorus:instances): 49 subdomains, ~88 consumers. Legacy read surface (subdomains_get / subdomains_list) live. chorus-instance-explorer.html queries Fuseki direct, bypassing the API.
  • v2 — tree.json + the generated Domain API at :3360 live, 34 domains + CORS. chorus-v2.ttl + shapes landed. v1 and v2 are bridged.
  • Operational layer — MCP tools (chorus_tree_get / chorus_ownership_lookup / chorus_blast_radius), SessionStart envelope injection, the rendered tree page, the card/code/test tagging convention.

TO-BE

  • One generator covers Product / Service / ValueStream (today: Domain only). Shapes are the generator's input; every class gets the same generated API + page treatment.
  • DAL writes (#3257) — the API becomes the only write path; SHACL the only gate; the graph whole by construction (the wholeness invariant in §5).
  • Doc corpus ingested as chorus:Document instances — design docs become graph citizens, cited by IRI.
  • Pages generated — the §6 flow (landing → step → product → domain → service) emitted by the generator; the flow mock is the spec until then.
  • Overlays — authority / ownership / deploy-route / customer-happiness as first-class queryable annotation predicates.

Path to close

#3350 (done) → #3373 (done) → #3351 (ValueStream/Step shapes + instances + generated API + landing page) → DAL #3257 (write path) → crawler.

Live constraints the path must honor (compressed from the v1 doc's edge-handling / requirements / implementation sections):

  • Schema → API → data order — each layer constrains what gets added above it (v1's data-first order is how sloppiness became normal). Fresh write, not rewrite: v2 lives under its own namespace; v1 stays as legacy reads, 308-redirects at cutover, retires wholesale.
  • Bootstrap order — Role, ValueStream, ValueStreamStep, seed Documents must populate before any Product/Domain/Service can validate (every required edge needs an existing target).
  • Write attribution — every mutation emits a spine event with role + card_id + triple provenance; uniform error taxonomy across endpoints; comprehensive logging (no silent 4xx).
  • Single contract source — OpenAPI/JSON-Schema generated from SHACL, never hand-maintained in parallel; shape changes ship with their backfill migration, never alone.
  • Inferred dependencies — dependency edges the structural graph already implies are derived (SPARQL), not stored; stored dependsOn/integratesWith only for what the graph can't express.
  • Five test layers — SHACL fixtures (valid + one invalid per violation type) → integration round-trips → API contract → UI renders → scheduled live-graph data-quality scan emitting violation spine events (also Loom's learning corpus: recurring incoherence → candidate principles).

Gaps (honest)

  • v1 predicate layer: hasDomain double-declared, consumes overloaded, belongsTo used-but-undeclared — the API bottleneck; a traversal verb can't return one consistent type.
  • Service layer missing: service-named nodes still typed SubDomain; chorus:Service never instantiated; type-migration not run.
  • ValueStream / ValueStreamStep: bare class declarations — no shape, no instances, no generated API (closes via #3351).
  • No write path yet: mutations ship in a later wave; until the DAL + pre-commit hook land, direct TTL edits remain possible.
  • Two vocabularies (chorus: / fw:) with no equivalence bridge — graph-wide, every traversal verb hits it.
  • Borg placement unresolved: Product in the instances graph vs SubProduct under Chorus in the tree — affects every traversal starting from "Chorus."
  • ServiceDesign vs Service — fold into Service metadata, or keep as a separate artifact-type class?
  • Customer-happiness overlay home — Loom (measurement) or Athena (structural annotation)?
  • stepOrder per-stream uniqueness — carried to #3351 (SHACL sh:sparql vs DAL-at-mint).
  • Migration ordering — Service type-migration first (smaller, isolated), then recursive-Domain collapse.
