Gate ceremony at agent velocity

How to scale quality gates for our actual use case — 3 agents coding frequently into 1 environment with high standards. Research → recommendation, not options menu.

v3 (2026-04-28 17:51) integrates peer review across two passes: Silas (Move 2 cleavage, tiebreaker, Move 4 reframe, §9 endorsement, then a second pass with five sharper pushes); Kade (taxonomy refinement, Move 3 two-layer framing, parent-arc pairing). Silas's second pass added Move 7 (distribute gate-product), the context-freshness reframe of Move 4, the honest "Move 1 is codification not behavior-change" softening, and the N=3 assumption flag.

Recommendation (one-line)

Codify the tiering that's already happening de facto, run independent gates concurrently, and add a narrow self-attest path for low-risk cards — but the bigger discovery is that gates are not the actual bottleneck.

The audit shows roles are already adapting gate scope ad hoc by card type. The compounding cost isn't gate count per card; it's that every card pays full ceremony regardless of whether it needs it, and gate-product is single-threaded through Wren. Codifying the existing practice + parallel execution + a self-attest path closes ~60–70% of the ceremony tax without dropping safety.

The honest find: the larger pattern Jeff named — bottleneck — is mostly upstream of gates. It's authorship discipline (follow-on compounding), card hygiene (scope drift), and integration mechanics (PR serialization). Those are different cards.

The gate ceremony cost is not uniform 5×N. It's already approximately 3.1×N (mean of sample). The structural fix is to make that explicit, not to invent new lightness.

2. The actual constraints

Wren-as-gate-product is the single-thread

Every product gate in the sample passed through me. Three agents producing → three queues converging → one me. No amount of personal attention solves this; it's a structure problem. When I'm the builder I self-gate, which subverts the gate's purpose (the audit-of-other shape).

Sequential gates serialize what's independent

code, quality, arch, ops, product are mostly pairwise independent reviews of the same artifact. Today they run in series via nudges-and-waits. Wall time is dominated by waiting for the next gate to start, not by the gates themselves. Concurrent execution is on the table.

Card-type and gate-set are not formally linked

The card type taxonomy (type:fix, type:enhance, type:new, type:chore) is too coarse to drive gate selection. We need a finer taxonomy that names card-shape (substrate / surface / coordination / investigation / refactor / doc) so /pull can pick the gate set automatically.

Gate ceremony is agent-judgment correction, not deployment readiness

This is the framing that matters. With humans, code review catches cognitive errors and design oversights. With agents, gates catch different patterns — false confidence in self-tests, plausible-but-unprincipled code (per the SlopCodeBench paper), structural erosion. Our gates exist because we don't trust agent self-assessment alone, not because we're shipping to customers. That reframes them: targeted catches for known agent failure modes, not bureaucratic checkpoints.

3. External references — what the field is doing

Read four pieces this week directly relevant. Synthesizing what each says about our problem.

Fowler / Zhang — Structured Prompt-Driven Development (Thoughtworks, 2026-04-28)

Workflow: prompts as version-controlled artifacts, REASONS canvas, three skills (alignment, abstraction-first, iterative review). One review loop, not five gates. Their use case is one engineer + AI; lighter ceremony works because the human is in-the-loop continuously. What we take: the iterative review pattern is closer to /pull + commit-time hooks than to card-end gates.

Hatoum — Auto / "Claude with Ketchup" (LinkedIn)

LLM-powered guardrails on every commit. "Encode mistakes as rules; the next Claude can't repeat them." Per-commit, not per-card. Continuous detection, not gate ceremony. Closer to our PreToolUse hooks + memory feedback files. What we take: the leverage is shifting detection earlier, not adding more end-of-card review.

Boeckeler — Harness Engineering (Thoughtworks)

Computational sensors with the agent — static analysis, test quality. Sensors run continuously. Same idea: detection upstream of gate boundaries. What we take: our gate-quality is doing what a continuous sensor should do; we could shift much of it left.

Lindenberg — agentic-runtime 10-layer breakdown (LinkedIn)

Distinguishes orchestration runtime (session, tool-execution policy, observability) from gate ceremony. We may be conflating two layers. Some of what our gates do is runtime (does the test pass? does the build clean?) which should be automated, not human-reviewed. What we take: separate runtime checks from judgment checks.

Mason / SlopCodeBench (UW–Madison + WSU + MIT)

80% of agent-coded trajectories show structural erosion; 90% show verbosity increase; 2.2× more verbose than human code; code passes tests but quietly becomes harder to extend. Pass-rate benchmarks systematically miss extension robustness. What we take: our gates should specifically test for structural erosion patterns that automated tests miss. This is the biggest validation of why we need judgment-gates at all.

TBD-at-scale practice (Google, Meta, etc.)

Continuous integration + mandatory pre-commit + 1 reviewer per CL. Not 5 gates. Their scale (hundreds of devs) drives them to minimum-viable-ceremony. What we take: 5 gates is unusually heavy. But our scale is also unusual — 3 agents, no customer pull, judgment substituting for trust.

4. What makes our use case unusual

None of the references above quite fits. Our combination is novel — and worth flagging that this entire design optimizes for N=3 agents; if Jeff spins a 4th role or splits an existing surface, bottleneck patterns shift (gate-product single-thread gets worse, parallel-gate coordination harder, the owner-pair cleavage in Move 2 needs re-examination). Move 7 is N-robust by design; Moves 1–4 are not. Honest assumption to name (Silas's call).

  • 3 autonomous agents writing concurrently, each capable of full-stack work, no human between authorship and PR.
  • 1 environment — local Mac = dev = "production." No staging buffer; no deploy step that creates a natural gate boundary.
  • High quality bar with no customer pull. The bar exists because Jeff cares about it, not because users will complain. So gates substitute for customer feedback as the quality signal.
  • Continuous high-volume output. Cards land hourly across three queues. CI runtime + gate latency × N cards becomes the dominant cost line.
  • Cross-role dependencies through the live environment. Silas's hook change can break Kade's session boot in the same minute the commit lands. This is unique to the 1-env constraint.

No prior art for "3 agents, 1 env, high quality, no customer." We're inventing the harness as we go. Gates are the mechanism we've used for "is this good enough to live with" — but they were designed assuming a human-paced authorship rate.

5. Candidate moves

Move 2 — Run gates concurrently at owner-pair granularity adopt

Refined per Silas's review. The right cleavage is Kade-pair (code + quality) || Silas-pair (arch + ops), not full 4-way decomposition. Within each pair, the owner serializes internally (Kade runs code → quality back-to-back; Silas runs arch → ops). Across pairs, no real dependency — code never blocks arch, quality never blocks ops. Gate-product runs last.

Evidence backing parallelization: Silas audited 5 substrate cards from the sample (#2549, #2550, #2510, #2516, #2522) for retroactive-fail patterns where one gate's finding required re-running an already-passed gate. Zero retro-fail patterns found. arch findings are namespace/domain-boundary issues independent of code; ops findings are health/log/disk independent of quality.

Tiebreaker rule (Silas's contribution — handles the only real overlap): when arch and quality disagree on structural fit, arch wins on "system fit," quality wins on "within-card hygiene." Document in TEAM_PROTOCOL.

For substrate / coord:ci cards (full 5 gates), this collapses wall time from ~30min to ~8–10min — bounded by the slower of the two pairs.

Tradeoff: 4-way decomposition was simpler to describe but added risk for marginal gain (Silas's call). Owner-pair is more honest about who's actually doing the work. Hidden retro-fail patterns: audit said zero historically, but instrument the gate spine events to track if any emerge after the change.

Move 3 — Self-attest only retires the judgment layer, never the machine layer adopt

Refined per Kade's framing — sharper than the original. We have two layers of fail-loud:

  • Machine layer — pre-commit hooks (#2481 ratchet, #2526 wave 2 narrow), CI required-checks (DEC-2525). Compile, lint, typecheck, hermetic test pass. Fires always, on every commit, regardless of card shape.
  • Judgment layer — peer gates. "Does this make sense? Right pattern? Right scope? Right product framing?" Catches a different class of bug than the machine layer.

Self-attest only touches the judgment layer. For narrow card shapes (artifact with low scope: typo-fix, lint-only, doc-only), the builder declares the judgment-layer gates pass without queueing peer review. The machine layer fires unchanged on every commit.

Spine event tagged gate.self-attested. Random sample audited weekly. Start with the narrow tail and expand only on audit data.

Tradeoff: drift risk on the judgment layer is real but bounded by the audit. The two-layer framing is the load-bearing piece — "true self-attest = no automated check either" would be a regression because the auto-sensors catch mechanical bugs (different class) than peer review catches (judgment bugs).

Move 4 — Catch issues while context is fresh (continuous sensors) deferred to post-#2528

Reframed per Silas's two passes. The actual leverage of continuous-over-periodic isn't "earlier detection" — it's detection while the author still has the model in their head. Card-end gates fire after the role has context-switched; the cost of fixing a finding scales with how far the author has moved on. Boeckeler / Hatoum are saying the same thing in different words.

The substrate is already here post-#2526:

  • Signal-only CI jobs (continue-on-error roster from wave 2)
  • Nightly drift lane (wave 3 schedule trigger)
  • Per-job spine events on every workflow run
  • Hermeticity audit data (#2523 JSON, refreshable on demand)
  • Cross-language test categorization (#2524)

What's actually missing per Boeckeler / Hatoum:

  • Aggregation surface — Borg quality/fitness/replay pages are 80% there but not wired to the CI-signal stream
  • Alert-tuning mechanism — signal-to-noise governance ("the sensor fired 47 times this week, what now")
  • Sensor consumer registry — #2528. The load-bearing piece. DEC-2525's "criteria for adding a required check" explicitly requires citing a consumer; without the registry, that criterion is unenforceable.
  • Static-analysis-as-sensor for SlopCodeBench-shape erosion (verbosity-creep, extension-fragility) — open R&D, custom sensor needed; not bundled with this work

Right framing: not "needs infrastructure we don't have" but "1–2 month arc after Move 1+2+3 land + #2528 lands." Sequence #2528 alongside the gate-tiering work and Move 4 becomes actionable, not someday. Owner of the SlopCodeBench-shape sensor sits with whoever owns #2528.

Tradeoff: still defer for THIS card's recommendation — Move 1+2+3 + #2528 must land first. But naming the actual gating dependency (sensor consumer registry) instead of vague "sensor framework" makes the next step concrete.

Move 5 — Drop a gate skip

Considered: do gate-arch and gate-ops both pay rent when Silas writes both for the same card? Could collapse to gate-infra.

Why no: the audit shows arch and ops fire independently with different findings on substrate cards (#2549, #2550, #2510). They're catching different things. Collapsing them loses signal. Move 1 (card-shape tiering) already drops arch+ops from cards that don't need them.

Move 7 — Distribute gate-product (retire Wren-single-thread + self-gate) adopt

New in v3 per Silas's second pass — the structural fix this doc was avoiding. Two things the data plainly shows that v2 named-and-parked instead of fixing:

  • gate-product is single-threaded through Wren on every card sampled (the actual choke point)
  • When Wren is the builder, Wren self-gates own product (#2549, #2550) — performative by construction, same shape as code-reviewing your own commit

Both are the same structural problem: one reviewer for product. The fix is to distribute. Two candidate splits, pick one:

  • (a) Any peer can run gate-product against a defined rubric (AC complete, demo evidence present, scope-vs-promise check, domain visibility, spine contract honored). Silas + Kade run it on Wren cards; Wren still runs it on theirs.
  • (b) Split gate-product in two: AC verification (peer-runnable mechanical) + experience integration (Wren/Jeff only judgment). The mechanical half goes parallel; the judgment half stays Wren-pace.

Both leans converged on (b). Refined rubric (per Kade's second pass — load-bearing corrections):

Tradeoff: (b) requires the rubric to hold tight under pressure. Move 6 — Lighten gate-product to "is this what was asked for" maybe

Currently gate-product re-checks AC, demo evidence, domain visibility, spine contract. Some of that overlaps with code/quality. Could narrow gate-product to: did this card deliver the experience promised? Lighter, faster, less single-threaded.

Why maybe: the overlap is real but the integrative check is also real — gate-product is the only gate that holds the whole. Narrowing risks missing scope drift. Want to revisit after Move 1 lands and we see what actually changes.

6. What's NOT a gate problem

Pattern-matching by shape, not symptom

Several frictions get blamed on "gates are slow" but are actually different problems. Naming them so we don't fix the wrong thing:

  • Card hygiene — fuzzy AC, scope drift, badly-shaped cards. Lives at /pull time, not gate time. Card #2433 (cascade-signal) and #2091 (card-type hygiene) are upstream of this.
  • Follow-on compounding — cards generating cards. Authorship discipline issue, named yesterday. Gate ceremony actually helps here (each gate is a chance to scope-correct), so reducing gates doesn't help.
  • Integration mechanics — PR ordering, merge conflicts, branch coordination. CI/branching domain. Card #2556 already addressed the entangled-branch failure mode; #2195 (worktree isolation) is the next move.
  • Cross-role nudge latency — waiting for the other role to respond. Attention contract issue (DEC-1571), not a gate issue.
  • Single env / cross-role breakage — Silas's hook change breaking Kade's session. Different problem: needs better isolation (worktrees), not lighter gates.

7. Recommendation in detail

Adopt Move 1 + Move 2 + Move 3 + Move 7 as one coherent change. They reinforce each other:

  • Move 1 codifies the gate-set-by-shape mapping (mostly labeling work that skills already do, plus the new shapes Kade surfaced).
  • Move 2 reduces wall-time for the cards that still pay full ceremony (owner-pair concurrency).
  • Move 3 drops the judgment layer to self-attest for the narrow tail; machine layer fires unchanged.
  • Move 7 distributes gate-product so it stops being a single-thread and ends the self-gate-on-own-work pattern. The structural fix the data demanded.

Together: ceremony tax drops from ~3.1 gates × N cards × serial-time to ~2.0 gates × N cards × parallel-time + distributed product gate, with a small zero-gate tail. Estimated 60–70% wall-time reduction on gates, with structural relief on the bottleneck pattern (Wren-single-thread + self-gate), with no drop in coverage of the failure modes that gates actually catch.

Sequence the implementation:

  1. File card: codify card-shape taxonomy in board labels — substrate / surface / coord:ci / coord:process / platform-tooling / refactor / artifact (small Wren card, ~1h).
  2. File card: update /pull skill + /demo skill to read shape and pick gate set (medium card, Wren+Silas).
  3. File card: parallelize /demo Step 5 nudges at owner-pair granularity (small Silas card).
  4. File card: define self-attest narrow categories + spine event + weekly audit script (small Wren card).
  5. File card: split gate-product into AC-verification (peer-runnable) + experience-integration (Wren/Jeff-only); add the "no self-gate on own product" rule (medium Wren+Silas card).
  6. Run for 2 weeks, re-audit with same method as this card, see if the prediction held.

Skip Moves 5, 6 for now. Move 4 is the right long-term direction; gating dependency named (#2528). Move 5 lacks evidence. Move 6 is subsumed by Move 7 (distributing product gate is the lighter-product question handled correctly).

8. Honest gaps in this analysis

  • Sample size is 10 cards over 36 hours. Real audit should run on 30+ cards over 30 days. The patterns I named are real but the percentages are rough.
  • Cycle-time numbers conflate gate latency with author latency. A card that took 22h spent most of that on authorship, not gates. Need finer instrumentation (gate-fired timestamps in spine) to separate.
  • I did not audit failed gates. Sample is all Done cards. The cases where a gate caught something and the card got reshaped are missing — and those are the cases where the gate's value is highest. Future work.
  • "Wren as builder gates own product" frequency not quantified. Smelly pattern; needs separate audit. Move 7 addresses the structural shape; the audit would size the actual incidence.
  • Design assumes N=3 agents. Moves 1–4 are tuned for current team shape; Move 7 is N-robust. If Jeff adds a fourth role or splits a surface, gate-product single-thread gets worse, parallel-gate coordination harder, owner-pair cleavage in Move 2 needs re-examination. Worth flagging now (Silas's call).
  • External references are mostly authorship-side; less is published on multi-agent gate ceremony. We may genuinely be inventing this. The Boeckeler / Hatoum direction (continuous sensors) is the closest external analog; others are different shapes.

9. The bigger frame, surfaced honestly

Going into this card my hypothesis was "gates are the bottleneck." The audit doesn't support that as the dominant story. The dominant story is more uncomfortable:

We have an unusual use case and we're inventing the harness as we go. Most of our investment so far has been on authorship-side discipline (TDD gate, AC quality, role gates) — which is well-tuned. The actual leverage is on the integration side, where we're under-invested and where the bottleneck pattern Jeff named lives.

Gates are part of the integration loop, but they're not the whole. Worktree isolation, merge serialization, CI runtime, cross-role attention contracts, and the absence of a dedicated integration role are all in this same neighborhood and probably weigh more.

This card's deliverable (Move 1+2+3) is a real improvement. But the next research card — and I'd file it — is "Integration as a first-class team activity at agent velocity", which would treat gates, worktrees, CI, branch protocol, and integration ownership as one connected system instead of as separate discipline cards. That's the parent arc this lives under.
