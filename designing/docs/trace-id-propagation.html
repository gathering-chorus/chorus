trace_id propagation contract (#2839)

Silas · 2026-05-08 · substrate design. Implementation lands per-surface in follow-on cards. Sources: spike #2841, Kade's MCP-mint feedback, Wren's rename direction.

Headline

Every transaction-shaped flow (a /pull, /acp, /demo, gate invocation, building-pipeline trigger) is one trace. One trace = one trace_id minted at the entry point and carried through every downstream emission until the flow terminates. Today there's no transaction-scope id; the spike found 13 distinct ids per /acp run. This contract collapses that to one.

Vocabulary

Two concepts already coexist; the contract names them distinctly so they stop being confused (per Wren #2841 PM call: rename over qualify, OTel-aligned).

OpenTelemetry alignment: trace_id is the cross-service flow id; span_id is one operation within it. Chorus is taking the same shape without going full OTel. The vocabulary alignment earns its keep when (not if) Chorus emits to OTel collectors.

Format

trace_id is a UUIDv7 string (already what mintTraceIdV7 emits at platform/api/src/server.ts:1835). 36-char canonical form, lowercase, hyphenated. UUIDv7 is time-ordered so traces can be range-scanned by approximate time without a separate timestamp index.

trace_id format:
  ^[0-9a-f]{8}-[0-9a-f]{4}-7[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$
  example: 019e09a5-2f3a-7c3c-91c1-500190aa2e5e

Mint rule (the load-bearing primitive)

Mint at the MCP entry handler. Pass to SpineEmitter. Every emit() inside the call inherits it automatically. This mirrors the existing chorus_nudge_message path — proven shape, no new substrate. Per-call wiring is forbidden because it drifts; the SpineEmitter constructor is the seam.

Failure mode if not enforced: the spike found chorus_acp emits 11 internal step events (chorus_acp.*.{started,completed}) all carrying card_id but no trace_id — exactly because the emitter is per-call wired. Fold the mint into the emitter constructor, and 11 events get the field for free.

Per-surface handoff

MCP entry handlers (TS, chorus-api)

The primary surface. Six handlers in scope today (chorus_acp, chorus_commit, chorus_pull_card, chorus_unpull_card, chorus_cards_done, chorus_cards_set); chorus_nudge_message already conformant.

Today's shape (platform/api/src/server.ts:93): SpineEmitter is a function type (event, fields) => void; defaultSpineEmitter at line 499 is already a factory returning a closure. The contract grows that into createSpineEmitter(traceId): SpineEmitter — closure captures trace_id, every emit() inside automatically attaches it. No type-system fight, no API break for handlers that don't pass trace_id (the field is omitted when undefined).

Recommended (a) — emitter construction inside each handler:

// at handler entry (executeAcp / executeCommit / executePullCard):
const trace_id = mintTraceIdV7();
const emit = createSpineEmitter(trace_id);
// every emit(event, fields) inside this call carries trace_id
return await runFlow(args, emit);

This is the right long-term shape: each handler owns its trace lifecycle, no shared mutable state.

Acceptable migration shortcut (b) — wrap the existing singleton:

// at handler entry:
const trace_id = mintTraceIdV7();
const tracedEmit = (e, f) => emitSpineEvent(e, { ...f, trace_id });
// pass tracedEmit instead of emitSpineEvent

Smaller diff, gets to first-green faster, but leaves the singleton emitter as a coupling point. Use (b) if step 3 (the six MCPs) needs to ship before step 1's full refactor lands. Otherwise (a). Both produce identical wire-level behavior; the difference is how cleanly future handlers compose.

Returned to caller in the response shape so the calling skill can store it and run follow-on queries: { role, card_id, sha, pr_url, trace_id }.

Bash CLIs (chorus-log, cards, git-queue, chorus-werk, chorus-build, chorus-deploy)

Two cases:

  • Bash invoked by an MCP handler: trace_id rides on env (CHORUS_TRACE_ID) set by the MCP's execFile options. chorus-log reads $CHORUS_TRACE_ID and adds it to every emit. No argv pollution.
  • Bash invoked from a top-level skill (Claude bash, cron, launchd): the bash entry point mints (UUIDv7 via uuidgen | tr A-Z a-z for now; or call chorus-log mint-trace if we add a subcommand) and exports CHORUS_TRACE_ID for child processes.

chorus-log change: today it accepts event role kv... argv. Add: if $CHORUS_TRACE_ID is set, prepend trace_id=$CHORUS_TRACE_ID to the kv list before forwarding to the shim binary. One-line change in the wrapper.

Rust hooks (chorus-hook-shim, chorus-hooks daemon)

The shim reads stdin (PreToolUse JSON) which already carries session id but not trace_id. Two cases:

  • Hook fired during a known transaction: trace_id is in the env (set by the calling skill's bash). Shim reads $CHORUS_TRACE_ID and includes it in every event it emits.
  • Hook fired outside a transaction (file save, ad-hoc tool call): mint a per-hook trace_id. These hooks aren't part of a flow; their trace stands alone. Tag them flow_kind=hook-standalone so query surfaces can distinguish.

TS handlers outside MCP (chorus-api routes, pulse worker, observer)

HTTP requests carry X-Chorus-Trace-Id header. Middleware reads or mints, attaches to req.locals.trace_id, and any SpineEmitter constructed in the handler reads from there. Pulse worker mints per-cycle (one trace per pulse run); observer fan-out inherits the trace_id from the triggering event when known, mints when not.

External boundaries (gh, git, npm, launchctl)

External tools have no trace awareness. The MCP brackets the call:

emitter.emit('chorus_acp.gh-merge.started', { pr_number });
const result = await execFile('gh', ['pr', 'merge', ...]);
emitter.emit('chorus_acp.gh-merge.completed', { pr_number, exit_code: result.code });

The interior of the call is opaque to spine. Documented gap: gh/git/npm log lines won't carry trace_id; the bracket events are the correlation surface.

spine-events.json schema change

Add trace_id to the global event envelope (not per-event). Today the schema declares per-event fields; trace_id is universal.

// designing/schemas/spine-events.json — top-level addition
"envelope": {
  "trace_id": {
    "type": "string",
    "format": "uuid",
    "pattern": "^[0-9a-f]{8}-[0-9a-f]{4}-7[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$",
    "required": true,
    "description": "Transaction-scope correlation id, UUIDv7. Minted at flow entry, propagated through every emission until terminal event. Distinct from span_id (per-log-line)."
  },
  "span_id": {
    "type": "string",
    "format": "uuid",
    "required": false,
    "description": "Per-log-line / per-request id. Pino's current 'trace_id' field renamed to span_id when this contract lands."
  }
}

Validator (existing scripts/validate-spine-event.sh if present, or new) refuses any emission whose envelope lacks trace_id once the migration completes. During migration, validator emits a warning, not an error, until every surface conforms.

Termination

A flow ends with one of three terminal markers:

  • flow.completed — happy path, MCP returned success
  • flow.failed — typed refusal returned to caller
  • flow.aborted — handler crashed, no clean exit

The terminal event carries trace_id + flow_kind (acp, pull, demo, gate, build, etc) + duration_ms. Without a terminal event, query surfaces can't tell whether a trace is in-flight or orphaned. Adding terminal events to the existing 6 transactional MCPs is a sub-deliverable of this contract.

Cross-cycle stitching (/acp → building-pipeline)

One trace_id per MCP call gives "all events for one /acp" but not "all events for one building cycle" — the building-pipeline triggered by the merged commit runs as its own flow, minted by launchd's wakeup, not by /acp.

Resolution: commit-trailer carries trace_id across the cycle. /acp's commit step writes Chorus-Trace-Id: <trace_id> into the commit message trailer (alongside existing trailers like Co-Authored-By). Building-pipeline reads git log -1 --format='%(trailers:key=Chorus-Trace-Id,valueonly)' on the commit it's about to build. If present, it reuses the trace_id; if not (commits made outside /acp), it mints fresh.

This makes one /acp + the building-pipeline it triggers + the deploy events the pipeline emits all share one trace_id. Loki query {job=~".+"} |~ "<trace_id>" returns the full cycle.

Channels for cycle stitching, in priority order:

  1. Commit trailer — primary, persists with the commit, durable across re-fetches and replays.
  2. launchd plist environment — when launchd is the trigger (e.g., scheduled crawler runs), the plist EnvironmentVariables dict can carry a fresh trace per scheduled invocation; the script reads $CHORUS_TRACE_ID as documented above.
  3. Webhook header X-Chorus-Trace-Id — for any future GitHub-Actions / external-CI handoff.

What this contract does NOT do

  • Does not retrofit gh/git/npm/launchctl logs. External tool interiors stay opaque; MCP brackets are the correlation surface.
  • Does not backfill historical events. trace_id arrives forward-only. Audit queries against pre-contract events will continue to use card_id + temporal joins.
  • Does not standardize trace_id across machines. Library and Bedroom mint independently; cross-machine flows (library-health-probe SSH'ing to Library) carry trace_id over SSH env. UUIDv7 collision probability is zero in practice.
  • Does not define card_id propagation. That's #2838, sequenced after this. card_id rides the same envelope but has different mint/derivation rules.

Test fixtures

The spike's per-hop ledger (designing/docs/trace-id-card-id-threading-spike.html) doubles as the acceptance test for this contract. Re-run /acp on a card after migration steps 1+3+4 land; assert one trace_id across all 40+ events. The spike's "13 trace_ids per flow" finding becomes "1 trace_id per flow" — that's the headline metric.
