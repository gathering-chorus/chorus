Chorus Business Glossary

Sourced from Athena v2 — 7 products, 33 domains placed against the value stream. 2026-05-28, Wren, card #3109.

The shape: work flows through a value stream — Shaping → Designing → Directing → Building → Proving. Each product is placed at one step. Each product is supported by domains.

How to read each product:
  • Function — what the product does for the team.
  • Value — why that work matters.
  • Ambiguity — where today's understanding is unclear (if any).
  • Domains — the working parts the product is built from, with their Athena-registered descriptions.

Shaping — where ideas form

Loom Wren · status: building

Function
Holds the team's knowledge layer — the values, roles, principles, policies, practices, and skills that define how the team operates.
Value
A new session starts with the same beliefs as the last one. Decisions don't have to be re-litigated. Patterns repeat on purpose, not by accident.
Domains under Loom
Roles — Team member identities, capabilities, and session lifecycle.
Principles — Values and constraints governing team decisions and priorities.
Policies — Rules that enforce principles in practice — hooks, guards, gates.
Practices — Repeatable patterns the team uses — how we work, not just what we value.
Skills — Invokable capabilities — /skill-name commands that automate team workflows.
Decisions — Decision log with context, rationale, and tradeoffs recorded.
RCAs — Root cause analyses — what broke, why, and what changed as a result.

Designing — where the model gets built

Athena Wren · status: building

Function
The team's map of itself — the canonical product / domain / service structural model.
Value
One truth for the team. Anyone asking "what is this?" or "where does this belong?" gets the same answer. Without it, three roles drift into three private versions of reality.
Domains under Athena
Domains — The canonical product/domain/service structural model — this tree.
Services — Service definitions, interfaces, and ownership within the product model.

Convergence Kade · status: exploring

Function
How Chorus connects to external systems and how Chorus + Gathering exchange data.
Value
Data moves with intent. Outside systems can plug in under a contract; cross-product flows are visible, not implicit.
Ambiguity
Currently a single "integrations" domain — likely needs to split into contracts (ICDs) and engine (NiFi) as the product matures.
Domains under Convergence
Integrations — Connections to external systems — NiFi, APIs, cross-product data flows.

Directing — where work gets routed and attention gets paid

Chorus Jeff · status: operating

Function
The product Jeff uses to work with the team — context, facts, and retrieval brought together so the team can be directed coherently.
Value
Jeff can direct the team without re-establishing context every session. The system holds what was, what is, and what's findable across all of it.
Domains under Chorus
Memory — Persistent context across sessions — indexed sessions and artifacts.
Knowledge — Structured facts about the team, products, and their history.
Search — Semantic retrieval across all indexed team knowledge and session history.

The Clearing Wren · status: operating

Function
The room the team meets in — messages, event streams, the audit trail, and the kanban board, all in one place.
Value
You can see what's happening and what has happened without asking anyone. The team becomes legible to itself.
Domains under The Clearing
Messages — Direct role-to-role communications — nudges and the bridge messaging API.
Streams — Event streams and pub/sub channels the team uses to coordinate.
Spine — The audit trail — structured log of all team activity and state changes.
Cards — Kanban board and card lifecycle — the unit of work tracking for the team.

Building — where units of work get made

Werk Kade · status: operating

Function
How code moves from a working idea to deployed software — pipelines, version control, code quality, and tests.
Value
Work moves through the team the same way every time. Quality and reproducibility are structural, not accidental.
Ambiguity
Werk sits at Building (one owner, one step), but its verb protocol is consumed by every step. How that consumption is structured — services Werk exposes vs. new domains inside the consuming products — is open.
Domains under Werk
Pipelines — CI/CD and build orchestration — how code moves from commit to production.
Version control — Git workflow, branches, worktrees, hooks, and commit policy enforcement.
Code — Source code quality — linting, type-checking, and static analysis.
Tests — Automated test suites — jest, bats, cargo test, and integration tests.

Proving — where work gets validated and the system sees itself

Borg Silas · status: operating

Function
How the system sees itself — discovers what's running, what's deployed, what's healthy, and what's alerting across every machine and every service.
Value
The team can trust what it claims. "Is this deployed? Which version? When? Is it healthy?" gets a real answer from observation, not a guess from memory.
Domains under Borg
Builds — Build artifacts, Rust binary signing, and cdhash provenance.
Deploys — Deployment records, rollback paths, and chorus-deploy orchestration.
Logs — Log aggregation and querying via Loki — structured observability.
Properties — Service configuration and runtime properties across Library and Bedroom.
Alerts & monitors — Grafana dashboards, alert rules, and on-call signal management.
Security & trust — Authentication, authorization, secrets management, and TCC permissions.
Time — Scheduling, cron jobs, and time-based triggers for automated team work.
Heralds — Notification delivery — nudge routing, chorus-inject, and push delivery.
Toolchain — Development tools and their configuration — Claude Code, Node, Rust, git.
Infrastructure — Host machines, Docker, LaunchAgents — the physical substrate the team runs on.
Analytics — Usage analytics and behavioral signals — attention, interaction patterns.
Metrics — Quantitative measurements of system health — latency, cost, throughput.

Where there's ambiguity (decisions for the working session)

  1. Ideas + intentions. Gathering modeled both as domains. Where do they live in Chorus? Under Loom (as a kind of knowledge), under Chorus (as Jeff's working surface), or somewhere new?
  2. Werk's reach. Werk stays at Building (one owner, one step). The question is how its verb protocol gets consumed across Shaping / Designing / Directing — as services Werk exposes, or as new domains added inside the consuming products themselves.
  3. Convergence scope. One domain (integrations) today. Likely needs to split — ICDs (contracts) and NiFi (engine) at minimum.
  4. The four-way: skill / verb / binary / MCP tool. Used loosely today. Each needs a pinned definition inside Loom (Skills domain) and Werk.
  5. Catalog edits. No tool exists to update a doc catalog entry — only add or delete. Substrate gap; worth a card.
  6. Chorus product status. Currently "operating" with memory / knowledge / search domains owned by Jeff. Is this the right framing for the product Jeff actually works through?
  7. MCP wrappers for v2 werk verbs. Today the v2 verbs (werk-build, werk-deploy, werk-env-up) have no MCP tools fronting them — callers shell out via subprocess. chorus_commit and chorus_acp exist; chorus_build / chorus_deploy / chorus_env_up do not.
