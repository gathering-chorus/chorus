Werk — Value-Stream Design — the Building-step instance of the api-driven value-stream template

Kade · 2026-07-12 · #3421 · INSTANCE of value-stream-template.html, peer to the athena / borg / clearing / convergence / loom instances. Registered in the live graph as value-stream-werk.

Werk is how a card becomes proven, running software. The value stream spans Building (pulled → merged source on origin/main) and Proving (shown working, accepted on Jeff's GO, deployed, torn down clean). Its engine is the one pipeline — chorus_werk → act → werk.yml → the twelve verb binaries — and its ledger is the spine: one trace from /pull to the running cdhash, so a green result can't lie. The product frame lives in werk-product-design.html; this doc is the flow.

1 · The werk-* cycle

Twelve verbs, one order, one trace. Each verb is a zero-external-dependency Rust binary with its own flock, typed refusal taxonomy, witness (ops/logs/werk-<verb>.jsonl), and spine emits — the verb IS the contract (ADR-032/ADR-037).

Recovery verbs off the main line: werk-sync repair|recover (canonical recovery), werk-unpull (the /pull inverse). The shared werk-teardown lib (#3431) is the one native teardown implementation under accept/unpull, with the two-tier squash-merge proof (#3014) and orphan propagation (#3498).

What crosses each seam: pull hands a clean worktree off origin/main on <role>/<card>; commit hands a rebased local sha; push hands the branch to origin + a chorus/push status; build hands signed werk-slot binaries (cdhash); test hands a BLOCKING verdict from the model-derived plan (#3634); review hands advisory floor findings; deploy-werk + env-up hand a live per-card variant; demo hands the witness (gates + gathers + test result + presented round) and STOPS for the human; on GO, merge hands a content-verified main sha, sync ffs canonical, deploy-canonical installs from main and verifies running==built, accept flips the board and tears down. Resume is first-class (#3632): a land that dies mid-run re-invokes and skips completed steps via idempotent dispositions instead of dying at them.

2 · The engine — chorus_werk → act → werk.yml

One trigger, one runner, one workflow file — the single generative mechanism of the stream:

  • chorus_werk (MCP) — the ONE pipeline trigger. Spawns act DETACHED (nothing held across the human wait), writes a run record (~/.chorus/werk-runs/<card>.json + .log); re-invoking POLLS: it attaches to a live run, or reads the WERK_EXIT sentinel and reports the terminal phase (#3458). go:true supplies the GO at invoke.
  • act — runs the workflow locally (self-hosted map). No orchestration logic beyond "run the steps in order, stop on first failure."
  • werk.yml — ONE job (#3499). Steps invoke verbs two ways today: through chorus-mcp-call.sh (commit/push/build/deploy/merge) or as direct binaries (test/review/demo/accept/sync) — a known asymmetry, named as pipeline debt in the product doc. The land steps gate on steps.demo.outputs.proven; the GO is patch-keyed on the witness, so a content-preserving rebase survives and a superseded patch re-demos.

The stop: werk-demo presents (cockpit: verbs · gates · tests · review · variant), notifies, and exits — a stopped pipeline, not a held connection. Jeff's go at the demo IS acceptance (GO=accept, #3311, DEC-048 authority in werk-accept).

3 · Werk's model & the domains it supports

The model layer is roles/kade/ontology/werk-domains.ttl (rendered in the product doc §6): 18 classes across the shared spine (VerbRun / TypedRefusal / Witness) and the four domains — pipelines · version-control · code · tests. The authoring rule is query-backwards: a class that cannot answer a query from real graph data gets cut.

The loop is now closed in one domain and the pattern for the rest: the tests domain serves 4,617 Test instances with covers edges from owl-api; werk-test derives its run plan from them and writes TestSuiteRun instances back (#3634) — declare → consume → record → reflect. Pipelines and version-control are graph-ready (rich witness events per verb run); their ingestion sweep is the open leg.

5 · Sequencing — spine first, legs derive

  1. Spine (done): the verb binaries + one-run pipeline + witnesses + resume dispositions — stable and observable, everything else relies on it.
  2. Model legs (in flight): tests domain closed the full loop (#3634); pipelines/version-control ingestion sweeps derive next, then owl-api generates their domain APIs the same way.
  3. Flow hardening (owned tails): deploy self-clobber health-check + auto-rollback and the fail-closed recovery carve-out (Silas); the yml invocation-shape unification + inline-bash retirement (product doc, pipeline debt); pure blast-radius test selection once the superset week proves out.
