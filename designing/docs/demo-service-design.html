Demo Service Design — the proving ceremony

2026-06-04 · Wren · #3116 · SUPERSEDES the #2864 cascade design (the 7-step / 5-hook orchestration this doc previously held). Aligns with Kade's #3211 flat-verb pipeline.

Demo is the proving ceremony, and nothing else. The artifact is already built and running in the role's werk slot (the prior verbs did that). Demo presents it, initiates the gates as subagents and routes each result to its owning role for review, fires the product/domain feedback gather verbatim, invokes one prover for the verdict — Jeff's eyes or a machine test — and records a single demo.verdict. It builds nothing, deploys nothing, and never asks a peer to bless the work by running its gate for it.

What demo does today (the hole). In werk-mcp.sh the "werk-demo boundary" is just steps 3–4: chorus_build → chorus_deploy target=werk → chorus_env_up. It stands up a running instance in the slot and stops — "waits-for-humans-to-browse." Nobody proves anything. The gate chain runs by manual cross-role go-run-your-gate nudges (the coordination tax). The verdict is hollow; accept rubber-stamps. This design makes demo finally do its one job.
What the prior #2864 design did (the over-correction). It encoded demo as an orchestrator: a 7-step chain (validate → AC-preflight → 5-role gate-chain → smoke → signal → show → accept-enforcement) enforced by 5 cascading hooks, with accept gated on a 4-event demo.*.completed chain. The 5-role gate-chain-by-comment is the agents-grading-agents stamp-mill. That model is retired here; the gate checks survive, relocated.

The flow

commit→ push→ build→ deploy → werk→ env-up ‖ PRESENT→ GATE→ FEEDBACK→ PROVE→ RECORD ‖ merge→ accept

Grey = other atomic verbs (not demo). The ‖ marks the demo boundary: it touches only the running werk instance, never main.

Precondition (NOT demo — prior verbs, unchanged)

werk-commit → werk-push → chorus_build → chorus_deploy target=werk → chorus_env_up — the change is built and running as an instance in the role's werk slot. "Prove in the slot, never touches main." Demo does none of this; it assumes a live instance to point at.

StepWhat demo doesOutput
1. PRESENT Point the prover at the running werk instance — the announced URL / endpoint / command. The artifact is already up; demo surfaces it. card.demo.started + the variant address
2. GATE The demoer initiates each of the 5 gates (product / code / quality / arch / ops) as an in-process subagent. The demoer runs the gate; the result is routed to the owning role for review. The role reviews the produced result (applies domain judgment) — it does not run the gate. No go-run-your-gate nudge anywhere. per-gate {gate, result, reviewer}
3. FEEDBACK GATHER Fire the product/domain feedback nudge to peers — verbatim (see below). Initial round + re-nudge unacked peers within the 10-min ACK window (#3100 AC#2). This is the value gather (the PM-lens questions), distinct from the gate review. peer acks
4. PROVE Exactly one prover, never a peer. Machine: run the card's Experience test (DEC-1674) against the deployed werk instance → pass/fail. Jeff: Jeff watches/uses the running instance; his input in the demo window is the verdict. Selection: machine-checkable assertion → machine; visual/UX/judgment → Jeff's eyes. pass / fail + prover
5. RECORD Emit one verdict event. That is demo's entire output — it does not call accept. demo.verdict 
demo.verdict {
  card,
  pass | fail,
  prover: "machine" | "jeff",
  gates:    [ { gate, result, reviewer } ],
  feedback: [ { role, acked } ]
}
Three separate concerns, never collapsed. Gates review quality. Feedback gather collects value judgment. The verdict proves it works — and is Jeff-or-machine, never peer-blessing. A green gate is not a verdict; a peer ack is not a verdict.

The three nudge classes

The kill is narrow. Only one nudge class dies.

NudgeFateWhy
Gate initiation — "go run your gate"DEADThe orchestration medium was the waste — it cost Jeff's coordination budget and was the agents-grading-agents relay. The demoer runs the gate as a subagent instead.
Gate result → owning roleKEPTThe good nudge: it carries a finished result to review, not work to do. Async, non-blocking.
Product/domain feedback gatherKEPT — verbatimThe value gather. Load-bearing PM artifact; exact framing matters (below).

The product/domain feedback gather — KEPT VERBATIM

Source today: platform/services/werk-demo/src/lib.rs → send_mcp_nudge() / signal(). Do not reword. The message to each peer is exactly:

[feedback #<card> — ACK REQUIRED]
From: <from>
Read the card. Read the code. Then reply.
(1) How does this impact your products?
(2) How does this impact you and your domain?
(3) Am I over-building or under-planning?
(4) Does this strengthen Loom, or just please the room?
Ack: substantive reply or blocked-on-X within 10 min.
Framing rationale (preserve — do not "improve"). Neutral: pointers + ask, no editorializing that biases the reply. Sender + ack-required up front; "read the card and the code" (the recipient forms their own read); the three questions as the actual ask. No "before /acp" pressure, no "narrow / clean / delivered" pre-framing — those just inherit the builder's satisfaction to the reviewer. A test pins this wording byte-for-byte so a refactor can't quietly soften it.
Who the users are (#3116) — contingent on demoer/demoee. The feedback nudge already fires to the peer roles; those recipients are the users. So Q2 addresses each recipient directly — "you and your domain" — and the contingent set (jeff / wren / silas / kade, minus the demoer) answers "who" automatically. Vague "your users" let a role answer without saying anything true. Loom — the discriminator — can't reply to a nudge and no peer can speak for it, so it gets its OWN question (Q4): does this strengthen the discriminator, or just please the room? That line is the direct counter to indiscriminate, locally-rational suboptimization — a change that pleases the room but does not feed Loom is a local win mistaken for progress.

The key transfer (load-bearing)

Demo proves the werk instance pre-merge. werk-merge then content-verifies werk ≡ main (squash). That content identity is what lets a pre-merge proof authorize the post-merge / prod accept without re-proving against main. Proving the slot — not main — is the whole point of the boundary: it's why demo "never touches main," and it sidesteps the merged≠live cost class.

Accept contract

werk-accept (atomic, finalize-only, #3175) gates finalize on one thing: demo.verdict = pass for the card_id. This replaces the prior 4-event demo.*.completed chain and the demo:preflight-pass evidence comment. DEC-048 (builder cannot self-accept a code card; Jeff exempt as human authority) stays enforced at werk-accept, unchanged.

Deleted / Kept / Out of scope

Deleted / moved out

  • Gate-NUDGE relay (nudge each role to run its gate) — deleted.
  • build / deploy / env-up inside demo — they are their own verbs, run before demo. Demo never builds or deploys. (Today's werk-demo binary calls werk-build + werk-deploy internally — that "act" step comes out.)
  • #2864 cascade — validate → preflight → gate-chain → smoke → show → accept-enforcement as 5 cascading hooks. Retired as the model.
  • accept-chain enforcement (accept_gate reading 4 demo.*.completed events) — replaced by the single demo.verdict=pass read.
  • smoke / stakes-brief / signal-mechanics — smoke is the machine prover now; the rest was ceremony theater. (The product/domain feedback nudge is NOT in this list — it is kept verbatim.)

Kept (do not delete)

  • The 5 gate checks — relocated as demoer-initiated subagents; results routed to the owning role for review.
  • The product/domain feedback gather — verbatim.
  • The owning role's participation — as reviewer of a produced result, not as runner.
  • DEC-048 at werk-accept.

Out of scope (don't conflate)

  • The deploy-from-main / prod-deploy gap (werk-mcp.sh steps 6–7 hard-stop) — separate blocked work, not demo's concern. Demo proves the werk instance; prod deploy is downstream.
  • Kade's #3211 flat-verb frame (no orchestrator, act=trigger-only, werk-acp retired) is correct and stays. This card only trims werk-demo from "build+deploy+gate-chain+wait" to "present+gate+feedback+prove+record" and moves accept-gating to werk-accept.

Open design decisions (confirm before build)

  1. Gate subagents — one per gate via the Agent tool inside the demo verb, run concurrently; demoer collects and routes each result to its role. Lean: yes.
  2. Gate-result → role mechanism — result-carrying review-nudge, async, non-blocking (carries a produced artifact, not a go-do ask) vs. the role's pulse/inbox for async pickup. Lean: result-carrying review-nudge.
  3. Machine test placement — inside the demo verb (one Experience check against the deployed instance), not a sibling verb. Lean: inside (one test invocation is the proving act; a sibling verb re-introduces sequencing).
  4. Jeff-prover signal — jeff.input.delivered in a tight window after card.demo.started; explicit accept-keystroke as a P2 hardening. Lean: windowed input now.
