Demo Service Design — unskippable hooks

2026-05-09 · Wren · #2864 · status: draft (post-reconciliation; SKILL.md + sequence-demo.html align; skill-lifecycle.html is stale)

The /demo skill is a 7-step proving gate (DEC-048). Today the skill is honored aspirationally — agents skip steps, post demo:preflight-pass comments without showing Jeff, and the comment-trail substitutes for the actual moment of shared attention. This design names each step as a structural hook (Rust in chorus-hooks or settings.json) that fires unskippably — including in auto-mode. If a hook should fire and didn't, the next transition refuses with a typed reason. Self-attestation in card comments is not a substitute path.

Design principle. The MCP / skill / hooks form a triangle. The MCP is the typed-refusal contract. The skill is the orchestrator the human invokes. The hooks are the structural enforcement that fires regardless of which surface the agent reaches for. Demo correctness lives in the hooks, not in the skill body or the MCP refusal taxonomy alone.

Hook decomposition

Five hooks, not one-per-step. The pattern in chorus-hooks today is "one hook per logical gate, multi-step preconditions bundled in the gate script." Smoke check, stakes brief, and signal are orchestration details — they're proof-of-side-effects, not gates with independent preconditions. They fold into demo_show's precondition set rather than firing as separate hooks. This collapses 8 → 5 with no loss of safety and a tighter cognitive surface (one fewer thing to hold in your head when something refuses).

Hooks fire as PreToolUse on the transition that depends on them — the validator blocks the next move, not the current one. Refusal taxonomy is shared across hook + MCP + skill so the failure surface is consistent.

demo_show signal — the load-bearing detail

What evidence proves Jeff actually watched? Three options surveyed:

OptionInstrumentable today?Source
(a) peer role's gemba tail event during demo windowNo — gemba is human-initiated, doesn't emit spine events; would require new observation.gemba.tail event—
(b) Bridge-side jeff.input.delivered spine event during demo windowYes — fires today from Clearing/server.ts on every Jeff messagejeff.input.delivered
(c) Explicit accept keystrokeNo — no infrastructure captures keystrokes/focus/session input—

Recommendation: use (b) as primary signal. Pass condition: jeff.input.delivered with timestamp within a window of card.demo.started (e.g. 30s–10min). Reject (a) — passive observability is the trap that produced today's hollow shape.

Concrete failure mode: Jeff starts a demo, role emits card.demo.started, Jeff steps away. Stale jeff.input.delivered events from earlier in the Clearing buffer get matched, demo_show passes, Jeff never watched. Mitigation: require timestamp within a tight window AFTER card.demo.started (not before), AND require at least one input matching the card_id or demo brief content. If even tighter trust is needed, fall back to (c) as a P2 add — capture explicit "accept" keystroke once Bridge supports it.

Auto-mode behavior

Hooks fire and enforce identically in auto-mode. Auto-mode is the default substrate today and amplifies the skip-rate because velocity is the default. The hooks must not branch on auto-mode — they fire, they refuse, they log spine events the same way regardless of harness mode. Refusal output is the same shape; the only difference is whether the agent has a chance to course-correct mid-loop or fails the transition outright. Both produce identical spine evidence for downstream readers.

Concretely:

  • No if (auto_mode) { skip } branches anywhere in the hook code.
  • Refusal in auto-mode = transition aborted + typed-refusal logged + agent receives the refusal string (per existing hook contract).
  • Refusal in interactive mode = same.
  • The spine record of the refusal is identical in both modes; observers can't distinguish.

Comment-shape decommission

Today demo:preflight-pass ac=N/N comments are the de-facto evidence anyone reads. Tomorrow they keep being written (Step 1.5 still posts one) but they are no longer trust-relevant:

  • accept_gate.rs stops reading demo:preflight-pass comments as evidence; reads demo.*.completed spine events for the card_id.
  • quality-health fitness function (#2862) updates its 28.7% gate-skip metric to read spine events instead of comment grep — the metric becomes structural, the number gets honest.
  • The card comment continues to post for human-readable card history. It documents what happened; it doesn't authorize transitions.

Open questions

  • demo_show timestamp window — recommended 30s–10min after card.demo.started. Tight enough to prevent stale buffer matches, loose enough that Jeff has room to actually watch. Tune from observation once shipped.
  • demo_show input-match — should the jeff.input.delivered match contain the card_id or some content from the demo brief, to prevent unrelated buffered messages from passing? Recommend yes for v1; relax if too strict in practice.
  • Smoke staleness window — how long after smoke-check.sh runs is the "passed" state honored? Recommend until-next-commit so re-running for unchanged code doesn't double-tax.
  • Multi-card demos — SKILL.md handles them with consolidated brief. Each card needs its own demo.*.completed chain, OR the chain fires once with multiple card_ids. Recommend per-card chain so the spine binding is clean.
