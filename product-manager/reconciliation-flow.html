Chorus Reconciliation Flow

How a role processes raw interaction data back into coherent context

Trigger: Jeff switches from Slack to Claude
Jeff finishes a Slack conversation. Switches to a terminal tab. Types /chorus or starts a new session (SessionStart hook fires automatically).

The role has been offline. It doesn't know what happened in Slack, what other roles did in their sessions, or whether its state files are stale.
↓
Phase 1: Ingest — Fetch what's new
1a

Index new Slack messages

Incremental fetch from all 7 channels since last watermark. ~3 seconds.

200 new messages from #all-gathering, #wren, #silas, #kade
1b

Index new Claude session transcripts

Scan ~/.claude/projects/ for new JSONL data. Parse user + assistant messages.

Kade had a 90-minute session. Silas had a 30-minute session.
1c

Index new artifacts

Check briefs/, ADRs, decisions.md, activity.md for modifications since last indexed.

2 new briefs, decisions.md modified, 1 new ADR.
↓
Phase 2: Classify — What type of signal is each message?
Raw messages are a mix of clarity and confusion. Classification separates signal types:

DECISION Explicit choice made. "Let's do SQLite, not Elasticsearch."
Signal: high. Action: update decisions.md if not already recorded.

DIRECTION Jeff gave direction without formal decision. "I want cockpit to be deterministic only."
Signal: high. Action: update understanding, may need to formalize as decision.

IDEATION Exploration, brainstorming, trying framings. "Maybe like Tim's Star Trek thing? No wait, not that..."
Signal: context. Action: note the exploration but don't act on abandoned branches. The last framing usually wins.

COMMITMENT Someone committed to do something. "I'll write the data contracts brief."
Signal: trackable. Action: add to nudge log or card.

CONFUSION Contradictory signals, unresolved tension. Jeff said X in Slack, but his session with Silas implies Y.
Signal: needs resolution. Action: flag for Jeff, don't assume either is correct.

CONTEXT Background, stories, preferences, emotional state. "I think on walks, in the garden..."
Signal: durable. Action: absorb into memory. These shape future decisions.

↓

Phase 4: Reconcile — Compare new data against my current state
This is where the intelligence lives. Compare what the index shows against what my state files say:
✓

Confirmed

My state matches the index. No action needed.

"My backlog shows cockpit as P1. Slack confirms Jeff approved it. ✓"
⚠

Stale

My state is outdated. The index shows something changed that I haven't captured.

"My decisions.md says DEC-028 is 'awaiting build'. But Kade's session shows he shipped it. → Update needed."
!

Conflicting

The index contains contradictory signals. Can't auto-resolve.

"Slack: Jeff said 'Silas leads cockpit architecture.' Session: Jeff said 'each role owns their vertical.' → DEC-029 vs DEC-030. Flag for Jeff."
+

New

Something happened that I have no state for. New work item, new direction, new constraint.

"Jeff described a 'context service' concept. No card exists. No decision recorded. → Create card, capture direction."
↓
Phase 5: Surface — Tell Jeff what matters
The output should be structured by action needed, not by data source:
⚠ Needs your attention:
• DEC-029 and DEC-030 may conflict — which governs Chorus work ownership?
• Kade committed to cockpit build but has no brief from Silas yet (48h)
✓ State updated:
• DEC-028 → marked shipped (was stale in my records)
• Context service → carded as Chorus #17
📋 Since you were gone:
• @team: 3-turn conversation about cockpit scope (you + 3 roles)
• Silas session: 45 min, shipped communication-flows.md
• 2 new briefs landed in Kade's inbox
↓
Phase 6: Resume work with coherent context
The role now operates with shared memory, not private memory. When the session ends, its transcript gets indexed into the shared memory for other roles to reconcile against later.

The loop: Jeff talks in Slack → index captures it → role reconciles on session start → role works with full context → session transcript gets indexed → next role reconciles against that.
