How This Was Built

Data Source

All 1,604 tasks from Vikunja API across 3 projects (Gathering: 1,542, Chorus: 48, Self: 14). Fetched via paginated REST calls.

Classification Method

  1. type: label (ground truth) — Cards with type:fix or type:swat → reactive. Cards with type:new or type:enhance → reflective. 316 cards have type labels.
  2. type:chore → title inspection — Chores are ambiguous. Classified by scanning title for reactive keywords (fix, broken, stale, missing, error, etc.). 55 cards.
  3. Untagged → title classification — 1,226 cards with no type label. Classified by title keyword matching against a broad reactive pattern set.
  4. Noise filter — Cards with titles matching test artifact patterns (BDD, CLI test, Test card) excluded from ratios. 48 cards.
  5. Date bucketing — Used done_at when available, else created date.

Known Limitations

  1. Title-based classification undercounts reactive by ~30%. Validated against 7-day ground truth: cards with type:fix often describe the solution ("Cloudflare tunnel monitoring") not the problem. Title pattern sees "monitoring" and calls it reflective.
  2. 1,226 of 1,604 cards (76%) have no type: label. These are older cards from before the type taxonomy was enforced. All classification for these cards is inferred.
  3. Chore is a judgment call. "Extract shared hook utilities" is reflective (planned refactor). "Remove all background nudge polling" is reactive (it was broken). Both are type:chore.

Confidence

The direction is reliable: reactive work is trending up. The exact percentages should be read as ±5-8% for recent data (type-labeled) and ±10-15% for older data (title-only).

The origin:reflective / origin:reactive labels added today will make future analysis exact — no inference needed.

What This Means

The system is demanding stabilization. Reactive work rose from 17% (Feb) to 51% (this week). Half the team's capacity is going to things that broke. The "stop starting, start finishing" directive is the right call — the framework sequence Silas is running targets the root causes (duplicated code, gate false positives, broken nudges) that spawn fix cards.
