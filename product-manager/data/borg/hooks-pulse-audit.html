Chorus Hooks & Pulse Logging Audit

Date: 2026-03-30 | Author: Wren (PM) | Source: Code review of chorus-hooks/src/

Related cards: #1854 (hook tracing, shipped), #1855 (pulse logging, carded), #1772 (hardening, shipped), #1715 (code review, Later)

Executive Summary

  1. 33 hook modules, 8,033 lines, 221 tests. The system is substantial and well-tested for its age. This is a real codebase, not a prototype.
  2. Pulse logging (#1854) is live but incomplete. Every hook entry/exit is logged to hooks.log with timestamp, tool, role, and decision. Missing: module name, duration_ms, reason on blocks, session_id.
  3. No circuit breakers. A broken hook (like the grep lobotomy) blocks all three roles with no auto-disable. This is the highest-risk gap.
  4. Subprocess calls in the hot path. 7 hooks shell out to board-ts, role-state, or find synchronously. One slow Vikunja response blocks the entire hooks pipeline.
  5. Session JSONL scanning is a recurring pattern. 5 hooks scan the session JSONL file for evidence (tests, pairs, demos, synthesis). Each re-reads the file independently. No caching, no shared reader.

2. Pulse Logging Assessment

Current State (hooks.log, shipped #1854)

The log_hook() function in main.rs logs every hook intercept:

FieldFormatExample
timestamp%H:%M:%S10:37:33
hook_type20-char paddedpre_tool_use
tool6-char paddedRead
role5-char paddedwren
decision6-char paddedallow / BLOCK / DENY / WARN
detail100-char truncatedFirst line of stderr/stdout

What's Working

  • Every hook entry is logged with enter phase showing tool input
  • Every hook exit is logged with decision (allow/BLOCK/DENY/WARN)
  • File: ~/Library/Logs/Gathering/hooks.log -- clean, human-readable, printable
  • Log rotation: append-only, no rotation configured (will grow unbounded)

What's Missing

Missing FieldWhy It MattersPriority
module nameWhen a hook blocks, the log shows "BLOCK" but not which hook module made the decision. Currently the detail field has a source prefix from log_return(), but not all paths use it. You can't tell if infra_guardrails or app_state_guard blocked.P1
duration_msNo way to detect slow hooks. demo_preflight shells out to board-ts + smoke-check.sh -- could take 5+ seconds. Without timing, performance regression is invisible.P1
reason on blocksThe detail field captures the first 100 chars of stderr/stdout, but blocking hooks put their message in stdout (permission_deny_json). The deny reason gets truncated or lost.P2
session_idCan't correlate hooks to specific sessions. Multiple roles running concurrently = interleaved log lines with no session context.P2
date in timestampOnly HH:MM:SS -- no date. Logs spanning midnight are ambiguous.P3
log rotationAppend-only file grows forever. No rotation, no size cap.P3

Two-Log Architecture

The system currently writes to two logs with different audiences:

LogFormatAudienceContent
hooks.logHuman-readable pipe-delimitedJeff (printed, marked up)Every hook entry/exit, decision, tool input preview
permission-prompts.logJSONLBorg (analysis)All tool calls with role, tool, detail. Error fingerprints in command-errors.log.

This is the right architecture. hooks.log is for Jeff, JSONL is for Borg. Don't merge them.

4b. Structural Concerns

ConcernDetailImpact
Subprocess calls in hot path 7 hooks shell out synchronously: accept_gate (board-ts, find), demo_preflight (board-ts x3, smoke-check.sh), demo_provenance (board-ts, chorus-log.sh), nudge_blast_radius (role-state), pair_enforcement (nudge.sh), nifi_discipline (board-ts), icd_write_gate (npx, curl x3, python3). These block the hooks pipeline. One slow subprocess blocks all hooks for all roles. No timeouts on subprocesses.
Session JSONL scanning (5 hooks) tdd_gate, demo_gate, pair_gate, memory_gate, autonomy_guard all independently read and scan the session JSONL file. No caching, no shared reader. Each scans the last 200-300 lines. 5x I/O on the same file per tool call (when all hooks fire). File can be large (100MB+).
Overlap between app_state_guard and infra_guardrails Both block docker compose, kill commands, terraform. Both fire on every Bash command. Duplicate regex evaluation. Confusion about which blocked. Maintenance burden. Consider consolidating.
Inconsistent response conventions Some hooks use permission_deny_json() (stdout), others use block_with_stderr() (stderr + exit_code 2). The log_return() function tries to detect decision from both stdout and stderr, but the logic differs. Pulse log decision field can be wrong. The "DENY" detection checks stdout for "deny" string, but some blocks use exit_code + stderr.

4c. What's Working Well

  • Architectural clarity. One binary, Unix socket, axum router. Clear dispatch from main.rs to hook modules. Easy to add new hooks.
  • Fire-and-forget for non-blocking hooks. tool_telemetry, observer, jdi_detector, handoff_logger all use tokio::spawn for async operations that shouldn't block the response.
  • Shared state management. AppState with Arc> for cross-hook coordination (search blocks, chorus search cooldown, session init markers).
  • Guard exemptions are well-designed. stop_on_error has a carefully curated exemption list (grep, diff, cargo test, board-ts). infra_guardrails exempts heredocs and git-in-heredoc.
  • The autonomy guard is remarkably sophisticated. Three trigger paths (PreTool, UserPrompt, Stop), 7 preference patterns, legitimate signal detection, response scanning with code block stripping. 18 tests.
  • chorus_log.rs (shim) is properly tested. 8 tests including roundtrip, alias resolution, special characters, error handling.

5. Three Architecture Lenses

Every hook operates at the intersection of three systems. A hook that's correct in one lens can be broken in another.

5a. Memory Architecture — What the Team Knows

Hooks are the system's enforced memory. Each encodes a decision, feedback, or pattern from Jeff's history. When a hook fires, it's the system remembering something.

Key insight: MEMORY.md has 200+ entries. Hooks encode ~15 of them. The rest depend on roles reading and remembering — which is the pattern that produced the Docker fiction.

5b. Attention Architecture — What the Team Sees

Hooks control what roles can see and do. A broken hook is a blindfold (lobotomy). A noisy hook is a distraction (Jeff overriding false positives). Both cost Jeff attention.

Attention PatternHooks InvolvedCost to Jeff
Blindfold — hook broken, roles can't seegrep lobotomy (2 days), any hook with 0 tests + no circuit breakerJeff becomes the monitoring system. Highest cost.
False positive — hook blocks correct actionnudge_blast_radius warning on --force, autonomy_guard blocking legitimate cautious responsesJeff has to explain why the hook is wrong. 6 touches per message.
Silent pass — hook allows incorrect actionNo hook validates nudge syntax (garbage delivered). No hook verifies Fuseki is responding.Jeff discovers the problem hours later. Trust erosion.
Correct block — hook prevents real mistakeinfra_guardrails blocking docker exec, write_scrubber blocking credentialsZero Jeff attention cost. System handles it.

Key insight: The ratio of Jeff-attention-required hooks to self-handling hooks determines system health. Today: ~12 hooks are self-handling, ~5 create false positives or silent passes, and any of the 21 untested hooks could become a blindfold. Jeff should need zero attention on hooks. He currently needs ~3-4 interventions per session.

5c. Urgency & Expectations — When Context Matters

Not every hook should fire the same way. Jeff's current interaction pattern (SWAT vs reflection vs demo) should influence hook behavior.

Key insight: The interaction pattern detection (interaction.pattern.detected spine events) exists but doesn't feed into hook behavior. Wren detects patterns, hooks ignore them. Wiring pattern → hook urgency would eliminate most false-positive attention cost.

6. Recommended Acceptance Criteria for Hooks Hardening

Card: "Hooks Circuit Breaker + Pulse Hardening"

Priority: P1 | Owner: Silas | Estimate: 2-3 sessions

AC1: Circuit Breaker (prevents next lobotomy)

  • Each hook module wrapped with a 2-second timeout (tokio::time::timeout)
  • Per-module failure counter: if a hook panics or times out 3x in 5 minutes, it auto-disables
  • Auto-disabled hooks log: CIRCUIT_BREAK | module_name | reason to hooks.log
  • Auto-disabled hooks emit Bridge notification
  • Hooks re-enable on service restart (stateless circuit breaker)
  • Test: mock a panicking hook, verify it circuit-breaks after 3 failures

AC2: Pulse Enrichment (hooks.log gets useful)

  • Every log line includes: YYYY-MM-DD HH:MM:SS.mmm (full timestamp with date and millis)
  • Every log line includes the hook module name that made the decision
  • Every log line includes duration_ms
  • BLOCK/DENY lines include the full reason (not truncated to 100 chars)
  • Add session_id to log lines (truncated to 8 chars for readability)
  • Add log rotation: max 10MB, rotate to hooks.log.1

AC3: Test Coverage Floor

  • Every hook module has at least 2 tests: one allow path, one block/warn path
  • Priority modules for new tests: session_init_gate, csc_guard, demo_preflight, accept_gate, sensitive_paths
  • One integration test: send a mock request through the full pre_tool_use pipeline and verify the correct hook fires

AC4: Subprocess Timeout

  • All subprocess calls (Command::new) wrapped with a 3-second timeout
  • Subprocess timeout logs: SUBPROCESS_TIMEOUT | module | command
  • Test: verify hook returns allow (not hang) when subprocess times out

Card: "Session JSONL Cache" (future, P2)

  • Shared session JSONL reader in AppState: read once per prompt cycle, cache results
  • tdd_gate, demo_gate, pair_gate, memory_gate, autonomy_guard all use the cached reader
  • Cache invalidates on new tool_response (not on every check)

Card: "Guard Consolidation" (future, P3)

  • Merge app_state_guard into infra_guardrails (they overlap on docker, kill, terraform)
  • Standardize response conventions: all blocks use exit_code 2 + stderr message + stdout deny JSON
  • Single log_decision() function that all hooks call, guaranteeing consistent pulse format
