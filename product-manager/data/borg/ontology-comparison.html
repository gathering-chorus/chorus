Which modeling standard should back Borg's system ontology? Evaluated three candidates against Borg's requirements: topology, health, resources, dependencies, current-vs-target state.

Comparison Matrix

Recommendation: Hybrid Stack

None alone covers Borg's surface. The natural combination:
  1. SOSA/SSN for the observation layer — already OWL, fits Fuseki, models "what did we measure about what, when." Low adoption cost. Start here.
  2. TOSCA-inspired topology vocabulary — borrow node/relationship type system for service dependencies. Define lightweight OWL classes (borg:Service, borg:HostedOn, borg:DependsOn), not the full 400-page spec.
  3. K8s spec/status pattern — every managed resource gets borg:desiredState and borg:observedState. This is the reconciliation loop.
What this means for the patent: The hybrid stack aligns with US9552400B2 — RDF/OWL + SPARQL + workflow gates. SOSA observations flow through SPARQL queries that trigger workflow state transitions. The topology graph is the "ontology-driven workflow" the patent describes.

Borg Questions the Ontology Must Answer

Refactor or Rewrite?

Given a service's dependency graph, health history, and resource trend — is incremental change viable or does the blast radius force a rewrite?

What breaks if X changes?

Traverse borg:DependsOn edges from a service to enumerate downstream impact. The topology graph IS the blast radius calculator.

Current vs target drift

SPARQL query: select resources where borg:observedState ≠ borg:desiredState. That's the work backlog.

Where is capacity being consumed?

SOSA observations of resource properties (disk, memory, ports) aggregated by host. The Activity Monitor view Jeff wants.
