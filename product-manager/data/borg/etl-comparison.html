Recommendation: NiFi

NiFi wins on the three things that matter most:
  1. Provenance is architectural. Every FlowFile carries its full history — creation, transformation, routing, delivery. This IS the ICD verification layer. Jeff's Staples pattern: the ARB approved architecture, the CIA team made it trustworthy. NiFi's provenance is that trust layer.
  2. It already works. Installed on the 32GB Bedroom Mac. Prototype flow succeeded (#1662). The REST API means Chorus can orchestrate flows programmatically.
  3. ICD validation fits structurally. Source → ExecuteScript (ICD validation) → Transform → InvokeHTTP (Fuseki). Failed validation routes to a failure queue with full provenance. You can see it, query it, prove it.

What NiFi Costs

  • Memory: 1-2GB on the 32GB machine — acceptable, but don't run it on Library (16GB, 88% disk)
  • Learning curve: Processor model is medium complexity, but visual. Jeff's EIP background (Hohpe) maps directly to NiFi concepts.
  • Java ecosystem: JVM maintenance, version management. Homebrew handles this on Mac.

Why Not the Others

Camel — reference EIP implementation, Jeff would recognize every pattern from the Hohpe book. But it's code-first with no built-in provenance. You'd build governance tracking yourself. Right choice if this were a Java shop building reusable integration libraries. It's not.
Singer/Meltano — lightest footprint, Python native. But no community taps for Apple Photos, Google Takeout, or iPhone backups, so you're writing custom Python anyway. At that point the framework constrains your transform model without giving provenance or visualization. No win.
Custom scripts — total flexibility, zero structure. Every pipeline is bespoke. When it works, it works. When it fails, you're reading bash at 10pm with no provenance trail. This is what we have and why we're here.

ICD Pipeline Pattern in NiFi

For any data source, the NiFi flow follows:
  1. Source Processor — GetFile, ExecuteSQL, or ListFile depending on source type
  2. ExecuteScript (ICD validation) — calls ICD validator, routes pass/fail
  3. TransformJSON/ExecuteScript — map source fields to canonical RDF
  4. InvokeHTTP — POST to Fuseki Graph Store Protocol
  5. Failure queue — failed records with full provenance for debugging
Every record that reaches Fuseki has a provenance chain proving it passed ICD validation. No record skips the gate.
