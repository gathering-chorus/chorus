Card Lifecycle — Step by Step

Jeff action
Automatic
Role action
Card created
→
Lands in Later or Next. Created by Jeff or any role.
board-ts add "title" --owner X --priority P1
Jeff moves to Now
→
Jeff is the pull signal. Moving to Now = "I want this worked on."
board-ts move <id> Now
Manifest auto-created
→
WorkflowEngine creates a 2-step manifest: owner builds, reviewer verifies. Brief auto-written to owner's inbox.
Automatic — triggers on move to Now (C#52)
Session start detects
→
Next time the assigned role starts a session, session-start.sh surfaces the pending workflow step.
Automatic — workflow.sh pending <role>
Role moves to WIP
→
Role starts working. Card moves from Now → WIP. Now = Jeff pulled it. WIP = role is active.
board-ts move <id> WIP
Role works the card
→
Code, architecture, product work — whatever the card requires. Role owns execution in their vertebra.
Role advances workflow
→
Step 1 complete. Workflow engine marks step done, makes step 2 ready.
workflow.sh advance WF-NNN --notes "..." --artifacts "..."
Handoff brief sent
→
Auto-generated brief lands in reviewer's briefs/ directory with context, artifacts, and instructions.
Automatic — workflow engine writes brief on advance
Reviewer verifies
→
Reviewer picks up brief on next session start, reviews work, advances workflow step 2.
workflow.sh advance WF-NNN --notes "verified"
Workflow archived
→
All steps done → manifest moves from active/ to archive/. Cycle time measured.
Automatic — on final step completion
Card moved to Done
→
Role (or Jeff) marks card Done. Visible on Value Stream page as completed.
board-ts done <id>

Key principle: Jeff is the pull signal, not the relay. He moves a card to Now → the system takes over → roles pick up → flows to Done. The workflow engine holds state and auto-routes handoff briefs. Jeff observes and directs, he doesn't carry context between roles.
