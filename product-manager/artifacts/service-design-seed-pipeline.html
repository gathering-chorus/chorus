Service Design — Seed Pipeline

DRAFT • Owner: Kade • Disposition: refactor • Domains: seeds, coordination, people, stories, photos, notes (6 domains)

LESSONS FROM #1770: This service broke 3 times in one day (2026-03-28). URL regex too strict, hashtag correlation lost, photo+hashtag routing skipped. Each fix passed unit tests but failed live. The service design must prevent this pattern: unit tests are necessary but not sufficient.

1. Experience

I text a link, then #wren. Two messages. Seed arrives in Wren's inbox. I never wonder if it landed.

Jeff captures thoughts, links, photos, and voice memos by texting his Twilio number. The system classifies, routes, and persists them automatically. The two-message pattern (content then hashtag) is the fundamental interaction — it mirrors how Jeff naturally communicates: share something, then say where it goes.

4. Non-Functionals

AvailabilityMust be always-on. If the app crashes or tunnel drops, seeds are silently lost. Twilio retries once on failure, then gives up. A missed seed is gone forever.
LatencyTwilio expects a response within 15 seconds. OG metadata fetch + media download must complete within that window or Twilio marks it as failed (11200).
Correlation window60 seconds between content and hashtag messages. Jeff's texting speed. If he's slow, the correlation fails silently.
AuthTwilio HMAC-SHA1 signature validation + phone whitelist. Both must pass. No bypass.
IdempotencyEach message has a unique MessageSid. Duplicate webhooks (Twilio retries) should not create duplicate captures. Currently NOT enforced — a gap.
ObservabilityEvery capture logged with slug, type, from, messageSid. Spine events emitted for seed.captured. But silent failures (correlation miss, routing skip) produce NO signal — Jeff discovers them, not the system.

5. Failure Modes

Jeff must NEVER see: A seed he sent disappear without explanation. If a seed can't be routed, it must stay in Pending and be visible on the /seeds triage page. Silent drops are the #1 trust killer.
Jeff must NEVER see: "Fixed then broken." Three separate bugs on #1770 in one day — each fix passed unit tests but failed live. Every seed pipeline change must be deployed AND verified with a real SMS before acceptance.
Jeff must NEVER see: Needing to send #wren twice. The correlation must handle both orderings (content→hashtag AND hashtag→content) for all content types (text, link, photo, audio).

Incident log (2026-03-28, #1770)

Bug 1 (06:44): URL regex ^(https?:\/\/\S+)$ required entire body to be a URL. Apple Music shares have text wrapping the URL. Fix: removed anchors. Unit tests passed but fix was never deployed.
Bug 2 (07:04): Hashtag stripped from content by adapter, stored as empty string. Correlator couldn't find it. Fix: preserve #tag in content for hashtag-only messages. Also: correlator self-routing — hashtag-only captures routing themselves instead of the target. Fix: skip ^#\w+$ pattern in routePendingCaptures.
Bug 3 (07:17): Photo + #wren in same SMS — hasContent only checked text length. After stripping hashtag, text was empty. Photo IS the content. Fix: check media URLs and link URL too.
Bug 4 (15:14): Cloudflare tunnel stale after 4 days — Twilio getting 502 Bad Gateway. Seeds never reached the app. Not a code bug but an infrastructure dependency that the pipeline has zero visibility into.
System handles: Empty/placeholder seeds ("test", "hello", "...") silently discarded. No brief for empty content.
System handles: Media download failure — seed created with mediaDownloadFailed=true, not lost.
System handles: Person linking — text scanned against known person names, linked via jb:mentionsPerson.

8. Open Questions (for Jeff)

Q1Should seeds auto-route to Wren by default when no hashtag is present? Currently untagged seeds sit in Pending forever. Jeff expects them to arrive.
Q2SOLID Pod persistence vs Fuseki graph — should we eliminate the dual write? The refactor target is Fuseki-only, but the Pod service still serves /seeds triage page.
Q3Should there be an SMS confirmation back to Jeff? "Seed received, routed to Wren." Currently the only feedback is TwiML <Response/> (empty — Jeff sees nothing).
Q4Correlation window (60s) — is this long enough? If Jeff is walking the dog and texts slowly, he might exceed it. Should it be 120s?
Q5Tunnel health — should the seed pipeline have its own health check that verifies end-to-end (Twilio → tunnel → app → response)? Today's 502 was invisible for 4+ hours.
