Domain — Memory

Memory • Team recall • Chorus product

Memory is how the team recalls what happened. Not search infrastructure — recall. "What did Jeff and Wren talk about this morning?" The data exists: 97K messages indexed from Claude sessions, Clearing transcripts, and artifacts. The experience doesn't: getting an answer requires knowing query syntax, field names, timestamp formats, and often writing code. Chorus captures everything. Chorus surfaces almost nothing.

Jeff depends on the engineering team. The engineering team depends on Memory to do their job. When Memory doesn't surface conversations, Jeff becomes the memory system — carrying context between roles manually.

Gaps — Why It Doesn't Work Today

Gap 1: Jeff's messages aren't first-class. Session JSONL stores Jeff's words as type: "user" content blocks, often mixed with system-reminder tags and skill loads. The indexer may skip them or bury them. When a role searches Chorus, Jeff's half of the conversation is missing or requires parsing raw JSONL.
Gap 2: No conversation view. Search returns 20 disconnected hits ranked by relevance. There's no way to say "show me the conversation between Jeff and Wren from 10am to 2pm." Results are fragments, not threads.
Gap 3: Time is UTC, not Boston. All timestamps stored in UTC or mixed formats. Jeff thinks in Boston time. Querying "this morning" requires manual UTC conversion.
Gap 4: No session-level query. The API searches messages, not sessions. There's no concept of "the conversation Jeff and Wren had" — only individual messages that happen to be from the same session file.
