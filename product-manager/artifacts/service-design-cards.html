Card Lifecycle

Wren creates card with Experience + AC → card.item.created Wren moves card to Now GATE: description with Experience + AC (hard block) → card.item.moved, workflow.manifest.created auto-assigns: builder + reviewer Builder pulls card to WIP GATE: AC section exists (hard block) GATE: Experience section exists (hard block) GATE: chunk tag exists (hard block) GATE: blast radius — code card has file paths (hard block) WARN: WIP overlap — same domain already in WIP → card.item.moved, card.pulled Builder builds, tests, completes → card.item.completed Builder demos to Jeff (via Wren) GATE: demo evidence exists (hard block on accept) GATE: self-accept blocked for code cards → card.demo.started Jeff accepts (/acp) → card.accepted, card.pm.briefed auto-unblocks downstream cards auto-archives workflow

Mandatory Fields by Transition

Principle: If a field is required at a gate, make it mandatory at the API level for that transition. Don't let cards accumulate without specs and rely on late enforcement.

Current gap: --quick bypasses type, chunk, and description. Cards enter unclassified, then hit hard blocks at WIP. Proposed: --quick only exempts description, not classification. All cards get type + chunk + domain + priority at creation.

BDD Scenarios

Card Creation

Feature: Card creation enforces mandatory fields

  Scenario: Create card with all required fields
    Given Wren runs cards add "Fix login bug" --owner silas --priority P2 --chunk ops --type fix --domain chorus
    Then the card is created in Later
    And a card.item.created event is emitted

  Scenario: Create card without type is rejected
    Given Wren runs cards add "Fix login bug" --owner silas --priority P2 --chunk ops --domain chorus
    Then the command exits with error
    And the error message says "Cards require a type"

  Scenario: Create card without chunk is rejected
    Given Wren runs cards add "Fix login bug" --owner silas --priority P2 --type fix --domain chorus
    Then the command exits with error
    And the error message says "Cards require a chunk tag"

  Scenario: Create card without domain is rejected
    Given Wren runs cards add "Fix login bug" --owner silas --priority P2 --chunk ops --type fix
    Then the command exits with error
    And the error message says "Cards require a domain"

  Scenario: Create card without priority is rejected
    Given Wren runs cards add "Fix login bug" --owner silas --chunk ops --type fix --domain chorus
    Then the command exits with error
    And the error message says "Cards require a priority"

  Scenario: Quick card still requires classification
    Given Wren runs cards add "Quick fix" -q --chunk ops --type fix --domain chorus --priority P2
    Then the card is created without description
    And type, chunk, domain, and priority are set

Move to Now

Feature: Now gate requires description with Experience and AC

  Scenario: Move card without description to Now is blocked
    Given card #100 exists with no description
    When Wren runs cards move 100 Now
    Then the command exits with error
    And the error says "Cards entering Now require Experience + AC in description"
    And a card.quality.blocked event is emitted

  Scenario: Move card with Experience + AC to Now succeeds
    Given card #100 has description with "## Experience" and "## AC" sections
    When Wren runs cards move 100 Now
    Then card #100 is in Now
    And a workflow.manifest.created event is emitted
    And builder and reviewer are auto-assigned

  Scenario: SWAT card bypasses description gate
    Given card #100 is type swat with no description
    When Wren runs cards move 100 Now
    Then card #100 is in Now

Pull to WIP

Feature: WIP gate enforces spec quality

  Scenario: Pull card without AC to WIP is blocked
    Given card #100 is in Now with description but no AC section
    When Silas runs cards move 100 WIP
    Then the command exits with error
    And the error says "Missing acceptance criteria"
    And a card.quality.blocked event is emitted

  Scenario: Pull card without Experience to WIP is blocked
    Given card #100 is in Now with AC but no Experience section
    When Kade runs cards move 100 WIP
    Then the command exits with error
    And the error says "Missing Experience section"

  Scenario: Pull code card without file paths to WIP is blocked
    Given card #100 is in Now with AC and Experience
    And card #100 title contains "Fix" (code indicator)
    And card #100 description has zero file paths
    When Kade runs cards move 100 WIP
    Then the command exits with error
    And the error says "Code card must list affected files"

  Scenario: Pull fully specced card to WIP succeeds
    Given card #100 is in Now with Experience, AC, chunk, and file paths
    When Kade runs cards move 100 WIP
    Then card #100 is in WIP
    And a card.pulled event is emitted

  Scenario: WIP overlap warns but does not block
    Given card #200 is in WIP with domain=chorus chunk=ops
    And card #100 is in Now with domain=chorus chunk=ops
    When Silas runs cards move 100 WIP
    Then card #100 is in WIP
    And a card.wip_overlap.detected event is emitted

Accept

Feature: Accept gate enforces demo and prevents self-accept

  Scenario: Accept without demo evidence is blocked
    Given card #100 is in Done
    And no demo brief or demo spine event exists for #100
    When Wren runs /acp for card #100
    Then the command is blocked
    And the error says "Demo evidence required"

  Scenario: Builder cannot self-accept code card
    Given card #100 is a code card built by Kade
    When Kade runs /acp for card #100
    Then the command is blocked
    And the error says "Builder cannot accept their own code card"

  Scenario: Jeff can always accept
    Given card #100 is in Done built by any role
    When Jeff runs /acp for card #100
    Then card #100 is accepted
    And a card.accepted event is emitted

  Scenario: Chore card skips demo gate
    Given card #100 is type chore in Done
    And no demo evidence exists
    When Wren runs /acp for card #100
    Then card #100 is accepted

  Scenario: Accept auto-unblocks downstream cards
    Given card #200 is blocked by card #100
    And card #100 is the only blocker of #200
    When card #100 is accepted
    Then card #200 is moved from Later to Next
    And a card.unblocked event is emitted for #200

Description Update

Feature: Cards CLI can update description (#1964)

  Scenario: Update card description with Experience + AC
    Given card #100 exists with no description
    When Wren runs cards update 100 --desc "## Experience\nJeff sees...\n\n## AC\n- [ ] Item one"
    Then card #100 description contains "## Experience"
    And card #100 description contains "## AC"

  Scenario: Description preserves markdown
    Given card #100 exists
    When Wren runs cards update 100 --desc with headings, checkboxes, and lists
    Then the description renders correctly in cards view
