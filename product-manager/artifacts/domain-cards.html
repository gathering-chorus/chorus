Domain — Cards

Operating • Work items from idea to acceptance • Wren is the primary actor

No work without a card. The cards CLI is the only interface to the board. No direct Vikunja API calls from roles. All mutations through cards — the CLI is the product surface. Jeff's target: two touches per card — direction + acceptance. Everything in between is the team's job.

Card Policy

  • Every piece of work has a card before it starts
  • Wren writes Experience + AC on every card — no card enters WIP without a spec
  • Cards require type, chunk, domain, and priority at creation
  • WIP limit: 3 total, 1 per role is healthy (DEC-051)
  • Equal priority → smallest first (DEC-049 WSJF)
  • Builder cannot self-accept code cards (DEC-048)
  • Demo before acceptance — Jeff sees every demo

Actor Flow

Jeff gives direction. Wren cards it. Builder builds it. Jeff accepts it.

JEFF (direction) Gives intent via conversation, seed, or triage Two touches: direction + acceptance WREN (spec — primary actor) Creates card with Experience + AC Sets type, chunk, domain, priority Sequences work by domain grouping Moves card to Now → workflow auto-assigns builder + reviewer BUILDER (Silas or Kade) Pulls card to WIP — gates enforce AC, Experience, chunk, blast radius Builds, tests, completes Demos to Jeff via Wren ACCEPT Demo evidence required (hard block) Self-accept blocked for code cards Jeff accepts → card.accepted event Auto-unblocks downstream cards

What Does Not Exist Yet

Gaps between the design and reality (2026-04-02):
  • --quick bypasses type, chunk, domain — cards enter system unclassified (#1966, Silas WIP)
  • Now gate is a warn, not a block — cards enter Now without Experience + AC (#1966)
  • Domain and priority are optional at creation — should be mandatory (#1966)
