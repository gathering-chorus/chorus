3. Conventions & Standards

Commit Messages

Format: <role>: <action> #<card> — <description>

Example: wren: acp #1839 — experience-driven card format, all open cards retrofitted

Git Workflow

  • All commits through git-queue.sh — serializes across roles with lockf
  • Direct git commit blocked by hook in team repo
  • Single branch (main) — no feature branches
  • Push after each commit

Naming

  • RDF types: Real nouns — Person, Photo, Track. Not aspirational prefixes (DEC: URI naming discipline)
  • Graph URIs: http://localhost:3000/pods/jeff/<domain>/
  • LaunchAgents: com.chorus.* or com.gathering.* (#1778 documents convention)
  • Card IDs: Monotonic integers from Vikunja

Language Rules (DEC-100)

ContextLanguage

Hooks, gates, performance-criticalRust
Operational scripts (glue)Bash (migrating to Rust/TS)

NeverPython (DEC-100: consolidate engines)

4. Test Strategy

TDD Discipline (DEC-1674)

Tests describe Jeff's experience, not implementation details. Every code card follows:

  1. Read AC → write test cases
  2. Tests verify what Jeff sees — UI behavior, API responses, delivery confirmation
  3. Run tests — must fail (red)
  4. Write minimum code to pass
  5. Run tests after each change
  6. All tests green before demo
GATE: TDD hook blocks cards done if no test runs detected in session.

Test Types by Card

Card TypeWhat to Test
Hook cardsInput classification, block/allow decisions, error messages
Bridge cardsMessage attribution, delivery, session filtering
Nudge cardsRole-pair delivery, WIP state detection, cross-domain warnings

10. Quality Guide

From engineer/quality-guide.md (10K). Key gates from code change to production:

  1. Card exists — no work without a card
  2. Experience section — blocks WIP entry
  3. Blast radius — files and domains documented
  4. Tests written first — TDD (DEC-1674)
  5. Tests pass — blocks done declaration
  6. Lint ceiling — ratchet, only moves down (#1282)
  7. Demo to Jeff — proving gate (DEC-048)
  8. Jeff accepts — /acp
