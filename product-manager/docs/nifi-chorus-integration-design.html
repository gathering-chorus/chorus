Engine vs Driver

NiFi handles the data plumbing. Chorus handles the intelligence. · 2026-03-23

"I don't want to solve a DSL problem for integration. I do want to for AI."
— Jeff, 2026-03-23

The Problem

Today we build custom harvest scripts for every domain — bash, Python, TypeScript, JXA. Each script is a one-off integration with no provenance, no retry, no flow visualization, no progress emission. When they break, Jeff finds out by noticing the system is slow or the data is wrong. 15+ custom scripts solving a problem that was solved 20 years ago.

Anti-pattern: Building custom integration when the engine already exists. Every hour spent writing harvest scripts is an hour not spent on Chorus — where the real IP is.

The Split

Buy: Data Integration (Engine)

  • Source connectors (file, HTTP, JDBC, SFTP)
  • Transform processors (JSON→TTL, EXIF, dedup)
  • Record-level provenance tracking
  • Visual flow design and monitoring
  • Retry, backpressure, error routing
  • Progress emission — built in, not bolted on
  • Batch and streaming modes

Solved problem. Use NiFi, Camel, or equivalent.

Build: AI Coordination (Driver)

  • Role-based task orchestration
  • ICD-governed pipeline design
  • Human-AI steering (Jeff → roles)
  • Quality validation against ICD
  • Demo/accept/prove workflow
  • Feedback → gates (memory → hooks)
  • Cross-role attention and observation

Unsolved problem. This IS Chorus.

The AI doesn't write integration code. It assembles flows from existing processors, validates results against the ICD, and reports to Jeff. The engine does the work. The driver makes the decisions.

Constraints

16GB RAM on Library Mac LaunchAgent-native, no Docker Fuseki as primary target ICD governs all flows Must run alongside 3 Claude sessions

NiFi's JVM footprint is the main risk. Default heap is 512MB–1GB. On a 16GB machine already running Fuseki (1.4GB JVM), Grafana, Loki, Prometheus, and three Claude sessions, NiFi needs to fit in ~1GB. Possible with tuning, but tight.

Alternative: Run NiFi on Bedroom Mac (32GB, M2 Pro). Data sources are mostly there anyway (Gathering NFS mount, music, photos). Processing happens on the storage machine, results load to Library Fuseki over the network. Separates compute from serving.

Migration Path

PhaseWhatOutcome
1. Spike Install NiFi on Bedroom. Prototype Apple Photos source → ICD validation → Fuseki load. Prove it works. Measure memory, throughput, provenance quality.
2. One domain Migrate Photos harvest pipeline to NiFi. Three sources, era-scoped merge, canonical rebuild. Replace 5+ scripts with one visual flow. Validate against 3-way comparison.
3. All domains Music, Documents, People, Social, Stories — each domain becomes a NiFi flow. Zero custom harvest scripts. Every pipeline visible, traceable, governed.
4. Chorus integration AI roles assemble and validate NiFi flows via NiFi REST API. ICD→flow generation. The driver steers the engine. Jeff sees the pipeline, roles manage it, NiFi executes it.

Prior Art

Staples ICD Pattern (2010–2015): Jeff's team used ICDs + ARB mapping tool + Anzo as semantic hub. The pattern was: define the contract, map the sources, validate before loading. NiFi replaces the custom middleware that executed those mappings. The ICD and the validation are Chorus artifacts.
Enterprise Integration Patterns (Hohpe/Woolf): Message routing, content-based routing, dead letter channels, wire tap. MuleSoft implemented these as a product. NiFi implements them as a flow engine. The patterns are the same — the question is which implementation fits a Mac mini.
