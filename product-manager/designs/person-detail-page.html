Design Decisions

  • Photo as avatar — if face cluster exists, use the best face crop. Otherwise initials.
  • Source badges — show where data comes from. Multiple badges = richer person.
  • Relationship state — Active (green), Dormant (orange), Passed Away (gray). Visual at a glance.
  • Stories are the depth — the stories section is the most important part. Properties give handles; stories give meaning.
  • Contact info masked — show last 4 digits of phone, masked email. Privacy default.
  • Family as links — family members link to their own Person pages. Navigate the family graph.
  • Empty sections hidden — don't show "Social Posts: none". Only render sections with data.
  • Photo grid — small thumbnails, click to expand. "+N more" for large collections.

Implementation Notes for Kade

  • Route: GET /people/:slug in people.handler.ts
  • View: views/person-detail.ejs (new)
  • Query service: extend people-query.service.ts with getPersonDetail(slug)
  • SPARQL: federated query across people, photos, stories, social graphs
  • Slug resolution: URI slug from urn:jb:people/{slug}
  • Link from collection page: change person name click from LinkedIn URL to /people/:slug
  • Responsive: single column on mobile, two columns on desktop
