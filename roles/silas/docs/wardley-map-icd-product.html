Wardley Map — ICD-Driven Integration Architecture

Product positioning analysis · Silas, 2026-03-18 · Using Wardley's evolution axis: Genesis → Custom-Built → Product → Commodity

1. The User Need

Enterprise CTO/VP Engineering says: "We need to retire our legacy integration platform (webMethods / Informatica / SSIS / DataStage) and migrate to a modern stack without losing business logic, breaking data flows, or spending 3 years and $20M on it."

This need is universal and urgent. Every enterprise with 10+ years of integration systems faces this. Gartner predicts 70% cost reduction via GenAI by 2027. Datafold calls it a "$10B opportunity." The need is visible — it's the budget line item that keeps getting deferred because the risk is too high.

5. Strategic Plays

5.1 Core IP (Genesis — Protect and Invest)

Three components are in Genesis — nobody else has them:
  1. ICD Layer — machine-readable, human-browsable integration contracts with severity tiers that drive both code generation and validation from one source of truth
  2. Quality Gates derived from ICD — the contract IS the test spec. Change the ICD, the gate updates automatically. No drift between spec and validation.
  3. Attention Architecture — formalized agent attention loops (Observer/Navigator/Driver) that maintain quality during long-running migration work
These are the moat. Everything else is either commodity or can be acquired/partnered.

5.3 Commodity Layer (Don't Build)

These are utilities — use existing solutions, don't differentiate here:

  • XML parsing, SPARQL engines, triple stores, cloud providers, serverless runtimes, AI model APIs
  • Partner with Datafold for data reconciliation (they're best in class)
  • Use open-source SHACL engines (Apache Jena)

  • Open-source the ICD HTML format specification (data-* attribute schema)
  • Open-source vendor parsers (SSIS, Informatica, webMethods)
  • Open-source the code generator framework
  • Build community around the ICD standard

Phase 2: Commercial Product

  • Hosted visual mapper (SaaS) with team collaboration
  • Enterprise knowledge graph (hosted, SPARQL API, blast radius analysis)
  • Attention architecture SDK (agent loops for migration projects)
  • Multi-target code generation (choose your stack)
  • Quality gate as CI/CD integration (GitHub Action / Azure Pipeline step)

Phase 3: Platform

7. Competitive Moat Summary

What we have that nobody else does:
  1. The contract IS the spec IS the test. One HTML document with data-* attributes drives visual display, code generation, SHACL shapes, and quality gates. Change one, all update.
  2. Severity tiers inferred from legacy code. NOT NULL → Violation. DEFAULT → Warning. Nullable → Info. No other tool maps legacy constraints to validation severity automatically.
  3. The ICD lives in a knowledge graph. Not a file. Not a database. A graph that supports blast radius queries, cross-domain impact analysis, and continuous evolution. Migration is not a project — it's an operating capability.
  4. Attention architecture for agents. Observer/Navigator/Driver loops that keep AI agents focused, quality-checked, and accountable during long-running migration work. Nobody else has formalized this.
  5. Platform-independent by design. The ICD is HTML. The graph is RDF. The targets are pluggable. No cloud vendor lock-in. No new lock-in replacing the old one.
What could kill us:
  • Prophecy pivots to ICD-first. They have $47M and a visual mapper. If they add severity tiers and quality gate generation, they're us with more funding.
  • Datafold adds contract-driven generation. They have the reconciliation layer we don't. If they add ICD → code generation, they close the loop from the other direction.
  • Cloud vendors build it in. Azure already has Legacy Modernization Agents (open source). If Microsoft adds ICD-driven generation to Azure Data Factory, the enterprise market follows.
  • The ICD format doesn't get adoption. If nobody else uses data-* attributes in HTML for integration contracts, it's a proprietary format with a pretty face.
