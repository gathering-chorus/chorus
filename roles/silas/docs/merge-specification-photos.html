Photos Domain — Era-Scoped Merge Specification

Card: #1643   Author: Silas   Date: 2026-03-24   Status: Draft

Purpose: Define the exact merge algorithm for every canonical photo field, per era, so Kade can implement it mechanically. This document is the contract between architecture (Silas) and implementation (Kade).

Depends on: #1642 (era-scoped provider authority — shipped). Consumed by: CrossSourceReconcilerService rebuild.

Principle: Richness before volume. The merge algorithm never counts records — it evaluates per-record field coverage within each era. A source with 7 fields per record beats one with 60K records at 3 fields each.

2. Merge Strategies

Every canonical field uses exactly one of these strategies:

StrategyBehaviorExample
era-precedence Take value from era's golden source. If null, fall through to supplementary, then volume-fill. Never take a value from a source where the field has a metadata ceiling. dateTaken: Apple in Eras 1–3, iPhone in Era 4
era-precedence + override Same as era-precedence, but a specific field overrides the default order for that era. Overrides are declared in icd:EraFieldOverride. location: Takeout preferred in Era 2, iPhone preferred in Era 3
merge-all Union values from all sources. Deduplicate by key. True if true in ANY source. albums, isFavorite, hasSourceRecord, people
exclusive Only one source has this field. Take it or leave it null. sceneLabels (Apple only), webViewLink (Drive only)
computed Derived from image content or merge output, not directly from any provider field. canonicalId (phash), servingPath, thumbnailPath

3. Field-by-Field Merge Rules

3.1 Identity Fields (Violation severity — reject record if missing)

FieldStrategyEra 1: Pre-digitalEra 2: CameraEra 3: iPhone PrimaryEra 4: ModernConflict Rule
canonicalId computed Perceptual hash (phash) from image content. Requires image file access — not available from metadata-only sources. Reconciler computes this. If two records produce the same phash, they are the same photo. Merge all source records into one canonical.
dateTaken era-precedence Apple EXIF Apple EXIF Apple EXIF iPhone EXIF Fallback: golden → supplementary → volume-fill.
NEVER use Google creationTime or Drive createdTime.
Gate: if date is within 7 days of harvest date, reject (Takeout date-fallback bug).
filename era-precedence Apple Apple Apple iPhone Camera original filename. Used as dedup join key across sources. Normalize case.
hasSourceRecord merge-all Accumulate all source IDs: apple:{ZUUID}, google:{filename}, drive:{fileId}, iphone:{ZUUID} Array of prefixed IDs. Every source record that contributed to this canonical gets listed.
servingPath computed Best available renderable path: thumbnailPath if exists, else bestResPath. Prefer local (Library) over remote (Bedroom). Computed after merge. Must resolve to an actual file on disk.
thumbnailPath computed Generated via sips on Bedroom. Path: public/thumbnails/photos/{YYYY-MM}/{uuid}.jpg Generate from highest-quality source image available.
dimensions era-precedence Apple Apple Apple iPhone Takeout ceiling: never has dimensions.
Apple or iPhone only. If neither has the record, dimensions is null.
mediaType era-precedence Apple Apple Apple iPhone Apple ZKIND + ZKINDSUBTYPE is most granular (photo, video, panorama, slo-mo, timelapse, live, screenshot). Fall back to MIME type from filename extension.

3.2 Location (era-precedence with field overrides)

EraPrimary GPS SourceFallbackRationale
Pre-digital No GPS possible — metadata ceiling. All sources 0% GPS. Pre-2006 cameras had no GPS hardware.
Camera (2006–2012) Takeout (override) Apple (44.8%) Takeout GPS fills Apple gaps. Union of both improves total coverage beyond either alone.
iPhone Primary (2013–2019) iPhone (override, 87.5%) Apple (55.0%) → Takeout (54.9%) iPhone has 87.5% GPS vs Apple 55%. Prefer iPhone when record exists in both sources.
Modern (2020+) iPhone (golden, 89.7%) Takeout (89.7%) → Apple (27.2%) iPhone is golden for everything in this era. Takeout GPS is equally good — true fallback.
Location merge rule: First non-null GPS from the era's location precedence chain. Filter Apple sentinel value -180.0 → null before comparison. Validate lat/lon range: lat ∈ [-90, 90], lon ∈ [-180, 180].

3.3 Enrichment Fields (Enrichment severity — accept without, track gap)

FieldStrategySource(s)Merge Rule
sceneLabels exclusive Apple (osxphotos enrichment) 1,538 unique labels, 96% coverage. Only available via osxphotos Python API reading Photos.sqlite. No other source provides this. Ceiling: only photos in Apple Photos library get labels.
faces exclusive Apple Face cluster IDs from ZDETECTEDFACE. Currently 48/50 clusters named. Ceiling in Era 4 (Modern): Mac library frozen — no face detection for post-2020 photos.
albums merge-all Apple + Takeout Union of album names from all sources. Deduplicate by slug (lowercase, strip whitespace). A photo can appear in albums from different platforms.
isFavorite merge-all All sources true if favorited in ANY source. Logical OR across all source records.
description exclusive Takeout User caption. Rare (<5% coverage) but valuable when present. Apple Photos doesn't expose descriptions in SQLite. Only Takeout JSON has this.
fileHash era-precedence Apple / iPhone ZORIGINALSTABLEHASH from Apple/iPhone SQLite. Not available from Takeout or Drive. Used for dedup verification (secondary to phash).
fileSize era-precedence Apple / iPhone From Apple/iPhone SQLite. Takeout ceiling: never has fileSize. Drive has it but Drive records were purged (phantom records).
bestResPath computed Merge output Highest-quality image file path. Prefer: iPhone originals (Bedroom) → Apple originals (Library) → Takeout files (Bedroom NFS). Include machine prefix: library: or bedroom:.
deviceType exclusive Takeout Camera device type from Takeout JSON. ~10% coverage. Apple SQLite doesn't expose camera model directly. Low priority.
webViewLink exclusive Drive Google Drive URL. Drive provider purged (phantom records) — this field will be null for most records. Retain if matched record has it.
people merge-all Apple + Takeout Union of people. Apple: face cluster → jb:Person URI via jb:faceClusterId. Takeout: people tags from JSON sidecar (18K/68K coverage). Deduplicate by person URI after name normalization.

4. Metadata Ceilings Per Era

A metadata ceiling means the field is impossible at that source in that era — not missing, impossible. The merge algorithm must never attempt to fill a ceiling field from a ceilinged source.

EraSourceCeilinged FieldsReason
Pre-digitalApplelocation, faces, sceneLabelsNo GPS hardware. Scanned imports don't trigger ML classification.
Takeoutlocation, dimensionsNo GPS hardware. Takeout JSON never includes dimensions.
CameraTakeoutdimensions, fileSizePersistent Takeout JSON format limitation.
iPhone PrimaryTakeoutdimensions, fileSizeSame format limitation.
ModernTakeoutdimensions, fileSizeSame format limitation.
ApplefacesMac library frozen since 2020 — no new face detections.

5. Provenance Tracing

Every canonical record carries full provenance — which source contributed which value.

Provenance model: Each canonical record includes a jb:mergeProvenance block that records:
  1. Era assignment: which era the photo belongs to (derived from dateTaken)
  2. Source records: all contributing source IDs (via hasSourceRecord array)
  3. Field attribution: for each non-null canonical field, which source provided the winning value
  4. Merge timestamp: when the canonical record was last rebuilt

5.1 Provenance Triple Pattern

For each canonical photo, emit provenance as RDF reification or named graph annotations:

# Canonical record
<urn:jb:photos/abc123> a jb:Photo ;
    jb:dateTaken "2018-06-15T14:30:00"^^xsd:dateTime ;
    jb:location "42.3601,-71.0589" ;
    jb:hasSourceRecord "apple:ABC-DEF-123" ;
    jb:hasSourceRecord "google:IMG_4521.jpg" ;
    jb:hasSourceRecord "iphone:XYZ-789" ;
    jb:mergeEra <icd/era/photos/iphone-primary> ;
    jb:mergeTimestamp "2026-03-24T16:00:00"^^xsd:dateTime .

# Field attribution (in provenance graph)
<urn:jb:photos/abc123/provenance> {
    <urn:jb:photos/abc123> jb:fieldSource [
        jb:field "dateTaken" ;

        jb:field "location" ;

        jb:sourceRecord "iphone:XYZ-789" ;
        jb:overrideApplied <icd/era-override/photos/iphone-primary/iphone-gps>

}

5.2 Provenance Requirements for Kade

  1. Every canonical photo MUST have jb:mergeEra pointing to its era URI
  2. Every canonical photo MUST have at least one jb:hasSourceRecord
  3. Field attribution is SHOULD (implement for dateTaken, location, and people first — these are the fields where source matters most)
  4. When a field override was applied, record jb:overrideApplied so we can audit era authority decisions

6. Merge Algorithm (Pseudocode)

function mergePhoto(sourceRecords: SourceRecord[]): CanonicalPhoto {
  // Step 1: Determine era from best available date
  const bestDate = resolveDateTaken(sourceRecords);
  const era = classifyEra(bestDate);

  // Step 2: Get era authority chain
  const chain = getAuthorityChain(era);
  // e.g., Era 3: [apple (golden), iphone (supplementary), takeout (volume-fill)]

  // Step 3: Get field overrides for this era
  const overrides = getFieldOverrides(era);
  // e.g., Era 3: { location: [iphone, apple, takeout] }

  // Step 4: Merge each field
  const canonical = {};

  // Identity fields — era-precedence
  canonical.dateTaken = eraPrecedence('dateTaken', sourceRecords, chain, era);
  canonical.filename  = eraPrecedence('filename', sourceRecords, chain, era);
  canonical.dimensions = eraPrecedence('dimensions', sourceRecords, chain, era);
  canonical.mediaType = eraPrecedence('mediaType', sourceRecords, chain, era);
  canonical.fileHash  = eraPrecedence('fileHash', sourceRecords, chain, era);
  canonical.fileSize  = eraPrecedence('fileSize', sourceRecords, chain, era);

  // Location — era-precedence with override
  const locationChain = overrides['location'] || chain;
  canonical.location = eraPrecedence('location', sourceRecords, locationChain, era);

  // Merge-all fields
  canonical.hasSourceRecord = mergeAll('hasSourceRecord', sourceRecords);
  canonical.albums = mergeAllDedupe('albums', sourceRecords, slugify);
  canonical.isFavorite = mergeAny('isFavorite', sourceRecords);
  canonical.people = mergeAllDedupe('people', sourceRecords, normalizePersonURI);

  // Exclusive fields — single source only
  canonical.sceneLabels = exclusive('sceneLabels', sourceRecords, 'apple');
  canonical.description = exclusive('description', sourceRecords, 'google');
  canonical.deviceType  = exclusive('deviceType', sourceRecords, 'google');
  canonical.webViewLink = exclusive('webViewLink', sourceRecords, 'drive');
  canonical.faces = exclusive('faces', sourceRecords, 'apple');

  // Computed fields — after all other fields resolved
  canonical.canonicalId = computePhash(bestImageFile(sourceRecords));
  canonical.servingPath = canonical.thumbnailPath || canonical.bestResPath;
  canonical.thumbnailPath = computeThumbnailPath(canonical.dateTaken, canonical.canonicalId);
  canonical.bestResPath = computeBestResPath(sourceRecords, chain);

  // Step 5: Record provenance
  canonical.mergeEra = era;
  canonical.mergeTimestamp = now();
  canonical.fieldProvenance = recordFieldSources(canonical, sourceRecords);

  return canonical;
}

function eraPrecedence(field, records, chain, era) {
  const ceilings = getCeilings(era);
  for (const provider of chain) {
    if (ceilings[provider]?.includes(field)) continue;  // skip ceilinged sources
    const record = records.find(r => r.provider === provider);
    if (record && record[field] != null) {
      // Date-specific gate: reject if within 7 days of harvest
      if (field === 'dateTaken' && isNearHarvestDate(record[field])) continue;
      return record[field];
    }
  }
  return null;
}

function classifyEra(dateTaken) {
  if (!dateTaken) return 'unknown';  // requires manual triage
  if (dateTaken < '2006-01-01') return 'pre-digital';
  if (dateTaken < '2013-01-01') return 'camera-era';
  if (dateTaken < '2020-01-01') return 'iphone-primary';
  return 'modern';
}

7. Dedup / Record Matching

Before merge can happen, source records must be matched across providers. The reconciler uses a tiered matching strategy:

TierJoin KeyConfidenceNotes
1Perceptual hash (phash)DefinitiveContent-addressable. Same image = same phash regardless of source. Requires file access.
2filename + dateTaken (±2 sec)HighCamera filenames are unique per session. Date tolerance handles timezone/rounding differences.
3filename onlyMediumSufficient when only one record per filename per source. Watch for IMG_NNNN collisions across years.
4dateTaken + dimensionsLowFallback for renamed files. Many false positives — use only to suggest manual review.
Known risk: Google Takeout date fallback bug. ~15% of Takeout records have photoTakenTime set to the upload date (2026-03-12) instead of the actual capture date. These records will fail Tier 2 matching. Use Tier 3 (filename only) as fallback, then flag for manual review.

9. Implementation Checklist for Kade

  1. Era classifier function — classifyEra(dateTaken) with 4 era boundaries. Add to CrossSourceReconcilerService.
  2. Authority chain loader — read era authority rules from ICD TTL (via Fuseki SPARQL or compile-time extraction). Return ordered provider list per era.
  3. Field override resolver — check icd:EraFieldOverride for location field per era. Override the chain for that field.
  4. Ceiling filter — before attempting to read a field from a source, check if that source has a metadata ceiling for that field in this era. Skip if ceilinged.
  5. Date rejection gate — dateTaken values within 7 days of any known harvest date are suspect (Takeout bug). Reject and fall through.
  6. Merge-all implementation — for albums, isFavorite, hasSourceRecord, people: accumulate from all sources, deduplicate.
  7. Provenance emission — every canonical photo gets jb:mergeEra + jb:hasSourceRecord[]. Field-level provenance is Phase 2.
  8. Face cluster boundary — do not merge face IDs across the iPhoto → Photos.app migration (2015-04). Treat as separate cluster namespaces.
  9. HEIC filename handling — normalize .HEIC/.heic alongside .JPG/.jpg in filename join keys.
  10. Unknown era triage — photos with no dateTaken from any source go to a manual review queue, not discarded.

10. Validation Criteria

The merge rebuild is correct when:

  1. Every canonical photo has a jb:mergeEra — no unclassified records
  2. No canonical field value comes from a ceilinged source
  3. Golden source values are preferred over supplementary in >95% of cases (remaining 5% are field overrides or null golden values)
  4. Total canonical count approximates: ~24K (Apple unique) + ~44K (Takeout-only) + ~47K (iPhone-only) minus ~15K overlap ≈ ~100K canonical photos
  5. Zero records with dateTaken within 7 days of harvest date survive into canonical
  6. Albums and favorites from all sources are preserved (merge-all count >= max single-source count)

11.2 Merge Strategy → NiFi Processor Mapping

Merge StrategyNiFi ProcessorConfiguration Approach
era-precedence JoltTransformJSON Per-era Jolt spec: shift operation that selects field from golden → supplementary → volume-fill. Four spec files, one per era. Era attribute routes to correct spec.
era-precedence + override JoltTransformJSON Same as above, but the Jolt spec for that field uses a different source order. E.g., Era 3 location spec: iphone.location → apple.location → takeout.location.
merge-all JoltTransformJSON modify-overwrite-beta operation: concatenate arrays from all sources, then deduplicate via ScriptedTransformRecord (Groovy) for slug normalization.
exclusive JoltTransformJSON Simple shift: take field from designated source only.
computed ExecuteScript (Python/Groovy) Phash requires image file access — ExecuteScript calls phash library. ServingPath/thumbnailPath are derived from other fields — JoltTransformJSON modify-default-beta.

11.4 Jolt Spec Example (Era 3: iPhone Primary)

// era-3-iphone-primary-merge.json
[
  {
    "operation": "shift",
    "spec": {
      // Identity fields — Apple is golden
      "apple": {
        "dateTaken":  "canonical.dateTaken",
        "filename":   "canonical.filename",
        "dimensions": "canonical.dimensions",
        "mediaType":  "canonical.mediaType",
        "fileHash":   "canonical.fileHash",
        "fileSize":   "canonical.fileSize"
      },
      // Location override — iPhone preferred (87.5% GPS)
      "iphone": {
        "location": "canonical.location"
      },
      // Fallback: if golden field was null, supplementary fills
      "iphone": {
        "dateTaken":  "canonical.dateTaken_fallback",
        "filename":   "canonical.filename_fallback",
        "dimensions": "canonical.dimensions_fallback"
      },
      // Exclusive fields
      "apple": {
        "sceneLabels": "canonical.sceneLabels",
        "faces":       "canonical.faces"
      },
      "google": {
        "description": "canonical.description",
        "deviceType":  "canonical.deviceType"
      }
    }
  },
  {
    "operation": "modify-default-beta",
    "spec": {
      "canonical": {
        // Fill nulls from fallback chain
        "dateTaken":  "${canonical.dateTaken_fallback}",
        "filename":   "${canonical.filename_fallback}",
        "dimensions": "${canonical.dimensions_fallback}",
        // Location fallback: Apple → Takeout
        "location":   "${apple.location}"
      }
    }
  }
]
For Kade: Each era gets its own Jolt spec file. The NiFi flow routes by era attribute to the correct spec. Merge-all fields (albums, isFavorite, people, hasSourceRecord) use a shared spec that runs after era-specific merge. The phash computation runs as a separate ExecuteScript processor since it requires image file access via SSH to Bedroom.
