The first two are true long-running daemons (port-bound). The last two are subprocess-invoked binaries that have launchd registrations for TCC permission stability — they aren't always running.

Per-role demo variants (when the role has an active werk)

Each role gets its OWN running variant of the long-running daemons, on its own port, running from its werk's dist. Activated by a marker file at $ROOT/.werk-<svc>/active; the role's session reads the marker via chorus-env-setup.sh and routes its calls to its own port instead of canonical.

Routing — how a session finds its services

chorus-env-setup.sh runs at session start and sets env vars based on CHORUS_ROLE + presence of marker files:

  • PATH: prepend $WERK_<ROLE>_BIN, then $CHORUS_BIN — CLI verbs resolve to the role slot first
  • CHORUS_MCP_PORT: if $ROOT/.werk-mcp/active exists → role's werk-mcp port; else CHORUS_MCP_PORT_CANONICAL (3341)
  • CHORUS_API_PORT (#3092 new): if $ROOT/.werk-api/active exists → role's werk-api port; else CHORUS_API_PORT_CANONICAL (3340)
  • .mcp.json interpolates $CHORUS_MCP_PORT into the MCP server URL

Result: when the role is mid-card with a deployed demo env, its session resolves to its OWN werk-api/werk-mcp. Other roles + Jeff's browser + all cron/scripts stay on canonical :3340 / :3341.

env build = the plist generation + bootstrap step at first /demo. env deploy = subsequent /demos that re-kickstart the variant against new dist. env destroy = the bootout + marker-removal + role-bin cleanup wired into chorus-werk remove (so /acp success and /unpull both trigger it).

What this delivers vs what it doesn't

CapabilityStatusWhy

Role can test its own code in demoYESRole session routes to its werk port; tests exercise the role's new code
Other roles unaffected by a role's demoYESThey keep using canonical or their own werk variant
Jeff's browser sees the role's demoNOBrowser still hits :3340 canonical; Jeff can curl the werk port to verify
State isolation (DB, Fuseki, Vikunja)NO — by designShared state is the intent (#3092 decision); a role's test against real data is the point
Per-event binaries (hooks/inject) isolatedNOClaude Code invokes them by absolute path; per-role wouldn't be exercised
Crash recovery / rollback for demoYESVariant is independent; if it crashes, role re-deploys; canonical never touched
