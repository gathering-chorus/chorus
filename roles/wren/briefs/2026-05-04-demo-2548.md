# Demo brief — #2548 (signing fix for nudge delivery)

**Card:** #2548 ad-hoc rebuild of chorus-inject silently revokes TCC AppleEvents grant, kills nudge delivery
**Builder:** Wren
**Date:** 2026-05-04
**Branch / sha:** wren/2548 / 35a52b53

## Why this matters

This is the bug Jeff has been telling the team about for weeks: nudges silently fail and the team doesn't notice until he names it. Reproduced today as: every `cargo build --release` of chorus-inject changes the binary's cdhash, macOS TCC's AppleEvents grant is bound to cdhash, so rebuilds silently revoke the grant. The settings panel keeps showing "ON" because TCC.db is correct — the System Events daemon's in-memory cache is what answers stale-deny. Effect: panel shows everything granted, runtime answers `osascript is not allowed to send keystrokes (1002)`, role doesn't get the prompt.

After this card lands, rebuilding either binary keeps the cdhash stable, the TCC grant follows the binary identity, and nudges deliver consistently.

## What changed

- `platform/scripts/build-signed.sh` upgraded from ad-hoc (`--sign -`) to keychain-identity signing (`Chorus Local Signing`). Stable identifiers: `com.chorus.inject`, `com.chorus.hook-shim`, `com.chorus.hooks`. The `chorus-hooks` shortcut now also signs the second binary `chorus-hooks` in the same crate.
- `platform/scripts/werk` deploy step switched from raw `cargo build --release --bin chorus-hook-shim` to `bash build-signed.sh chorus-hooks`. Dev mode keeps raw cargo (signing not load-bearing for non-deploy runs).
- `platform/scripts/shim-wrapper.sh` error-message hint updated to point at build-signed.sh.
- `platform/tests/chorus-inject-signed-stable.bats` (new) — 7 gates asserting both binaries get stable identifier, keychain authority (not ad-hoc), and cdhash equivalence across two consecutive build-signed runs.
- Memory: `feedback_target_release_is_deployed_code` — rebuild of these binaries IS redeployment; CI sensors must use a separate `CARGO_TARGET_DIR`.

## Risk / what's NOT fixed

This fixes the cdhash-rebuild-revocation class. It does NOT fix:
- Initial TCC grant. The first time the keychain identity signs a binary, macOS prompts for permission. That's a one-time setup. Subsequent rebuilds inherit the grant via stable cdhash.
- System Events daemon cache becoming stale for OTHER reasons (e.g., panel toggle, TCC.db update by other apps). The cache is still a known intermittence source. Workaround: `osascript -e 'tell application "System Events" to quit'` forces a fresh cache read.
- The 2,520-message backlog Silas surfaced on Kade's pending nudges (going back to 2026-04-09). Different surface; flagged for separate look.
