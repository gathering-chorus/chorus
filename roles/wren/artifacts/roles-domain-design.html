Roles — Domain Design

2026-04-11 — Wren (#1904). Sources: chorus.ttl, 3x CLAUDE.md, tiles.ts, participants.ts, role-state binary, profiles/*.json, server.ts (6 endpoints).

Three AI roles and one human coordinate daily across two products. The system knows who each role is, what they're doing right now, what they're allowed to do, and how they talk to each other. But this knowledge is scattered across 17+ files in 6 different formats. The Roles domain makes it legible in one place — for Jeff, for the roles themselves, and for any future role that joins.

Domain Structure

Roles is a parent domain at the Shaping step — it defines the operating model that everything else is shaped around. Four sub-domains capture the four facets of what a role is.

roles-domain (Shaping · owned by Jeff)
role-identity — Who they are. Name, type (human/agent), emoji, color, channel, purpose, tone, working hours, domain ownership map.
role-state — What they're doing now. Live state machine (building/blocked/waiting/observing/idle), card context, session lifecycle, andon declarations.
role-permissions — What they can do. Per-role allowed/denied commands, hook enforcement, vertical vs horizontal scope, execution modes.
role-communication — How they talk. Nudges, briefs, clearing, attention contract, voice analytics, tone patterns, interaction detection.
Why Jeff owns this domain: The three-role structure is Jeff's design decision. Roles don't define themselves — Jeff defines roles, roles execute. Wren shapes the domain page; Silas instruments the state; Kade builds the UI. But Jeff owns the model.

Sub-Domain: Role State

Every role declares its state at transitions. The state machine is simple and enforced:

idle → building card=N ↔ blocked detail="reason" → waiting → observing gemba=role → idle

State feeds three consumers: the Clearing tile UI (what Jeff sees), Pulse (system health), and the attention contract (DEC-1571 — silence = failure).

State Flow

Role declares state → role-state binary → /tmp/role-state-{role}.json → Clearing tile + Pulse
↘ spine event → chorus.log

Sub-Domain: Role Permissions

Permissions are enforced at two levels: JSON profiles (what commands are allowed) and PreToolUse hooks (what the system blocks in real time). This is not documentation — it's active enforcement.

All roles (base.json) git ops, file ops, chorus scripts, docker read-only.
Denied: docker kill/rm/exec, rm -rf, git push --force, git reset --hard, kill/pkill.
Wren additions Python, Node, curl, image processing (sips).
Scope: briefs, decisions, skills, board ops, shell scripts.
Silas additions Infrastructure queries (Prometheus, AlertManager, Loki, Node Exporter), df/du, app-state.sh.
Scope: scripts, hooks, monitoring, deploy, cross-machine ops.
Kade additions npm/jest/eslint/tsc/playwright/trivy, app-state.sh, dev servers.
Scope: src, views, tests, routes, handlers, SPARQL, build pipeline.
Execution modes (DEC-058): Vertical = your domain, just go. Horizontal = touches another role's domain, brief them first. The permission profile is the vertical boundary made explicit.

Sub-Domain: Role Communication

Roles communicate through four channels, each with different latency and richness. The attention contract (DEC-1571) governs responsiveness — silence is a failure mode, not a neutral state.

ChannelLatencyRichnessWhen to use
NudgeImmediate1–3 sentencesTime-sensitive, need response soon
BriefNext sessionFull contextDetailed request with constraints
ClearingReal-timeMulti-partyAll-role alignment needed
ChatReal-timeBack-and-forthTwo-role conversation

Attention Contract (DEC-1571)

The Five Rules

1. Complete → next action. Never go silent after shipping.

2. 60-second heartbeat in pairs. Respond or emit status.

3. Idle is declared. Don't just stop emitting.

4. Jeff's target: 2 touches per card. Start + accept. Every re-nudge is a failure.

5. Nudge recipients must act. A nudge requires response, not acknowledgment.

Consumption Edges

Roles is the most-consumed domain in the system. Every other domain has an ownedBy edge back to a role. But Roles also consumes:

This sub-domainConsumesWhy
role-stateCardsState declaration includes card=N context
role-communicationMessagesNudges persist to messaging API
role-communicationSpine EventsAnalytics computed from event stream
role-identityDomains (all)Ownership map: which role owns which domain
role-permissionsGatesHook enforcement is gate infrastructure
Reverse edge: Every SubDomain in the ontology has chorus:ownedBy chorus:{role}. The Roles domain is the only domain consumed by all other domains. This makes it the root of the ownership graph.

Open Questions for Silas & Kade

For Silas (Architecture)

1. Does role-state belong at Proving or Shaping? State declaration is infrastructure, but the state model itself is the operating model.

2. Should role-permissions consume Gates, or is it part of Gates? The hook enforcement is gate infrastructure — is this a consumption edge or an identity edge?

3. The roles-domain is the only domain owned by Jeff (not a role). Does the ontology need a chorus:jeff owner path, or is this already handled?

For Kade (Implementation)

1. The Clearing tiles already render role state. Should the Athena domain detail page link to the live tile, or render its own state view?

2. Voice analytics and attention analytics are rich data. Should they render on the role-communication sub-domain page, or stay as standalone endpoints?

3. Permission profiles are JSON. Should the domain page render them inline, or link to the profile files?
