How three AI roles and one human coordinate across products, domains, and contracts.

The Core Problem

Jeff's correction rate climbed from 7% to 28% over 8 weeks. Repeat instructions up 19x. Cards shipped code but not reference propagation. Roles reported things as "broken" without searching for moved files. Jeff became the monitoring system.

The root cause wasn't shallow reasoning. It was organizational: everything was framed as infrastructure, which made one role the bottleneck. There was no ownership model, no contracts between roles, and no way to see from the filesystem who owned what.

The street crew analogy. Jeff's neighborhood has construction crews that tear up the road, fill it with asphalt, then come back 1–5 days later to tear it up again. Each crew only sees their own work order. Nobody owns the whole road. The result: the patient gets opened and sewn up 3–5 times instead of once. That's what happened with cards #1308 → #1791 → #2328 → #1794. Four restructures, each leaving the next one a mess.

Two Axes of Responsibility

Every role has both vertical ownership (products they build and maintain) and horizontal responsibilities (concerns that cut across all products). The org design makes both visible.

Vertical: Products

  • You own the code, the tests, the alerts, the domain knowledge
  • One owner, one namespace, one place in the filesystem
  • You define what "healthy" means for your product
  • You instrument your own code — nobody writes your health checks for you

Horizontal: Contracts

  • Cross-cutting concerns that every product must satisfy
  • Observability, quality, code standards, design coherence
  • Product conforms to the contract; consumer reads from it
  • No coupling — the contract is the interface

Two directories per product per owner: products/ (code) and domains/ (knowledge). Both live under the owning role. Everything about a product in one tree.

Horizontal Contracts

Each contract follows the same pattern: product conforms, horizontal owner consumes. The contract is the interface. No coupling in either direction.

Observability Contract

Product owner → emits → spine events (domain.noun.verb_past) → consumed by → Silas

Product owner defines health and instruments their code. Silas provides the observability platform (Grafana, Loki, spine). Silas consumes events for cross-product correlation. Product-specific alerts live in the product directory, not in Silas's domain.

Quality Contract

Product owner → declares → AC + tests + demo → validated by → Wren

Product owner defines acceptance criteria, writes tests (TDD), runs the demo ceremony. Wren validates AC coverage. Silas runs gate hooks. The quality definition belongs to the product, not to the validator.

Each product emits to the spine event contract. Silas is the only role that reads all of them.

The fan-in position is Silas's unique value: cross-product pattern detection. Cards healthy + Clearing healthy + nudge delivery failing = a system problem only visible from this vantage point. But the fan-in role consumes. It doesn't produce each product's monitoring. That distinction prevents the bottleneck.

Operating Rules

Pair as default, solo as exception. The 28% correction rate proves solo builds drop cross-cutting concerns. Pair on anything that spans products or repos. Solo only for vertical work within your own product.
Done means propagation done. A card is not done when the code compiles. It's done when downstream references, docs, plists, configs, and tests all reflect the change. Don't sew up the patient between steps.
Products conform, consumers read. Every horizontal contract follows the same pattern. Product doesn't know how it's consumed. Consumer doesn't know product internals. The contract is the interface. No coupling.
Ownership is the namespace. If you can't read who owns it from the path, it's in the wrong place. No lookup tables, no metadata, no asking. The filesystem is the org chart.

What Changes

Before

  • Everything is "infrastructure"
  • Silas writes all monitoring
  • Solo builds are default
  • Cards close when code ships
  • Ownership requires a lookup
  • Each restructure opens and closes the road 3–5 times

After

  • Everything is a product made of code
  • Product owners instrument their own code
  • Pair is default
  • Cards close when propagation is done
  • Ownership is the path
  • One owner, one pass, one close
The transition risk. Changing the org model is itself a restructure. If we rename directories, move files, and reassign ownership all at once, we create the same 333-stale-reference problem we just fixed. Pilot on Cards first. Prove the pattern. Then roll it out product by product. Don't open the whole road.
