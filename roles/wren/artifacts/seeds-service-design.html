Seeds — Service Design

2026-04-08 — Wren (#1759). Sources: seed.handler.ts, seed-sparql.service.ts, sms-seed.adapter.ts, domain-context-seeds.md, seed-pipeline.feature.

Jeff sends an SMS at any hour — a thought, a link, a photo. The system receives it, classifies it, stores it, and routes it to the right place. No thought lost. No manual copy-paste. The seed arrives, and one prompt cycle later, the right role sees it and acts. Seeds is the intake nerve — the first touch between Jeff's mind and the system.

Cross-Product Boundary

Seeds is the only domain that spans both products. Gathering receives and stores; Chorus monitors and routes to roles. Chorus API (:3340) is canonical for team reads. Gathering (:3000) is the write path for Twilio webhooks. Two endpoints, two purposes, both stay.

Gathering (Kade owns code)Twilio webhook, SMS adapter, SPARQL persistence, triage UI, pipeline dashboard, routing to 12 destinations, media handling, link metadata.
Chorus (Wren owns domain)Intake policy, routing rules, monitoring (seed-probe, check-seeds, /cs), BDD specifications, spine events, alerting, role nudges.

Critical Dependency: Cloudflare Tunnel

Twilio delivers SMS via HTTPS webhook through a Cloudflare tunnel to localhost:3000. If the tunnel dies, seeds silently stop arriving. Jeff has no indication his SMS failed — Twilio retries for a while, then gives up. The daily seed-probe.sh tests this hop, but there's no real-time tunnel-down alert that fires immediately.

Data Flow

Jeff sends SMS → Twilio → POST /api/seed/sms :3000 → HMAC + Whitelist → SmsSeedAdapter → Fuseki (urn:jb:seeds/)
spine: seed.received → seed-probe (daily) → check-seeds (/cs)
Role triages at /seeds → POST /api/seed/:slug/route → Target domain (ideas, projects, etc.) → Status: Routed

Two-Message Pattern

How Jeff Sends Seeds

Jeff sends two SMS messages in sequence:

MessagePurposeExample
1. ContentThe thought, link, or photo"What if we mapped the garden light coverage by season"
2. HashtagRouting marker"#idea"

Rules: Hashtag-only messages do NOT create seeds. Content without hashtag defaults to wren. The adapter extracts the hashtag and correlates with the preceding content message via Twilio MessageSid.

Gaps

Gap: No Bridge confirmation when a seed arrives. Jeff has no proof his SMS landed until he opens the triage page. (Card #1937)
Gap: No role nudge on seed arrival. Seeds sit in pending until a role happens to check.
Gap: Hashtag-only messages still appear in triage view, despite not creating seeds. Visual noise.
Gap: No close-the-loop. Jeff never sees what a role did with his seed. The routing is invisible to the sender.
Gap: BDD step definitions are broken. 3 feature files exist (seed-pipeline, seed-media, seed-alert) but cucumber-js hangs on execution. Step defs need wiring after repo restructure (#1791). Verified by Silas 2026-04-08.
Gap: Auto-classifier is orphaned. seed-capture.service.ts has story/photo/note/discard classification built (250 lines) but nothing imports it. The handler goes straight to SPARQL. Classification is entirely manual via the triage UI. Verified by Kade 2026-04-08.
Gap: Cloudflare tunnel is a silent single point of failure. If it dies, Twilio can't reach the webhook and seeds stop arriving with no immediate alert. Daily probe catches it at 5:55am but not in real time.
Resolved: Chorus API (:3340) is canonical for team seed queries. Gathering (:3000) is the write path for Twilio webhooks. Two endpoints, two purposes, both stay. Confirmed by Silas 2026-04-08.
Resolved: Twilio signature validation + phone whitelist prevents unauthorized SMS. Dedup via MessageSid prevents double-processing.
Resolved: Daily seed-probe.sh tests all 5 hops end-to-end. LaunchAgent triggers at 5:55am.
