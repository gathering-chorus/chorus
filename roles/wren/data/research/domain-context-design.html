The Problem

Three AI roles (Wren/PM, Silas/Architect, Kade/Engineer) work on cards tagged with domains — photos, music, infrastructure, and more. Each domain has accumulated knowledge across multiple repos, sessions, and roles. But roles repeatedly start building without knowing what already exists.

Knowledge is scattered across the system

  • ICDs live in architect/
  • Tests live in engineer/
  • Decisions live in product-manager/
  • Constraints live in session transcripts, memory files, and people's heads

No single entry point tells a role: "here is everything you need to know about this domain before you touch it."

The Cost of Not Having This

40% data loss — Photos canonical rebuild

Kade built a photo merge pipeline without reading the ICD. Missed that Apple Photos uses UUID filenames internally. Matched across sources by filename, which is meaningless for Apple-sourced photos. Jeff caught the 40% data loss during verification. Root cause: Kade didn't know a constraint that was already documented in the ICD.

Duplicate test work — Nudge integration

Silas wrote new nudge integration tests from scratch instead of running the existing suite that Kade had shipped the day before. The tests already existed in engineer/tests/. Root cause: Silas didn't know what test files existed in the domain.

Pattern: knowledge exists but isn't found

These aren't knowledge gaps — they're discovery failures. The information existed. The role just didn't find it. A system that forces discovery before building eliminates this entire class of failure.

The Solution

One file per domain. Automatic injection. Hook-enforced.

Each domain gets a single domain-context-{name}.md file that serves as the entry point. When a role pulls a card tagged with domain:photos, the domain context file is automatically injected into their session. The role reads it before building — enforced by hook, not by discipline.

Domain Context File Structure

domain-context-{domain}.md
📜
1. ICD — Interface Control Documents
File paths to .ttl and .html specs that define the domain's data contracts. The source of truth for field mappings, source systems, and canonical schema.
✅
2. Tests — Existing Test Coverage
All test files related to this domain. File paths plus what each test covers. Prevents duplicate test work and ensures new work builds on existing coverage.
💾
3. Persistence — Where Data Lives
Fuseki graph URIs with record counts, NiFi flow names and which machine they run on, file system paths, scripts. The physical topology of the domain.
⚖
4. Key Decisions — Constraints from Above
Decision records (DEC-nnn) that constrain work in this domain. One-line summaries with enough context to avoid violating them.
⚠
5. Constraints — Hard-Won Knowledge
Non-obvious rules learned the hard way. The things that aren't in the ICD but will bite you if you don't know them. Domain-specific gotchas and edge cases.

Constraints

Photos Domain — Non-Obvious Rules

  • Apple Photos uses UUID filenames internally. Don't match across sources by filename alone. UUIDs are meaningless outside Apple's database.
  • iPhone reuses IMG_NNNN across years. Dedup by filename+date, not filename. IMG_0001.HEIC from 2021 and 2024 are different photos.
  • Source priority shifts by era. Apple primary pre-2020, iPhone primary 2020+, Takeout supplementary all eras.
  • Era boundaries matter. Pre-2006 (no digital), 2006–2012 (camera era), 2013–2019 (mixed), 2020+ (iPhone primary).
  • Cross-graph SPARQL joins timeout on full datasets. Use per-era queries to stay under timeout thresholds.

Hook Integration Flow

1 Role runs /pull 1644 — card has tag domain:photos
↓
2 Pull skill reads card metadata → finds domain:photos tag
↓
3 Hook injects domain-context-photos.md into session context
↓
4 Role reads domain context before building — enforced by ICD pre-read hook
↓
5 At close-out: if-touched check asks "did you update domain context?"

The hook is the enforcement mechanism. Roles don't need to remember to check — the system injects the context before they can start building. The close-out check ensures context stays current after work is done.

Maintenance Model

TriggerOwnerWhat Changes
Code ships that changes persistence, adds tests, or discovers constraints Kade Tests section, Persistence section, Constraints section
Decisions made that affect the domain Wren Key Decisions section
ICDs or infrastructure changes Silas ICD section, Persistence section

Stale domain context is worse than no domain context

A role that reads outdated constraints will make confident wrong decisions. The if-touched check at close-out is the primary defense: if you changed anything in the domain, you update the context file before your session ends.

Design Properties

✓ One file per domain, not per card
Domain knowledge accumulates across cards. The file is the single entry point regardless of which card you're working.
✓ Human-readable markdown, not config
Roles read it as prose. No parsing, no schema, no tooling dependency. A new team member could read it cold.
✓ Injected automatically, not fetched manually
The pull skill handles injection. Roles never need to remember "go check domain context." The system does it for them.
✓ Enforced by hook, not by discipline
The ICD pre-read hook blocks building until domain context is acknowledged. Discipline fails; hooks don't.
✓ Updated incrementally, not regenerated
Each builder adds what they learned. The file grows with the domain's knowledge, session by session.
✓ Close-out enforced freshness
The if-touched check at session close catches stale context before it can mislead the next role.
