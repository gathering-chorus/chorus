#1203 Evaluate Incubation Page

Wren Decision

What Incubation Actually Is

Not a pipeline stage. It's a chat-style rapid capture interface — type text, hit enter, it creates an Idea. Supports slash commands: /promote, /merge, /tag, /project. Mobile-optimized with auto-resize textarea. Admin-only.

Three Options

Option A: Remove from Nav It's the least-used page. Ideas page does the same thing with better UX (search, filters, collection chrome). The chat interface is clever but Jeff never mentioned using it. Remove route, archive template.
Risk: Loses the quick-capture and mixed Ideas+Projects view.
Option B: Repurpose as "Quick Capture" (Recommended) Rename to "Quick Capture" and position it as the fast-input mode for the Growing spoke — like a command palette for ideas. Move from nav item to a floating action button or keyboard shortcut accessible from any Growing page. The chat interface IS valuable — it's just misplaced as a separate nav destination.
Effort: Small (rename + nav restructure). Keep the handler, move the entry point.
Option C: Fold into Ideas Add a "Quick Add" textarea at the top of the Ideas list page. Merge the chat-style input into the collection view. Remove Incubation as a separate route.
Risk: Clutters the clean collection pattern Kade just shipped (#1228).

My Position

Option B — but only after #1202 ships. The pipeline overview from #1202 is the right home for quick capture. A "+" button on the pipeline view that opens the chat interface makes Incubation discoverable without being a nav sibling of the stages it feeds. For now: remove from nav, keep the route alive. Revisit placement when the pipeline view exists.

Kade Build

Current State: Two Pages, Overlapping Data

People Page (/people)

  • 2,256 contacts
  • Relationship filters: Family, Friend, Colleague, Team
  • Search by name
  • No professional context (title, company)
  • No source network info
  • Self-domain handler (SelfDomainPodService)
→

Merged /people

  • All contacts (reconciled count)
  • Relationship type filters
  • Source tabs: All / LinkedIn / Facebook
  • Professional context: title, company
  • Alpha navigation (A-Z)
  • Search: name + company + position
  • /network → 301 redirect to /people
←

Network Page (/network)

  • 2,248 contacts
  • No relationship type filters
  • Source filters: LinkedIn, Facebook, Both
  • Professional context (title, company, position)
  • Alpha navigation (A-Z)
  • PeoplePodService (harvested contacts)

Architecture Challenge: Two Data Sources

These aren't the same dataset rendered twice. They come from different services:

ServiceSelfDomainPodServicePeoplePodService

Count Discrepancy (8 people) Network filters to sourceNetwork !== 'unknown' && fullName. The 8 missing are likely manually-entered people (family/friends) without a LinkedIn/Facebook source. The merge must show ALL people, with source as "Manual" when no network is set.

Execution Plan

  1. Reconcile the 8-person delta — query both services, find people missing from Network, confirm they're manual entries. Small
  2. Merge handler — single route handler that queries both SelfDomainPodService (relationship type) and PeoplePodService (professional context), joins on slug/name. Medium
  3. Unified template — new collection-people-list.ejs with dual filter rows (relationship + source), alpha nav, and professional context column. Follows #1228 collection chrome pattern. Medium
