The Borg Assimilation Pattern

Legacy system knowledge extraction → ICD normalization → regeneration on new stack · Silas, 2026-03-18

"We are the Borg. Your technological distinctiveness will be added to our own."
The Borg don't care what species built the technology. They care about the function. The ICD captures function — not implementation, not vendor, not runtime. Once the knowledge is in the ICD, the vendor lock-in breaks.

1. The Problem: Knowledge Trapped in Vendor Lock-In

Every enterprise has legacy integration systems that represent years of accumulated business knowledge:

PlatformKnowledge FormatLock-In Mechanism
webMethodsFlowService XML, IS document types, triggers, pub/subProprietary runtime, commercial license, accumulated adapters
InformaticaMapplets, sessions, workflows, parameter filesPowerCenter server, repository metadata, enterprise licensing

SSIS.dtsx packages, data flows, control flowsSQL Server ecosystem, Windows-only execution
Stored ProceduresSQL, PL/SQL, T-SQL, business logic in DBDatabase vendor, accumulated tribal knowledge

The business logic — which fields map where, what transforms apply, what validation rules exist, what the canonical model looks like — is embedded in vendor-specific formats. Retiring the platform means extracting that knowledge, or losing it.

The traditional approach: Hire 150 people. Manually inspect every integration flow. Document what each one does in spreadsheets. Build the new system from the spreadsheets. Hope the spreadsheets are right. Duration: 2-3 years. Cost: tens of millions. Success rate: mixed.

2. The Assimilation Pattern

Phase 1: OBSERVE — Agent reads the legacy system

Input: Vendor-specific artifacts (XML, SQL, config files, metadata exports)
Output: Structured extraction of fields, transforms, routing, validation
Agent role: Read every integration flow. For each: identify source fields, target fields, transformation logic, conditional routing, error handling, and validation rules. Emit as structured JSON.

The agent reads what exists — not what the documentation says exists. This is the Borg scanning a ship: comprehensive, systematic, no assumptions about what matters.

webMethods IS Doc TypeField names, types, constraints, cardinality, nested structureConsumer cf-rows with data-datatype, data-cardinality

SQL Stored ProcedureParameters, SELECT columns, JOIN logic, WHERE filters, INSERT targetsProvider fields (SELECT) + consumer fields (INSERT)
DataWeave TransformInput schema, output schema, mapping expressionsField-rows with data-transform expressions

Phase 2: ASSIMILATE — Normalize into ICD

Input: Structured extraction from Phase 1
Output: SEMANTIC_MAPPER.html with all data-* attributes populated
Agent role: For each integration flow, emit a domain in the mapper with: provider ICDs (one per source system), consumer ICD (the canonical model), field-row mappings with transform types, severity tiers (derived from existing validation rules or NULL checks).

Phase 3: ADAPT — Human reviews and refines

Input: Agent-generated ICD
Output: Confirmed, refined ICD that passes lint
Human role: This is the Staples manual inspection step — now assisted by the agent's extraction. The human confirms: Are these the right fields? Are the severity tiers correct? Did the agent miss any business rules? Are there implicit transforms the code doesn't capture?
This is where domain knowledge matters. The agent reads code. The human knows intent. The agent might correctly extract that ORDER_DATE maps to canonicalOrder.createdAt with a date format transform. But only the human knows that ORDER_DATE is actually the ship date when the order type is "dropship" — a business rule that lives in tribal knowledge, not in the code.

The adapt phase uses the same tools as the build phase:

  • ICD lint — does every provider have Purpose, Risks, Non-Functionals, Interface, Diagrams?
  • Map integrity — does every provider data-maps-to have a matching consumer data-canonical?
  • Severity coverage — does every consumer have at least one Violation, Warning, Info?
  • Nudge review — architect reviews ICD, flags concerns, builder confirms before coding

Phase 4: REGENERATE — Generate on new stack

Input: Confirmed ICD (lint-passing)
Output: Pipeline code on target stack + SHACL shapes + quality gate
Agent role: Read the ICD data-* attributes. Emit code for the target platform. Run the quality gate against migrated data. Report pass/fail per severity tier.

The ICD is the invariant. The generated code is the variable. Change the target platform, regenerate. The contract stays the same.

5. Why This Works

The Borg are efficient because they separate knowledge from implementation. They don't preserve the Klingon ship — they extract what it can do and integrate that capability into the collective. The ship is irrelevant. The function persists.

The same principle applies to legacy retirement:

  • The webMethods runtime is the ship. It's irrelevant. What matters is the integration logic it executes.
  • The ICD is the extracted knowledge. Platform-independent, visual, machine-readable, human-reviewable.
  • The new stack is the new ship. TypeScript, Python, Java, whatever. Generated from the same ICD.
  • The quality gate is the assimilation verification. Did we capture everything? Does the new implementation match the old? Required = 100%.

7. The Commercial Engine Illusion

The SSIS proof reveals what commercial integration engines actually do at the data flow level:

What the Vendor SellsWhat Actually ExecutesWhat You Need
Visual drag-and-drop mapperXML config encoding column mappingsICD with data-* attributes
OLE DB Source adapterSELECT with parametersSQL query + connection config
OLE DB Destination adapterINSERT with column mappingSQL insert + column map from ICD
Data Flow pipelineRead → transform → writePython/Node script with streaming
Expression transformsString manipulation, casting, conditionalsGenerated functions from data-transform-type
Error handlingRedirect rows, log errorstry/catch + quality gate severity tiers
Runtime orchestrationExecute tasks in sequenceShell script, cron, or LaunchAgent
Monitoring dashboardExecution logs, row countsProgress to stderr + Grafana/Loki
The vendor lock-in was never the runtime. A $500K/year webMethods license, an SSIS installation, an Informatica PowerCenter — they all execute the same operations: read, transform, write, validate. The lock-in is the accumulated mapper decisions — which fields map where, what transforms apply, what validation rules exist. Once those decisions are in the ICD, the runtime is commodity.

What a commercial engine provides that you DO need:

  • Visual mapper for field-level decisions — SEMANTIC_MAPPER.html
  • Validation at ingest — SHACL shapes / quality gate
  • Monitoring and alerting — Grafana + Loki + incremental progress
  • Retry and error handling — shell wrapper with exit codes

What a commercial engine provides that you DON'T need:

  • Proprietary runtime — SQL is SQL. Python reads and writes it.
  • Vendor-specific adapters — every database has a driver
  • Enterprise licensing — the mapper decisions are the IP, not the engine
  • Proprietary config format — HTML with data-* is open, visual, version-controlled

8. Product Vision

A tool that:

  1. Points at a legacy system (webMethods, Informatica, MuleSoft, stored procedures, flat files) and reads every integration artifact
  2. Produces a visual ICD mapper with provider/consumer mappings, severity tiers, transform rules, and diagrams — all browsable in HTML
  3. Lets the architect refine the ICD interactively — add tribal knowledge, adjust severity tiers, confirm or correct agent inferences
  4. Generates pipeline code on any target stack from the confirmed ICD
  5. Runs quality gates on migrated data — severity-tiered, auditable, pass/fail
  6. Tracks retirement progress per domain — which integrations have been assimilated, which are pending, which are decommissioned

The attention architecture (observer/navigator/driver) manages the agents doing the work. The ICD discipline ensures the work is correct. The quality gate proves it.

Resistance is futile. The legacy system's knowledge will be assimilated. The question is whether it happens in 3 months or 3 years.
