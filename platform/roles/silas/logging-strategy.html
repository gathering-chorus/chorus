1. Diagnosis: What We Have vs What We Need

The system has strong infrastructure observability — Prometheus scrapes 23 targets, Loki ingests 18 log sources, Grafana serves 13 dashboards, and 40+ alert rules cover compute, network, and service health. This is the Splunk-at-Staples pattern applied to infrastructure.

What's missing is the integration layer — the seams where data moves between components. When Jeff texts a seed, when a nudge fires, when a card moves through the pipeline — these are the flows where failures are silent. The system tells you when a service is down but not when a piece of data dropped between services.

Strong
Infrastructure Layer
  • CPU, RAM, disk on both Macs (node-exporter)
  • 23 ICMP probes (router, mesh, devices)
  • HTTP probes for 7 services
  • Docker container status
  • Deploy SLA (build time, deploy time, staleness)
  • Fuseki query performance baselines
  • Disk space with APFS-aware thresholds
Partial
Application Layer
  • Custom JSON logger in app (good format, inconsistent coverage)
  • Correlation IDs on requests (middleware)
  • 9 of 65 handlers have NO logger instance
  • No HTTP request metrics exported (relying on blackbox probes)
  • No per-route latency histograms
  • Messaging service: startup log only, zero operational logging
Weak
Integration Layer
  • Seed pipeline: arrival logged, routing silent, person-linking silent on failure
  • Nudge pipeline: send logged, delivery confirmation missing, API persist failures swallowed
  • Context cache: generates markdown, no events on success or failure
  • Daily reviews: run and post to Bridge, no spine events
  • Card lifecycle: 15 events defined in schema, only 8 ever emitted
Dark
Data Lineage
  • No trace from "Jeff texted a link" to "seed on /seeds page"
  • No trace from "nudge sent" to "role acted on nudge"
  • No trace from "harvest started" to "data in Fuseki"
  • No delivery confirmation on Bridge messages
  • SPARQL update errors swallowed silently
  • OG metadata fetch failures invisible

The adapter had zero logging — the only signal was Jeff saying "where's my seed?" This is the pattern: the integration seam between Twilio and the graph has no instrumentation.

Messaging Service: The Dark Heart
The messaging tier handles 181K messages, all nudge persistence, all chat state, and role state declarations. It logs exactly one line: "Messaging service listening on http://localhost:3475". No request logging, no error logging, no queue depth, no delivery tracking. This is the most critical operational gap — it's the backbone of role-to-role communication and it's completely invisible.

The Pattern

This is the Staples integration architecture pattern applied to observability. At Staples, the ICD defined what data crosses each boundary and how to verify it. Here, the spine event schema is the ICD for operational events — but half the events are defined and never emitted. The schema is the blueprint; the implementation hasn't caught up.

The fix follows the same sequence: define (schema exists) → map (this document) → build (wire the events). We're at step 2. The strategy is not "add more logging" — it's "emit what we already defined, then instrument the seams we missed."

The Splunk lesson: instrument at the integration layer, share visibility. We did layer 1 (infrastructure). Now we need layer 2 (integration seams) and layer 3 (data lineage).
