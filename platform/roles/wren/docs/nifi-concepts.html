Apache NiFi

Core concepts and capabilities · Reference for Gathering/Chorus evaluation

Origin: Built by the NSA for intelligence data routing. Open-sourced in 2014. Now an Apache top-level project. Designed for the problem of "data comes from many places in many formats and needs to get somewhere reliably." The name stands for NiagaraFiles — the internal NSA project name.

The Mental Model

NiFi is a flow-based programming system. You don't write code — you place processors on a canvas and connect them with queues. Data flows through the processors like water through pipes. Each processor does one thing: read a file, parse JSON, filter records, write to a database. The connections between them are queued, buffered, and monitored.

Think of it as a visual circuit board for data. Each component has inputs and outputs. You wire them together. Data flows. If something breaks, the queue backs up visibly — you see it, not infer it from silence.

Core Concepts

FlowFile
The unit of data moving through the system. Every FlowFile has content (the actual data — a photo record, a JSON document) and attributes (metadata — filename, source, timestamp, any key-value pair you add).
Like: a record in a pipeline, but it carries its own metadata everywhere it goes.
Processor
A unit of work. Reads files, parses JSON, transforms records, queries databases, writes output. NiFi ships with 300+ built-in processors. You configure them — you don't code them.
Like: a step in a pipeline, but visual, configurable, and reusable across flows.
Connection
A queue between two processors. Has backpressure thresholds (max queue size, max age). When a queue fills up, the upstream processor pauses. No data lost, no silent overflow.
Like: a conveyor belt with a limit switch — if the next station is full, the belt stops.
Process Group
A container for related processors and connections. Encapsulates a sub-flow. Can be nested, reused, versioned. Each domain (photos, music, documents) would be its own process group.
Like: a module or namespace — keeps the canvas organized.
Controller Service
Shared infrastructure — database connections, SSL contexts, lookup tables. Configured once, used by many processors. Fuseki's SPARQL endpoint would be a controller service.
Like: a connection pool or shared config that processors reference.
Provenance
NiFi records every event for every FlowFile: created, modified, routed, dropped, sent. Queryable by time range, processor, FlowFile ID. Full lineage from source to destination.
Like: git blame for every record — where it came from, every step it went through, when.

Key Capabilities

  • Visual canvas Browser-based UI. Drag processors, draw connections, see live throughput (records/sec), queue depth, and errors. The flow IS the documentation.
  • 300+ processors File (local, SFTP, S3), HTTP (REST, SOAP), database (JDBC, MongoDB), message (Kafka, JMS), format (JSON, CSV, XML, Avro), scripting (Python, Groovy, JavaScript). Custom processors via Java API.
  • Record-level provenance Every FlowFile event stored and queryable. "Where did this photo record come from? What processors touched it? When?" Answerable in the UI or via API.
  • Backpressure Queues between processors have configurable thresholds. When full, upstream pauses. No lost data, no silent overflow. Visible in the UI as queue counters.
  • Error routing Every processor has a "failure" relationship. Failed records route to a separate path — log them, retry them, dead-letter them. Not lost, not silently dropped.
  • Scheduling Each processor has its own schedule: timer-driven (every N seconds), cron-driven, or event-driven (run when data arrives). Mix batch and streaming in one flow.
  • Flow versioning NiFi Registry stores flow versions. Roll back a flow to yesterday's version if a change breaks it. Like git for data pipelines.
  • REST API Full API for everything: create/update flows, start/stop processors, query provenance, check status. This is how Chorus would drive it — AI roles assemble flows via API.
  • Clustering Multiple NiFi nodes share the workload. Not needed for personal scale, but means the same architecture scales if the use case grows.
  • Expression Language Built-in expression language for attribute manipulation: ${filename:substringBefore('.')}, ${dateTaken:toDate('yyyy-MM-dd')}. Transforms without code.

How Chorus Would Drive NiFi

The key insight: NiFi has a complete REST API. Every operation you can do in the UI, you can do via HTTP. That means Chorus roles can create flows, configure processors, start/stop pipelines, and query provenance — all programmatically. The AI doesn't replace NiFi; it drives it.
