Chorus Protocol Stack

Multi-Agent Coordination Over Claude Code · Positioning Document · March 2026

Thesis: Claude Code's hook system (PreToolUse, PostToolUse, UserPromptSubmit) is an undocumented IPC layer. Chorus uses it to build persistent, role-based, governed multi-agent coordination — something no other framework does over Claude Code. The protocol stack is a general solution; Chorus is the first implementation.

1. The Problem Nobody Is Solving

Multi-agent AI systems exist. AutoGen, CrewAI, LangGraph, MetaGPT — they all orchestrate multiple AI agents. But they share three assumptions that limit them:

1. Disposable agents. Agents are spun up per task and discarded. No persistence, no accumulated context, no memory across sessions. Every interaction starts cold.
2. Programmatic orchestration. A Python/JS program defines the agent graph, routes messages, and manages state. The human writes code to coordinate agents, not direct them interactively.
3. No human in the loop. The human starts the swarm and reads the output. There's no model for a human directing agents in real time — steering, interrupting, observing, accepting or rejecting work mid-stream.

Chorus solves all three. Roles are persistent (37 days of accumulated context). Coordination is interactive (Jeff directs from Bridge, not from Python). And the human is the irreplaceable center — providing judgment, energy, and direction that the AI cannot.

2. The Protocol Stack

Chorus layers on top of Claude Code using its hook system as the coordination primitive. Five layers, from hardware to human:

L5
Human Direction
Jeff directs via Bridge, terminal, voice. Pattern detection (9 interaction modes). Gemba observation. Demo/accept workflow. The human layer that no framework automates because it can't be automated.
L4
Role Coordination
Named roles with vertical domains. Briefs as cross-domain messages. Nudge as instant delivery. Cards as work units. Attention contract (roles monitor each other). Pipeline sequencing. This is Chorus's core — the coordination protocol.
L3
Gate Enforcement
Compiled Rust hooks: ICD validation on write, demo pre-flight, accept gates, autonomy guard, search hierarchy, blast radius. PreToolUse blocks actions. PostToolUse validates results. Hooks are the enforcement layer — memory suggests, hooks enforce.
L2
Session Persistence
CLAUDE.md as role configuration (v73, evolves daily). Memory files for cross-session context. Skills as repeatable protocols. UserPromptSubmit hooks inject context on every prompt. The session knows who it is and what happened before.
L1
Claude Code Runtime
The AI execution engine. Tool use (Bash, Read, Write, Edit, Grep, Glob). Background tasks. Subagents. Cron scheduling. The raw capability layer that Chorus orchestrates but doesn't modify.
Key insight: L1 and L2 are Claude Code features (provided by Anthropic). L3 is the bridge — hooks that Anthropic provides but doesn't use for coordination. L4 and L5 are pure Chorus — the protocol that turns individual AI sessions into a team. The innovation is L3→L4: using hooks as IPC to enable role coordination.

3. Hooks as IPC

Claude Code's hook system was designed for safety gates (blocking dangerous commands, enforcing permissions). Chorus repurposes it as an inter-process communication layer:

HookIntended UseChorus Use
PreToolUse Block dangerous file writes ICD validation gate, autonomy guard, search hierarchy enforcement, blast radius check, demo pre-flight
PostToolUse Log tool usage ICD auto-reload + lint on write (Athena pattern), spine event emission, deploy verification, context injection
UserPromptSubmit Validate user input Team scan (check other roles' state), nudge consumption, brief routing, interaction pattern detection, context synchronization

This is the unsupported but working pattern. Anthropic didn't design hooks for multi-agent coordination. But the hook contract (fire on event, return allow/block/inject) is exactly what IPC needs: intercept, validate, route, inject. Chorus proves it works at scale — three roles, 1,660+ cards, 37 days of continuous operation.

6. The Anthropic Angle

Chorus demonstrates that Claude Code's hook system is more powerful than its current documentation suggests. The hooks are positioned as safety features. Chorus uses them as a coordination backbone.

What We'd Tell Anthropic

Your hooks are an IPC layer. We're running three Claude Code sessions as a coordinated team — persistent roles with named identities, vertical domain ownership, compiled enforcement gates, and a human director. The hook contract (PreToolUse, PostToolUse, UserPromptSubmit) is sufficient for multi-agent coordination. We've been running this daily for 37 days with 1,660+ cards shipped.

What We'd Ask For

RequestWhy
1. Background task progress API Background tasks are silent. A progress callback or polling API would solve the "chained to desk" problem without custom hooks. This is the most acute daily pain — three times in one session, the human sat watching a blank screen while a job ran silently.
2. Hook-to-hook messaging Today, roles communicate via file writes + nudge scripts (osascript injection). Native hook-to-hook messaging (Session A fires event → Session B receives it) would eliminate the fragile bridge and make cross-role coordination reliable.
3. Session identity API Roles derive identity from CLAUDE.md. A session identity primitive ("I am Silas, architect role") would make role awareness native instead of inferred from file paths and prompt content.
4. Cross-session state Memory files work but are fragile. A shared state layer across sessions (key-value, queryable) would replace file-based coordination with something atomic and consistent.
5. Hook performance metrics Hooks add latency. We can't measure it. Hook execution time as a reportable metric would help optimize the enforcement layer — currently we can't distinguish "Claude is thinking" from "the hook is slow."
