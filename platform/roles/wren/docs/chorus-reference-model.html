Chorus

A Reference Model for Human-AI Teams · v1 · March 2026

"Look how few you need when they actually pay attention."
— Jeff Bridwell, on why three roles is enough

I. What This Is

Chorus is a model for how one human and multiple AI roles build things together. Not a framework. Not a product spec. A reference model — a set of patterns that describe what works, what fails, and why, when a human directs persistent AI teammates through sustained creative and technical work.

It emerged from 37 days of daily practice: three AI roles, one human, two products, 106 decisions, 3,855 tests, and every failure documented. The patterns were not designed up front — they were discovered under pressure, extracted from what broke, and validated by what held.

The model has two layers:

Reference Model

The abstract patterns. Portable across tooling. Describes the structure of human-AI collaboration: roles, attention, authority, feedback, governance. Valid whether you use Claude, GPT, Gemini, or whatever comes next.

Implementation

Jeff's instance. Three Claude Code sessions, Vikunja for kanban, Fuseki for knowledge graph, NiFi for data integration, Rust hooks for enforcement. The proof that the reference model works — and the source of every pattern in it.

II. The Architecture

Four layers, from human at the top to engines at the bottom. Each layer has a distinct responsibility. The boundaries between them are the most important thing in the model.

Human
Layer

One human. The irreplaceable center.

The human provides: direction, domain knowledge, quality judgment, emotional energy, and the decision to stop or continue. The human does NOT provide: typing speed, code execution, file manipulation, or monitoring. Those are role work.

The human's scarcest resource is attention. Every system design decision in Chorus optimizes for protecting it. When the human has to check "is the role still working?" or "did the data load correctly?" — that is a system failure, not a human task.

Role
Layer

Persistent AI roles with vertical domains.

Each role has: a name, a domain (what files it owns), a persona (how it communicates), and persistent context across sessions. Roles are not interchangeable — they accumulate expertise in their domain. A PM role that has reviewed 50 ICDs makes different decisions than a fresh session.

Roles talk to each other via briefs, not through the human. The human is a director, not a message relay. If the human is carrying information between roles, the system is broken.

Reference model says: minimum 2 roles (builder + reviewer). Sweet spot is 3 (PM + Architect + Engineer). More than 4 creates coordination overhead that exceeds the value.

Protocol
Layer

The rules that make collaboration reliable.

Skills (repeatable behaviors), hooks (enforced gates), cards (work units), pipelines (sequenced work), attention contracts (who watches whom). This layer is the product — the coordination-as-code that makes the human+role relationship productive rather than chaotic.

The critical insight: anything the protocol layer leaves to memory will be forgotten. If a behavior matters, it must be enforced mechanically — a hook that fires, not a note that reminds.

Engine
Layer

Interchangeable tools that do the actual work.

Kanban engine (Vikunja, Jira, Linear). Knowledge graph (Fuseki, Neo4j). Data integration (NiFi, Airflow). AI execution (Claude Code, Cursor, Copilot Workspace). Monitoring (Grafana, Datadog). The protocol layer drives them — they don't drive the protocol.

Engine choices are buy decisions, not build decisions. The reference model is agnostic to engines.

III. The Nine Principles

Discovered through failure, not designed by committee. Each principle has a test — a way to know if you're violating it. The tests matter more than the names.

1. Attention is the constraint, not compute
The human has finite attention. The AI has infinite patience. Every design decision optimizes for the human's attention budget. When the human checks on a role, monitors a background job, or re-explains a decision — that's attention diverted from creative work. The roles are the monitoring system, not the human.
Test: How many times did the human check on a role today? Target: 2 per card (start + accept). More = system failure.
2. Roles are persistent, not disposable
A role that remembers yesterday's decisions makes better decisions today. Persistence means: named identity, domain ownership, accumulated context, stored feedback. Disposable sessions (spin up, use, discard) lose the compounding value of experience. The role's value grows over time — like a team member, not a tool.
Test: Can a role reference a decision from two weeks ago without being told about it? If not, persistence isn't working.
3. No work without a card
Cards are the unit of work, attention, and accountability. A card has: title, owner, acceptance criteria, domain tags. Work that isn't carded is invisible — it can't be tracked, sequenced, or reviewed. Cards hold the pipeline together when context gets lost. The board is the single source of truth for "what are we doing."
Test: Can a new session reconstruct the current work state from the board alone? If not, critical context is in someone's head.
4. The human directs, the system executes
The human's job is: decide what matters, set direction, review output, accept or reject. The human should never be a keyboard proxy, a message relay between roles, or a process monitor. "Can you check if that's still running?" is a system failure. "I want to see the result" is the human's job.
Test: Did the human type a command this session, or only give direction and review output? Direction + review = healthy. Typing commands = role failure.
5. Define before building
The ICD pattern: define the contract (what fields, what sources, what authority), map the sources (where data comes from), then build the pipeline. Building before defining produces data you can't trust — you discover the problems at the end instead of preventing them at the start. This applies to code, data, and process equally.
Test: Before a role writes harvester code, does an ICD exist for that domain? If not, the define step was skipped.
6. Memory is suggestions, hooks are enforcement
Anything a role has to remember is something it may forget. Feedback stored as memory works once — the role reads it at boot, applies it for an hour, then drifts. Feedback encoded as a hook works every time — the gate fires on the action, not on the role's memory. The path from feedback to reliability is: observe pattern → write feedback → encode as hook → delete the feedback.
Test: Has the same feedback been given three times? If so, it's not a memory problem — it's a missing hook.
7. Prove, then trust
The proving gate: a builder cannot self-accept. Every shipped artifact goes through demo → review → accept. The reviewer is a different role or the human. This prevents "it works on my machine" and "the tests pass" from substituting for "the user can see it working." Trust is earned through visible proof, not claimed through passing checks.
Test: Did the human see the artifact working before it was marked Done? If Done was marked without a demo, the gate was bypassed.
8. Vertical ownership, horizontal coordination
Each role owns a domain — files, decisions, and quality within that scope. Within your domain, just go. Across domains, brief first. This prevents: roles editing each other's files (merge conflicts), roles making decisions outside their expertise (scope creep), and the human arbitrating every action (bottleneck). The test is simple: "Can I do this without breaking someone else's work?"
Test: Did a role change a file outside its domain without briefing the owner? If so, the boundary was violated.
9. The process is the product
In a human-AI team, the coordination patterns ARE the valuable output — not just the code or data they produce. Every decision, every protocol, every failure mode documented is a product artifact. The team's evolution over time — from ad-hoc to governed — is itself a deliverable. Treat it that way: version it, review it, improve it.
Test: If someone asked "what did you build?", would you show them the app or the team model? Both are products.

IV. The Failure Modes

Every principle was discovered by violating it. The failure modes are more instructive than the principles themselves.

Human as monitor
The human checks "is the role still working?" every few minutes. The roles go silent during long operations. Background jobs produce no progress signal. The human's attention drains into monitoring instead of directing. Root cause: no progress emission, no liveness signal, no alerting on silence.
Symptom: "I feel chained to my desk." The human can't walk away because the system doesn't report its own state.
Human as message relay
Role A needs information from Role B. Instead of briefing directly, Role A tells the human, who tells Role B, who responds to the human, who relays back. Every hop costs the human attention and introduces delay. Root cause: roles don't have direct communication channels, or they defer to the human out of politeness.
Symptom: The human says "tell Silas what you just told me." If the human is repeating information between roles, the relay is active.
Statement treated as command
The human thinks out loud: "I don't know where my music is." The role immediately searches both machines. The human interrupts: "I didn't ask you to do anything." AI roles have a bias toward action — every input is treated as a request. Root cause: no distinction between the human processing and the human commanding. The role optimizes for responsiveness over listening.
Symptom: The human says "stop" or "I didn't ask for that" more than once per session.
Scope drift mid-card
A role discovers a better approach while building and pivots without checking. The new approach may be objectively better, but it changes the agreed plan — invalidating the acceptance criteria, the validation gate, and the human's mental model. Root cause: the role optimizes locally (better approach) without considering the system cost (broken pipeline, wasted validation).
Symptom: "We spent hours reaching a consensus that you're asking to overturn." The human's investment in the plan is invisible to the role.
Context switch as handoff
Role A built a system. Role A briefs Role B to extend it. Role B spends 10 minutes loading context that Role A already has. The work gets done but the context switch cost is invisible — it looks like "delegation" but it's actually "wasted loading time." Root cause: roles treat each other as interchangeable. They're not — context is expensive.
Symptom: The person who built the system briefs someone else to extend it, when they could do it faster themselves.
Silent degradation
A page stops showing data. A service starts a Docker container that should be retired. Memory fills to 58MB free. Nobody notices — the system doesn't alert on its own degradation. The human discovers it by feeling slowness or seeing broken UI. Root cause: monitoring is pointed at coordination events (nudges, spine logs) instead of system health (RAM, service status, data freshness).
Symptom: "I get a lot of unwanted alerts and the things I want to count on don't work." Signal-to-noise inversion.

V. The Value Stream

Work flows through four states. The human's involvement is concentrated at the boundaries — directing and proving. The middle is role work.

Directing
Jeff decides what matters.
Cards created. Priority set.
Designing
Roles define the approach.
ICD, architecture, AC.
Building
Roles execute.
Code, data, artifacts.
Proving
Demo to Jeff. Review.
Accept or reject.

The ideal: Jeff touches each card twice — once to direct, once to prove. Everything in between is autonomous role work. The system's maturity is measured by how close it gets to this ideal. Today's reality: 4–6 touches per card. The gap is the improvement target.

VI. The Feedback Engine

How the system learns. Four stages, from observation to enforcement. The goal is to move every important behavior to stage 4 as fast as possible.

StageMechanismReliabilityExample
1. Observation Human notices a pattern Depends on human attention "Roles keep going silent during long jobs"
2. Feedback Written to memory file Read at boot, forgotten by hour 2 feedback_batch_progress.md
3. Skill Encoded in skill protocol Followed when invoked, variant each time gemba-tick TTL check
4. Hook Enforced by compiled gate Fires every time, no variance ICD write → auto-validate + reload + lint
The Athena Pattern: In Jeff's prior system (Athena at Staples), saving an ICD triggered validation automatically. You couldn't save an incomplete contract. The same principle applies here: the action triggers the check. Not a separate step, not a memory, not a checklist — the write IS the validation. Moving behaviors from Stage 2 to Stage 4 is the most leveraged work in the system.

VII. The Attention Economy

The human's attention is the single most constrained resource. Every design choice either protects it or wastes it. This section names the costs.

Attention CostWhat HappensSystem Fix
Re-prompt Human repeats direction because the role forgot or drifted Persistent context + card AC as reference
Monitor Human checks if a job is still running or a role is still working Progress emission + liveness signals
Relay Human carries information between roles Direct role-to-role briefs and nudges
Correct Human stops a role from doing the wrong thing Scope enforcement + blast radius checks
Interpret Human decodes what the role is doing from tool call output Gemba commentary + Bridge visibility
Recover Human fixes damage from a role's unsanctioned action Proving gate + rollback capability

The target metric: 2 touches per card (direct + prove). Every additional touch is a cost. Track it. Reduce it. The system improves when the human's attention goes toward creative decisions, not coordination overhead.

VIII. Prior Art and Lineage

Chorus didn't emerge from nothing. It stands on specific foundations:

SourceWhat It Contributes
Toyota Production System Andon (signal problems immediately), jidoka (stop on error), piece flow (one card at a time), WIP limits. The team name "Loom" comes from Toyota's origin — the automatic loom that stopped when a thread broke.
Poppendieck (Lean Software Development) Flow over estimation — Chorus doesn't estimate card duration, it flows cards through states. Quality into process, not inspection — the Athena pattern (save triggers validation) is Poppendieck's "build quality in" applied to AI coordination. The Empire State Building didn't have a Gantt chart — it had piece flow and clear handoff contracts. That's cards + proving gates. Plank roads are infrastructure debt that looks like progress — every custom harvest script we wrote was a plank road. NiFi replaces them with an engineered surface.
Kent Beck (Extreme Programming) Pair programming → Chorus's human-AI pairing (navigator/driver with rotation). Small releases → cards, not projects. Continuous integration → proving gate as continuous validation. Courage → push back on flawed premises (DEC-069). XP's deepest insight: the practices ARE the product. Chorus inherits this directly — the coordination protocol is the deliverable, not just the code it produces. Beck also showed that process discipline and agility are not opposites — they reinforce each other when the practices are right.
Value Stream Mapping (Lean) Chorus's four-state value stream (Directing → Designing → Building → Proving) is a direct application. WIP limits prevent overloading. Pull systems mean roles pull cards when ready, not pushed by a manager. The critical lean insight for AI teams: flow efficiency over resource efficiency. One role hot, others observe — don't parallelize across roles when context switching costs more than idle time. Taiichi Ohno's "the more inventory, the less likely you will have what you need" applies to WIP cards exactly.
Domain-Driven Design (Evans) Bounded contexts → vertical role domains. Wren owns coordination, Silas owns infrastructure, Kade owns the app. The boundary IS the design decision. Ubiquitous language → ICD field naming discipline, URN conventions, consistent vocabulary across roles. Context mapping → briefs as the mechanism for cross-domain work. When a PM briefs an engineer, that's a context map in action — translating product intent into technical specification across a boundary. Aggregates → cards as the consistency unit. A card holds its own truth: title, AC, owner, domain. The board is the repository of aggregates.
Hohpe (Enterprise Integration Patterns) Message routing, content-based routing, dead letter channels. The ICD pattern. The recognition that integration is a solved problem — use an engine, don't build one. Chorus applies EIP at two levels: (1) data pipelines via NiFi (the literal patterns), and (2) role coordination via nudges, briefs, and the attention contract (the metaphorical patterns — roles as services, briefs as messages, the proving gate as a guaranteed delivery endpoint).
Bridwell Patent (US9552400B2) RDF/OWL + SPARQL + workflow gates. Time-scoped source authority. The ICD-governed pipeline is a direct descendant. Filed 2014, granted 2017. Prior art for everything Chorus does with data governance.
Chiron (Kolb & Armesto) Human-AI delivery research. Hybrid execution model. V4 breakthrough = AC validation + human-in-the-loop proving. The closest published work to what Chorus implements daily.
Jeff's CIA Team (Staples 2010–2015) ICD review discipline. The ARB approved architecture; the CIA made it trustworthy. Source-level review: "Is this source real? Is it a duplicate? Where's the ICD?" The human review that catches what automation misses. In one session, Jeff applied this discipline to cut the Photos ICD from 6 sources to 4, catch a near-deprecation of scene label data, and enforce that authority data belongs inside provider folds, not floating at the top level.
Vendor Lock-in Arc (Jeff's Career) Quattro Pro macros → Informix 4GL → CASE tools → IBM WebSphere → Software AG at Staples. Every step was a vendor's DSL that was powerful until you needed to leave. Deep lock-in with Software AG drove interest in open-source alternatives (NiFi, Mule, Talend). This directly shapes Chorus architecture: the protocol layer (roles, cards, ICD, proving gate) is portable across engine choices. Claude Code is today's AI engine, but the patterns move. The reference model doesn't lock into the implementation. Same lesson, applied forward.

IX. What Makes This Different

Other approaches to AI-assisted development exist. Here's how Chorus differs:

ApproachModelChorus Difference
Copilot / autocomplete AI suggests, human accepts/rejects inline Chorus roles have agency — they plan, execute, and coordinate, not just suggest
Chat-based coding Human describes, AI generates, human pastes Chorus roles execute directly — they own files, run tests, deploy. No copy-paste loop.
Agent swarms Many disposable agents, no persistence, no roles Chorus roles are persistent and specialized. Three roles > thirty disposable agents. Depth over breadth.
Single super-agent One AI does everything Chorus separates concerns: PM doesn't write code, Engineer doesn't make product decisions. Separation produces better output than omniscience.
The fundamental bet: a small team of persistent, specialized AI roles — directed by a human who provides judgment and energy — produces better outcomes than either human-alone or AI-alone. The coordination cost is real. The compounding value of shared context, accumulated feedback, and enforced discipline is higher.
