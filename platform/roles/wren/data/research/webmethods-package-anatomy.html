1. Package Structure & Directory Layout

A webMethods package is a directory (or .zip archive of that directory) that lives under the Integration Server's packages/ folder. The Integration Server identifies a package by the presence of a manifest.v3 file and an ns/ directory. Everything else is convention.

MyPackage/
  manifest.v3            ← Package descriptor (version, deps, startup/shutdown)
  ns/                     ← Namespace: ALL integration logic lives here
    MyPackage/            ← Top-level namespace folder (matches package name)
      services/           ← Folder containing flow/java services
        processOrder/     ← One directory per service
          node.ndf        ← Service metadata: I/O signature, ACLs, audit settings
          flow.xml        ← Flow logic (XML). Only for flow services.
        transformData/
          node.ndf
          flow.xml
      docs/               ← Document types (data schemas)
        OrderRequest/
          node.ndf        ← Document type definition (field schema)
        OrderResponse/
          node.ndf
      triggers/           ← Messaging triggers
        onNewOrder/
          node.ndf        ← Trigger config: conditions, routing, JMS/Broker binding
      connections/        ← Adapter connections (JDBC, JMS, HTTP, etc.)
        myJDBCConn/
          node.ndf
      notifications/      ← Adapter notifications (polling listeners)
        pollNewRecords/
          node.ndf
  code/                   ← Compiled Java
    classes/              ← .class files for Java services
    source/              ← .java source (if included)
    jars/                ← Third-party JARs bundled with the package
  config/                 ← Package-specific config files
  lib/                    ← External libraries (JARs, native libs)
  pub/                    ← Web content: DSP pages, HTML, images
  resources/              ← Flat file schemas, WSDL, XSD, etc.
  templates/              ← Output templates for services
  web/                    ← JSP web apps (served via embedded Tomcat)
Key insight for reverse engineering: The ns/ directory IS the integration logic. Every service, document type, trigger, and adapter connection is a subdirectory under ns/ containing a node.ndf file (and optionally flow.xml for flow services). Walk this tree and you have the complete inventory.

Key File Types

FileFormatContains
manifest.v3XML (IData serialization)Package metadata, version, dependencies, startup/shutdown services
node.ndfXML (IData serialization)Node definition: type, I/O signature, ACLs, audit config, field schemas
flow.xmlXML (Flow language)Flow service execution steps: INVOKE, MAP, BRANCH, SEQUENCE, LOOP, EXIT
node.idfXMLInterface definition file for namespace folder nodes (folder metadata)
java.fragTextPointer to compiled Java class for Java services

2. The Manifest (manifest.v3)

The manifest.v3 is the package descriptor. It uses Software AG's IData XML serialization format — a <Values> root with <value>, <record>, and <array> child elements.

Real Example (from WmVCS package)

<?xml version="1.0" encoding="UTF-8"?>

<Values version="2.0">
  <value name="enabled">yes</value>
  <value name="system_package">yes</value>
  <value name="version">10.3.0.12</value>
  <value name="publisher">Software AG</value>
  <value name="description">webMethods Version Control System Module</value>

  <!-- Startup services: run when package loads -->
  <record name="startup_services" javaclass="com.wm.util.Values">
    <null name="wm.server.vcsadmin:startup"/>
    <null name="wm.server.vcsui:addSolutions"/>
  </record>

  <!-- Dependencies: other packages required -->
  <record name="requires" javaclass="com.wm.util.Values">
    <value name="WmRoot">10.3</value>
  </record>

  <!-- Shutdown services: run when package unloads -->
  <record name="shutdown_services" javaclass="com.wm.util.Values">
    <null name="wm.server.ui:removeSolution"/>
    <null name="wm.server.vcsadmin:shutdown"/>
  </record>
</Values>

Manifest Fields Reference

FieldTypeMeaning

3. Flow Services (flow.xml)

Flow services are webMethods' visual programming language serialized as XML. Each flow service directory contains a flow.xml (the logic) and a node.ndf (the metadata & I/O signature). The flow.xml uses a distinct XML vocabulary — not IData format.

Core Flow Elements

ElementPurposeAnalogy
FLOWRoot elementFunction body
INVOKECall another serviceFunction call
MAPData transformation stepVariable assignment block
BRANCHConditional routingswitch/case
SEQUENCEGroup steps with error handlingtry/catch block
LOOPIterate over arrayfor-each
REPEAT / RETRYRetry on failureRetry with backoff
EXITExit flow/parent with signalreturn / throw
COMMENTDeveloper commentCode comment

Map Sub-Elements (Data Transformation)

ElementPurposeKey Attributes
MAPCOPYCopy value from source to targetFROM, TO, CONDITION
MAPSETSet a field to a literal valueFIELD, OVERWRITE
MAPDELETERemove a field from pipelineFIELD
MAPINVOKEInline service call (transformer)SERVICE

Example: Complete Flow Service

<FLOW VERSION="3.0" CLEANUP="true">

  <!-- Branch on /message value -->
  <BRANCH TIMEOUT="" SWITCH="/message">

    <!-- Case: message contains "nok" -->
    <SEQUENCE NAME="/nok/" TIMEOUT="" EXIT-ON="FAILURE">
      <INVOKE SERVICE="pub.flow:debugLog"
              VALIDATE-IN="$none" VALIDATE-OUT="$none">
        <MAP MODE="INPUT">
          <MAPSET FIELD="/level;1;0" OVERWRITE="true">
            <DATA ENCODING="XMLValues">
              <Values version="2.0">
                <value name="xml">Error</value>
              </Values>
            </DATA>
          </MAPSET>
        </MAP>
      </INVOKE>
    </SEQUENCE>

    <!-- Case: message contains "ok" -->
    <SEQUENCE NAME="/ok/" TIMEOUT="" EXIT-ON="FAILURE">
      <INVOKE SERVICE="pub.flow:debugLog"
              VALIDATE-IN="$none" VALIDATE-OUT="$none"/>
    </SEQUENCE>

    <!-- Default case -->
    <SEQUENCE NAME="$default" TIMEOUT="" EXIT-ON="FAILURE">
      <EXIT FROM="$parent" SIGNAL="FAILURE"
            FAILURE-MESSAGE="Unknown message type"/>
    </SEQUENCE>

  </BRANCH>
</FLOW>

Example: Data Mapping with Transformers

<MAP NAME="transformOrder" MODE="STANDALONE">

  <!-- Copy field: source path → target path -->
  <MAPCOPY FROM="/customerName;1;0"
           TO="/order;4;0;acme.docs:Order/name;1;0"/>

  <!-- Conditional copy -->
  <MAPCOPY FROM="/products[1];2;1/unitPrice;3.5;0"
           TO="/price;3.5;0"
           CONDITION="%productType% == /standard/"/>

  <!-- Set literal value -->
  <MAPSET FIELD="/status;1;0" OVERWRITE="true">
    <DATA ENCODING="XMLValues">
      <Values version="2.0"><value name="xml">PENDING</value></Values>
    </DATA>
  </MAPSET>

  <!-- Delete a field from pipeline -->
  <MAPDELETE FIELD="/tempData;1;0"/>

  <!-- Inline transformer: uppercase a city name -->
  <MAPINVOKE SERVICE="pub.string:toUpper">
    <MAP MODE="INVOKEINPUT">
      <MAPCOPY FROM="/address;4;0;acme.docs:Address/city;1;0"
               TO="/inString;1;0"/>
    </MAP>
    <MAP MODE="INVOKEOUTPUT">
      <MAPCOPY FROM="/value;1;0"
               TO="/address;4;0;acme.docs:Address/city;1;0"/>
    </MAP>
  </MAPINVOKE>

</MAP>

Reading Typed Paths

The semicolon-delimited paths in FROM/TO/FIELD attributes encode type information:

# Format: /fieldName;typeCode;dimensionCode
/customerName;1;0       # string (type 1), scalar (dim 0)
/order;4;0              # record/document (type 4), scalar
/items;4;1              # record (type 4), array (dim 1)
/products[1];2;1        # object (type 2), array — with index
/price;3.5;0            # float/numeric (type 3.5), scalar

# Nested path through a document type reference:
/order;4;0;acme.docs:Order/name;1;0
#  ^record ^ref to doc type       ^string field inside it
Type CodeMeaning
1String
2Object
3, 3.5Numeric / Float
4Record (IData / Document)

4. Document Types (node.ndf)

Document types are webMethods' data schemas. They define the structure of data flowing through the pipeline. Each document type is a directory under ns/ containing a single node.ndf file in IData XML format.

Real Example: OrderResponse Document Type

<Values version="2.0">
  <record name="record" javaclass="com.wm.util.Values">
    <value name="node_type">record</value>
    <value name="node_nsName">FLOW_Tutorial.docs:OrderResponse</value>
    <value name="node_pkg">FLOW_Tutorial</value>
    <value name="is_public">false</value>
    <value name="field_type">record</value>
    <value name="field_dim">0</value>

    <!-- Fields defined as array of records -->
    <array name="rec_fields" type="record" depth="1">

      <!-- Nested record: Response -->
      <record javaclass="com.wm.util.Values">
        <value name="field_name">Response</value>
        <value name="field_type">record</value>
        <value name="field_dim">0</value>

        <array name="rec_fields" type="record" depth="1">

          <!-- String field: Customer -->
          <record javaclass="com.wm.util.Values">
            <value name="field_name">Customer</value>
            <value name="field_type">string</value>
            <value name="field_dim">0</value>
            <value name="nillable">true</value>
          </record>

          <!-- Array of records: Orders (field_dim=1 means array) -->
          <record javaclass="com.wm.util.Values">
            <value name="field_name">Orders</value>
            <value name="field_type">record</value>
            <value name="field_dim">1</value>  <!-- 1 = array -->
            <value name="rec_closed">true</value>

            <array name="rec_fields" type="record" depth="1">
              <record javaclass="com.wm.util.Values">
                <value name="field_name">OrderID</value>
                <value name="field_type">string</value>
                <value name="field_dim">0</value>
              </record>
              <record javaclass="com.wm.util.Values">
                <value name="field_name">AmountDue</value>
                <value name="field_type">string</value>
                <value name="field_dim">0</value>
              </record>
            </array>
          </record>

        </array>
      </record>
    </array>
  </record>
</Values>

Document Type Field Properties

PropertyMeaning
field_nameField name
field_typestring, record, object (corresponds to Java String, IData, Object)
field_dim0 = scalar, 1 = array (list)
nillableWhether null values are allowed
rec_fieldsChild fields (for record types)
rec_closedtrue = no extra fields allowed (closed schema)
field_content_typeXSD type reference (xmlns + ncName)
node_nsNameFully-qualified namespace name (e.g., MyPkg.docs:OrderResponse)
is_publicWhether the type is visible to other packages
Schema extraction insight: The rec_fields array is recursive. Walk it depth-first and you get a complete field tree. Each leaf has field_name + field_type + field_dim — that's your schema. Document types with field_content_type referencing XSD types can be converted to XSD using pub.schema:createXSD (on a running server) or parsed directly from the XML.

5. Triggers & Adapters

Triggers and adapter connections are also stored as node.ndf files under the ns/ tree. The node_type field in the NDF distinguishes what you're looking at.

Node Types You'll Find

node_type valueWhat It IsAlso Contains
service (+ flow.xml)Flow serviceI/O signature, pipeline logic
service (+ java.frag)Java serviceI/O signature, class reference
recordDocument typeFull field schema
webServiceDescriptorWeb service descriptorWSDL binding info
triggerMessaging triggerConditions, document types, routing service
adapter (various subtypes)Adapter connection/notificationConnection params, polling config

Trigger Structure (node.ndf)

A trigger's node.ndf specifies which document types it listens for, filter conditions, and which service to invoke when a matching message arrives.

<!-- Simplified trigger node.ndf structure -->
<Values version="2.0">
  <record name="record" javaclass="com.wm.util.Values">
    <value name="node_type">trigger</value>
    <value name="node_nsName">MyPkg.triggers:onNewOrder</value>

    <!-- Trigger type: JMS or Broker -->
    <value name="trigger_type">jms-trigger</value>

    <!-- JMS destination -->
    <value name="jms_destination">queue:///order.incoming</value>
    <value name="jms_connection_alias">MyJMSConnection</value>

    <!-- Conditions: which doc types + filter + routing service -->
    <array name="conditions" type="record">
      <record>
        <value name="document_type">MyPkg.docs:OrderRequest</value>
        <value name="filter">%orderType% == 'STANDARD'</value>
        <value name="service">MyPkg.services:processOrder</value>
      </record>
    </array>
  </record>
</Values>

Adapter Connection Types

Adapter PackageConnects ToKey Config Fields in node.ndf
WmJDBCAdapterDatabases (JDBC)Driver class, connection URL, credentials (often Base64-encoded), pool size
WmJMSAdapterJMS providersJNDI factory, provider URL, queue/topic names
WmHTTPAdapterHTTP/REST endpointsBase URL, auth method, timeout
WmFlatFileFlat files (CSV, fixed-width)Schema definition, delimiter, record types
WmSAPSAP R/3RFC connection, BAPI names, IDoc types
WmMQAdapterIBM MQQueue manager, channel, queue names
Adapter notification services (polling triggers) also live as node.ndf files. A JDBC adapter notification, for example, will contain the SQL query used for polling, the document type for the result set, and the processing service name. These are gold for understanding what data is being pulled from external systems.

6. Package Dependencies

Dependencies form a graph. Understanding it tells you which packages you need to analyze together.

Where Dependencies Are Declared

  1. manifest.v3 requires block — explicit package-level dependencies with minimum version
  2. INVOKE elements in flow.xml — runtime references to services in other packages (implicit)
  3. Document type references in node.ndf — fields referencing doc types from other packages
  4. Adapter connection references — services referencing connections defined in other packages

Common Base Packages

PackagePurposeServices You'll See Referenced
WmPublicCore public servicespub.flow:*, pub.string:*, pub.list:*, pub.math:*, pub.date:*, pub.io:*, pub.xml:*
WmRootCore IS serviceswm.server.* (admin, internal)
WmARTAdapter runtimeConnection management, pooling framework
WmJDBCAdapterJDBC adapterpub.art.jdbc:*
WmFlatFileFlat file parsingpub.flatFile:*

Building a Dependency Graph from Files

# 1. Extract explicit dependencies from manifest
grep -o 'name="[^"]*">[^<]*' manifest.v3 | grep -A1 requires

# 2. Extract all INVOKE targets (implicit service dependencies)
grep -rh 'SERVICE=' ns/ --include='flow.xml' | \
  sed 's/.*SERVICE="\([^"]*\)".*/\1/' | \
  sort -u

# 3. Extract document type cross-references
grep -rh 'node_nsName' ns/ --include='node.ndf' | \
  sed 's/.*>\([^<]*\)<.*/\1/' | \
  sort -u

7. Reverse Engineering — Offline Extraction

You do NOT need a running Integration Server to extract the integration logic from a webMethods package. Everything is stored in XML files on the filesystem. Here's the approach.

Step 0: Unpack

# Packages exported from IS are usually .zip files
unzip MyPackage.zip -d MyPackage/

# Or if delivered as part of a full IS filesystem:
ls IntegrationServer/packages/MyPackage/

Step 1: Inventory All Nodes

# Find every node definition in the package
find MyPackage/ns -name 'node.ndf' | while read ndf; do
  dir=$(dirname "$ndf")
  name=$(basename "$dir")
  type=$(grep -o '<value name="node_type">[^<]*' "$ndf" | head -1 | sed 's/.*>//')
  echo "$type | $name | $ndf"
done | sort | column -t -s'|'

This produces a complete inventory like:

record   OrderRequest    ns/MyPkg/docs/OrderRequest/node.ndf
record   OrderResponse   ns/MyPkg/docs/OrderResponse/node.ndf
service  processOrder    ns/MyPkg/services/processOrder/node.ndf
service  transformData   ns/MyPkg/services/transformData/node.ndf
trigger  onNewOrder      ns/MyPkg/triggers/onNewOrder/node.ndf

Step 2: Extract Document Type Schemas

Parse the rec_fields arrays recursively from document type node.ndf files. Here's a Python script that does it:

#!/usr/bin/env python3
"""Extract field schema from a webMethods document type node.ndf."""
import xml.etree.ElementTree as ET
import sys, json

def extract_fields(parent_element, path=""):
    """Recursively extract fields from rec_fields arrays."""
    fields = []
    for rec_fields in parent_element.findall(".//array[@name='rec_fields']/record"):
        name_el = rec_fields.find("value[@name='field_name']")
        type_el = rec_fields.find("value[@name='field_type']")
        dim_el  = rec_fields.find("value[@name='field_dim']")
        nil_el  = rec_fields.find("value[@name='nillable']")

        if name_el is None or type_el is None:
            continue

        field = {
            "name": name_el.text,
            "type": type_el.text,
            "array": dim_el is not None and dim_el.text == "1",
            "nullable": nil_el is not None and nil_el.text == "true",
            "path": f"{path}/{name_el.text}" if path else name_el.text,
        }

        # Recurse into nested records
        if type_el.text == "record":
            field["children"] = extract_fields(rec_fields, field["path"])

        fields.append(field)
    return fields

tree = ET.parse(sys.argv[1])
root = tree.getroot()

ns_name = root.find(".//value[@name='node_nsName']")
print(f"Document: {ns_name.text if ns_name is not None else 'unknown'}")
print(json.dumps(extract_fields(root), indent=2))

Step 3: Extract Flow Service Logic

#!/usr/bin/env python3
"""Extract the call graph and data mappings from a flow.xml."""
import xml.etree.ElementTree as ET
import sys

def walk_flow(element, depth=0):
    """Walk flow elements and extract logic."""
    indent = "  " * depth
    tag = element.tag

    if tag == "INVOKE":
        service = element.get("SERVICE", "?")
        print(f"{indent}CALL {service}")
        # Extract input/output mappings
        for m in element.findall("MAP"):
            mode = m.get("MODE", "")
            for copy in m.findall("MAPCOPY"):
                src = copy.get("FROM", "").split(";")[0]
                dst = copy.get("TO", "").split(";")[0]
                print(f"{indent}  {mode}: {src} -> {dst}")

    elif tag == "BRANCH":
        switch = element.get("SWITCH", "")
        print(f"{indent}BRANCH on {switch}")

    elif tag == "SEQUENCE":
        name = element.get("NAME", "")
        exit_on = element.get("EXIT-ON", "")
        print(f"{indent}SEQUENCE {name} (exit-on={exit_on})")

    elif tag == "MAP" and element.get("MODE") == "STANDALONE":
        print(f"{indent}MAP (standalone)")
        for copy in element.findall("MAPCOPY"):
            src = copy.get("FROM", "").split(";")[0]
            dst = copy.get("TO", "").split(";")[0]
            print(f"{indent}  COPY {src} -> {dst}")
        for delete in element.findall("MAPDELETE"):
            field = delete.get("FIELD", "").split(";")[0]
            print(f"{indent}  DELETE {field}")
        for s in element.findall("MAPSET"):
            field = s.get("FIELD", "").split(";")[0]
            print(f"{indent}  SET {field}")

    elif tag == "LOOP":
        arr = element.get("IN", "")
        print(f"{indent}LOOP over {arr}")

    elif tag == "EXIT":
        signal = element.get("SIGNAL", "")
        msg = element.get("FAILURE-MESSAGE", "")
        print(f"{indent}EXIT {signal}: {msg}")

    for child in element:
        walk_flow(child, depth + 1)

tree = ET.parse(sys.argv[1])
walk_flow(tree.getroot())

Step 4: Extract Service I/O Signatures

Every service's node.ndf contains its input and output signature — the pipeline contract. These are stored in svc_sig (or sig) sections with the same rec_fields structure as document types.

# Quick extraction: find all service signatures
find MyPackage/ns -name 'node.ndf' -path '*/services/*' | while read f; do
  svc=$(basename $(dirname "$f"))
  inputs=$(grep -c 'field_name' "$f")
  echo "$svc: $inputs fields in signature"
done

Step 5: Build the Call Graph

# Extract all service-to-service calls from flow.xml files
find MyPackage/ns -name 'flow.xml' | while read f; do
  caller=$(echo "$f" | sed 's|.*/ns/||;s|/flow.xml||' | tr '/' '.')
  grep -o 'SERVICE="[^"]*"' "$f" | sed "s/SERVICE=\"//;s/\"//" | while read target; do
    echo "$caller -> $target"
  done
done | sort -u

This produces a directed graph you can visualize with Graphviz or feed into your ICD tooling:

MyPkg.services.processOrder -> pub.flow:debugLog
MyPkg.services.processOrder -> MyPkg.services.transformData
MyPkg.services.processOrder -> pub.db:call
MyPkg.services.transformData -> pub.string:toUpper
MyPkg.services.transformData -> pub.date:formatDate

8. Mapping to ICD / Canonical Model

This is where webMethods reverse engineering meets your ICD pipeline. The goal: extract the implicit data contracts and field mappings from webMethods packages and express them as explicit, auditable ICD entries in your canonical model.

What to Extract for ICD

webMethods ArtifactICD EquivalentWhere to Find It
Document type fieldsInterface field definitionsnode.ndf → rec_fields recursion
Flow MAPCOPY operationsSource-to-canonical field mappingsflow.xml → MAPCOPY FROM/TO
MAPSET operationsDefault values / enrichment rulesflow.xml → MAPSET FIELD + DATA
MAPINVOKE operationsTransformation rulesflow.xml → MAPINVOKE SERVICE
BRANCH conditionsRouting rules / business logicflow.xml → BRANCH SWITCH
Trigger conditionsEvent contracts / message filtersTrigger node.ndf → conditions
Adapter connectionsSystem connection inventoryConnection node.ndf → URLs, drivers, queues
Service I/O signaturesInterface contracts (input/output)Service node.ndf → signature sections

Recommended Extraction Pipeline

  1. Unpack all packages into a working directory
  2. Inventory every node.ndf by node_type (script from Step 1)
  3. Extract schemas from all document types (Python script from Step 2)
  4. Extract flow logic from all flow.xml files (Python script from Step 3)
  5. Build call graph to understand service dependencies (Step 5)
  6. Map MAPCOPY operations to source/target field pairs — these are your field mappings
  7. Cross-reference trigger document types with service inputs to trace message flow end-to-end
  8. Output as ICD-format field mapping tables (or directly as RDF/TTL if your pipeline supports it)

What You Won't Get Without the Runtime

  • Runtime variable values — global variables, session properties, environment-specific config
  • Dynamic service invocation — calls via pub.flow:invokeService with service name as a variable
  • Java service internals — compiled .class files need a decompiler (use jad, cfr, or fernflower)
  • Scheduled tasks — configured in the IS admin UI, not stored in the package filesystem
  • Credential values — passwords in adapter connections are Base64-encoded IData blobs, often encrypted with the IS master password
