ICD Integration Layer Design

Extending the Gathering ICD ontology with integration contract concepts grounded in Enterprise Integration Patterns (Hohpe & Woolf, 2003).

Author: Wren (PM) Date: 2026-03-25 Status: Proposal Ontology: urn:gathering:icd# Pattern vocabulary: EIP

1. Problem Statement

The current ICD ontology models data contracts well: domains, providers, canonical fields, field mappings, era-scoped authority, metadata ceilings, migration events, harvest runs. This is the right foundation and it works.

What's missing is the integration contract layer. Jeff Bridwell designed the original ICD pattern at Staples (2010-2015) with an ARB template that included integration patterns, directionality, roles, and endpoints. The Gathering ontology dropped that layer because the initial use case was one-way batch harvest from local sources. No pub/sub, no real-time, no API consumers.

Two things made the gap visible:

  • NiFi flows — the iPhone extraction and canonical rebuild introduce triggered and batch patterns that the ontology can't describe.
  • webMethods package analysis — reverse-engineering OAGIS verbs (Sync, Get/Show, Load, Confirm) showed exactly what integration contract metadata looks like at enterprise scale. Our ontology is missing all of it.

Rather than invent a custom pattern vocabulary, this proposal adopts Enterprise Integration Patterns (Hohpe & Woolf) as the canonical naming convention. EIP is the shared language of every integration stack we've touched — webMethods, NiFi, Apache Camel, Spring Integration, MuleSoft. Using EIP names means the ICD speaks the same language as the tools and the practitioners.

Design principle All new classes are additive. No breaking changes to existing ontology. Existing Provider instances gain new optional properties. This is an extension, not a migration.

2. Prior Art: Enterprise Integration Patterns

Enterprise Integration Patterns Gregor Hohpe & Bobby Woolf. Enterprise Integration Patterns: Designing, Building, and Deploying Messaging Solutions. Addison-Wesley, 2003. ISBN 0-321-20068-3.

The canonical vocabulary for integration architecture. 65 patterns organized into messaging channels, message routing, message transformation, messaging endpoints, and system management. Most integration stacks (webMethods, MuleSoft, Apache Camel, NiFi, Spring Integration) implement EIP patterns by name. Using EIP as our pattern vocabulary means the ICD is immediately readable by any integration architect who has worked with these tools.

The EIP catalog gives us precise names for what we previously described loosely. "Batch" becomes a composition of PollingConsumer + MessageTranslator + IdempotentReceiver. "Pub/Sub" maps to PublishSubscribeChannel. This is not added complexity — it's added precision. The patterns were always there; now they have names.

4. EIP Pattern Catalog for the ICD

The following EIP patterns are relevant to integration contracts in both the Staples/webMethods context and our Gathering/NiFi context. Each becomes a named instance of icd:IntegrationPattern.

Messaging Patterns (channel-level)

EIP PatternDescriptionCategory
PublishSubscribeChannelOne sender, many receivers. Each subscriber gets a copy of every message.Messaging
PointToPointChannelOne sender, one receiver. Exactly-once consumption.Messaging
DeadLetterChannelCaptures messages that fail processing for later inspection or replay.Messaging
GuaranteedDeliveryPersists messages to survive crashes. Delivery is assured even if the receiver is down.Messaging

Message Routing Patterns

EIP PatternDescriptionCategory
ContentBasedRouterRoutes a message to a different channel based on message content.Routing
MessageFilterRemoves uninteresting messages from a channel based on criteria.Routing
SplitterBreaks a composite message into individual messages for independent processing.Routing
AggregatorCombines related individual messages into a single composite message.Routing
RecipientListRoutes a message to a list of dynamically determined recipients.Routing

Message Transformation Patterns

EIP PatternDescriptionCategory
MessageTranslatorConverts a message from one format to another (source schema to canonical).Transformation
ContentEnricherAugments a message with additional data from an external source.Transformation
ContentFilterRemoves unwanted data from a message, keeping only what the receiver needs.Transformation
ClaimCheckStores message data externally and passes a reference, reducing message size during routing.Transformation

Endpoint Patterns

EIP PatternDescriptionCategory
PollingConsumerActively polls for messages on a schedule. The consumer controls timing.Endpoint
EventDrivenConsumerReacts to message arrival. The producer controls timing.Endpoint
IdempotentReceiverSafely handles duplicate messages. Processing the same message twice produces the same result.Endpoint

System Management Patterns

EIP PatternDescriptionCategory
ControlBusManages and monitors the integration system. Configuration and health in a single channel.Management
WireTapInspects messages flowing through a channel without affecting the flow. Logging and auditing.Management
MessageHistoryAttaches a list of all components the message has visited. Full audit trail.Management

6.4 icd:hasDirection (Property on Provider)

Not a class, but a new property on icd:Provider:

New Property: hasDirection
hasDirection : xsd:string — "inbound" | "outbound" | "bidirectional"
rdfs:domain icd:Provider
  • Inbound: source → canonical (almost all current Gathering providers)
  • Outbound: canonical → consumer (future Photos API, future Garden page API)
  • Bidirectional: both directions (future SOLID pod sync)

8. Validation: webMethods OAGIS Mapping (EIP-aligned)

Every webMethods OAGIS verb maps to a composition of EIP patterns. This validates that the EIP vocabulary is sufficient to express enterprise integration contracts.

webMethods Verb EIP Patterns Used Role Direction
Sync
SyncSalesOrder		 PublishSubscribeChannel + MessageTranslator Publisher Inbound
Get
GetQuote		 PointToPointChannel + ContentBasedRouter Requestor → Responder Bidirectional
Show
ShowShipment		 PointToPointChannel + ContentFilter Responder Outbound
Load
LoadPayable		 PollingConsumer + MessageTranslator + IdempotentReceiver BatchProducer Inbound
Confirm
ConfirmCNHServiceRequest		 PublishSubscribeChannel + GuaranteedDelivery Publisher Inbound
Create
CreateCNHServiceRequest		 PointToPointChannel + MessageTranslator Requestor Outbound
Update
UpdateCredit		 PublishSubscribeChannel + ContentEnricher Publisher Inbound
Result: Full coverage with richer semantics. Every OAGIS verb decomposes into EIP pattern compositions. The previous model mapped each verb to a single custom pattern; the EIP model captures the channel, transformation, and endpoint patterns independently. This is the difference between "it's a batch job" and "it's a PollingConsumer feeding a MessageTranslator with an IdempotentReceiver."

The canonical rebuild isn't just "batch" — it's a PollingConsumer feeding a Splitter-Translator-Aggregator chain with a DeadLetterChannel for failures. This is the level of precision that makes integration contracts auditable.

11. Impact on Existing System

ComponentImpactEffort
icd-ontology.ttl Append new classes + properties + EIP instances. No changes to existing definitions. Small
icd-instance-*.ttl Existing instances gain new optional triples. No migration required. Add integration metadata as providers are updated. Incremental
Convergence page Gains an "Integration" section per provider showing EIP patterns, role, direction, endpoint. Medium
ICD gate hook No change — integration metadata is optional, not gated. None
webMethods extraction script Can auto-populate EIP pattern composition from OAGIS verb prefix. Verb → EIP patterns is deterministic (see Section 8 mapping table). Small
NiFi flow documentation Each NiFi process group maps to an EIP pattern chain. Processor types have direct EIP equivalents (RouteOnAttribute = ContentBasedRouter, ConvertRecord = MessageTranslator, etc.). Small
Not in scope (intentionally deferred)
  • Flow/Orchestration class — EIP's ProcessManager and RoutingSlip patterns exist for this, but our NiFi flows are simple enough that the pattern composition on Provider is sufficient. Defer until we have multi-step orchestrations that need their own lifecycle.
  • Contract SLA class — the existing providerLatency free-text field covers current needs. EIP's GuaranteedDelivery is captured as a pattern instance; formal SLA enforcement (retry policies, circuit breakers) deferred.

13. Recommendation

Ship the three new classes (IntegrationPattern, InterfaceRole, Endpoint) plus the four new Provider properties (hasEndpoint, playsRole, usesPattern, hasDirection) and the two new IntegrationPattern properties (eipReference, eipCategory) as an additive extension to icd-ontology.ttl.

Sequence:

  1. Append classes, properties, and EIP pattern instances to icd-ontology.ttl
  2. Reload Fuseki and verify no parse errors
  3. Add integration metadata to one provider instance (Apple Photos — photos domain) as proof
  4. Update convergence page to render integration section with EIP pattern names
  5. Wire webMethods extraction script to auto-populate EIP pattern composition from OAGIS verb

No migration. No breaking changes. The data contract side stays exactly as-is. The integration contract layer fills the gap the Staples ICD template always had — and now it speaks the same language as every integration tool and every integration architect who has read the book.
