The Problem

Jeff tails 3 terminal windows 8-10 hours a day. Every nudge, every role-to-role handoff, every brief routing — he sees all of it. The cognitive load is in the filtering, not the reading. He's doing the work of a message router in his head.

Today: Jeff → 3 terminals → reads everything → mentally filters → responds where needed
Goal: Jeff → 1 window → sees only what's directed at him → roles handle the rest

Design Principle

The glance is the heartbeat. osascript is the defibrillator.
The Bridge is where the captain stands.

Jeff is the pilot. He sees instruments (role tiles), hears comms directed at him (filtered messages), and gives orders (typed input). The crew (roles) talk to each other on a separate frequency he doesn't monitor.

Role Tiles

Each tile shows three things at a glance:

  • Role name + emoji — identity
  • Active card # — what they're working on (links to board)
  • State dot — ● building / ● waiting / ● blocked / ● idle

Tile updates live from role-state.sh output — same data as team-scan, rendered visually.

Click a tile → expands to show last 3 actions (from chorus log). No terminal tailing needed.

Clearing vs Bridge

Clearing (exists)

  • All voices visible
  • Roles talk to each other in Jeff's view
  • Session-based (start/stop)
  • Uses Claude Haiku for role responses
  • Good for: alignment, decisions, debates

Bridge (new)

  • Jeff's view filtered
  • Roles coordinate behind the glass
  • Always-on (replaces terminal tailing)
  • Proxies to real terminal sessions
  • Good for: daily operation, directing
Recommendation: Build as a separate service, not an extension of Clearing. Different purpose, different message model. Clearing = meeting room. Bridge = cockpit. Share the Socket.IO infrastructure and UI components.

Technical Approach

Architecture

  • Server: Node.js + Express + Socket.IO on localhost:3470
  • Message source: Tail chorus log + role state files — not a new messaging layer
  • Jeff's input: Typed in browser → routed via nudge.sh to the appropriate role terminal
  • Role responses: Chorus log events tagged jeff-facing surface to the channel
  • Tile data: Poll /tmp/andon-*.json every 5 seconds
  • Hidden messages: Role-to-role nudges still flow via nudge.sh, just not displayed

Message Classification

The routing decision: is this message for Jeff or for another role?

  • Nudge with --from and --to both being roles → hidden
  • Nudge to Jeff or from Jeff → visible
  • Demo ready / accept request → always visible (surfaced as alert)
  • Blocked state → always visible (surfaced as alert)
  • State transitions → tile update only, no message

Input Routing

Jeff types in the Bridge. Where does it go?

  • @wren / @silas / @kade → direct to that role's terminal
  • No @mention → Wren gets it first (PM is the default recipient)
  • Reply to a message → routes to the role that sent it

What This Doesn't Replace

  • Terminal sessions — roles still run in Claude Code terminals. The channel is a view, not a replacement.
  • Clearing — still needed for all-hands alignment where Jeff wants to see the debate.
  • Direct terminal access — Jeff can still open a terminal and talk to a role directly when he wants full context.
