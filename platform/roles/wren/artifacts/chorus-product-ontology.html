Chorus Product Ontology

Value Streams × Domains × Services • v0.1 • Card #1783 • 2026-03-28

1. Ontology Rules

Rule 1: Every Domain maps to at least one Value Stream.
Rule 2: Every Service maps to 1..n Domains and 1..n Value Streams.
Rule 3: Value Streams are stable (change = product pivot). Domains are bounded contexts (change = product expansion). Services are implementations (change = engineering decision).

2. Classes

ValueStream Core Class

A named flow of value from signal to outcome. Stable — when a value stream changes, the product has fundamentally changed.

PropertyRangeDescription
namestringHuman name: "Personal", "Life", "Chorus"
stagesStage[]Ordered sequence of stages within this stream
rhythmstringTemporal cadence: "minutes-hours", "weeks-years", "hours-sessions"
productstringWhich product this stream serves: "Gathering", "Chorus", "Both"
Stage Core Class

A phase within a value stream. Each stage is a stance — a posture that determines what you see, value, and produce.

PropertyRangeDescription
namestring"Capturing", "Directing", "Designing", "Building", "Proving"
stancestringThe posture: "Receiving", "Prioritizing", "Architecting", "Shipping", "Verifying"
ownerRole[]Primary role(s) in this stance
producesstring[]Artifacts this stage must emit
belongsToValueStreamParent value stream
orderintegerPosition in the stream sequence
Domain Core Class

A bounded context with its own data, handlers, ontology, and routes. Domains rarely change — when they do, the product has expanded or contracted.

PropertyRangeDescription
namestring"seeds", "music", "photos", "coordination", "convergence"

primaryStageStageThe stage where this domain has strongest affinity

statusstring"active", "partial", "aspirational"
ontologyClassURIRoot class in jb-ontology for this domain
Service Core Class

A running piece of infrastructure with a language, data model, integration contract, and non-functionals. Services implement domain capabilities.

A repeatable discipline that applies across domains. Convergence is a practice, not a domain.

8. What This Ontology Enables

  • Impact analysis: "Which services are affected if I change the seeds domain?" → Seed Pipeline, Gathering App
  • Gap detection: "Which domains have no service design artifact?" → Most of them. That's the #1781 work.
  • Coverage queries: "Which value stream stages have no services?" → Reflecting has zero services. Connect has zero.
  • Blast radius: "What breaks if the Chorus API goes down?" → coordination, search, knowledge-graph domains affected
  • Design enforcement: "Does this new service declare its domains and value streams?" → Gate check before Building
