Cookbook — Substrate-class Domain Treatment

A reusable recipe for turning a documentation surface into a contract. Abstracted from the principles arc; refined by the decisions arc.
v2 · 2026-04-25 · Wren · second-instance refined

A substrate-class domain is content that ought to be a contract, not documentation — referenced by name across role files, briefs, and code; queried more than authored; shape definable. Examples: principles (shipped), decisions, practices, policies, services, roles.

The principles arc (#2447 → #2477) put the first such domain through five moves that each remove a path back to documentation-as-truth. The shape that emerged is the recipe below. See /loom/principles-reference-impl.html for the worked example. The decisions arc (#2485) was the second instance — most of v2's refinements come from what that arc surfaced.

Two-phase model

Each substrate-class arc has two phases. Phase 2 doesn't open until Phase 1 closes. Restructuring a domain on top of dirty data compounds drift; refactor decisions made on broken data carry the brokenness forward.

  • Phase 1 — Data correctness. The five moves below. Graph populated, schema canonical, instances in the right place per ADR-025, drift test holds, citations clean, MCP + hook close the affordance.
  • Phase 2 — Refactor / placement. A separate card opens after Phase 1 closes: move the domain's code to its right repo + folder, fix its validation surface, settle structural placement.

Cross-cutting refactors (e.g., a shared validation surface like the Athena domain page) span all domains and are independent arcs from any single domain's two-phase work.

When to reach for this recipe

Apply when all of:

  • Referenced by name across role files, briefs, or code.
  • Has a definable shape (predicates you could write down).
  • Read-heavy after the initial seed.
  • "Stop being documentation, start being a contract" is the goal.

Don't apply when:

  • Content is transient or single-use.
  • Edit cadence is rare and readership is small (over-engineering).
  • The domain is a UI surface, not a content surface.

The five moves

Each move closes a path the previous left open. Order is non-negotiable.

Move 1 — Graph as the single source
Deliverable
chorus:<Domain> instances in urn:chorus:instances (per ADR-025), one per source-of-truth entry. URI scheme parallel to siblings.
Acceptance
Instance count in graph = entry count in source files. Cross-graph SHACL boundary check (label-required → 400) wired.
Precedent
#2314 — principles ontology→instances migration.
Pre-Move-1
Query the existing graph first. If the domain has prior data in any other graph, read its predicate set and reuse — don't invent a thin schema. Decisions arc cost: 30 min rework when discovered urn:chorus:decisions already had a rich schema (decisionDate, decisionStatus, enforcementLevel, hasDomain×N) that the new thin schema (chorus:id, decisionType DEC|ADR) ignored.
Pre-Move-1
Audit the source corpus for ID collisions. Different content sharing one ID is a Move 1 blocker — populate cannot satisfy "instance count = source count" with collisions. Decisions found 3 (DEC-093, DEC-096, DEC-101) and ate a swat (#2492) before Move 1 could land.
Pre-Move-1
Declare the URI scheme before populate. Probe the existing graph for any prior instance URI shape (e.g., urn:chorus:decision:dec_NNN vs https://jeffbridwell.com/chorus#dec-NNN). If found, adopt it. If not, declare one in this cookbook's addendum for the domain and pin it as canonical. Decisions arc cost: ~30 min when v1 backfill wrote on the wrong URI shape. Cross-tool consistency is non-negotiable — seed scripts, hand-authored TTL, MCP tools, and Athena handlers must all emit and accept the same shape.
Pre-Move-1
Declare predicate names once. When adding any derived-from-graph rule, declare the predicate ONCE in this cookbook's addendum and check that all consumers reference that exact name: writer (discover-*), page facet, completeness count, MCP tool. Today's hasContract-vs-hasEndpoint mismatch (writer wrote one, reader expected the other) cost half an afternoon. Doneness sub-test: grep <predicate> across handlers + completeness + page client returns matching reference counts.
Pre-Move-1
Audit discover-* scan paths before Move 1 closes. A new substrate-class id needs entries in all four discoverers it touches: discover-pages (for /loom/<slug>.html), discover-code (alias override for the source root), discover-endpoints (route URL match), discover-tests (test filename match). Decisions arc hit gaps in all four. One-line per discoverer in the addendum: "this domain registers in scan paths X/Y/Z."
Move 2 — API as the only write path
Deliverable
Athena handlers GET/POST/PUT/DELETE /api/athena/subdomains/<id>/<plural>, reusing ENTITY_SECTIONS. Any pre-existing parallel write path retired (308 redirect; full removal is a later card).
Acceptance
Every write goes through the API. POST without required predicates → 400. Handlers emit spine events.
Precedent
#2314.
Move 3 — Drift test as the gate
Deliverable
Static baseline (book/<domain>-reconstructed.html or equivalent) + a CI assertion holding the baseline against the live graph. Divergence fails CI.
Acceptance
Mutate one graph instance without updating the baseline → CI red. Mutate baseline without graph → CI red.
Precedent
#2447 — 12/12 API regressions + 6/6 drift assertions.
Move 4 — Cite-by-ID, never inline
Deliverable
Every caller (role CLAUDE.md fragments, briefs, decision logs, code comments) that paraphrases entries rewritten to cite by URI/ID. A lint or hook that fires if inline restatement reappears.
Acceptance
grep for inline-restatement pattern across all callers returns 0.
Precedent
#2449 — role principles.md fragments became citation-only.
Move 5 — Hook backstops the affordance
Deliverable
(a) MCP tools <domain>_list / <domain>_get — single-verb-single-target per #2476. Defer _create until cadence proves out. (b) Pre-commit hook that blocks the old path of least resistance (direct TTL edit, or HTTP on the retired endpoint).
Acceptance
A role asking Claude in natural language reaches for the MCP tool, not curl. The hook blocks an attempted regression to the old path.
Precedent
#2476 (MCP tools), #2314 (TTL-edit hook).

Order discipline

MUST Land moves serially, not in parallel. Move 5's hook only makes sense after Moves 1–4 have made the typed path more attractive than the path being blocked. Skip the order, ship a hook that punishes roles for using the only path they have.
MUST Ship Move 3 (drift test). It's the most-skipped move. The drift test is what makes the contract a contract; without it, paraphrase grows back.
MUST Be atomic per move, not across moves. Each card commits one move end-to-end. Cross-class atomicity ("decisions and practices together") is sacrificed for testability and reversibility.
MUST NOT Open Phase 2 (refactor / placement) before Phase 1 closes. Restructuring on dirty data carries the dirt forward. Two-phase rule.

Doneness test

The domain is Phase-1 treated when all six are true:

  1. Every instance lives in the graph.
  2. Every write went through the API.
  3. CI fails if graph and baseline diverge.
  4. Every caller cites by ID; lint enforces it.
  5. MCP tools exist and the hook bites the old path.
  6. Lifecycle gate measures truth. The completeness model classifies each facet as derived / required-authored / optional-authored, and done.pass reflects "all required-authored present + all derived attempted." Hand-authored proxies for completeness percentage are no longer accepted as closure evidence.

If any one is missing, you have a half-treated domain — and the path you didn't close is where drift will start.

Phase-1 closure does NOT mean placement is right. A reference implementation isn't valid until the code is in the right repo, the data is in the right graph, AND the validation surface is in the right product. Decisions arc surfaced this: #2485 closed Phase 1 with the Athena page (which validates every domain) still living in the gathering app, not chorus. That's a Phase-2 fix; Phase 1 closes regardless, but the contract isn't whole until Phase 2 lands.

Facet classification (lifecycle gate, post-decisions arc)

Each facet on the Athena domain page gets one classification:

  • derived — populated by discover-* handlers or graph traversal. Empty count = "we asked, found nothing." Examples: pages, code, tests, services, endpoints, decisions, cards, release-history, prior-art, logs.
  • required-authored — must be hand-authored for the domain to be design-complete. Missing = card not done. Examples: actors, scenarios, contract. Substrate-class subdomains add prior-art to required.
  • optional-authored — adds quality but doesn't gate. Examples: gaps, integrations, persistence, pipeline.

Lifecycle pass criteria:

  • create.pass: label + comment + owner + step.
  • wip.pass: edges + ≥1 required-authored present (proof design started).
  • done.pass: ALL required-authored present (≥1 each) AND all derived facets had derivation attempted. Optional-authored ignored.

Sizing & cost

  • ~5–7 cards per arc for Phase 1. Principles took 7: #2447, #2314, #2157, #2473, #2449, #2476, #2477. Decisions took 1 parent (#2485) + 1 swat (#2492) + 3 follow-ons (#2486, #2502, #2503) over a single intense session.
  • ~1–3 weeks elapsed at current cadence.
  • Lowest-blast-radius move first. For most domains: Move 1 (graph populate) is safest; Move 5 (hook) is most disruptive — last.
  • Plan a swat budget. Move 1 surfaces latent corpus issues — collisions, schema mismatches, misfiled entries, dual-graph drift. Decisions arc consumed an extra ~2h on cleanup that wasn't in the original AC. Treat Move 1 as a discovery move, not just a populate.

Rules & anti-patterns

MUST NOT Skip the drift test. "Tests can come later" means the contract is fictional.
MUST NOT Invent a new schema when the graph already has a rich one for this domain. Query before designing. Decisions arc cost: 30 min when the existing urn:chorus:decisions schema was richer than the thin one being added.
MUST NOT Hand-author per-domain chorus:hasX edges to satisfy a completeness percentage. The percentage measures the wrong thing if it gates on edges instead of derivation. Fix the model, not the score.
MUST NOT Pass gate-product performatively when the validation surface is broken. If the domain page lies (missing sections, hidden empty folds, hand-authored proxies), the PASS is fiction. Retract, reshape, pass for real.
MUST NOT Refactor placement before data is clean. Two-phase model. Restructuring on dirty data carries the dirt forward.
MUST NOT Edit other-product files under this card's WIP. If a chorus card needs a gathering-app edit, that's cross-product scope-grow — file a gathering card.
MUST NOT Close Phase 1 quietly when Phase 2 is needed. Phase-1 doneness ≠ "the domain works in production." Open the Phase-2 placement card explicitly so the half-state is visible.
MUST NOT Stop at Move 4. Citations look clean, but without Move 5 the next regression is a paraphrase one keystroke away.
MUST NOT Use cross-class atomicity ("migrate decisions and practices together"). Per-class atomicity makes failure recoverable.
SHOULD Plan a swat budget for Move 1 cleanup. Move 1 surfaces latent corpus issues — collisions, schema mismatches, misfiled entries, dual-graph drift. Treat Move 1 as a discovery move, not just a populate.
SHOULD Generate the per-domain reference-impl page only after the shape settles. Static authored pages beat dashboards while learning the shape; generation comes after the second instance.
MAY Defer _create MCP tools until _list/_get have proven cadence. Anti-affordance until reach is established. Decisions arc deferred per ADR cadence (monthly-ish).
MAY Auto-sync caller files when canonical entries change — but only if the editorial layer is null. For domains with role emphasis (principles), auto-sync erases role judgment. For domains without it (decisions), sync is safer.

Domain-specific divergence template

Fill in for each new domain as you start. These are where arcs differ:

  • Source-of-truth shape: single file? per-role fragments? per-instance files? (decisions: DEC-NNN log + per-ADR files; principles: TTL + per-role emphasis lists)
  • Editorial layer: does the domain have role-specific weighting/emphasis? (principles: yes — emphasis IDs per role. decisions: probably no — DEC-NNN are team-level.)
  • Ordering: is there canonical order? (principles: Hemenway book order. decisions: chronological + numbered. practices: probably not.)
  • Multi-source intake: content authored in multiple places needing a single graph view? (decisions: yes — DEC log + ADR files. principles: no.)
  • _create cadence: day-one or deferred? (principles: shipped in #2476 — writes weekly. decisions: deferred — ADR cadence is monthly-ish.)

How to use this recipe on the next domain

  1. Pick the domain. Apply the "when to reach for this" test.
  2. Audit the corpus first. Query the existing graph for any prior schema on this domain (different graph, prior modeling, partial migration). Reuse rich predicates; don't invent thin ones. Scan source files for ID collisions and resolve before Move 1 populate.
  3. Fill in the divergence template below.
  4. Open one parent card titled <domain> — five-move treatment (Phase 1). AC organized against the five moves + the placement-deferral note (Phase 2 is its own card). Mirror #2485.
  5. Land each move as a sub-card or sub-AC. In order. Demo each before pulling the next. Plan a swat budget for cleanup that emerges.
  6. After all five land + lifecycle done.pass=true per the new model: write /loom/<domain>-reference-impl.html mirroring the principles page. That's the case-study artifact.
  7. Open the Phase-2 placement card when Phase 1 closes. Move the domain's code/page to its right structural home. Do NOT do this before Phase 1 closes.
  8. The diff between your <domain>-reference-impl and the principles one tells the cookbook maintainer what's universal vs. accidental. After three instances, abstract this recipe again.
