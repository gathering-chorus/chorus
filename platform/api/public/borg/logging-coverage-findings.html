Definition (Jeff's): the harness = our automated invariant-execution protocol — the verbs, gates, hooks, and guards that automatically enforce the workflow's invariants (AC must exist · no commit on main · no canonical write · demo-before-accept · no secrets). The matrix below audits that harness. The one row that is not the harness is the agent-runtime layer (Claude Code's classifier/permission machinery) — the substrate the harness runs on.
The harness automatically executes its invariants — but records only 13% of the times it does.
Across 238 failure paths in 14 surfaces: 31 VISIBLE (13%) reach the error metric as counted, card-bound events · 88 PARTIAL (37%) emit something the rollup can't/won't count · 119 BLIND (50%) emit nothing. An invariant fires — a commit refused, a write blocked, a deploy rolled back — and 87% of the time the enforcement leaves no trace the metric can see.

BUT — corrected this layer: the failures are captured (in the tool_telemetry stream — every command + output + error-regex). They aren't surfaced, because the pain rollup reads named spine events, not the telemetry stream. Captured ≠ surfaced. Not data absence — the metric reads the sparse source while the rich one sits right there.

The onion — layers peeled so far

  • Lived experience — the workflow feels like far more friction than the metric reports. (Jeff's instinct — the thread that started this.)
  • Trace reader works — /borg/trace.html?card=N renders any card's events end-to-end (the #3122 tool I'd lost the URL to). Proved the per-card data exists.
  • Error count suspiciously low — first workflow found only 20 card-bound errors across 20 cards. Smelled wrong.
  • Undercount diagnosed — per-card endpoint caps at 1d (old cards return 0 → false zeros), counts only card-bound events, and 90% of all errors (2345/2591 over 7d) carry no card_id → invisible to per-card analysis.
  • Coverage audit — 14 auditors read real source, classified 238 failure paths: 13% visible / 37% partial / 50% blind.
  • tool_telemetry layer (this layer) — every tool command (PreToolUse) + output with error-regex (PostToolUse) is logged via append_log. So the audit measured the wrong thing: emit-coverage (13%), not observability-coverage. The friction is captured; it just isn't read by the metric.

Blind-spot categories (from the audit)

HIGH

1 · Failure emits go to a stream promtail never tails

werk-pull writes pull.refused/rolledback only to a jsonl the source itself says (L109) isn't scraped; emit_spine fires on success only. Same for werk-accept. → why #3152's pull-rollback logged nothing.

Fix: promote failure-path jsonl to a real spine emit, or point promtail at the file.

HIGH

2 · Silent Err exits with no event — the ACT failures

werk-build and werk-deploy have no *.failed event anywhere. Compile/sign/cdhash, build-subprocess, demo deploy-fails all ?-propagate to exit-1 with zero emit. The merged≠live class.

Fix: one top-level error handler per binary emits <verb>.failed{reason} before exit.

HIGH

3 · Rollbacks fire but aren't counted (success-but-wrong)

deploy.rolledback{cdhash-mismatch} carries card_id + reason, but the rollup counts only .failed/.refused/.error — so every stale-deploy rollback is invisible, and a degraded deploy.completed logging the old cdhash counts as clean success.

Fix: count .rolledback + a degraded flag in the rollup rule.

HIGH

4 · Guard/hook bites are decisions, not errors

Every PreToolUse deny = permissionDecision:deny, which the rollup explicitly doesn't count. canonical_write_guard emits nothing on deny. The git-on-main / canonical-write blocks (most frequent friction) count as zero.

Fix: each deny emits gate.<name>.refused/guard.<name>.blocked with card_id + decision=deny; rollup counts deny.

MED

5 · Gates fail OPEN on infra error

accept_gate defaults to PASS when its subprocess errors, with only a warn! — a gate that should block silently allows. The protection is gone, not just unlogged.

Fix: fail-closed by default + emit gate.<name>.error{reason=infra}.

I confirmed the code captures it via append_log; I did not confirm the scrape path end-to-end. This single fact decides the whole fix strategy — if the stream is queryable, the cheap read-side fix is real; if not, we're back to per-verb emits.
  1. Verify the sink (above). One check: where does append_log write, and is that path in a promtail job / queryable in Loki?
  2. If queryable → read-side fix first (cheapest): point the error analysis at the telemetry stream that already does ERROR_RE + severity on every command's output. Surfaces the whole BLIND mass without touching a single verb. (Silas — observability)
  3. Rollup-rule fix (the multiplier): change from event-name-suffix gate → count level=error/warn OR deny/refuse/rollback disposition. Reclassifies everyone's failures at once. (Silas)
  4. Per-owner emit-on-failure (refinement, clean typed signal): Silas build/deploy · Kade pull/commit/push · Wren demo/accept · gate-hooks by hook. Each adds a card-bound *.failed/*.refused on its Err/deny paths.
  5. Agent-runtime denials (the one non-harness slice): classifier/permission blocks — capturable only via a wrapper around tool invocation, or accept-blind. Decide later.
