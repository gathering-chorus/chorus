Model + Data

Everything here is model-driven. Ontologies define the domains, RDF graphs hold the data, and views render from queries. The model is the source of truth — not the code, not the UI.

The approach

Each product starts with an ontology — classes, properties, and relationships that describe what exists and how it connects. Data enters as RDF triples stored in named graphs. Views are projections of the graph, not hand-coded pages.

This means the model can grow without rewriting the application. Add a class to the ontology, load data into a graph, and the existing views discover it. The same approach powers both products below.

Shared principles

Ontology first

Define the domain before writing code. The model drives the application, not the other way around.

Named graphs

Data lives in graphs that can be loaded, replaced, and queried independently. Each source gets its own graph.

SPARQL views

Pages render from queries, not hardcoded data. Add to the model and the UI discovers it.
